# 1. 为什么要使用消息队列？MQ 的核心作用是什么？

MQ 的核心作用有三个：**异步**、**解耦**、**削峰填谷**。

- **异步**：将同步请求转为异步处理，提升系统响应速度。
- **解耦**：上下游系统通过 MQ 通信，一方变更不影响另一方。
- **削峰填谷**：在高流量场景（如秒杀）下，MQ 充当缓冲，按消费者处理能力消费，防止系统被突发流量压垮。秒杀场景中可控制人数，超过阈值直接丢弃订单，应用程序按自身最大处理能力消费。

# 2. 引入 MQ 会带来哪些问题？

- **系统可用性降低**：引入的外部依赖越多，越容易挂掉。MQ 挂了会导致整套系统崩溃。
- **系统复杂度提高**：需要处理消息重复消费、消息丢失、顺序保证等问题。
- **一致性问题**：生产者处理成功返回，但下游部分系统写库失败，导致数据不一致。

# 3. MQ 有哪些核心概念？

- **Topic**：消息类别，用于消息的归类与隔离。
- **Partition（分区）**：物理层面的数据文件。**同一个消费组中，一个分区只支持一个消费线程**消费消息。分区过少会导致消费速度远落后于生产速度。多分区模式支持多个消费者并发消费，提高吞吐量。
- **Broker**：部署了 MQ 实例的服务器节点。
- **Consumer Group（消费者组）**：包含一个或多个消费者。**多分区 + 多消费者**可极大提高下游处理速度。
- **Offset**：记录消费者消费的位置。Kafka 中 **Consumer 自己维护 Offset**，每个 Topic 有多个 Partition，Consumer 需记录每个 Partition 的消费位置。好处是 Consumer 可以重新消费数据（如修复 bug 后重放）。

# 4. Kafka 为什么由 Consumer 维护 Offset，而不是 Broker？

如果由 Broker 维护 Offset：

- Broker 将消息立即标记为已消费，**Consumer 宕机时正在消费的消息就会丢失**。
- 若标记为"已发送但未消费"，等待 ACK：
  - Consumer 处理完但未发 ACK 即宕机，消息会被**消费两次**。
  - Broker 需为每条消息维护多个状态，带来**性能问题**。

因此 Kafka 交由 Consumer 自行管理 Offset，Consumer 可在修复后重新消费数据。

# 5. 什么是重复消费？Kafka 中出现重复消费的原因是什么？

RabbitMQ、RocketMQ、Kafka 都**可能出现重复消费**。**重复消费问题通常不是由 MQ 保证，而是由消费方自行保证**。

Kafka 重复消费场景：Consumer 消费了数据后，在提交 Offset 之前挂了，重启后 Consumer 会从上次提交的 Offset 继续消费，导致收到重复数据。

# 6. 如何保证消息不被重复消费（消费幂等性）？

根据业务类型选择以下方案：

- **数据库操作**：给消息设置唯一主键（或 UUID），重复消费时主键冲突，避免脏数据。
- **数据库设置唯一键约束**，重复插入时报错而非插入重复数据。
- **Redis**：**Set 操作天然幂等**，无论 set 几次结果一样。
- **全局唯一 ID + Redis 查重**：生产者为每条消息加全局唯一 ID，消费时先查 Redis 是否存在，不存在则消费并存入 ID。
- **内存 Set**：维护内存中的 Set，收到的消息先查 Set，已存在则丢弃。
- **第三方接口**：确认第三方是否有去重能力，若无则自行实现去重。

# 7. 如何保证消息的顺序性？

消息队列中的若干消息如果**对同一数据进行操作**，且操作具有前后依赖关系，必须按顺序执行。

- **Kafka 只能保证 Partition 内有序**，Partition 间无法保证。
- 保证同一业务数据落入同一 Partition：
  - **按 Key 路由**：`abs(hash(主键/key)) % partition_num`，保证同一订单/用户落入同一 Partition。
  - **直接指定 Partition**：所有消息发往同一 Partition。
- **消费端**：Kafka 保证 **1 个 Partition 只能被 1 个 Consumer 消费**，内部单线程消费，保证同一 Partition 内的有序性。

# 8. MQ 有哪些典型使用场景？

MQ 的四大特点——**FIFO、发布订阅、持久化、分布式**——支撑了以下场景：

- **应用解耦（异步）**：上游投递到 MQ 后立即返回，下游消费。MQ 接收性能极高，不影响上游性能。
- **通知**：下游监听 MQ，**先进先出、逐条消费**，保证及时性且降低耦合。
- **限流**：MQ 中**一条数据只有一条在使用中**，保证并发场景下的数据一致性。
- **数据分发**：MQ 支持**一对多或广播模式**，下游根据规则选择是否接收数据，扩展性强。
- **分布式事务**：MQ 作为**二次提交的中间节点**，存储请求数据，失败时重试或回滚，兼顾性能与一致性。

**电商系统示例**：

1. 用户注册后发送邮件、赠送优惠券、同步客户数据——通过 MQ 异步通知，保证注册性能。
2. 商品发布后通过 MQ 生成 Elasticsearch 索引数据。
3. **延迟 MQ** 实现订单 24 小时未支付自动取消，避免定时任务频繁查库。
4. 支付完成后通过 MQ 通知子系统（库存、积分、短信），保证核心支付流程高效。

# 9. 使用 MQ 有哪些注意事项和常见误区？

- **验证码短信、邮件等场景**更适合多线程异步处理，不需要 MQ（可容忍几十秒延迟）。
- **秒杀库存控制**依赖高性能缓存（如 Redis）和强制锁，**不要依赖消费者最终返回结果**。
- **上游-MQ-下游**效率低于**上游-下游**。MQ 的高效在于上游投递到 MQ 完成即视为成功。

# 10. Kafka 创建 Topic 时如何将分区放置到不同的 Broker 中？

- **副本因子不能大于 Broker 的个数**。
- 第一个分区（编号 0）的第一个副本位置从 brokerList **随机选择**。
- 其他分区的第一个副本位置相对于分区 0 **依次后移**。例如 5 个 Broker、5 个分区，分区 0 在第 4 个 Broker 上，则分区 1 在第 5 个、分区 2 在第 1 个，依次类推。
- 剩余副本的位置由 **nextReplicaShift**（随机生成）决定。
