# 1. Kafka、RabbitMQ、RocketMQ 各自的核心定位和特点是什么？

**Kafka**：基于 Pull 模式消费，追求**高吞吐量**，最初用于日志收集和传输。**不支持事务**，对消息的重复、丢失、错误没有严格要求，适合**产生大量数据的互联网服务的数据收集业务**。

**RabbitMQ**：Erlang 开发，基于 **AMQP 协议**，面向消息、队列、路由（点对点和发布/订阅）、可靠性、安全。更多用在**企业系统内，对数据一致性、稳定性和可靠性要求很高的场景**，对吞吐量要求不高。

**RocketMQ**：阿里开源，纯 Java 开发，具有**高吞吐量、高可用性、适合大规模分布式系统**的特点。思路起源于 Kafka，但对**消息的可靠传输及事务性做了优化**，广泛应用于交易、充值、流计算、消息推送、日志流式处理、binlog 分发等场景。

# 2. 三种 MQ 的消息发送方式有什么差异？

- **Kafka**：发送端和消费端均**按批次发送/消费**消息。发送端将多个小消息合并批量发向 Broker，消费端每次取一个批次批量处理。**消息极大化**。
- **RabbitMQ**：不批量发送，消息粒度较小。
- **RocketMQ**：消费端可以**批量消费**（可配置每次消费数量），但发送端**不是批量发送**。消息体较大。
- **ActiveMQ**：消息体比较大。

# 3. Kafka 为什么吞吐量高？

两个核心原因：**顺序写入磁盘**和 **Broker 无状态**。

**顺序写入磁盘**：磁盘线性写的速度远大于随机写。在一个 7200rpm SATA RAID-5 阵列上线性写约 **600MB/秒**，随机写仅 **100KB/秒**，相差近 **6000 倍**。操作系统还可利用 read-ahead 和 write-behind 技术提升性能。

**Broker 无状态**：Offset 由 **Consumer 自行控制**，Broker 不需要标记哪些消息被哪些 Consumer 消费过，也不需要通过 Broker 保证同一个 Consumer Group 只有一个 Consumer 能消费同一条消息，因此**不需要锁机制**。

# 4. 如何根据业务场景选择 MQ？

- **日志收集、大数据场景**：选 Kafka，高吞吐、可重放，允许少量丢失。
- **企业级应用、金融交易**：选 RabbitMQ，AMQP 协议成熟，稳定可靠，一致性要求高。
- **阿里系、Java 技术栈、高并发交易**：选 RocketMQ，高吞吐 + 事务消息 + 可靠性强，适合交易、充值、流计算等场景。
- **已有 Kafka 又想降成本**：可以用 Kafka 替代部分 RocketMQ 场景，但 RocketMQ 在事务消息和可靠性上更优。

# 5. RocketMQ 如何保证消息的可靠性？

从消息完整链路分为三个阶段来保证：

- **生产阶段**：消息在 Producer 端创建，经网络传输发送到 Broker。通过**同步发送 + 重试机制**保证生产端消息不丢失。同步发送会在收到 Broker 确认后才返回成功。
- **存储阶段**：消息在 Broker 端**持久化存储**。主备或多副本架构下，消息会被复制到其他节点。采用**同步双写/异步刷盘**策略：同步双写保证主备数据一致，异步刷盘提升性能。Broker 宕机时可从副本恢复。
- **消费阶段**：Consumer 从 Broker 拉取消息消费，通过**重试机制**最大限度保证消费成功。消费失败后 RocketMQ 支持**定时重试**，达到最大重试次数后进入**死信队列**，人工处理。
