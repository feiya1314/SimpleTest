# 1. 分区（Partition）是什么？物理层面如何存储？

Topic 可拆分为多个 Partition，Partition 在服务器上就是一个一个的文件夹，每个 Partition 文件夹下有多组 **segment 文件**，每组包含 `.index` 文件、`.log` 文件、`.timeindex` 文件。`.log` 文件实际存储消息，`.index` 和 `.timeindex` 为索引文件用于检索消息。

- 文件以该 segment **最小 offset** 命名，如 `000.index` 存储 offset 为 0\~368795 的消息。
- Kafka **顺序写入磁盘**（效率远高于随机写入），结合 **分段+索引** 解决查找效率问题。

# 2. 副本机制是如何工作的？

Kafka **分区是多副本的**，用于保证高可用。每个分区可有多个副本（通过 `replication-factor` 指定），副本集合中**一个为 Leader，其余为 Follower**。

- **Leader**：所有**读写请求**都由 Leader 处理，Producer 和 Consumer 只与 Leader 交互。
- **Follower**：从 Leader **拉取**消息进行同步，写入本地磁盘。
- 当 Leader 不可用时，从 ISR 中选举一个新的 Leader。
- 分区副本在 Broker 间分配遵循以下原则：
  - 在所有 Broker 上**均匀分布**。
  - 确保分区的每个副本分布在**不同的 Broker** 上。
  - 如果配置了机架信息（`broker.rack`），尽可能将副本分配到**不同机架**的 Broker 上。

# 3. 什么是 ISR 机制？Follower 满足什么条件才能加入 ISR？

**ISR（In-Sync Replica）** 是分区中所有与 Leader **保持同步的可用副本**的列表，Leader 必然是 ISR 成员。Follower 需满足两个条件才能被认为是同步副本：

- 与 Zookeeper 之间有活跃会话，**定时发送心跳**。
- 在**规定时间内从 Leader 低延迟地拉取过消息**。

不满足条件的 Follower 会被从 ISR 列表中移除，直到恢复满足条件才重新加入。

# 4. Kafka 生产者的 acks 参数有哪些取值？

`acks` 参数在 Producer 端设置，控制消息写入的可靠性：

- **acks = 0**：Producer 把消息发送出去即认为成功，不等待任何确认。吞吐量最高，但**可能丢失消息**（Leader 写入前宕机）。
- **acks = 1**（默认）：Leader **接收到消息并写入本地磁盘**即认为成功，不等待 Follower 同步。如果 Leader 写入后、Follower 同步前宕机，消息**可能丢失**。
- **acks = all（-1）**：Leader 需等待 **ISR 列表中所有 Follower 都同步完消息**才认为成功。可靠性最高，**不会丢失消息**。

# 5. Kafka 的 Leader 是如何选举的？什么是 HW 截断机制？

Kafka 采用 **ISR 机制选举 Leader**，而非多数选举法。Leader 宕机后，**只能从 ISR 列表中选择新的 Leader**，因为 ISR 中的副本都知道 HW（High Watermark）之前的数据，能保证数据一致性。

**HW 截断机制**：

- 选出新 Leader 后，其不能保证拥有旧 Leader 的所有数据，**只能保证 HW 之前的数据是同步过的**。
- 所有 Follower **将数据截断到 HW 位置**，再与新 Leader 同步。
- 当宕机的旧 Leader 恢复后，发现数据与新 Leader 不一致，会**将自己的数据截断到宕机前的 HW 位置**，然后像 Follower 一样同步新 Leader 的数据。

# 6. 什么是 Consumer Group？分区和消费者的关系是什么？

**Consumer Group** 是 group.id 相同的消费者集合，组内所有消费者**协调消费订阅主题的所有分区**。

- **一个分区只能被同一个消费组内的一个消费者消费**，但可被不同消费组同时消费。
- 如果 Consumer 比 Partition 多，**多余 Consumer 闲置**（一个 Partition 不允许并发消费）。
- 如果 Consumer 比 Partition 少，**一个 Consumer 消费多个 Partition**。
- **Partition 数目最好是 Consumer 数目的整数倍**，否则会导致消费不均。
- Consumer 从多个 Partition 读数据时，**不保证跨 Partition 的数据顺序性**，Kafka 只保证 Partition 内有序。

# 7. 什么是 Rebalance？Coordinator 如何确定？

**Rebalance** 是在增减 Consumer、Broker、Partition 时，重新分配 Consumer 与 Partition 对应关系的机制。Rebalance 后 Consumer 对应的 Partition **会发生变化**。

Kafka 通过 **Coordinator** 管理 Consumer Group：

- Consumer Group 的**第一个 Consumer 启动时**，会向 Kafka 集群发送请求，确定该组的 Coordinator。
- 消费者向任意 Broker 发送 `GroupCoordinatorRequest`，服务端返回**负载最小的 Broker 节点 id** 作为 Coordinator。
- 组内所有成员与 Coordinator 协调通信。

# 8. Kafka 有哪些分区分配策略？

通过 `partition.assignment.strategy` 参数设置，有三种策略：

- **Range（默认）**：对每个主题，将分区按序号排序，消费者按字母排序，分区数除以消费者数，**除不尽时前面消费者多消费一个分区**。
- **RoundRobin（轮询）**：将所有 Partition 和 Consumer 线程列出，按 HashCode 排序后**轮询分配**。如果所有消费者订阅相同主题，分区会**均匀分布**。
- **StickyAssignor（粘性，0.11.x+）**：两个目标——**分配尽可能均匀**，且**尽可能与上次分配保持一致**。第一个目标优先于第二个。实现比前两种复杂。

# 9. Zookeeper 在 Kafka 中扮演什么角色？

早期 Kafka 重度依赖 Zookeeper 管理集群元数据：

- **Broker 注册**：每个 Broker 启动时在 `/brokers/ids` 下创建**临时节点**，写入 broker.id。
- **Controller 选举**：多个 Broker 竞争在 Zookeeper 上创建 `/controller` 节点，**创建成功的 Broker 成为 Controller**，负责管理主题分区及副本状态。Controller 宕机时触发新的选举。
- **Topic 注册**：Topic 分区信息及与 Broker 的对应关系由 Zookeeper 维护。
- **消费者注册与监听**：消费者启动后在 `/consumers/[group_id]/ids/[consumer_id]` 创建临时节点，并监听消费者分组变化和 Broker 列表变化，**触发负载均衡**。
- **消费进度 Offset 记录**：消费者将 Offset 记录到 `/consumers/[group_id]/offsets/[topic]/[broker_id-partition_id]`，以便重启后从之前进度继续消费。
- **分区与消费者关系记录**：在 `/consumers/[group_id]/owners/[topic]/[broker_id-partition_id]` 记录消费者的消费权。
- **消费者注册流程**：
  - 在 `/consumers/[group_id]/ids/[consumer_id]` 创建临时节点，写入订阅的 Topic 信息。
  - 对消费者分组变化注册监听（`/consumers/[group_id]/ids`），触发负载均衡。
  - 对 Broker 列表变化注册监听（`/brokers/ids/[0-N]`），触发负载均衡。
  - 进行消费者与消息分区的重新分配。

# 10. Zookeeper 如何实现生产者的负载均衡？

Kafka 支持两种生产者负载均衡方式：

- **四层负载均衡**：根据生产者的 IP 和端口为其关联一个 Broker，生产者只与该 Broker 维持单个 TCP 连接。逻辑简单，但**无法做到真正的负载均衡**（不同生产者消息量差异大时会导致 Broker 负载不均），且无法感知 Broker 的新增或删除。
- **Zookeeper 方式**：生产者通过监听 `/brokers/ids` 节点的变化，**动态感知 Broker 列表变更**，实现动态负载均衡。

# 11. Kafka 新版本中如何减少对 Zookeeper 的依赖？

新版本（0.9+）的 Consumer 使用 **Kafka 内部的 Group Coordination 协议**，不再将消费状态、Group 管理和 Offset 存储在 Zookeeper，降低了 Zookeeper 的单点风险。新版本中**逐渐弱化了 Zookeeper 的作用**。

# 12. Kafka 的分区数据保留机制是怎样的？

Kafka 不会永久保留数据，通过以下参数配置数据保留期限或数据量：

- `log.retention.bytes`：删除数据前允许的最大数据量，默认 -1（无限制）。
- `log.retention.ms`：保存数据的毫秒数。
- `log.retention.minutes`：保存数据的分钟数。
- `log.retention.hours`：保存数据的小时数，默认 **168（一周）**。
- 参数优先级：`log.retention.ms` > `log.retention.minutes` > `log.retention.hours`，未设置时依次降级使用。

Kafka 将分区分为多个片段，**当前写入的片段（活跃片段）永远不会被删除**。当按照保留策略删除时，会删除最老的片段。
