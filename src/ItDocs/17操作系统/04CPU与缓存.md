## 1. CPU由哪些部分组成？从内存读写数据需要哪三种信号？

CPU由**运算器和控制器**组成。

CPU要从内存中读写数据，必须具备三种信号：

- **地址信号**：定位从内存的哪个地址开始读写。通过**地址总线（Address Bus）**传送，地址只能从CPU传向外部存储器或I/O端口，**地址总线是单向的**。地址总线的位数决定了CPU可直接寻址的内存空间大小，若地址总线为n位，则可寻址空间为 **2^n 字节**。8位微机的地址总线为16位，最大可寻址空间为64KB；16位微型机的地址总线为20位，可寻址空间为1MB。
- **数据信号**：确定读还是写数据。通过**数据总线（Data Bus）**传送，是**双向三态**形式的总线，既可以把CPU的数据传送到存储器或I/O接口，也可以将其它部件的数据传送到CPU。数据总线的位数通常与微处理器的字长一致，决定了CPU和外界的数据传送速度。8根数据总线一次传送8个比特位，16根一次传送两个字节。
- **控制信号**：控制数据怎样传送。通过**控制总线（Control Bus）**传送，用于传送控制信号和时序信号。控制信号包括CPU送往存储器和I/O接口的读/写信号、片选信号、中断响应信号，以及其它部件反馈给CPU的中断申请信号、复位信号、总线请求信号等。**控制总线的传送方向是双向的**，具体由控制信号而定。**控制总线的位数决定了CPU对外部器件的控制能力**。

## 2. 什么是指令集？指令集存储在CPU哪里？如何获取CPU信息？

- **指令集**：从软件意义上是指**规范汇编文件格式的规范**，描述"CPU能使用哪些机器码"。从硬件意义上，**指令集就是CPU的电路设计逻辑**，由晶体管组成，**CPU本身就是CPU指令集**。指令集规定CPU能做什么事，CPU就是实现这些事的工具。
- **存储位置**：狭义的CPU指令集存放位置是CPU中的**译码电路**。
- **获取CPU信息**：绝大多数CPU信息都可以通过**CPUID指令**读取，BIOS就是通过CPUID了解CPU信息的。CPUID信息本身不可造假，但不良商家通过**修改BIOS**来汇报错误信息给操作系统，甚至直接修改操作系统注册表来造假。判断CPU信息不能只看操作系统结果，要用**CPU-Z等直接读取CPUID的软件**，而不是读取SMBIOS或操作系统信息的软件。

当所有进程处于休眠或挂起状态时，CPU无事可做，进入**IDLE状态（CPU空转）**。每个CPU核心都有一个idle进程，作用是不使用CPU，达到省电的目的。

## 3. 什么是总线？总线如何分类？总线有哪些基本特性？

**总线**：主板上专门把CPU和其他芯片电路连接的导线。计算机中的所有设备都通过总线连接。

**单总线结构**：只有一条系统总线，计算机中的所有设备都连接在这条总线上。

**总线的基本特性**：
- **发送的互斥性**：某一时刻只允许一个功能部件往总线上发送数据。
- **接收的共享性**：允许多个功能部件在同一时刻接收总线上传输的信息。

**总线的基本作用**是传输信号，其基本原理是**多路复用技术**（多个用户共享公用信道），主要有**时分多路复用、频分多路复用和码分多路复用**。

一般x86处理器都用**PCI总线**来管理各种设备（包括主板芯片组、内存控制器等）。PCI的地址总线和数据总线复用地址和数据位，通过不同的命令编码/特殊地址来区分。

**总线的生命周期**：主从设备完成一次完整信息通信的过程，包括三个阶段：
1. **申请分配阶段**：主设备申请总线控制权。
2. **寻址阶段**：总线控制器寻找到底是哪个设备发起的控制请求。
3. **数据传输阶段**。
4. **结束阶段**。

## 4. 总线的判优控制有哪几种方式？

总线的判优控制用于**保证在同一时间内只能有一个申请者使用总线**。能够控制总线并启动数据传送的设备称为**主控器**（通常CPU），能够响应主控器指令的设备称为**受控器**（存储器），I/O设备可以为主控器也可以为受控器。

**集中式判优控制**的三种方式：

1. **链式查询**：
   - 使用三根线：**BS（总线忙）**、**BR（总线请求）**、**BG（总线同意）**。
   - 设备检测到总线空闲，通过BR线发出占用请求。
   - 总线控制部件收到请求后，通过BG线逐个查询，直到找到请求设备。
   - 设备优先级按**位置确定**，离总线控制部件最近的设备优先级最高，最后一个设备可能永远无法占用总线。设备串联，BS线某处损坏则查询无法进行。

2. **计数器定时查询**：
   - 总线接收到BR请求后，计时器开始计数，通过设备地址线发送地址。
   - 设备地址与请求设备一致时，设备获得总线控制权，计数器停止。
   - 计数器值可通过**软件设定**，人为确定设备优先级。

3. **独立请求方式**：每个设备有独立的请求线和同意线，控制逻辑集中在总线控制器中。

## 5. 总线的通信控制方式有哪些？

1. **同步通信**：通信双方由**统一的时钟**控制数据传输，时钟信息通常由CPU发出，送到总线上所有部件，在规定的总线周期内只有通信双方可以收发数据。

2. **异步通信**：通信双方通过**"握手"信号**实现数据传送，通常用于不同速度部件之间的数据传送。

3. **并行通信**：各位**同时传输**（8/16/32/64位）。位数越多传输越快，但传输距离短，只能在一个机柜内使用。
   - **并行通信的缺陷**：
     - **信号时滞**：所有位同时传输，但不能保证同时到达，先到等后到，且随传输距离拉长越发明显。
     - **串扰**：并排线缆间互相干扰，导致更高的误码率。
     - **影响机箱散热**：依赖大连接器和宽带状传输电缆，挤压散热空间。

4. **串行通信**：一位一位传输。速度较慢，但**通信线路简单、容易实现双向传输，特别适合远距离传输**。

5. **串行异步通信**：串行通信和异步通信的结合，需要使用具有**起始位、停止位和奇偶校验位**的特殊数据格式。

## 6. 如何保证总线传输的可靠性？总线带宽如何计算？

**可靠性是评定总线最关键的参数**。提高可靠性的措施：
1. 数据帧发送前发送器对总线**侦听**，总线空闲时才传送数据帧，避免数据冲突。
2. 采用**双绞线差分信号**传送数据，降低单线电压升降幅度，减小信号边沿产生的高次谐波。
3. 适当让数据边沿具有一定的**斜坡**。
4. 增加**匹配电阻和电容**，减少信号反射和平衡分布电容。
5. 采用合适的**网络拓扑结构和屏蔽技术**，减少其他信号干扰。

**总线带宽计算**：由总线的**位宽**和**工作频率**决定。
- **总线带宽**：单位时间内总线上传送的数据量（每秒传送MB的最大数据传输率）。
- **总线位宽**：总线能同时传送的二进制数据的位数（如32位、64位）。
- **工作频率**：控制总线时钟信号的频率（Hz），即一秒钟被分成的节拍数。

## 7. 什么是DMA？DMA的数据传输过程是怎样的？

**DMA（Direct Memory Access，直接内存访问）**：实际上是**DMA控制器将数据从一个设备拷贝到另一个设备的过程**。DMA控制器的初始化需要CPU参与，但**数据传输过程不需要CPU参与**。

**DMA适用的数据传输场景**：
1. **内存 → 内存**
2. **内存 → 外围设备**
3. **外围设备 → 内存**
4. **外围设备 → 外围设备**

有两种方式可以引发DMA数据传输：

**软件对数据请求的传输过程**：
1. 进程调用read，驱动程序函数分配一个**DMA缓冲区**，并让硬件将数据传输到这个缓冲区中，**进程进入睡眠状态**。
2. 硬件将数据写入DMA缓冲区，写入完成后产生一个**中断**（由DMAC产生）。
3. 中断处理程序获取到输入的数据，然后**唤醒进程**，进程可以读取数据。

**硬件异步数据传输过程**（以网卡为例）：
1. 硬件产生中断，宣告数据到来。
2. 中断处理程序分配一个缓冲区，告诉硬件向哪里传输数据。
3. 外围设备将数据写入缓冲区，完成后产生另外一个中断。
4. 处理程序分发新数据，唤醒相关进程。

网络数据最先到达的是**硬件网卡**，网卡接收到一定量的网络数据时产生中断，通过kernel处理，进入硬件异步数据传输过程。

## 8. 什么是CPU流水线？

CPU流水线是现代CPU为了提高指令执行效率而采用的一种技术。它将指令的执行过程分成多个阶段（如取指、译码、执行、访存、写回），每个阶段由独立的电路单元并行处理不同指令，从而实现**指令级并行**。

理想情况下，若流水线深度为k，则吞吐率可提升至原来的k倍。但流水线存在**结构冒险、数据冒险、控制冒险**等问题，需要通过前递、分支预测、流水线停顿等技术来解决。

## 9. 什么是CPU高速缓存？什么是缓存行（Cache Line）？

**CPU高速缓存**是为了缓解CPU执行速度与内存读取速度之间的巨大差距而引入的。CPU首先将数据从内存读取到缓存中，每次CPU都从缓存中取数据；修改数据时也是先将数据写入缓存，再由缓存刷新至内存。


CPU以**缓存行（Cache Line）**为读写单位，是内存与缓存数据交换的最小单元。目前主流CPU的Cache Line大小**通常是64字节**。一个Java的long类型是8字节，一个缓存行中可以存8个long类型的变量。也就是说哪怕只是从内存读入一个字节到缓存中，也会读取包含此字节的整个缓存行到缓存中。


并发的线程都分配在各个CPU上执行，每个线程都有自己的工作内存（相当于CPU中的Cache），共享变量的副本实际在cache line中。

## 10. 什么是局部性原理？

局部性原理：在CPU访问存储设备时，无论是存取数据还是指令，都趋于聚集在一片连续的区域中。

- **时间局部性（Temporal Locality）**：如果一个信息项正在被访问，在近期很可能还会被再次访问。比如程序中的循环、递归对数据的循环访问，主要体现在**指令读取**的局部性。
- **空间局部性（Spatial Locality）**：如果一个存储器的位置被引用，将来其附近的位置也会被引用。比如程序中对数据组的读取或对象的连续创建，对内存顺序读写，主要体现在对**程序数据引用**的局部性。

## 11. 什么是总线锁和缓存锁？它们有什么区别？

**CPU的锁主要解决多核心并发访问/修改同一块内存的问题**，关于CPU的锁有如下3种：
1. **处理器自动保证基本内存操作的原子性**
2. **使用总线锁保证原子性**
3. **使用缓存锁保证原子性**

多核CPU并发访问/修改同一块内存时，需要CPU锁来保证原子性。

多核CPU并发访问/修改同一块内存时，需要CPU锁来保证原子性。

**处理器自动保证基本内存操作的原子性**：处理器保证从系统内存中读取或写入一个字节是原子的。奔腾6和最新处理器能自动保证单处理器对同一个缓存行里进行16/32/64位的操作是原子的。但复杂的内存操作（跨总线宽度、跨多个缓存行、跨页表访问）不能自动保证原子性，需要总线锁和缓存锁机制。

**总线锁**：
- 使用处理器提供的**LOCK＃**信号，当一个处理器在总线上输出此信号时，其他处理器的请求将被阻塞，该处理器可以独占使用共享内存。
- **锁总线期间，其他CPU对内存的读写请求都会被阻塞，直到锁释放**。
- 总线锁定把CPU和内存之间的通信锁住，锁定期间其他处理器**不能操作其他内存地址的数据**，**开销比较大**。

**缓存锁**：
- 利用**MESI缓存一致性机制**来保证操作的原子性。
- 如果缓存的数据在处理器缓存行中，则在LOCK操作期间，处理器不在总线上声言LOCK＃信号，而是修改内部的内存地址，并允许缓存一致性机制来保证操作的原子性。
- 当其他处理器回写已被锁定的缓存行的数据时，会触发**缓存行无效**。
- **lock期间的写操作会回写已修改的数据到主内存，同时通过缓存一致性协议让其他CPU相关缓存行失效**。

**两种不使用缓存锁的情况**：
1. 当操作的数据**不能被缓存在处理器内部**，或操作的数据**跨多个缓存行（cache line）**时，处理器会调用总线锁。
2. 有些处理器**不支持缓存锁定**（如Inter486和奔腾处理器）。

**总结**：锁总线开销比较大，后来的处理器都采用**锁缓存替代锁总线**，在无法使用缓存锁时**降级使用总线锁**。

通过Intel处理器提供的**LOCK前缀的指令**可以实现以上机制，比如位测试和修改指令BTS、BTR、BTC，交换指令XADD，CMPXCHG和其他操作数和逻辑指令（如ADD、OR等），被这些指令操作的内存区域就会加锁。其中**xchg默认包含lock前缀**，实测**cmpxchg也包含lock前缀**。

## 12. 什么是MESI缓存一致性协议？MESI的四种状态是什么？

MESI协议是缓存一致性协议中最常见的一种。该协议要求在每个缓存行上维护两个状态位，使得每个数据单位可能处于M、E、S和I这四种状态之一：

1. **M（Modified，修改）**：表示共享数据只缓存在当前CPU缓存中，并且是被修改状态，缓存的数据和主内存中的数据不一致，且没有更新至主内存。**缓存行数据被修改且独占，未同步到主内存**。
2. **E（Exclusive，独占）**：表示缓存的独占状态，数据只缓存在当前CPU缓存中，并且没有被修改，与主内存中一致。其他CPU中没有缓存该数据。**缓存行数据独占且未被修改**。
3. **S（Shared，共享）**：表示数据可能被多个CPU缓存，并且各个缓存中的数据和主内存数据一致。**缓存行数据在多CPU中共享，与内存一致**。
4. **I（Invalid，无效）**：表示缓存已经失效。**缓存行数据无效**，需要重新从内存中加载。


每个CPU的处理过程是：先将计算需要用到的数据缓存在CPU高速缓存中，在CPU进行计算时直接从高速缓存中读取数据，计算完成之后写入到缓存中，在整个运算过程完成后，再把缓存中的数据同步到主内存。每个CPU有多个内核，每个内核有自己的缓存，同一个数据的副本可能存在于多个缓存中，导致数据不一致。

MESI协议通过**监听任务**来保证一致性：
- 状态为**M**的缓存行：必须时刻监听所有**试图读取**该缓存行对应主存地址的操作，如果监听到，必须在此之前把缓存行中的数据**写回主内存**。
- 状态为**S**的缓存行：必须时刻监听使该缓存行**无效**或**独享**的请求，如果监听到，必须把其缓存行状态设置为**I**。
- 状态为**E**的缓存行：必须时刻监听其他试图**读取**该缓存行对应主存地址的操作，如果监听到，必须把其缓存行状态设置为**S**。

## 13. MESI协议的消息类型有哪些？

| 消息名 | 类型 | 描述 |
|----|----|----|
| **Read** | 请求 | 通知其它处理器，当前处理器准备读取某个数据。包含待读取数据的内存地址 |
| **Read Response** | 响应 | 包含被请求读取的数据。可能是主内存提供的，也可能是嗅探到Read消息的其它Cache提供的 |
| **Invalidate** | 请求 | 通知其它处理器将其高速缓存中指定内存地址对应的缓存行状态置为I（无效） |
| **Invalidate Acknowledge** | 响应 | 接收到Invalidate消息的处理器回复此响应，表示已删除其高速缓存上的相应副本数据（逻辑删除，更新缓存行的Flag值） |
| **Read Invalidate** | 请求 | Read消息和Invalidate消息的组合。通知其它处理器发送方准备更新数据，请求其它处理器删除其高速缓存中相应的副本数据。接收方必须回复Read Response和Invalidate Acknowledge |
| **Writeback** | 请求 | 包含需要写入主内存的数据及其对应的内存地址 |

## 14. MESI协议的读写流程是怎样的？

**数据读取流程**（CPUA需要读取数据X）：
CPUA根据数据地址在自己的缓存L1A中找到对应的缓存行，判断缓存行状态：
- 状态为**M/E/S**：说明数据对当前读请求可用，**直接从缓存行中获取数据**。
- 状态为**I**：说明数据无效。CPUA向总线发送**Read**消息，其它处理器（CPUB）监听总线消息后处理：
  - 状态为**S/E**：CPUB构造**Read Response**消息将数据放入消息发送到总线，更新自己缓存行状态为**S**，CPUA收到后将数据存入缓存行并更新状态为**S**。
  - 状态为**M**：CPUB先将自己缓存行中的数据**写入主内存**，再响应**Read Response**消息，同时将状态更新为**S**。
  - 状态为**I**或不存在数据：主内存构造**Read Response**消息，从主内存读取数据返回。

**数据写入流程**（CPUA需要对地址A的数据X进行写操作）：
任何处理器执行内存操作时，必须拥有相应数据的**所有权**。CPUA根据缓存行状态处理：
1. 状态为**E/M**：CPUA已拥有数据所有权，直接将数据写入缓存行，更新状态为**M**，**不需要向总线发送任何消息**。
2. 状态为**S**：数据被共享，其它CPU可能存有副本。CPUA向总线发送**Invalidate**消息获取所有权，其它处理器收到后将缓存行状态更新为**I**并回复**Invalidate Acknowledge**。CPUA收到所有确认后写入数据，状态更新为**M**。
3. 状态为**I**：CPUA向总线发送**Read Invalidate**消息（我要读数据X，同时请其它处理器将包含该数据的缓存行置为无效）。收到所有Invalidate Acknowledge和主内存的Read Response后，将数据更新到缓存行，状态先为E再更新为M。

## 15. MESI协议存在什么问题？Store Buffer和Invalidate Queue是什么？

MESI的问题：MESI主要靠在总线上传递消息、对消息增加不同的监听来保证可见性。但消息传递需要时间，一个请求需要多个响应，每次涉及CPU的切换，对于CPU频繁读取来说，消息传递产生的时间是致命影响，会导致**引起缓存一致性流量风暴**，导致性能问题和稳定性问题。

**MESI的优化**：为了避免阻塞带来的资源浪费，CPU引入了**Store Buffer（存储缓存）**和**Invalidate Queue（无效队列）**。


**Store Buffer**：
CPU写数据时，先把数据写入store buffer中，然后马上返回，同时发送invalidate消息，之后由store buffer**异步执行**发送广播消息和写入cache的操作。CPU读取时会先从store buffer中读取，再从cache中读取。Store buffer的容量有限，如果其它CPU来不及处理invalidate消息，store buffer会不断堆积。

**Invalidate Queue**：
在Cache和总线间加入Invalidate Queue，CPU收到invalidate广播消息后，先将消息存储在invalidate queue中，**立即回复invalidate ack**，之后invalidate queue再异步执行将缓存行失效的操作。这样store buffer能及时清空。

**引入的问题**：Store Buffer和Invalidate Queue的改造提升了CPU执行效率（尤其是写效率），但将MESI由**强一致性**变成了**弱一致性（最终一致性）**。从store buffer发送invalidate广播到invalidate queue让缓存行失效期间，其它CPU读取到的都是老值，造成了短暂的不一致。

## 16. 什么是CPU内存屏障？X86提供了哪些屏障指令？

MESI缓存一致性协议可以解决各个CPU缓存的数据一致性问题，但为了避免阻塞引入Store Buffer和Invalidate Queue后，缓存一致性弱化为**最终一致性**，带来可见性问题。**CPU内存屏障就是为了使缓存一致性从最终一致性强化为强一致性**。（X86 lock指令有强缓存一致性的特点，lock前缀指令有内存屏障的作用）

**内存屏障**：将Store Buffer中的指令写入到内存，使其他访问同一共享内存的线程具有可见性。

**写屏障**：用于保证高速缓存间写事务的强一致性。当CPU执行写屏障指令时，必须强制等待存储缓存中的写事务全部处理完再继续执行后面的指令，相当于将存储缓存中异步处理的本地写事务做了强一致的同步。写屏障执行完后，当前核心位于写屏障前的本地写事务全部处理完毕，其它核心都已接收到当前所有远程写事件的写无效通知。

**读屏障**：用于保证高速缓存间读事务的强一致性。当CPU执行读屏障指令时，必须先将当前处于失效队列中的写无效事务全部处理完，再继续执行读屏障后面的指令。读屏障执行完后，当前核心位于读屏障前的远程写无效事务全部处理完毕，读屏障之后的共享数据读取会得到最新的值。

**Intel提供的内存屏障指令**：
- **sfence**：实现Store Barrier，将store buffer中缓存的修改刷入L1 cache中，使其他CPU核可以观察到这些修改。之后的写操作不会被调度到之前。
- **lfence**：实现Load Barrier，将invalidate queue失效，强制读取入L1 cache中。之后的读操作不会被调度到之前。
- **mfence**：实现Full Barrier，同时刷新store buffer和invalidate queue，保证mfence前后读写操作的顺序。mfence之后写操作结果全局可见之前，mfence之前写操作结果全局可见。
- **lock**：修饰当前指令操作的内存只能由当前CPU使用。让指令操作本身原子化，且自带Full Barrier效果。**Lock不是一种内存屏障，但能完成类似内存屏障的功能**。任何带有lock前缀的指令以及CPUID等指令都有内存屏障的作用。

X86-64一般情况根本不会需要使用lfence与sfence指令，除非操作Write-Through内存或使用non-temporal指令（NT指令，如movntdq、movnti、maskmovq），这些指令使用Write-Through内存策略，通常用于图形学或视频处理。

GNU中的三种内存屏障定义方法（结合了编译器屏障和三种CPU屏障指令）：
```c
#define lfence() __asm__ __volatile__("lfence": : :"memory")
#define sfence() __asm__ __volatile__("sfence": : :"memory")
#define mfence() __asm__ __volatile__("mfence": : :"memory")
```
代码中使用这些屏障是一种长远考虑，按照Interface写代码最保险。目前在X86下视为编译器屏障即可。

## 17. volatile关键字的底层原理是什么？

volatile被引入的成因：
1. 为解决CPU从内存读取数据的瓶颈，引入**Cache**，提高CPU执行效率。
2. 但又引入数据一致性问题，于是引入**MESI缓存一致性协议**。
3. 但如果CPU间严格遵守MESI协议，执行效率大幅降低，为妥协又引入**Store Buffer和Invalidate Queue**。
4. 这样数据由强一致性变成**弱一致性**，可能导致短时间内读取数据不一致，产生bug。
5. 于是又引入**内存屏障**来保证数据可见性。**volatile是用来在变量前后添加内存屏障以保障可见性的**。

**volatile的底层实现**：被volatile修饰的变量，在编译成机器指令时会在写操作后面加上一条特殊指令：**"lock addl #0x0, (%rsp)"**。这条指令会将CPU对此变量的修改**立即写入内存**，并**通知其他CPU将缓存行置为无效状态**。这里的lock主要用于锁总线，锁总线期间其他CPU的读写请求都会被阻塞，直到锁释放。

volatile在写操作时添加了**写屏障**，避免volatile变量写与之前其它变量写的排序；在读操作时添加了**读屏障**，避免volatile读与之后变量读的排序。其中隐含了**happens-before规则**：在时序上，如果对一个volatile变量的写操作先于后面的对这个变量的读操作执行，那么volatile读操作必定能读到volatile写操作的结果。

**既然有了MESI协议，为什么还需要volatile？**
1. 并不是所有硬件架构都提供了相同的一致性保证，**JVM需要volatile统一语义**（MESI只解决CPU缓存层面的问题，没有涉及其他层面）。
2. 可见性问题不仅局限于CPU缓存内，**JVM自己维护的内存模型中也有可见性问题**，volatile可以解决JVM层面的可见性问题。
3. **真·重排序**也会导致可见性问题，MESI无法解决。

## 18. 为什么说volatile解决死循环问题的分析不全对？JIT优化是怎么回事？

以下代码中t1线程陷入死循环：
```java
private static boolean running = true;
```

很多文章说是因为可见性问题——t2修改了running但对t1不可见，导致t1中的running始终为true，只需加volatile即可。

**结论（加volatile）没有错，但分析过程大错特错**。真实原因是**JIT的优化**导致的。由于线程t2开始前sleep了1s，t1不断执行while循环，到一定次数后，JIT将这段代码判定为**热点代码**，会将running直接替换为**true**，导致死循环。

证明方法有两种：
1. **关闭JIT优化**：执行时加上 `-Djava.compiler=NONE`。
2. **注释掉Thread.sleep(1000)**：t1的while循环次数还未被JIT判定为热点代码前，t2修改running为false，由于缓存一致性协议，running值同步到t1中，循环正常退出。

## 19. 单线程在多核CPU上执行会有可见性问题吗？

**不会**。因为线程在切换上下文时，会**保存上下文的东西**（包含CPU寄存器的所有内容），再切换到该线程时，会**恢复上下文保存的东西**。比如线程先在CPU0执行a=1，然后调度到CPU1执行a=2，再调度回CPU0时，恢复的内容包含a=2。

## 20. Hotspot VM是如何实现StoreLoad内存屏障的？

由于x86是遵循**TSO（Total Store Order）**的最终一致性模型，在data race情况下需要同步，尤其是在StoreLoad场景。其余场景由于store buffer的特殊性以及不存在invalidate queue的因素，可以不考虑重排序问题。因此**在x86平台下，除了StoreLoad Barrier以外，其余Barrier均为空操作**。

Hotspot VM选择**LOCK指令作为StoreLoad屏障**，源码中：
```c
void membar(Membar_mask_bits order_constraint) {
  if (os::is_MP()) {
    if (order_constraint & StoreLoad) {
      // 所有可用芯片支持"locked"指令作为屏障，且比cpuid快得多
      int offset = -VM_Version::L1_line_size();
      if (offset < -128) { offset = -128; }
      lock();
      addl(Address(rsp, offset), 0); // 声言lock#信号
    }
  }
}
```
使用 `lock(); addl(Address(rsp, offset), 0);` 实现，该操作除了触发flags和引入对目标内存位置的虚假依赖外，相当于无操作。选择`[esp-C]`作为操作地址是为了避免与当前方法栈上的数据冲突。
