# format-md：面试笔记 Markdown 格式优化器

本 skill 用于按本仓库面试笔记规范，**仅优化已有 `.md` 文件的排版格式**，不修改任何原始信息。

## 触发条件

当用户提出以下类型请求时调用本 skill：

- 「优化 xx.md 第 N 个问题的格式」
- 「把 xx 文件的方案二、方案三格式整理一下」
- 「美化这一段排版/排版优化/格式化」
- 「只调格式，内容不要动」

## 核心准则（必须严格遵守）

- **原始信息零丢失**：不增加、不删除、不改写任何事实、数据、术语、句子；只允许调整空白、标点周围空格、列表/加粗等纯排版结构。
- **作用域最小化**：用户指定哪个问题/哪一段，就只改哪一段；其余内容**逐字节保持不变**。
- **只处理 `.md` 文件**：遇到非 `.md` 文件直接拒绝并提示。
- **不执行编译、运行、测试**；不创建新的 md 文件、不写 README、不生成测试代码。
- **不修改文件结构层级**：全文只有 `#` 一级标题；不引入 `##`、`###` 等子标题；不引入「原理分析」「总结」等无意义二级标题。
- **保留问题编号顺序**：`# 1.` `# 2.` `# 3.` ... 编号和顺序不变。
- **问题之间不使用 `---` 分隔**，已有的不要加，没有的也不要加。

## 允许做的格式动作

- 把分号 `；` 拼接的多要点拆成 `- ` 无序列表，**每个要点独立成行**。
- 把「步骤 1/2/3」「第一步、第二步」之类的顺序流程改为有序列表 `1. 2. 3.`。
- 给关键词、关键结论加 `**加粗**`：仅加在原文已有的关键概念/结论词上，不要发明新词。
- 命令、SQL、字段名、配置项、文件名用反引号 `` ` `` 包裹（如 `ON DUPLICATE KEY UPDATE`、`expire_logs_days`）。
- 删除原文中**误用的、包裹普通正文的代码块**（` ``` ` 三反引号块），把里面文字还原为普通段落或列表项。
- 把扁平堆叠的「问题/风险/解决方案」结构改为「**风险：** 列表 + **解决方案：** 列表」的成对小节。
- 整理多余空行，保持段落之间一个空行；列表项之间不留空行。
- 修正中英文之间空格、`~` 误写成 `\~` 之类的转义残留。

## 禁止做的事

- 禁止改写句子表达、缩写、扩写、润色措辞。
- 禁止合并/拆分含义不同的两句话。
- 禁止增加自己理解的"小结""一句话总结"等新内容（除非原文已有，仅做格式调整）。
- 禁止删除任何包含信息的文字，哪怕看起来重复。
- 禁止引入二级标题 `##` 把单一问题拆成多块。
- 禁止把列表项内容用分号拼回一行。
- 禁止改动用户未指定的其他问题。

## 标准工作流

1. **先读后改**：用 Read 工具完整读取目标文件，定位用户指定的问题块（`# N. xxx` 到下一个 `# ` 之间）。
2. **二次确认作用域**：如果用户只说"格式化这个文件"未指定具体问题，先用 AskUserQuestion 确认是全文格式化还是某几个问题。
3. **逐题处理**：使用 Edit 工具，`old_string` 必须是该问题块的精确原文，`new_string` 是格式优化后的版本。
4. **信息守恒自检**：改完后，把新旧两段内容里的"事实关键词"心算对齐一遍——人名、数字、术语、API、SQL 关键字、步骤数都必须一一对应，没有遗漏才能完成。
5. **简短汇报**：完成后用 1-3 行说明具体做了哪些格式动作（如"把存在问题改为有序列表 + 加粗关键词"），不要长篇总结。

## 输出回答风格（如果同时需要回答内容）

- 回答面向技术面试口述场景，**无冗余书面语**。
- 多个并列要点必须用 `- ` 列表逐条列出，不要用分号拼一行。
- 关键概念和结论用 `**加粗**` 突出。

## 使用示例

用户：「07分库分表中第 9 个问题的方案二和方案三，只优化格式」

执行：
1. Read `04Mysql/07分库分表.md`，定位 `# 9.` 问题块中"方案二""方案三"两段。
2. 仅对这两段做：列表化、关键词加粗、清理误用代码块、命令加反引号。
3. Edit 替换两段；其余 9 个问题、方案一、方案四完全不动。
4. 用一句话汇报：「已格式化方案二、方案三：流程改为有序列表，问题/解决办法成对列出，关键词加粗。」
