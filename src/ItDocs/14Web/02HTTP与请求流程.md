## 1. RESTful接口中数据库查不到数据，statusCode应该返回200还是4xx？

**建议使用HTTP标准状态码**（400/404等），同时在响应体中附带自定义错误码和msg细化错误。

**返回200的弊端**：
- 前端主流框架均按HTTP状态码判定请求是否正常，返回200不会触发错误处理hook/callback，前端需要对每个请求额外检测自定义error code
- SLA监控等运维系统会放过这类异常请求，造成监控不准确
- 响应的职责不明确，将异常消息和正常数据糅合在一起，增加维护成本

**返回40X的好处**：
- 遵循HTTP协议语义（RFC 7807标准），客户端传参不足以找到对应数据，属于4xx客户端错误
- 充分利用框架原生错误处理能力
- 状态码不够用时可在响应体里附加自定义错误码细化，两者不冲突

**是否复用HTTP Status Code承载错误码，取决于架构层面是否将HTTP识别为独立层次**：
- **HTTP作为基础设施层**（公司有专门团队处理HTTP层）：只要不是HTTP层的问题，尽量返回200，业务层自己设计error体系，避免干扰运维对HTTP层的处理（如404被替换为公益页面）
- **HTTP与API在同一层次**（业务团队直接操控HTTP）：直接复用HTTP Status Code，很多框架原生支持

## 2. 接口设计中用HTTP状态码还是自定义code码？

两者各有优劣：

**HTTP状态码**
- 优点：符合HTTP协议规范，客户端和服务端交换信息方便，200/404/500等已定义好常见情况
- 缺点：非标准的业务逻辑错误（用户未登录、余额不足）难以用标准状态码表达

**自定义code码**
- 优点：灵活表示各种业务错误
- 缺点：不符合HTTP规范，客户端需额外解析code，放弃框架的hook设计

**建议**：以HTTP状态码为主，尽量用标准状态码；特殊业务逻辑用自定义code作为补充，并在文档明确说明。

## 3. 从浏览器点击按钮到服务端返回响应，整个网络请求流程是怎样的？

完整流程从**应用层→传输层→网络层→数据链路层→物理层**逐层拆解，再反向返回。

**前置基础概念**：
- **MAC地址**：二层物理地址，局域网内通信标识，跨网段无效
- **IP地址**：三层逻辑地址，全网路由寻址标识
- **TCP**：四层可靠传输，三次握手、确认应答、滑动窗口、拥塞控制
- **路由**：跨网段转发，依靠路由表、网关、公网路由节点
- **ARP协议**：IP→MAC解析，局域网核心映射

**一、客户端应用层**
1. 用户点击页面按钮，JS发起接口请求（fetch/axios）
2. 拼接HTTP/HTTPS请求报文（请求行、请求头、请求体）
3. **域名解析**：查浏览器缓存→系统缓存→本地hosts→DNS服务器递归查询，拿到服务端公网IP

**二、传输层（TCP封装）**
1. 浏览器选择随机客户端临时端口（1024~65535），目标端口HTTPS=443/HTTP=80
2. **TCP三次握手**：SYN → SYN+ACK → ACK，建立可靠连接
3. HTTPS在此后追加**TLS握手**：协商加密算法、交换证书、生成会话密钥，后续HTTP报文加密传输
4. TCP头部封装：源端口、目的端口、序列号、确认号、标志位、校验和

**三、网络层（IP封装&路由寻址）**
1. 封装**IP数据包**：源IP（客户端内网IP）、目的IP（服务端公网IP）、TTL、校验和
2. 判断目标IP是否同网段：同网段直接ARP找MAC；跨网段匹配路由表发给**默认网关**

**四、数据链路层（ARP→MAC→帧封装）**
1. 发起**ARP请求**（广播）："谁是网关IP？告诉我你的MAC"
2. 网关单播返回**ARP响应**，客户端缓存网关IP-MAC映射
3. 封装**二层数据帧**：源MAC（本机网卡）、目的MAC（网关路由器LAN口MAC）、帧头/CRC

**五、物理层→局域网→公网路由转发**
1. 网卡将二进制帧转为电信号，通过网线/Wi-Fi传输到路由器
2. **路由器处理**：拆二层帧取出IP包→查路由表→**NAT地址转换**（内网IP+端口替换为运营商公网IP+随机端口，保存NAT映射表）→重新封装二层帧
3. **公网多级路由转发**：每经一台路由器**二层MAC不断替换**（下一跳MAC实时变更），**三层IP全程不变**，**TTL每跳-1**防止环路

**六、服务端入网（公网→内网）**
1. 数据包到达机房防火墙/负载均衡SLB/云网关
2. 解二层帧，基于目标IP+端口转发：负载均衡分发到后端业务服务器
3. ARP解析后端服务器内网MAC，封装二层帧转发到业务服务器网卡

**七、服务端逐层解包**
1. 网卡收到二进制数据流→物理层解析
2. 数据链路层：拆帧头、校验CRC、取出IP包
3. 网络层：拆IP头部、取TCP报文段
4. 传输层：TCP协议栈校验序列号/ACK/校验和，根据目标端口将连接交给对应进程
5. HTTPS需完成TLS解密还原HTTP明文
6. Web容器接收HTTP请求，路由匹配接口路径

**八、服务端业务执行**
1. 网关/拦截器：鉴权、限流、日志、参数校验、跨域处理
2. 控制器层：解析请求参数/Header/Body
3. Service层：核心业务逻辑（查数据库、调RPC、缓存Redis、消息队列等）
4. 持久层：数据CRUD
5. 拼装**响应数据**（JSON/HTML），封装HTTP响应报文

**九、响应原路返回**
1. 应用层→传输层→网络层（源IP=服务端IP，目的IP=客户端公网IP）
2. 数据链路层：服务器查网关MAC，封装二层帧
3. 公网路由反向转发：每跳更换MAC，运营商路由器通过NAT映射表还原客户端内网IP+端口
4. 家庭路由器收到响应帧，根据NAT记录转发给客户端
5. 浏览器解析JSON/HTML，渲染页面

**核心总结（高频面试重点）**：
- **MAC每跳必变，IP全程不变**
- **TCP**：三次握手、可靠传输、流量控制、拥塞控制
- **ARP**：IP→MAC解析，局域网通信基础
- **NAT**：解决公网IP不足，内网IP统一转换为公网IP
- **跨网段**：靠网关+路由表转发

## 4. 服务器一般用内网IP，公网请求怎么到达内网服务器？

核心是 **DNAT（目的地址转换）+ 负载均衡（LB）** 两层转发。

完整链路：客户端→公网→边界网关/防火墙→负载均衡器LB→内网服务器

**数据包到达公网IP后的过程**：
1. 目的IP为公网IP（如220.181.38.251:80）的包被边界网关/LB接收
2. **DNAT转发**：LB做目的地址改写，将目的IP改为选中的内网IP（如10.1.2.3:80），同时记录DNAT会话表
3. 包在内网路由，交换机ARP找内网服务器MAC，送达网卡
4. **响应原路返回**：服务器回包（源IP=10.1.2.3，目的IP=客户端公网IP）→LB查DNAT表将源IP改回公网IP→返回公网→客户端

**百度真实架构**：公网VIP→负载均衡→成千上万内网服务器
- 对外只有几个公网IP（VIP），实际是负载均衡器的虚拟IP
- LB按轮询/权重/最少连接选一台内网Real Server，做DNAT
- 内网全部用私网IP（10.0.0.0/8），安全、易扩展、方便迁移

## 5. 查路由表时怎么判断下一跳？

**路由表只做一件事：根据最长前缀匹配找下一跳IP**，MAC只负责把包送到下一跳。

**三种路由**：
- **直连路由**：目标在同一局域网，不需要下一跳，直接ARP查目标MAC发以太网帧
- **下一跳IP（网关路由）**：跨网段访问，下一跳就是网关（出口路由器IP）
- **默认路由（0.0.0.0/0）**：所有找不到更精确路由的，全走这里

**IP从头到尾不变，MAC每跳都变**。

## 6. 百兆和千兆网线有什么区别？

千兆对线材要求更高，必须**8芯全通**，百兆只需要4芯。

限制因素：趋肤效应（Skin Effect）、信号衰减（Insertion Loss）、串扰（Crosstalk）。

## 7. 网卡、网卡驱动、TCP/IP协议栈、epoll、Java NIO分别工作在网络的哪一层？

**网卡（物理硬件）**：物理层+链路层
- 物理层：光电信号、电压、传输介质
- 链路层：MAC地址、帧封装、CRC校验、二层转发

**网卡驱动**：链路层内核驱动（内核态）
- 屏蔽硬件差异，内核协议栈和物理网卡的桥梁
- 负责网卡中断、收发队列、帧收发硬件控制

**TCP/IP协议栈（Linux内核）**：网络层+传输层
- 网络层：IP、ICMP、ARP
- 传输层：TCP、UDP、拥塞控制、滑动窗口、重传、连接管理

**epoll**：操作系统内核系统调用层，介于传输层与应用层之间
- Linux内核的IO多路复用事件监听机制
- 不处理网络协议，只监听socket文件描述符的可读/可写/异常事件

**Java NIO**：纯应用层
- Buffer、Channel、Selector是核心API
- Selector底层在Linux上封装调用epoll
- 关键区别：epoll是内核态OS能力，Java NIO Selector是应用层API封装

**完整数据流层级串联（一次TCP接收）**：
1. 网卡（物理+链路层）收到二进制帧
2. 网卡驱动（内核链路层）硬件中断、收取帧到内核缓冲区
3. 内核TCP/IP协议栈（网络层IP解析→传输层TCP拆包、校验、重组）
4. 内核把就绪socket事件放入epoll就绪队列
5. Java NIO Selector（应用层）调用epoll_wait拿到就绪事件
6. 业务代码通过Channel+Buffer读取应用层数据

## 8. 什么是跨域？实现跨域的方式有哪些？

跨域指浏览器**同源策略**限制，要求协议、域名、端口三者一致才能正常请求。

实现跨域的方式：
1. **反向代理**（推荐开发环境使用）
   - 通过自己的服务器（Nginx/Node）在后端请求目标服务器数据，返回给客户端
   - 前端开发环境可用webpack-dev-server的proxy配置（底层是http-proxy-middleware）
   - 开发体验好，无需修改业务代码

2. **JSONP**（仅支持GET请求）
   - 通过`<script>`标签不受同源政策限制的特性
   - 服务器将数据作为参数放在指定回调函数里返回
   - 需后端配合配置回调函数

3. **WebSocket**
   - 使用ws://或wss://协议，不实行同源政策
   - 请求中有Origin字段，服务器可根据Origin判断是否许可通信

4. **CORS**（根本解决方案）
   - 全称Cross-origin resource sharing，需要浏览器和服务器同时支持
   - 浏览器自动完成通信，对开发者透明，代码与同源AJAX无差别
   - **关键是服务器实现CORS接口**，添加相应的响应头（Access-Control-Allow-Origin等）
   - 支持任何类型的请求，IE需不低于IE10
