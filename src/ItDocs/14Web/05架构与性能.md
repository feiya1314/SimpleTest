## 1. 分布式Web系统架构演进的各阶段是怎样的？

- **单体架构**：所有功能部署在单一应用中，代码耦合，扩展困难
- **垂直拆分**：按业务模块拆分为多个独立应用（用户、商品、订单），通过API通信
- **分布式服务**：核心业务抽离为独立服务，引入RPC框架（Dubbo/Thrift），服务注册发现
- **SOA/微服务**：服务粒度更细，独立部署独立数据库，轻量级通信（REST/gRPC），容器化编排（K8s）
- **服务网格**：通信逻辑下沉到Sidecar代理，业务代码与基础设施解耦
- **中台架构**：沉淀通用业务能力为共享中台，前台应用灵活调用

## 2. 架构演进中引入了哪些核心组件？

- **负载均衡**：Nginx/LVS，分发请求提升吞吐
- **缓存**：Redis/Memcached，减轻DB压力
- **消息队列**：Kafka/RabbitMQ，异步解耦、削峰填谷
- **配置中心**：Nacos/Apollo，统一配置管理
- **注册中心**：Zookeeper/Nacos/Nacos，服务发现与健康检查
- **分布式数据库**：分库分表（ShardingSphere）、读写分离
- **链路追踪**：Skywalking/Pinpoint，监控调用链
- **容器编排**：Kubernetes，自动化部署扩缩容

## 3. 什么是QPS和TPS？有什么区别？

**QPS（Queries Per Second）**：每秒查询数，衡量读性能。
**TPS（Transactions Per Second）**：每秒事务数，衡量写/事务性能。

区别：
- 只读操作（如Nginx代理）→ 用QPS
- 涉及写操作（如Kafka）→ 用TPS
- 读写兼有（如MySQL）→ 两者都需关注
- 二者互相制约，分别侧重读/写性能

## 4. 提升系统QPS/吞吐量的方法有哪些？

**应用层**：
- 无状态设计，便于水平扩展
- 连接池复用（DB/HTTP连接池）
- 异步非阻塞编程（CompletableFuture/协程）
- 本地缓存（Caffeine）减少RPC

**缓存层**：
- 多级缓存（本地 → 分布式缓存 → DB）
- 缓存预热，防击穿/雪崩/穿透

**数据层**：
- 读写分离、分库分表
- 索引优化、慢查询治理
- 批量读写，减少网络IO

**架构层**：
- CDN加速静态资源
- MQ异步解耦，削峰填谷
- 限流降级熔断，保核心链路
- 水平扩缩容

## 5. 常见的限流算法有哪些？

- **固定窗口计数器**：固定时间窗口计数，超阈值拒绝。缺点：窗口边界流量翻倍
- **滑动窗口计数器**：窗口细分为小格滑动统计，解决边界突发，精度取决于格数
- **漏桶算法**：请求以固定速率流出，超桶容丢弃/排队，强制平滑流量
- **令牌桶算法**：固定速率添加令牌，请求需获取令牌，允许一定突发，最常用（Guava RateLimiter、Nginx limit_req）

## 6. 限流实现方式有哪些？

- **单机限流**：Guava RateLimiter（令牌桶）、AtomicInteger计数器
- **分布式限流**：Redis + Lua脚本实现滑动窗口或令牌桶
- **网关限流**：Nginx limit_req/limit_conn、Spring Cloud Gateway
- **框架限流**：Sentinel（限流/熔断/降级一体化）

## 7. 秒杀系统如何进行流量削峰？

**核心思路**：把百台服务器扛不住的问题简化为单台服务器轻松扛住，不让无效请求漏到后方。

![秒杀整体架构](../assets/14Web/a4ac777f22ce48229edccc59bed26ecc.png)

削峰手段：
- **业务隔离**：秒杀单独部署，避免影响其他服务
- **页面静态化/CDN**：静态资源提前推送到CDN
- **限流降级**：高于阈值直接返回失败，关闭非核心接口
- **队列削峰**：请求入队排队，异步处理，超出队列长度直接挡回
- **缓存原子扣减**：Redis原子操作扣库存，不直接操作DB
- **异步落库**：秒杀日志 + Worker异步计算，最终一致性写入DB
- **熔断**：超阈值接口返回默认数据或直接失败

## 8. 秒杀系统中库存如何管理？

- 活动创建时将库存写入Redis
- **下单锁定库存，支付减库存**
- 下单锁定需定时释放超时订单
- 付款减库存防止超卖但用户体验差
- Redis原子扣减保证同一时刻单线程操作库存
- 通过秒杀日志 + Worker异步落表保证最终一致性

## 9. 用户账号体系的核心目标是什么？

两个核心目标：
- **身份识别**：准确识别用户，物理用户与虚拟账户一一对应，不认错、不丢失、不被冒充
- **限权**：用户数据完整展示，不越权、不丢失

所有用户体系迭代都围绕这两个目标——降低身份识别成本，提升限权准确性。

## 10. 微信生态中openid和unionid有什么区别？

| ID | 唯一性范围 | 说明 |
|---|---|---|
| **openid** | 用户 + 应用（appid） | 不同公众号/小程序/App的openid不同 |
| **unionid** | 用户 + 开发者主体（公司） | 同一公司下所有应用的unionid相同 |

- openid = 用户微信号 + 应用的appid
- unionid = 用户微信号 + 开发者主体
- unionid用于关联同一用户在多个应用下的账号
- 微信网页授权获得的openid与对应公众号粉丝的openid一致
