## 1. Python与Java的核心差异有哪些？

- **静态方法翻译**：Java静态方法对应Python模块级函数，非类方法或静态方法；static final字段对应模块级常量。将Java静态方法翻译为classmethod会有额外内存分配开销（调用classmethod比staticmethod或function多一次内存分配），且会导致使用者输入Foo.Foo.someMethod而非Foo.someFunction
- **属性链开销**：Java中`.`分隔的名称由编译器在编译时查找，运行时无论多少个点都无开销；Python中每次运行时都要查找，每个点都有计算开销。应遵循"平铺比嵌套好"原则
- **switch语句**：Python用字典（哈希表）替代if-elif链。CPython的字典实现是已知宇宙中性能最佳的哈希表实现之一。如字典缺失键可用`dict.get(key, default)`优雅处理：`size_to_point = {'small':7, 'normal':9, 'large':12}; point_size = size_to_point.get(size, size)`
- **XML适用场景**：Python中XML仅用于互操作（导入/导出），不用做核心功能；Python代码比XML更易写，Python处理代码的速度远比你的代码处理XML快得多。Java中XML可以拯救你因为它能"不通过编码"实现DSL和扩展灵活性（编码意味着重新编译），但Python中编码更方便。例外：实现已有XML标准做互操作、创建导入/导出格式、或创建XML编辑器/处理工具
- **Getter/Setter**：Python从普通属性开始，后续可用`@property`替换，无需提前写getter/setter。Java中必须用getter/setter因为公共字段不允许你之后改变主意再改用getter/setter。Python中可以先使用普通特性，任何时间改变想法而不影响任何类调用方。不要先写getter/setter再包在property里
- **代码重复**：用闭包消除重复代码模板。外部函数接收变化参数，返回内部函数模板。如`def makeAdder(addend): def add_it(x): return x+addend; return add_it`，使用时`addOne = makeAdder(1)`。Java中因静态类型约束常需反复写相似方法，Python中没必要也不值得
- **核心理念**：习惯Java后不要相信本能（本能已为Java调节而非Python）。多问"Python如何已解决我的问题"和"哪个Python语言特性最接近我的问题"。善用语言特性可提升10倍生产力。如果怀念Java IDE，考虑是否你的Python程序比需要的更复杂
- **反射与内省**：Python中类本身就是对象，无需`.class`。`dir()`查看属性名，`inspect`模块检查函数签名，`type(name, bases_tuple, classdict)`创建新式类，`compile()`编译源码字符串，`exec`执行代码
- **猴子补丁 vs 子类化**：可考虑猴子补丁（替换标准库的功能）加速原型开发，但注意不具扩展性（多个模块同时monkey-patch同一对象会有冲突），泛型函数是更干净的长期替代方案

## 2. Go的GOPATH是什么？

GOPATH是Go的工作目录环境变量，指定Go寻找包、可执行程序等的位置，可设置多个目录（多目录用系统路径分隔符分隔）。

**作用**：
- 存放SDK以外的第三方类库
- 存放个人可复用的代码

**多GOPATH技巧**：设置两个GOPATH目录很有用——第一个放第三方包（go install会使用它作为默认位置），第二个放你自己的项目。在第二个GOPATH下使用go命令工作。

**目录结构示例**：每个GOPATH下应有src、pkg、bin三个子目录。

**多GOPATH问题**：
- 设置多个GOPATH可能导致当前项目错误引用其他目录的Go源码文件
- 例如：将projectA设为GOPATH，后续获取projectB源码但忘记修改GOPATH仍指向projectA，编译后会输出错误的工程版本
- 建议每个项目只设置一个GOPATH，不使用多个GOPATH和全局GOPATH

**go build/install**：
- `go build`：编译包及其依赖，生成可执行文件到当前目录
- `go install`：编译并安装，将可执行文件安装到`$GOPATH/bin`，已编译的包会缓存加速后续构建

## 3. Go Modules（go mod）包管理是什么？

Go 1.11+ 引入的官方包管理方案，Go 1.12进一步完善，Go 1.13起成为默认模式。

**旧版（GOPATH模式）问题**：
- 依赖包需手工下载，不使用额外工具时Go的依赖管理仅"仅仅可用"
- 第三方包无版本概念，作者做了不兼容升级会让开发者很难受
- 协作开发需统一所有成员本地`$GOPATH/src`下的依赖包
- 包地址转移后需手动修改源码引用
- 第三方包与自己的包的源码都在`src`下很混乱，混合技术栈项目目录存放有问题

**新模式优势**：
- 自动下载依赖包
- 项目不必放在`GOPATH/src`内
- 生成`go.mod`文件记录包依赖及版本号
- 依赖的第三方包准确指定版本号
- 可用`replace`替换已转移的包地址，无需修改代码
- 依赖下载到`$GOPATH/pkg/mod`，可保留同一包的不同版本（如`github.com/astaxie/beego@v1.11.1`）
- 生成`go.sum`文件维护每个模块版本内容的加密哈希，确保将来下载的位与第一次相同，防止恶意或意外篡改（不需手工维护，应检入版本控制）

**核心命令**：
- `go mod init <模块名>`：初始化模块，生成go.mod。子目录不需要init，所有子目录的依赖由根目录go.mod管理
- `go mod tidy`：添加缺失的模块并移除未使用的模块
- `go run/build`：自动解析代码中的依赖包并下载，将依赖关系和版本写入go.mod和go.sum
- `go get <包路径>`：获取或更新依赖（新模式下go run会自动处理）
- `go mod vendor`：将依赖复制到vendor目录

**go.mod文件结构**：
- 第一行声明`module <模块名>`
- `go 1.12`：Go版本
- `require github.com/astaxie/beego v1.11.1`：依赖及版本号（版本格式`vn.n.n`，与仓库release标签一致，未标记版本默认为`v0.0.0`）
- `replace golang.org/x/text => github.com/golang/text latest`：用指定包替代原包地址，原理是下载替代包到`$GOPATH/pkg/mod`原路径下

**模块名的用途**：项目不在`$GOPATH/src`内时，用`<模块名>/<子目录路径>`引用自己内部的包。如`go mod init hello`，子目录utils中的包用`import "hello/utils"`引用

**GO111MODULE 环境变量**：
- `auto`（默认）：项目在`$GOPATH/src`里使用旧模式依赖，在`$GOPATH/src`外使用go.mod
- `on`：强制模块模式，无论在何位置都使用go.mod的require包
- `off`：关闭模块模式，使用旧GOPATH方式

**老项目迁移**：
1. auto模式下把项目移到`$GOPATH/src`外
2. 进入目录运行`go mod init <模块名>`
3. 运行`go build`或`go run`一次即可

## 4. Go的context包是什么？

context包用于传递请求范围的值、取消信号和超时截止时间，主要解决goroutine间的协作取消和超时控制问题。

**核心概念**：
- 创建一个Context树，根节点由`context.Background()`或`context.TODO()`创建
- 派生Context：`WithCancel`、`WithDeadline`、`WithTimeout`、`WithValue`
- 当父Context被取消时，所有派生的子Context也会被取消

**使用场景**：
- HTTP请求处理链中传递请求范围内的值（如认证令牌、追踪ID）
- 控制goroutine的生命周期，当超时或取消时通知所有相关goroutine退出
- 数据库查询、RPC调用等设置超时截止时间

**最佳实践**：
- Context应作为函数的第一个参数传递，命名通常为`ctx`
- 不要将Context存储在结构体中
- 只传递请求范围的值，不要用Context传递可选参数

## 5. Go的build和install命令有何区别？

- **go build**：编译包及依赖，生成可执行文件到当前目录。只编译不安装，适合开发调试阶段快速验证。如果包已被go install安装过且源码未变，go build会重用缓存结果加速构建
- **go install**：编译并安装，将可执行文件安装到`$GOPATH/bin`（或`$GOBIN`），同时将编译后的包对象（.a文件）缓存到`$GOPATH/pkg`，加速后续构建。适合部署或长期使用
- **关键区别**：go install会把最终产物安装到bin目录而不是当前目录；已编译的包缓存可加速依赖该包的后续构建过程，而go build每次重新编译所有依赖（不过新版本Go也有构建缓存机制，两者差异已缩小）
