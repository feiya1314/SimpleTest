
# 1. 什么是 Java 动态追踪技术？有哪些应用场景？

Java 动态追踪指**在不重启 JVM 的前提下，动态获取运行态信息或修改类行为**的技术。底层基于 **Java Agent + Instrumentation + Attach API**。

应用场景：
- 线上问题排查：实时查看方法调用参数、返回值、异常
- 性能诊断：统计方法耗时、定位热点
- 动态日志：临时打开特定方法的 DEBUG 日志级别
- AOP 增强：热更新逻辑、埋点采集
- APM（SkyWalking、Pinpoint）：对业务代码无侵入的链路追踪

# 2. Java Agent 的原理是什么？如何实现？

Java Agent 是 JDK 1.5 引入的**字节码增强**机制，允许在 JVM 启动前（`premain`）或运行时（`agentmain`）修改类的字节码。核心接口是 **`java.lang.instrument.Instrumentation`**。

实现步骤：
1. 编写 Agent 类，实现 `premain` 或 `agentmain` 方法
2. 通过 **`Instrumentation.addTransformer()`** 注册 `ClassFileTransformer`
3. 在 `transform()` 方法中用字节码框架（ASM、Javassist、ByteBuddy）修改字节码
4. 打包成 JAR，`META-INF/MANIFEST.MF` 中指定 `Premain-Class` / `Agent-Class`

```java
public class MyAgent {
    public static void premain(String args, Instrumentation inst) {
        inst.addTransformer(new MyTransformer(), true);
    }
}
```

# 3. premain 和 agentmain 有什么区别？

| 对比项 | premain | agentmain |
|--------|---------|-----------|
| 加载时机 | JVM 启动前，`main()` 之前 | JVM 运行中，动态附加 |
| 启动方式 | `-javaagent:agent.jar` | Attach API 动态加载 |
| 类重转换 | 不能对已加载类做 retransform | 可通过 `retransformClasses()` 重新转换已加载类 |
| 典型场景 | 启动时全量增强（如 APM 探针） | 运行时诊断（如 Arthas attach） |

# 4. Attach API 是什么？如何实现运行时 Agent 加载？

Attach API 是 **`com.sun.tools.attach`** 包提供的机制，允许一个 JVM 进程向另一个目标 JVM 发送指令并加载 Agent。

流程：
1. 通过 **`VirtualMachine.list()`** 列出所有 Java 进程，获取目标 PID
2. 调用 **`VirtualMachine.attach(pid)`** 连接目标 JVM
3. 调用 **`vm.loadAgent(agentJarPath)`** 将 Agent JAR 发送到目标 JVM
4. 目标 JVM 收到后调用 Agent 的 **`agentmain`** 方法
5. 完成后调用 **`vm.detach()`** 断开连接

```java
VirtualMachine vm = VirtualMachine.attach("12345");
vm.loadAgent("/path/agent.jar");
vm.detach();
```

# 5. Instrumentation 的核心能力有哪些？

- **addTransformer()**：注册 ClassFileTransformer，可在类加载时拦截修改字节码
- **retransformClasses()**：对已加载的类重新转换（需要 `canRetransform=true`）
- **redefineClasses()**：用新字节码重新定义已加载类（无需 Transformer，直接替换）
- **getAllLoadedClasses()**：获取当前 JVM 所有已加载的类
- **getInitiatedClasses()**：获取由指定 ClassLoader 加载的类
- **canSetNativeMethodPrefix()**：是否支持设置 native 方法前缀

retransform 与 redefine 的区别：retransform 会重新走 Transformer 链，redefine 直接用新字节码覆盖。

# 6. Arthas 的原理是什么？它为什么能 attach 到任意 Java 进程？

**Arthas** 是阿里巴巴开源的 Java 诊断工具，底层使用 **Java Agent + Attach API + ASM 字节码增强**。

核心工作流程：
1. 启动时用 **VirtualMachine.attach(pid)** 连接到目标 JVM
2. 加载 Arthas 自己的 Agent JAR（`arthas-agent.jar`）
3. Agent 启动后通过 **Instrumentation.retransformClasses()** 对目标类进行增强
4. 在方法前后插入拦截逻辑，收集方法调用参数、耗时、异常等信息
5. 结果通过内嵌的 **HTTP/Telnet** 服务器回传给 Arthas 客户端

Arthas 最常用的命令：
- **watch**：观察方法调用，查看参数、返回值、异常
- **trace**：追踪方法内部调用路径和耗时
- **stack**：输出方法调用栈
- **tt**：时空隧道，记录方法调用快照用于回放
- **ognl**：在线执行 OGNL 表达式，查看任意变量值
- **sc/sm**：查看 JVM 已加载的类信息和方法信息
- **dashboard**：查看线程、内存、GC 等实时数据
- **thread**：查看线程状态、堆栈，定位死锁或 CPU 高线程

# 7. 如果目标 JVM 启动了 `-XX:+DisableAttachMechanism`，Arthas 还能 attach 吗？

不能。`-XX:+DisableAttachMechanism` 会关闭 JVM 的 Attach 机制，导致 `VirtualMachine.attach()` 调用失败。

但可以通过以下方式绕过：
- 启动时直接使用 `-javaagent` 参数预加载 Agent
- 通过 **JVMTI**（JVM Tool Interface）实现底层注入（需 JVM 层面支持）
- 修改目标 JVM 的启动参数，去掉该配置后重启

# 8. 自己实现一个简易版的 Arthas 需要哪些步骤？

1. 编写 Agent 类，实现 `agentmain`，注册 ClassFileTransformer
2. Instrumentation 设置 `canRetransform=true`
3. 利用 ASM/Javassist 在 transform 时拦截目标方法
4. 在方法前后插入计时和参数采集代码
5. 通过 socket、HTTP 等方式与客户端通信
6. 编写客户端接收命令，动态设置拦截规则

# 9. Java Agent 与 Spring AOP 的区别是什么？

- **Spring AOP**：基于动态代理（JDK Proxy / CGLib），只能增强 Spring 容器管理的 Bean，运行时生成代理对象。对 final 类、静态方法、私有方法无效
- **Java Agent**：基于 Instrumentation 字节码增强，可修改任意已加载类的字节码，**不限于 Spring Bean**，可修改 JDK 类库、第三方 Jar 中的类
- **灵活度**：Agent 可以在类加载前或加载后修改字节码，影响范围更大
- **侵入性**：Spring AOP 需要 Bean 走容器管理；Agent 对业务代码完全无侵入

# 10. 字节码增强框架 ASM、Javassist、ByteBuddy 有什么区别？

- **ASM**：直接操作字节码指令，性能最高、体积最小，但学习曲线陡峭，需要理解字节码指令集
- **Javassist**：提供源码级别的 API，可直接用 Java 字符串插入逻辑，易上手但灵活度不如 ASM
- **ByteBuddy**：封装了 ASM，提供优雅的 fluent API，类型安全，推荐在新项目中使用

```java
// ByteBuddy 示例
new AgentBuilder.Default()
    .type(ElementMatchers.named("com.example.Service"))
    .transform((builder, type, loader, module) ->
        builder.method(ElementMatchers.any())
               .intercept(MethodDelegation.to(MyInterceptor.class)))
    .installOn(inst);
```

# 11. 动态追踪中 ClassLoader 问题如何处理？

当 Agent 增强的类通过系统类加载器加载，但 Agent 自身的类通过自定义类加载器加载时，目标类无法引用 Agent 中的类，会抛出 `ClassNotFoundException`。

解决方案：
- **将增强后的代码注入到目标类的命名空间中**：使用 Instrumentation 的 `appendToBootstrapClassLoaderSearch()` 或 `appendToSystemClassLoaderSearch()` 将 Agent JAR 添加到 Bootstrap/System ClassLoader 的搜索路径
- **使用线程上下文类加载器（Thread Context ClassLoader）**：在增强代码中通过 `Thread.currentThread().getContextClassLoader()` 加载
- **使用辅助类隔离**：将需要目标类调用的逻辑提取到独立接口中，用 `Class.forName` 反射调用

# 12. retransform 会触发类初始化吗？有什么注意事项？

**不会触发类初始化（`<clinit>` 不会再次执行）**。retransform 只替换方法体，不重新执行静态代码块和静态变量赋值。

注意事项：
- 只能修改方法体，**不能增删字段、方法签名、父类、接口**
- 添加/删除方法在 JDK 7 之前不支持，JDK 7+ 可以新增方法（取决于实现）
- retransform 后原有类的实例不受影响，新创建的实例使用新字节码
- 修改后的类中的静态变量值保持不变
