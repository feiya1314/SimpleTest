
# 1. JDK1.6 对 synchronized 做了哪些锁优化？

JDK1.6在锁方面做了哪些改进？高效并发体现在什么技术上？

**原理分析**

JDK1.6之前，`synchronized`直接调用`ObjectMonitor`的`enter`和`exit`，这种锁称为**重量级锁**。从JDK1.5到JDK1.6，HotSpot团队对Java中的锁进行了重要优化，主要包含以下几项技术：

- **适应性自旋锁**：避免线程被挂起阻塞，让线程在CPU上"稍等一下"
- **锁消除**：JIT通过逃逸分析，取消不存在竞争的同步
- **锁粗化**：将多次对同一对象的加解锁合并为一次
- **轻量级锁**：无竞争时通过CAS操作避免互斥量
- **偏向锁**：同一线程重复获取锁时，消除同步操作

这些优化对开发者透明，**只需使用synchronized，虚拟机根据竞争情况自行决定采用哪种锁**。

# 2. 什么是自旋锁？和阻塞锁有什么区别？

自旋锁的原理是什么？为什么需要自旋锁？

**原理分析**

自旋锁的核心思想：当线程请求锁时，如果锁已被持有，**线程不放弃CPU执行时间**，而是循环等待（自旋），不断检查锁是否释放。

**为什么需要自旋锁：** 共享数据的锁定状态一般只会持续很短时间，为了这段时间去挂起和恢复线程并不值得。Java线程映射到操作系统原生线程，阻塞或唤醒需要从**用户态切换到内核态**，开销很大。

**自旋锁 vs 阻塞锁的区别：**

- **阻塞锁**：线程放弃CPU时间，进入等待区，等待被唤醒
- **自旋锁**：线程不放弃CPU时间，保持运行状态，时刻检查共享资源是否可访问
- **响应速度**：自旋锁更快，因为不涉及线程状态切换
- **适用场景**：自旋锁适合**锁持有时间短、线程竞争不激烈**的情况；如果锁持有时间长或线程数多，自旋会浪费大量CPU时间

自旋锁在JDK1.4中引入，JDK1.6中**默认开启**。

# 3. 什么是锁消除？JIT 如何判断可以消除锁？

JIT编译器如何优化不必要的同步？

**原理分析**

**锁消除**是JIT编译器在动态编译同步块时，通过**逃逸分析（Escape Analysis）**判断同步块所使用的锁对象是否只能被一个线程访问。如果锁对象**不会逃逸到其他线程**，JIT编译时会取消这部分代码的同步。

例如以下代码中，`hollis`对象的生命周期只在`f()`方法内，不会被其他线程访问，JIT会将同步消除：

```java
public void f() {
    Object hollis = new Object();
    synchronized(hollis) {
        System.out.println(hollis);
    }
}
```

优化后等效为：

```java
public void f() {
    Object hollis = new Object();
    System.out.println(hollis);
}
```

典型的应用场景是**局部变量使用StringBuffer**这种线程安全的类做字符串拼接，开发者可能忽略了append方法上的synchronized，JIT可以帮忙消除不必要的同步。

锁消除是JIT编译阶段的功能（非javac编译），无法通过反编译直接观察，需要使用`-XX:+PrintEliminateLocks`参数在Server模式下查看。

# 4. 什么是锁粗化？和减小锁粒度是否矛盾？

锁粗化的原理是什么？它和锁粒度控制有什么联系？

**原理分析**

**锁粗化**：当JIT发现一系列连续操作都对同一个对象反复加锁和解锁（甚至出现在循环体中），会将加锁同步的范围**扩散（粗化）到整个操作序列外部**，减少频繁加解锁的开销。

例如：

```java
for(int i=0;i<100000;i++){
    synchronized(this){
        do();
    }
}
```

会被粗化为：

```java
synchronized(this){
    for(int i=0;i<100000;i++){
        do();
    }
}
```

**锁粗化与减小锁粒度并不冲突：**

- **减小锁粒度**：强调将无关的准备工作放在锁外，锁内只处理并发相关的内容，好比去银行办业务前先准备好资料，不在柜台前占用时间
- **锁粗化**：建议同一个线程要执行多个同步操作时，一次性完成所有操作，而非反复取号排队，好比办理多个业务时在同一个窗口一次办完

两者目的都是减少不必要的锁竞争和上下文切换开销。

# 5. 什么是适应性自旋锁？

适应性自旋锁相比传统自旋锁有什么改进？

**原理分析**

**适应性自旋锁**是JDK1.6引入的改进。传统自旋锁的自旋次数是固定的（如10次），而适应性自旋锁的自旋次数是**动态变化的**：

- 如果在同一个锁对象上，自旋等待刚刚成功获得过锁，JVM认为这次自旋成功的可能性大，会**允许自旋更多次数**
- 如果对于某个锁，自旋很少成功获得锁，JVM会**减少自旋次数甚至直接省略自旋**，避免浪费CPU

这种自适应的机制使得自旋锁能够在竞争激烈时自动退化为阻塞，在竞争不激烈时保持高效。
