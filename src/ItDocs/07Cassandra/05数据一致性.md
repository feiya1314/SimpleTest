# 1. 最终一致性如何理解

**最终一致性（Eventual Consistency）** 是分布式系统中一种弱一致性模型，核心思想是：当没有新写入时，经过足够长的时间，所有副本最终会达到一致的状态。

- 允许数据在一段时间内出现不一致，但保证**在没有新更新的情况下**，所有副本最终会收敛到相同的值
- 相对于强一致性（要求实时一致）或读己之写一致性，最终一致性对延迟和可用性的要求最低

**Cassandra 把一致性级别的决定权交到了客户端手中**

- 客户决定每一次操作的一致性级别，即决定写入操作过程中集群内部必须有多少份副本完成才能响应读请求
- 如果客户设置的一致性级别的值小于设置的副本数量值，那么即便一些节点宕机，更新依然成功。

**在 Cassandra 中最终一致性的表现：**

- **写操作**：只要 Coordinator 节点收到 Quorum（或任意数量）副本的成功确认，就会向客户端返回成功。未被确认的副本通过 **Hinted Handoff**、**Read Repair** 和 **Anti-Entropy Repair** 在后台异步同步数据
- **读操作**：如果设置了 **ONE 或 LOCAL_ONE 一致性级别**，读请求只需要等待一个副本返回，该副本可能不是最新数据，导致读到旧值
- 多个副本之间的数据版本通过 **时间戳（Timestamp）** 和 **向量时钟（Vector Clock）** 解决冲突，采取 Last Write Win（LWW）策略决定最终值

触发最终一致的典型场景：

- **节点宕机恢复后**：宕机期间的数据通过 Hinted Handoff 或修复操作完成同步
- **网络分区恢复后**：分区期间的数据差异通过 Read Repair 和 Anti-Entropy 修复
- **写入一致性级别低于 ALL 时**：例如 CL=ONE 写入，部分副本未收到写入

最终一致性的优缺点：

- **优点**：写入延迟低、可用性高（即使部分副本不可用也能写入）、支持跨数据中心部署
- **缺点**：读取可能读到过期数据，应用层需要容忍短时不一致

面试追问点：

- 最终一致性与 **Quorum 机制** 的关系？Quorum 可以减少不一致窗口，但不能完全消除
- Cassandra 如何实现最终一致性后的冲突解决？**Last Write Win（LWW）** 基于时间戳，但 **时钟偏差可能导致问题** ，生产环境建议配置 NTP
- 如何权衡一致性级别？CL=ONE 延迟最低但有最终一致风险，CL=QUORUM 可保证强一致性但牺牲延迟

# 2. 如何保证最终一致性

Cassandra 通过以下四种机制协同保证最终一致性：（只要数据存在副本，就可以通过修复等手段达成最终一致性）

**1、读修复（Read Repair）**

- 读操作时，Coordinator 根据一致性级别向多个副本发起读请求，对比各副本返回的数据摘要
- 若摘要不一致，从数据最新的副本获取完整数据，并向其他过时副本发出写请求进行修复
- 该机制在**读路径上自动触发**，无需人工介入，读写负载均衡时修复效率很高

**2、提示移交（Hinted Handoff）**

当Cassandra写数据时，需要根据写一致性级别将数据写入到N个节点数据副本中，只有N个节点写入成功才会给用户返回操作成功

- 当目标副本节点宕机时，Coordinator 将写入数据以 Hint 形式**本地保存**到系统表 `system.hints`，并立即向客户端返回成功
- Gossip 协议检测到宕机节点恢复后，Coordinator 将保存的 Hint 重放给该节点
- Hint 默认保存 **3 小时**（由 `max_hint_window_in_ms` 控制），超时未重放的 Hint 会被丢弃，需通过反熵修复补齐

**3、反熵修复（Anti-Entropy Repair）**

- 使用 默克尔树**Merkle Tree** 比较各副本节点的数据范围，快速定位不一致的数据分区
- 对于不一致的数据，按时间戳用**最新版本覆盖旧版本**
- 需要手动触发或定期调度执行（`nodetool repair`），相比 Read Repair 是**主动、全量**的修复手段，开销较大但更彻底

**4、墓碑机制与分布式删除（Tombstone / Distributed Deletion）**

分布式删除的难点在于：

- 如果某对象的一个备份节点 A 当前不在线，而其他备份节点删除了该对象
- A 再次上线时，它并不知道该数据已被删除，所以会尝试恢复其他备份节点上的这个对象，
- 使得删除操作无效。

**cassandra:**

- 数据删除不会立即从磁盘擦除，而是写入一个特殊标记 **Tombstone（墓碑）**，并复制到所有副本
- Tombstone 在 **gc_grace_seconds**（默认 10 天）内保留，确保宕机节点恢复后能感知到此数据已被删除，防止被误恢复
- gc_grace_seconds 到期后，Compaction 过程中会真正清理掉 Tombstone 和被标记删除的数据
- 本质上是**将删除转化为一次写入操作**，借助已有的复制和一致性机制传播删除信息

**5、gossip协议通过在消息体中引入版本号**

- 节点只接收版本号更新的数据！通过版本号的比对实现数据在全集群达到一致。

**各机制的适用场景对比：**

- **读修复**：覆盖大多数日常不一致场景，但只在读操作触发时生效，冷数据（极少被读取）的不一致无法修复
- **提示移交**：应对短暂节点宕机（3 小时内），低延迟、零额外开销，但不能替换已超时的 Hint
- **反熵修复**：兜底方案，修复 Hint 超时、长期宕机或网络分区导致的数据差异，建议在低峰期定期执行
- **墓碑机制**：解决分布式环境删除操作的一致性问题，需合理配置 gc_grace_seconds，过大浪费存储，过小有数据误恢复风险

面试追问点：

- 读修复的性能影响？仅在摘要不一致时才拉取全量数据和发起写修复，对大多数一致场景几乎无开销
- Hinted Handoff 是否保证不丢数据？**不保证**，超过 3 小时或 Coordinator 自身宕机都会丢失 Hint，必须配合反熵修复
- gc_grace_seconds 设置过小会怎样？副本数据可能被误认为新数据而恢复已删除数据，出现 **Zombie（僵尸数据）**

# 3. 什么情况会数据不一致

**写入一致性级别低于 ALL 时**

- 使用 CL=ONE 或 CL=QUORUM 写入时，只要部分副本确认成功即返回客户端，未确认的副本可能长时间未收到更新
- 此时若读请求恰好落到未收到更新的副本，就会读到旧数据

**节点宕机**

- 目标副本宕机期间，Coordinator 将写入以 Hint 形式保存
- 若该节点在 **3 小时 Hint 有效期内**未恢复，Hint 被丢弃，数据永久丢失在该副本上，不一致持续存在直到反熵修复执行

**网络分区**

- 集群被网络故障分割为多个隔离的分区，各分区内的节点各自接受写入，数据产生**分支（Fork）**
- 分区恢复后，不同分区的数据可能基于不同的时间戳产生冲突，即使 LWW 机制也可能导致部分写入被覆盖丢失

**时钟偏差（Clock Skew）**

- Cassandra 依赖客户端或 Coordinator 的时间戳决定最新写入（LWW）
- 若各节点系统时钟不同步，时间戳较晚但实际更早的写入可能覆盖时间戳较早但实际更新的写入，导致数据丢失

**Hinted Handoff 自身故障**

- Coordinator 在保存 Hint 后尚未重放时自身宕机，Hint 随 Coordinator 丢失
- 该场景下不一致只能通过 Read Repair 或反熵修复来弥补

**gc_grace_seconds 配置不当**

- gc_grace_seconds 设置过小或未协调各节点时钟，Tombstone 在部分节点上被提前清理
- 宕机节点恢复后，已删除的数据被误当作有效数据同步给其他节点，出现 **Zombie 数据**

**Lightweight Transaction（LWT）部分失败**

- LWT 使用 Paxos 协议实现线性一致性，如果 Paxos 阶段（Prepare/Promise/Accept/Accepted）在某节点上超时或失败
- 可能导致部分副本提交成功而部分未提交，出现不一致，需要重试来收敛

**批量写入（Batch）跨节点失败**

- 一个 Batch 中的多个写入可能部分成功部分失败（即使使用了 LOGGED Batch）
- 失败的部分不会自动回滚已成功的部分，造成同一批次内的数据在副本间不一致

面试追问点：

- 如何最快检测数据不一致？**使用 Digest 查询**：读时各副本返回数据的 MD5 摘要，摘要不一致时再拉取全量对比
- Cassandra 和传统数据库的不一致有何本质区别？Cassandra **主动拥抱最终一致性**，而非意外错误，其副本同步机制本身就是异步设计的
- 生产环境中如何最小化不一致窗口？**定期执行 nodetool repair**、**配置 NTP 同步时钟**、**写操作使用 CL=QUORUM**、**监控 Hint 队列大小和 Tp 延迟**

# 3. 一定能保证数据最终一致性吗？

有了Hint、读修复、反熵修复是否一定能保证数据最终一致性

**答案：不能完全保证。** Cassandra 的最终一致性是一个**概率性保证**，依赖运维规范和配置合理，多种边界情况下这些机制可能失效或不足。

**Hint 的局限性：**

- **Hint 默认保存 3 小时**（`max_hint_window_in_ms`），超过时间窗口的 Hint 会被丢弃，长时间宕机的节点无法通过 Hint 恢复数据
- **Coordinator 宕机时 Hint 丢失**：Hint 保存在 Coordinator 本地，如果 Coordinator 自身宕机，它持有的所有 Hint 都会丢失
- **Hint 重放失败不会重试**：当前版本 Cassandra 不会为重放失败的 Hint 自动重新存储，数据丢失后只能依赖反熵修复
- **Hint 不处理冲突**：Hint 只是重放写入操作，如果数据在宕机期间被其他节点更新，Hint 重放可能覆盖更新版本

**Read Repair 的局限性：**

- **只修复读操作访问到的数据**：从未被读取的冷数据永远无法触发 Read Repair，不一致持续存在
- **后台 Read Repair 概率触发**：`read_repair_chance` 默认 0.1，不是每次读都触发修复，修复不保证收敛时间
- **无法修复写入时遗漏的数据**：Read Repair 只比对已存在的副本数据，如果某个副本从未收到写入（且 Hint 已丢失），Read Repair 无法发现缺失

**Anti-Entropy Repair 的局限性：**

- **需要手动触发或定时调度**：不自动执行，如果运维疏忽未执行 repair，不一致数据会持续存在
- **repair 只在执行时保证一致性**：repair 之后产生的新不一致需要下一次 repair 修复
- **gc_grace_seconds 配置不当导致 Zombie**：如果 repair 频率低于 gc_grace_seconds 周期，Tombstone 可能被提前清理，恢复已删除数据
- **repair 本身可能失败或中断**：网络问题、节点负载过高可能导致 repair 未完成，部分数据未修复

**无法保证最终一致性的典型场景：**

**1. 长时间多节点同时宕机**

- 多个副本节点同时宕机超过 3 小时，Hint 全部过期丢失
- 恢复后如果反熵修复未及时执行，数据永久丢失
- 即使执行 repair，如果宕机期间数据已被删除（Tombstone 清理），可能产生 Zombie 数据

**2. gc_grace_seconds 与 repair 频率不匹配**

- repair 执行周期 > gc_grace_seconds，Tombstone 在部分节点上被 Compaction 清理
- 宕机节点恢复后，已删除的数据被误当作有效数据同步回来，出现 Zombie 数据
- **这是最常见的无法保证最终一致性的场景**

**3. Coordinator 自身故障**

- Coordinator 在保存 Hint 前宕机，写入数据在该副本上永久丢失
- 即使其他副本有数据，该 Coordinator 持有的 Hint 无法恢复

**4. 时钟偏差导致 LWW 误判**

- 各节点系统时钟不同步，时间戳较晚但实际更早的写入覆盖时间戳较早但实际更新的写入
- LWW 机制导致正确数据被错误覆盖，这种覆盖是**不可逆的数据丢失**

**5. 批量写入部分失败**

- Batch 中的多个写入可能部分成功部分失败，失败部分不会自动回滚已成功的部分
- 同一批次内的数据在副本间不一致，需要应用层自行处理

**6. Repair 执行不当**

- repair 只在 gc_grace_seconds 窗口内执行一次，可能遗漏该窗口外的不一致数据
- repair 过程中网络分区导致部分修复失败，数据仍不一致

**如何最大程度保证最终一致性：**

- **合理配置 gc_grace_seconds**：确保 repair 执行频率 > gc_grace_seconds 周期
- **定期执行反熵修复**：生产环境建议每日或每周执行，低峰期执行 `nodetool repair`
- **配置 NTP 时钟同步**：所有节点时钟偏差控制在毫秒级，避免 LWW 误判
- **写操作使用 CL=QUORUM**：减少未确认副本数量，降低不一致概率
- **监控关键指标**：Hint 队列大小、repair 执行记录、Tombstone 扫描量
- **避免长时间多节点宕机**：通过多机架、多数据中心部署提高可用性

面试追问点：

- Cassandra 的最终一致性是"最终"还是"一定"？是**最终但非一定**，依赖运维规范和配置合理，存在边界情况无法保证
- 为什么不设计成强一致性？**为了可用性和低延迟**，强一致性需要同步等待所有副本确认，牺牲了分布式系统的核心优势
- 如何量化最终一致性？通过**不一致窗口大小**衡量，即从写入完成到所有副本一致的时间，取决于 Hint 重放速度、Repair 频率、网络延迟
- 生产环境中如何验证最终一致性？定期执行 repair 前后对比数据摘要，或使用 `nodetool verify` 检查数据完整性

# 4. 写一致性级别有哪些

**ANY**：写入至少到达一个节点即可成功。即使所有副本都宕机，Coordinator 会记录 Hint 到本地并返回成功，数据在该节点恢复前不可读。

- **用于对可用性要求极高、可以容忍数据短暂不可读的场景**，如日志收集、埋点数据

**ONE**：任意一个副本写入成功即返回。通常由离 Coordinator 最近的副本处理。

- **适用于对一致性要求不高、追求低延迟写入的绝大多数场景**，可用于非关键业务的数据写入

**LOCAL_ONE**：本地数据中心内任意一个副本写入成功即返回，**不跨 DC 通信**。

- **适用于多数据中心部署中只关心本地 DC 写入成功、减少跨 DC 延迟的场景**

**QUORUM**：所有数据中心整体超过半数副本写入成功即返回。计算公式 `(所有 DC 的副本总数 / 2) + 1`。

- **适用于需要强一致性的关键业务数据**，如订单、支付，可与 CL=QUORUM 读配合实现强一致性读写

**LOCAL_QUORUM**：本地数据中心内超过半数副本写入成功即返回。

- **适用于多数据中心中只保证本地 DC 一致性、避免跨 DC 延迟的场景**，常用配合 `NetworkTopologyStrategy` 使用

**EACH_QUORUM**：每个数据中心各自达到 Quorum 数量副本写入成功才返回。

- **适用于要求每个数据中心都强一致写入的场景**，跨 DC 复制场景中确保每个 DC 都写入了多数副本

**ALL**：所有副本（包括跨 DC）写入成功才返回。**提供最强一致性但可用性最低**，任一副本宕机写入即失败。

- **几乎不在生产环境使用**，更多用于测试或极端一致性需求

**SERIAL / LOCAL_SERIAL**：用于 **LWT（轻量级事务，IF NOT EXISTS / IF 条件更新）**。SERIAL 跨 DC 执行 Paxos，LOCAL_SERIAL 只在本地 DC 执行 Paxos。

- **适用于需要线性一致性（Linearizable Consistency）的 CAS 操作**

选择准则：

- **一致性要求高**：CL=QUORUM 或 LOCAL_QUORUM
- **延迟敏感**：CL=ONE 或 LOCAL_ONE
- **可用性优先**：CL=ANY（代价是写入数据可能暂时不可读）
- **多数据中心强一致**：EACH_QUORUM（注意延迟较高）
- **CAS 操作**：必须使用 SERIAL / LOCAL_SERIAL

重要补充：

- **Consistency Level 只决定返回前需要确认的副本数量**，而非写入的目标副本范围， **写操作始终发给该行所有副本节点**
- CL=ONE 写入后立即以 CL=ALL 读，仍可能读到旧数据，因为未确认的副本可能尚未完成写入
- 常用的读写一致性组合：**写 CL=QUORUM + 读 CL=QUORUM** 可实现强一致性；**写 CL=ALL + 读 CL=ONE** 能保证写入强一致但读取可能有延迟

面试追问点：

- 为什么不推荐 CL=ALL？任一副本宕机写入即失败，大量报错和重试，可用性严重下降，且写入延迟受最慢节点拖累
- CL=EACH_QUORUM 和 CL=QUORUM 的区别？EACH_QUORUM 要求每个 DC 都达到 Quorum，跨 DC 写入延迟叠加；QUORUM 只要求全局超过半数，可能仅某个 DC 的高副本数节点确认即返回
- 如何选择 ONE 和 LOCAL_ONE？单 DC 下无区别；多 DC 下 LOCAL_ONE 避免跨 DC 通信，延迟更低

# 5. 读一致性级别有哪些

**ONE**：由 Snitch 决定最近的节点返回数据即可成功，后台触发 Read Repair。

- **可用性最高、一致性最低，返回的结果不一定最新。**
- 适用于非关键查询、高并发读取场景

**TWO / THREE**：要求最近的 2 个或 3 个副本返回数据，取时间戳最新的结果。

- **比 ONE 更强的一致性保障**
- 适用于中等一致性要求的场景

**LOCAL_ONE**：本地数据中心内最近的一个副本返回即可，**不跨 DC 通信**。多 DC 部署下常用，避免跨 DC 读取延迟

**QUORUM**：所有 DC 总副本数的 `(总数 / 2) + 1` 个副本返回最新数据。

- **与 CL=QUORUM 写配合可实现强一致性读写**，
- 适用于订单、支付等关键业务

**LOCAL_QUORUM**：本地 DC 内 `(本地副本数 / 2) + 1` 个副本返回即可。

- **保证本地 DC 的强一致性读，不与跨 DC 延迟耦合**，
- 多 DC 环境下的推荐选择

**EACH_QUORUM**：每个 DC 各自达到 Quorum 数量的副本返回。

- **适用于跨 DC 读场景要求每个 DC 都读到一致数据**，
- 但延迟较高且极少使用

**ALL**：所有副本返回数据，任一节点不响应则读失败。

- **提供最强一致性，可用性最低**
- 几乎不在生产中使用

**SERIAL / LOCAL_SERIAL**：读取当前包括未提交（uncommitted）的事务数据，若发现未提交事务则先完成 commit 再返回结果。**用于 LWT 场景下的读取**，保证读到 Paxos 事务的最新状态

QUORUM 计算公式：**QUORUM = (所有 DC 副本数之和 / 2) + 1**（向下取整）

- 单 DC，RF=3 时 QUORUM = 2，可容忍 1 个节点宕机
- 双 DC，各 RF=3 时 QUORUM = 4，可容忍 2 个节点宕机

读写一致性组合的强一致保证公式：**写入节点数 + 读取节点数 > 副本因子（RF）**

- 例如 CL=QUORUM 写（2） + CL=QUORUM 读（2） > RF=3，可保证强一致性
- 若 CL=ONE 写（1） + CL=ALL 读（3），结果为 4 > RF=3，也能保证强一致性

读级别选择准则：

- **需要强一致性读**：CL=QUORUM 或 LOCAL_QUORUM（配合写 CL=QUORUM）
- **延迟敏感读**：CL=ONE 或 LOCAL_ONE
- **多 DC 只关注本地**：LOCAL_ONE 或 LOCAL_QUORUM
- **LWT 读取**：SERIAL / LOCAL_SERIAL

面试追问点：

- 为什么不推荐读 CL=ALL？一个节点慢或宕机整次读失败，可用性差；且写可能未发到所有副本，读 ALL 也不能保证绝对最新
- 读 CL=ONE 读到旧数据的概率和什么有关？**与副本不一致窗口大小有关**，即未同步副本的数量和延迟时间。写 CL=QUORUM + 读 CL=ONE 的不一致概率显著低于写 CL=ONE + 读 CL=ONE
- 后台 Read Repair 何时触发？读 CL=ONE 时默认也会后台异步检查其他副本摘要，但摘要不一致才修复，**不阻塞本次读返回**

# 6. Cassandra事务

Cassandra **不支持传统关系型数据库的 ACID 事务**，不支持跨分区多行事务（无 Rollback 机制），但提供了以下事务相关能力：

**轻量级事务（LWT）**

- 基于 **Paxos 共识协议**实现的 **CAS（Compare And Set）** 操作，语法为 `INSERT ... IF NOT EXISTS` 或 `UPDATE ... IF <条件>`
- 典型应用：**唯一键去重、账户扣减、库存扣减**等需要原子化条件判断的场景
- 使用方式：必须在语句中指定 `IF` 子句，且 Consistency Level 必须设为 **SERIAL** 或 **LOCAL_SERIAL**
- 示例：`UPDATE account SET balance = balance - 100 WHERE id = '123' IF balance >= 100;`

**LWT 的 Paxos 执行流程（4 轮 RTT）**

- **Prepare/Promise**：Coordinator 向分区所有副本提议一个时间戳，副本承诺在该时间戳之前不接受其他提议
- **Propose/Accept**：Coordinator 发起值提案，副本接受后返回确认
- **Commit/Accepted**：Coordinator 通知所有副本提交该值
- 相比普通写入只需 1 轮 RTT，LWT 需要 **4 轮 RTT**，延迟显著增加

**Batch（批量操作）**

- `BEGIN BATCH ... APPLY BATCH`，支持将多条 DML 语句打包
- **单分区 Batch**：保证原子性和隔离性（类似事务），高性能
- **多分区 Batch**：仅保证原子性提交（全部成功或全部失败），但**非隔离**，且 Coordinator 承担较大协调压力，不推荐高并发使用
- 使用 LOGGED Batch（默认）时，先写 `system.batchlog` 再执行语句；节点宕机后重启会从 batchlog 重放未完成的 batch

**Cassandra 不做传统事务的原因（设计哲学）**

- **去中心化架构**，无单点协调者，传统 2PC 事务在多节点间协调的延迟不可接受
- 优先保证 **可用性（A）和分区容忍性（P）**，弱化一致性（C）
- 数据模型为 **宽表 + 分区键**，单分区内的操作天然具有原子性，跨分区事务在 NoSQL 层面被视为反模式

面试追问点：

- LWT 的性能代价？**4 轮 RTT + 串行化 Paxos**，吞吐量远低于普通写入，TPS 可下降 5-10 倍，建议只在必要时使用
- LWT 在跨 DC 部署下支持吗？支持，但使用 **SERIAL** 跨 DC 执行 Paxos，延迟更高；LOCAL_SERIAL 仅在本地 DC 执行
- 多分区 Batch 的风险？Coordinator 成为瓶颈，一旦宕机 batch 会通过 `system.batchlog` 重放，但重放时机不确定，**不推荐替代事务使用**
- 如何实现类似事务的"回滚"效果？**应用层补偿**，Cassandra 本身不提供回滚能力

# 7. 介绍下墓碑机制

**Tombstone（墓碑）** 是 Cassandra 中标记数据已被删除的特殊记录，是解决分布式环境数据删除一致性的核心机制。

为什么需要墓碑：

- 分布式环境下无法保证所有副本同时执行物理删除，宕机节点恢复后会通过 Gossip 从其他节点同步数据
- 如果没有墓碑，已删除的数据可能被其他副本重新同步回来，出现 **Zombie（僵尸数据）**
- 本质上是 **将"删除"转化为一次写入操作**，利用已有的复制和一致性机制传播删除信息

墓碑的工作流程：

- **删除操作触发**：执行 `DELETE` 或 TTL 过期时，Cassandra 写入一条 Tombstone 记录，包含被删除列的列名、删除时间戳和一个特殊的删除标记
- **墓碑传播**：Tombstone 像普通写入一样被复制到所有副本节点，确保所有节点都知道该数据已被删除
- **读取时过滤**：读取数据时，Cassandra 会在内存中对比 Tombstone 时间戳和列数据时间戳，过滤掉已被标记删除的数据
- **Compaction 时清理**：当 Tombstone 存在时间超过 **gc_grace_seconds**（默认 10 天）后，Compaction 过程中才会真正物理清除 Tombstone 和对应的过期数据

gc_grace_seconds 的配置考量：

- 默认值为 **864000 秒（10 天）**，此时间内不清理 Tombstone，确保宕机最久的节点恢复后仍能收到删除信息
- **设置过小**：宕机超过 gc_grace_seconds 的节点恢复后可能恢复已删除数据，导致 Zombie 数据
- **设置过大**：Tombstone 长期占用磁盘空间，增加 Compaction 负担和读放大
- **建议设置**：根据最大预期宕机时间 + 余量设定。例如最大宕机 3 天，可设为 5-7 天

墓碑的性能影响：

- **读放大**：即使数据已被删除，读取时仍需扫描大量 Tombstone，增加 SSTable 扫描开销
- **Tombstone Overload**：短时间内大量删除时，单次读取可能扫描到超过 `tombstone_failure_threshold`（默认 10 万）个 Tombstone，Cassandra 会抛出异常并强制终止查询
- **Compaction 压力**：大量 Tombstone 使 Compaction 需要处理更多无效数据，I/O 和 CPU 开销上升
- 监控指标：`org.apache.cassandra.metrics.Table.TombstoneScannedHistogram`

常见问题和最佳实践：

- 避免频繁的 **大量范围删除**，尽量按 Partition Key 精确删除
- 使用 **TTL 自动过期** 替代手动 DELETE，TTL 过期生成的 Tombstone 更轻量
- 合理设置 **Compaction 策略**：TimeWindowCompactionStrategy（时序数据）比 SizeTieredCompactionStrategy 对 Tombstone 清理更高效
- 监控 **Tombstone 扫描量**，超过阈值时及时优化
- **双写时先删后写**的场景必须确保删除的 Tombstone 已通过 Compaction 清理，否则可能出现 Tombstone 覆盖新数据的"死胡同"

面试追问点：

- Tombstone 和 TTL 的区别？TTL 是列级别的自动过期，生成的行级或列级 Tombstone；DELETE 是显式删除操作，两者底层都通过 Tombstone 实现
- 为什么读取超过 tombstone_failure_threshold 会抛异常？防止大量 Tombstone 扫描耗尽内存和 CPU，这是一种**自我保护机制**
- Cassandra 为什么不直接物理删除？因为分布式环境必须保证**所有副本感知到删除事件**，直接物理删除会导致 Zombie 数据问题

# 8. 墓碑的缺点如何避免

墓碑的主要缺点：

**存储开销**

- Tombstone 本身占用磁盘空间，每条 Tombstone 包含列名、时间戳、删除标记等元数据
- 大量删除操作产生大量 Tombstone，在 Compaction 清理前持续占用存储

**读放大**

- 读取 SSTable 时，即使目标数据已被删除，仍需扫描所有 Tombstone 并逐条对比时间戳
- 分区的 Tombstone 越多，扫描开销越大，**读延迟显著上升**
- 扫描 Tombstone 数量超过 `tombstone_failure_threshold`（默认 10 万）时，Cassandra 直接抛出异常终止查询

**Compaction 压力**

- Compaction 需要处理大量 Tombstone 数据，增加 I/O 和 CPU 负载
- 无效 Tombstone 越多，Compaction 效率越低，消耗的系统资源越多

**写入开销**

- DELETE 操作本质上是一条写入（Tombstone），占用和普通写入相同的写入路径和内存

如何避免和优化：

**优先使用 TTL 替代手动 DELETE**

- TTL 过期自动生成 Tombstone，**不需要单独的执行 Delete 操作**
- 大量过期数据通过 Compaction 统一清理，减少随机删除带来的 Tombstone 碎片
- `INSERT ... USING TTL <seconds>`，适用于日志、消息等有过期时间的数据

**避免频繁的范围删除**

- 不推荐 `DELETE FROM table WHERE partition_key = ?` 删除整个分区，产生的 Tombstone 数量巨大
- 尽量按**精确的 Clustering Column** 删除单行

**按时间窗口分区**

- 对于时序数据，将 Partition Key 设计为时间窗口（如 `YYYYMMDD`），过期分区直接 `DROP` 整个 SSTable，**零 Tombstone 产生**
- 配合 TimeWindowCompactionStrategy（TWCS），旧窗口的 SSTable 超过 gc_grace_seconds 后直接删除

**合理配置 gc_grace_seconds**

- 根据最大预期宕机时间设置，不宜过大，否则 Tombstone 长期不被清理

**使用** `nodetool compact` 主动触发 Compaction

- 紧急情况下主动压缩，清理已过期的 Tombstone，但注意 Compaction 对线上 I/O 有冲击

**监控 Tombstone 指标**

- `nodetool tablestats` 查看表级别的 Tombstone 数量
- `nodetool cfhistograms` 查看 Tombstone 扫描分布
- 设置告警：`TombstoneScannedHistogram` 的 P99 超过阈值时及时介入

面试追问点：

- TTL 为什么比 DELETE 更优？TTL 在写入时就确定了过期时间，Tombstone 生成更规律，Compaction 清理效率高；手动 DELETE 分散不可控，Tombstone 易堆积
- 为什么 DROP 分区不产生 Tombstone？DROP 整个 SSTable 是**物理文件级别的删除**，不涉及数据逐行标记为删除，完全绕过 Tombstone 机制
- 已经产生了大量 Tombstone 怎么办？紧急措施：降低 `tombstone_failure_threshold`（临时规避异常），配合 `nodetool compact` 主动清理；长期方案：优化数据模型，改用 TTL 或时间窗口分区

# 9. 介绍下 Partition Key

**Partition Key** 是 Cassandra 数据模型中决定**数据分布位置**的核心字段，是每行数据唯一标识的第一部分，用于计算数据所属分区以及确定数据存储在哪个节点上。

Partition Key 与 Primary Key 的关系：

- **Primary Key = Partition Key + Clustering Columns（可选）**
- Partition Key 是 Primary Key 的第一个组成部分，用于**分区**
- Clustering Columns 用于**分区内排序**

Partition Key 的定位过程：

- Cassandra 对 Partition Key 的值计算 **Murmur3 哈希**（一致性哈希算法），生成一个 Token
- 根据 Token 值确定该行数据应落在哪个 **Token Range** 中，进而映射到对应的物理节点
- 同一 Partition Key 的所有数据**一定位于同一节点（及其副本节点）**，这是实现单分区原子性和局部性的基础

定义方式：

- **单列 Partition Key**：`PRIMARY KEY (user_id)`，user_id 是唯一分区字段
- **复合 Partition Key**：`PRIMARY KEY ((group_id, region), create_time)`，group_id + region 组成复合分区键，create_time 作为 Clustering Column
- 复合 Partition Key 使用**多列联合哈希**决定分区，适用于按多维度分布的访问模式

**Partition Key 的设计原则：**

- **均匀分布**：Partition Key 的取值应有足够的基数，避免热点。例如用 UUID 而非自增 ID 作为分区键
- **避免大分区**：单个分区内的数据量不宜过大（建议不超过几百 MB，行数不超过 10 万），否则造成读放大和GC压力
- **查询友好**：尽量让查询条件包含 Partition Key，避免 **跨分区全表扫描**。Cassandra 的查询效率：单分区查询 > 多分区查询 > 全表扫描
- **考虑数据模型**：将**最常按精确值查询的字段**作为 Partition Key

常见问题和最佳实践：

- **Partition Key 选择不当导致热点**：如使用按日期取模的字段（`YYYYMMDD % N`）比直接用时间戳更均匀
- **大分区问题**：使用 `nodetool tablehistograms` 监控分区大小，超限时通过增加 Partition Key 粒度来拆分
- **反向查询**：如果需要按非 Partition Key 字段查询，需建立**索引表或物化视图**，Cassandra 不支持单表灵活过滤

面试追问点：

- Partition Key 和 Clustering Column 的区别？Partition Key 决定**数据存在哪个节点**，Clustering Column 决定**数据在分区内如何排序和存储**
- 复合分区键在存储上如何组织？多列值拼接后计算哈希，所有列值同时参与路由，不是逐层路由
- 频繁查询无 Partition Key 的场景怎么办？使用**二级索引**或**物化视图**，但性能不如直接按 Partition Key 查询；或重新设计表结构，用查询字段做主键

# 10. 读修复如何实现的

读修复（Read Repair）是 Cassandra 在**读路径上自动触发的数据修复机制**，用于将过时副本更新到最新状态。

**核心流程（两阶段读）**：

第一阶段 - Digest 比对

- Coordinator 根据一致性级别向足够的副本发送 **Digest 请求**（仅请求数据的 MD5 摘要）
- 各副本计算并返回指定数据的摘要值
- Coordinator 比对收到的所有摘要
- 如果摘要全部一致，直接向客户端返回数据，流程结束

第二阶段 - 全量数据修复

- 摘要不一致时，Coordinator 向**所有副本**发送 **Data 请求**（拉取全量数据）
- Coordinator 对比各副本返回的完整数据，通过时间戳（LWW）确定最新版本
- 将最新数据返回给客户端
- 同时向**返回旧数据的副本**发起写请求（MUTATION 消息），将最新的数据写入这些过时副本
- 过时副本更新后，该行数据在集群中达到一致

**两种读修复模式**：

**Blocking Read Repair（阻塞式）** - 默认行为

- 发生在 Consistency Level > ONE 时（如 CL=QUORUM）
- Coordinator 需要等待多个副本响应，在等待过程中自然地进行 Digest 比对
- 修复完成后才返回客户端结果，**读延迟增加**

**Background Read Repair（后台式）** - 异步修复

- 发生在 CL=ONE 或 CL=LOCAL_ONE 时
- Coordinator 从一个副本获取数据返回客户端后，**不阻塞客户端**，后台异步检查其他副本的摘要
- 如果发现不一致，后台发起修复写请求
- 参数控制：`read_repair_chance`（默认 0.1）控制后台读修复的触发概率

**重要参数**：

- `read_repair_chance`：读操作触发后台读修复的概率（0 到 1），默认 0.1
- `dclocal_read_repair_chance`：本地 DC 内后台读修复概率，默认 0
- 生产环境建议将 `read_repair_chance` 调低（0.02-0.05），避免高并发下修复写请求对集群 I/O 的冲击

**读修复 vs 其他修复机制的区别**：

- **读修复**：按需修复，只修复**正在被读取的数据**，冷数据不会触发
- **反熵修复**：全量扫描，主动修复所有不一致分区，覆盖冷数据
- **提示移交**：写入时修复，针对短暂宕机节点，不覆盖数据冲突场景

**局限性**：

- 只修复读操作访问到的那行数据，从未被访问的（冷）数据不一致持续存在
- 后台读修复不保证一致性收敛时间，因为触发概率 < 1
- 修复写请求在**高并发读场景**下可能产生大量额外的写负载

面试追问点：

- Digest 请求为什么比 Data 请求轻量？Digest 只返回数据的 MD5 哈希值，通常远小于完整数据，减少网络传输和反序列化开销
- 读修复和 Hinted Handoff 是否可能同时工作？会。Hinted Handoff 处理写入时宕机的场景，读修复处理读时发现的不一致，两者互补
- 为什么 read_repair_chance 不能设为 1？每次读都触发后台修复写请求，在高并发下修复写的累加消耗会严重冲击集群 I/O 和 CPU

# 11. 反熵修复如何工作的

**反熵修复（Anti-Entropy Repair）** 是通过 **Merkle Tree** 主动比较各副本数据、修复不一致的机制，使用 `nodetool repair` 手动触发或定时调度执行，是最终一致性的**兜底保障**。

**核心流程（两阶段）**：

第一阶段 - 构建 Merkle Tree

- 每个参与修复的节点为指定 Token Range 构建一棵 Merkle Tree
- Merkle Tree 是**二叉树**，叶子节点是数据块的哈希值（行数据的 MD5 哈希），父节点是两个子节点哈希值的哈希（`hash(hash1 + hash2)`）
- Cassnadra 使用**深度为 15** 的紧凑树，共 `2^15 = 32768` 个叶子节点，对应 32768 个数据块
- 例如节点有 100 万分区，每个叶子覆盖约 30 个分区
- 构建过程需要全量读取 SSTable 并计算哈希，**消耗大量磁盘 I/O 和内存**

第二阶段 - 比较与修复

- 各副本节点交换 Merkle Tree 的**根节点哈希**，如果一致表示该 Token Range 内数据完全一致
- 根节点不一致则递归向下比较子节点，逐层定位到**不一致的叶子节点**
- 找到不一致的数据块后，只传输该叶子对应的**最小数据范围**进行全量数据对比
- 按时间戳（LWW）确定最新版本，更新过时副本
- 相比全量数据传输，Merkle Tree 大幅减少网络开销

**分段修复（Sub-range Repair）**：

- 默认一次 repair 处理节点上全部 Token Range，Merkle Tree 精度为 32768 块，每个块内含多处数据，修复时需传输整个块
- 使用 `-st`（start token）和 `-et`（end token）参数 将 Token Range 细分为子段
- 分段依据：`system.size_estimates` 表记录了每个 Token Range 的 `partitions_count`（分区数）和 `mean_partition_size`（平均分区大小）
- 当某段的分区数超过 32768 时，继续拆分为更小的子段，确保 Merkle Tree 精度**精确到单个或少量分区**
- 遍历所有子段完成后，整节点数据完成修复

**使用建议**：

- **定期执行**：建议在**低峰期**定期执行（如每日或每周），频率取决于数据变更量和一致性要求
- **避免同时触发**：不要在多个节点或整个集群同时触发 repair，分批次执行防止集群过载
- **修复类型**：`nodetool repair -pr` 只修复当前节点负责的 primary range；`nodetool repair -full` 执行完整修复（默认行为）；`-inc` 增量修复，只修复上次修复后的变更，速度更快

**反熵修复 vs 读修复**：

- 反熵修复是 **主动全量**，修复所有数据（包括冷数据）
- 读修复是 **按需增量**，只在读操作触发时修复
- 反熵修复开销大、耗时长，但能彻底解决不一致

面试追问点：

- Merkle Tree 深度为什么是 15？受限于**内存占用**，深度越大叶子越多精度越高，但树节点数呈指数增长，15 层（32768 叶子）是内存和精度的平衡选择
- 增量修复（-inc）的原理？通过对比**上次修复以来的 SSTable 变更**，只处理新增或修改的数据，大幅减少扫描量，但前提是在 gc_grace_seconds 内执行
- 修复过多会有什么风险？大量修复导致磁盘 I/O 饱和，影响线上读写延迟，甚至触发 OOM；建议用 `nodetool repair -j` 控制并行度，生产环境推荐 `-j 1` 或 `-j 2`

# 12. Token 段是什么

**Token 段（Token Range / Token Segment）** 是 Cassandra 一致性哈希环上的**连续 Token 区间**，每个节点负责一段或多段 Token Range。

Token 与一致性哈希：

- Cassandra 使用 **Murmur3 哈希** 将 Partition Key 映射为一个 **Token 值**，范围 `-2^63` \~ `2^63 - 1`
- 整个 Token 空间构成一个**环（Ring）**，每个节点负责环上的一段连续范围
- 例如节点 A 负责 Token Range (1, 100]，节点 B 负责 (100, 200]

Token Range 与副本关系：

- 每行数据根据 Partition Key 哈希到某个 Token，落在对应 Token Range 的节点即为该行的**主副本**
- 复制策略决定该行数据的副本节点：根据 **replication_factor**，从主副本开始在环上顺次取后续节点
- 因此 Token Range 决定了**数据分布边界**，副本范围由复制策略确定

Token 段在修复中的作用：

- `nodetool repair` 默认修复节点负责的全部 Token Range
- 使用 `-st`（start token）和 `-et`（end token） 参数可指定只修复某个子段
- 分段修复的价值：**缩小 Merkle Tree 覆盖的范围**，提高修复精度
- `system.size_estimates` 表记录了每个 Token Range 段的分区数和分区大小，用于决定如何拆分

为什么需要 Token 段：

- 默认 Merkle Tree 深度 15（32768 叶子），节点上数据量大时每叶子覆盖多个分区
- 一个叶子对应的数据块中只要有 1 条不一致，整个块的数据都需要传输对比
- **将 Token Range 切小** → Merkle Tree 精度提升 → 每个叶子覆盖分区数减少 → 修复传输量降低
- 理想情况：每个叶子覆盖 ≤ 1 个分区，实现精确修复

虚拟节点（Vnodes）下的 Token 段：

- 启用虚拟节点（`num_tokens > 1`）时，每个物理节点持有多个 **不连续** 的 Token Range 段
- 每个 vnode 是一个独立的 Token 段，各自维护副本关系
- 修复时需逐个 vnode 处理，修复粒度更细，但总修复时间可能更长

面试追问点：

- Token 段大小不一致怎么办？Cassandra 随机分配 Token 使各段分布均匀，但 vnode 个数配置不当仍可能倾斜，需监控 `system.size_estimates`
- 如何选择合适的 Token 段粒度进行修复？将分段后的分区数控制在 **32768 以内**，确保 Merkle Tree 一个叶子至多覆盖一个分区，实现精确修复
- Token 段的切分是物理的还是逻辑的？**逻辑的**，不改变物理存储，只影响修复时构建 Merkle Tree 的范围

# 13. Hint 如何工作的

**Hint（提示移交）** 是 Cassandra 在目标副本不可用时，由 Coordinator 节点代为保存写入数据并在目标恢复后重试的机制，主要用于**短暂节点宕机**场景。

**存储 Hint 的条件**：

- 写操作必须满足客户端指定的**一致性级别要求**（例如 CL=ONE 或 CL=QUORUM 已满足），协调节点才会为宕机副本存储 Hint
- 如果一致性级别无法满足（例如 CL=ALL 时任意副本宕机），Cassandra **不会存储 Hint**，直接返回写失败
- 可在 `cassandra.yaml` 中通过 `hinted_handoff_enabled` 全局开启或关闭

**Hint 的内容**：

- **宕机副本的位置**（目标节点地址）
- **元数据版本号**（schema 版本，确保重放时 schema 兼容）
- **实际的写入数据**（行数据、时间戳、TTL 等完整写入信息）

**Hint 的存储**：

- Cassandra 2.x：存储在系统表 `system.hints` 中
- Cassandra 3.x+：存储在**磁盘目录**（`hints_directory`，默认 `<data>/hints`），以文件形式保存
- 每个 Hint 文件按目标节点和时段组织，格式为二进制日志

**Hint 的重放（Replay）**：

- **Gossip 检测**：Gossip 协议探测到宕机节点恢复后，持有 Hint 的节点立即开始重放
- **定时轮询**：每 **10 分钟**检查一次是否有超时未响应的写操作，因为短暂停机可能短于 Gossip 故障检测时间
- 重放时向目标节点发送正常的写请求（MUTATION 消息），目标节点按时间戳判断是否接受

**Hint 的过期**：

- 默认保存 **3 小时**（由 `max_hint_window_in_ms` 配置，默认 `10800000` ms）
- 超过时间的 Hint 会被丢弃，因为长时间宕机更可能是永久故障，继续保存 Hint 浪费存储和重放开销
- 被丢弃的 Hint 对应的数据不一致需要通过 Read Repair 或反熵修复补齐
- 当通过使节点退役或使用nodetool removenode命令从集群中删除节点时，Cassandra会自动删除针对不再存在的节点的hint。Cassandra还删除了删除表的hint。

**典型场景示例**：

三个节点 A（协调节点）、B、C，RF=2，C 宕机，客户端以 CL=ONE 写入数据 K：

- Coordinator A 将 K 写入 B（成功），同时为 C 生成 Hint 存储到本地
- 客户端收到成功响应（1/2 副本确认，满足 CL=ONE）
- C 节点恢复后，A 通过 Gossip 感知到，将 Hint 重放给 C，C 写入 K

如果 C 在 3 小时内未恢复，Hint 被丢弃，K 在 C 上永久缺失 **，需通过反熵修复补齐**

**配置参数**：

- `hinted_handoff_enabled`：是否启用 Hint（默认 true）
- `max_hint_window_in_ms`：Hint 有效时间窗口（默认 3 小时）
- `max_hints_delivery_threads`：Hint 重放线程数（默认 2）
- `hints_flush_period_in_ms`：Hint 刷盘周期（默认 10 秒）

面试追问点：

- CL=ANY 和 Hint 的关系？CL=ANY 不要求副本确认，即使所有副本宕机也存储 Hint 返回成功，写入数据在节点恢复前**不可读**
- Hint 是否保证不丢数据？**不保证**。超过 max_hint_window_in_ms、Coordinator 宕机导致 Hint 丢失，都不可恢复
- Hint 重放失败会怎样？目标节点若再次宕机，当前 Cassandra 版本**不会重新存储 Hint**，数据丢失需通过修复补齐（Cassandra 4.0+ 已改进支持重新存储）
- 大量 Hint 堆积会有什么问题？占用磁盘空间、重放时产生大量写请求冲击集群，需监控 `HintedHandoffManager` 的 pending 数

# 14. Hint 对性能的影响

Hint 机制在**保证可用性**的同时，对性能有不可忽视的负面影响：

**写放大效应**

- 启用 Hint 后，即使部分副本宕机，Cassandra 仍会**继续接受全量写入**，同时为宕机节点存储 Hint，整体写入负载未降低
- 正常写操作是 1 轮 RTT，而 Hint 需要在写入路径上额外**序列化、写磁盘、记录元数据**，增加协调节点的 CPU 和 I/O
- 如果集群已在最大容量下运行，开启 Hint 可能导致协调节点过载

**Hint 存储开销**

- 所有发往同一目标副本的 Hint 存储在**单个分区键**下，重放是顺序读取，对性能影响小
- 但大量 Hint 堆积时占用磁盘空间，单节点可能积累 GB 级别 Hint 文件，影响正常写入的 Disk I/O

**重放对集群的冲击**

- 宕机节点恢复后，持有 Hint 的节点集中重放，产生**突发写流量**，可能压垮刚恢复的节点
- 多个协调节点同时重放 Hint 给同一恢复节点时，写负载叠加，加剧该节点的恢复难度

**协调节点内存压力**

- 如果故障检测器尚未检测到节点宕机，所有写请求会先尝试发给目标副本，等待 `write_request_timeout_in_ms`（默认 10 秒）超时后才转为 Hint
- 多个副本同时超时时，协调节点需要同时维护大量待超时请求和 Hint，**内存压力显著上升**
- 协调节点跟踪当前正在处理的 Hint 数量，超过阈值时抛出 `OverloadedException`，**暂时拒绝写入含异常节点的副本**，以此自我保护

**Hint 与 CL=ANY 的特殊代价**

- CL=ANY 下所有副本宕机仍存储 Hint 返回成功，协调节点承担全部写负载，集群实际写入能力为零但协调节点持续消耗资源
- 此时大量 Hint 堆积，恢复后重放压力极大

**最佳实践与缓解措施**：

- **不要将集群推到最大容量**，预留 20-30% 的冗余应对节点故障时的 Hint 写放大
- 监控关键指标：`HintedHandoffManager.pending`、`StorageProxy.totalHints`、`HintedHandoffManager.hintsTimedOut`
- 设置 `max_hints_delivery_threads` 限制重放并行度（推荐 1-2）
- 短时间大量节点故障时，可**临时关闭 Hint**（`nodetool disablehandoff`），待集群稳定后通过反熵修复补齐

面试追问点：

- 为什么 Hint 重放是顺序读取性能影响小？因为目标副本的所有 Hint 存储在**同一个分区键**下，SSTable 连续存储，读取时只需一次定位 + 顺序扫描
- Hint 和反熵修复在性能上的区别？Hint 是**写入时增量**存储，重放只涉及缺失数据，开销可控；反熵修复是**全量扫描比对**，磁盘 I/O 远大于 Hint 重放
- 如何判断 Hint 是否成为瓶颈？监控 Hint 文件大小持续增长 + 协调节点 GC 频繁 + `OverloadedException` 出现，就需要干预

# 15. 介绍Repair机制

Cassandra 的 **Repair（修复）** 是一组确保各副本数据最终一致的机制集合，覆盖从写入到读取、从自动到手动的全路径。

**Repair 的三大核心机制**：

**Hinted Handoff（提示移交）** - 写入路径修复

- 目标副本宕机时，Coordinator 本地保存 Hint，节点恢复后重放
- **修复写入时未送达的数据**，适用于短暂宕机（默认 3 小时内）
- 自动执行，无需人工干预，对性能影响最小

**Read Repair（读修复）** - 读取路径修复

- 读操作时通过 Digest 比对发现不一致，后台或同步修复过时副本
- **修复读取时发现的不一致数据**，覆盖日常读写负载下的不一致场景
- 自动触发，不阻塞客户端（后台模式），但冷数据无法覆盖

**Anti-Entropy Repair（反熵修复）** - 全量扫描修复

- 使用 Merkle Tree 主动比对副本数据，修复所有差异
- **兜底修复所有数据**，包括冷数据和 Hint 超时丢失的数据
- 需要手动触发（`nodetool repair`）或定时调度，开销最大但最彻底

**三种机制的协作关系**：

- Hinted Handoff 是**第一道防线**，处理写入时即时故障
- Read Repair 是**第二道防线**，在日常读写中逐步收敛不一致
- Anti-Entropy Repair 是**最后兜底**，确保集群最终完全一致
- 三者缺一不可：Hint 超时丢失的数据靠 Read Repair 或反熵修复补救，Read Repair 覆盖不到的冷数据靠反熵修复

**Repair 的执行方式**：

**增量修复（Incremental Repair）**

- `nodetool repair -inc`，只修复上次修复以来 SSTable 的变更
- 通过对比修复标记（Repair Flag），跳过已确认一致的 SSTable
- 速度快、开销低，但在 gc_grace_seconds 周期内必须至少执行一次完整修复

**完整修复（Full Repair）**

- 默认行为，重新扫描节点上所有 SSTable，构建 Merkle Tree 比对
- 全面但耗时，建议在低峰期执行（如每日凌晨）

**分段修复（Sub-range Repair）**

- `nodetool repair -st <start_token> -et <end_token>` 指定 Token 段范围
- 将修复任务拆分为多个子任务，减少单次修复的资源消耗
- 避免一个节点修复时全量扫描导致 I/O 打满

**Repair 的调度建议**：

- **时序数据**（TTL 自动过期）：增量修复 + TWCS，可降低修复频率（每周甚至每月）
- **关键业务数据**：每日增量修复 + 每周完整修复
- **多 DC 集群**：每个 DC 独立调度修复，避免跨 DC 修复的网络开销
- **避开高峰期**：修复消耗大量磁盘 I/O 和 CPU，建议在业务低峰执行

面试追问点：

- Repair 和 Compaction 的关系？**Repair 修复数据不一致**，**Compaction 优化 SSTable 布局并清理 Tombstone**，两者独立但互补
- 为什么增量修复不能完全替代完整修复？增量修复依赖**上次修复的标记**，长时间不执行完整修复可能导致标记失效或遗漏不一致数据
- 一个节点修复期间对其读写有影响吗？有，修复构建 Merkle Tree 时消耗大量 I/O，可能导致读延迟升高；写完路径不受影响，但建议控制修复并行度

# 16. 压缩机制和过程

**Compaction（压缩）** 是 Cassandra 后台异步执行的 **SSTable 合并与重写过程**，用于合并多个 SSTable、清理已删除数据（Tombstone）、提升读取性能。

**1、为什么需要 Compaction：**

Cassandra 采用 **LSM-Tree（Log-Structured Merge-Tree）** 存储模型，写入时数据先追加到 MemTable，MemTable 满后 Flush 到磁盘生成 SSTable。这种设计带来几个问题：

- **SSTable 不可变**：每次 Flush 都会生成新的 SSTable，随着时间推移，磁盘上会积累大量小文件
- **读放大**：查询时需要扫描多个 SSTable 查找同一行数据，读性能随 SSTable 数量线性下降
- **空间浪费**：更新和删除操作产生多版本数据和 Tombstone，占用大量无效空间
- **数据碎片化**：数据分散在多个文件中，范围查询需要跨多个 SSTable 合并排序

Compaction 就是为了解决这些问题，通过后台合并将多个小 SSTable 合并为大文件，清理无效数据，恢复读取性能。

**2、Compaction 的核心作用：**

- **合并 SSTable**：将多个小 SSTable 合并为更大的 SSTable，减少读取时需要扫描的文件数量
- **清理 Tombstone**：超过 gc_grace_seconds 的 Tombstone 在 Compaction 时被物理删除
- **去除覆盖数据**：同一行数据的旧版本被新版本覆盖后，旧版本在 Compaction 中被丢弃
- **优化存储布局**：按分区键排序重新组织数据，提升范围查询效率

**3、三种主要 Compaction 策略：**

**Size-Tiered Compaction Strategy（STCS）- 默认策略**

- **原理**：将大小相近的 SSTable 分为一组，当组内 SSTable 数量达到阈值（默认 4 个）时触发合并
- **触发条件**：相似大小的 SSTable 数量超过 `min_sstable_size` 配置
- **适用场景**：**写多读少**的日志型数据、时序数据
- **优点**：写放大低（约 10%），合并效率高
- **缺点**：读放大高（读取时可能需要扫描多个 SSTable），Tombstone 清理不及时
- **配置参数**：`compaction_threshold`（4-32）、`min_sstable_size`、`bucket_low/high`

**Leveled Compaction Strategy（LCS）- 读优化策略**

- **原理**：将 SSTable 组织为多层（Level），每层大小是上一层的 **10 倍**。Level 0 为初始层，数据从 L0 开始逐层合并
- **触发条件**：某层 SSTable 数量超过阈值时，与下层 SSTable 合并
- **层级结构**：L0 → L1（10 个 SSTable）→ L2（100 个）→ L3（1000 个）...
- **适用场景**：**读多写少**的业务数据，如用户信息、配置数据
- **优点**：读放大低（每个分区最多在 1 个 SSTable 中），空间放大大致为 10%
- **缺点**：写放大高（约 10 倍），因为每次合并需要重写整个 SSTable
- **配置参数**：`sstable_size_in_mb`（默认 160MB）

**Time-Window Compaction Strategy（TWCS）- 时序数据专用**

- **原理**：按时间窗口将 SSTable 分组，**窗口内合并、窗口间不合并**。当前窗口内的 SSTable 按 STCS 合并，旧窗口的 SSTable 直接标记过期
- **触发条件**：当前时间窗口结束且窗口内 SSTable 数量达到阈值
- **时间窗口设置**：`compaction_window_size`（窗口大小）+ `compaction_window_unit`（单位：HOURS/DAYS）
- **适用场景**：**时序数据**，如日志、IoT 数据、监控指标
- **优点**：写放大极低，旧数据按时间窗口整体淘汰，Tombstone 清理高效
- **缺点**：不适合非时序的随机读写场景
- **配置参数**：`compaction_window_size`、`compaction_window_unit`、`unsafe_aggressive_sstable_expiration`（直接删除旧窗口 SSTable）

**三种策略对比：**

| 维度 | STCS | LCS | TWCS |
| --- | --- | --- | --- |
| **写放大** | 低（\~10%） | 高（\~10x） | 极低 |
| **读放大** | 高 | 低 | 中（窗口内高） |
| **空间放大** | 高（临时） | 低（10%） | 低 |
| **适用场景** | 写多读少 | 读多写少 | 时序数据 |

**4、Compaction 的执行过程：**

- **后台线程执行**：Compaction 由后台线程异步处理，不阻塞前台读写
- **选择候选 SSTable**：根据策略算法选出待合并的 SSTable 文件
- **读取与合并**：逐行读取候选 SSTable，按主键排序并合并，去除重复和已删除数据
- **写入新 SSTable**：合并后的数据写入新的 SSTable 文件
- **原子替换**：新 SSTable 写入完成后，原子性地替换旧 SSTable，旧文件被删除
- **释放空间**：旧 SSTable 占用的磁盘空间被回收

**5、Compaction 的关键配置参数：**

- `concurrent_writes`：Compaction 并发写线程数（默认 2）
- `compaction_throughput_mb_per_sec`：Compaction I/O 限速（默认 16-64 MB/s）
- `compaction_warning_threshold`：单次 Compaction 超过阈值（默认 10 分钟）时打印警告
- `compaction_check_interval`：Compaction 检查间隔

**6、Compaction 的性能影响：**

- **I/O 开销**：Compaction 消耗大量磁盘 I/O，可能影响前台读写延迟
- **CPU 开销**：数据排序和合并需要 CPU 计算
- **磁盘空间**：合并期间需要额外的临时空间（约 SSTable 大小的 10-20%）
- **限流机制**：通过 `compaction_throughput_mb_per_sec` 控制 I/O 速度，避免影响线上服务

**7、最佳实践：**

- **监控 Compaction 指标**：`CompactionTask.pending`、`CompactionTask.completed`
- **高峰期避免触发**：合理调度 Compaction，避开业务高峰
- **调整 I/O 限速**：根据磁盘性能调整 `compaction_throughput_mb_per_sec`
- **时序数据用 TWCS**：配合 TTL 实现数据自动过期，减少 Compaction 负担
- **读多场景用 LCS**：牺牲写性能换取低读放大

面试追问点：

- Compaction 和 Repair 的区别？**Compaction 优化 SSTable 布局、清理 Tombstone**，**Repair 修复副本间数据不一致**，两者独立但互补
- STCS 为什么读放大高？因为多个 SSTable 可能包含同一分区的数据，读取时需要合并查询多个文件
- LCS 的空间放大为什么是 10%？每层大小是上一层的 10 倍，合并时旧 SSTable 和新 SSTable 同时存在，最大额外空间约一层大小
- TWCS 如何清理旧数据？超过 gc_grace_seconds 的时间窗口，整个 SSTable 直接删除，**无需逐行 Tombstone 清理**
- Compaction 配置不当会怎样？I/O 限速过高会拖垮线上服务，限速过低导致 SSTable 堆积、读性能下降

# 17. 数据是如何同步的

Cassandra 节点间的数据同步是一个**多层次、多机制协同**的过程，涵盖写入路径的实时同步、故障场景的异步修复、以及全量数据的一致性保障。

**1、写入路径的同步（实时复制）：**

当客户端执行写操作时，Coordinator 节点根据 Partition Key 计算 Token，确定目标副本节点，然后将写请求并行发送给所有副本节点：

- Coordinator 根据一致性级别等待指定数量的副本确认（如 CL=QUORUM 需要多数副本确认）
- 每个副本节点独立执行写入：先写 CommitLog（保证持久化），再写 MemTable（内存缓冲）
- MemTable Flush 后生成 SSTable，写入完成
- **所有副本节点独立执行相同操作**，不存在主从复制关系，这是 Cassandra 去中心化架构的核心
- 写操作始终发给**该行所有副本节点**，Consistency Level 只决定返回前需要确认的数量

**2、Gossip 协议（节点状态同步）：**

Gossip 是 Cassandra 的**节点发现和状态同步机制**，不直接同步用户数据，但为数据同步提供基础：

- 每个节点每秒随机选择一个节点交换信息，消息中包含**自己掌握的所有节点状态和版本号**
- 节点只接收**版本号更高**的状态信息，丢弃旧版本，通过这种机制最终所有节点感知到相同的集群状态
- Gossip 同步的信息包括：节点存活状态、负载信息、Token 范围、Schema 版本等
- 用于检测节点宕机、触发 Hint 重放、协调节点加入/退出

**3、Hinted Handoff（写入时临时同步）：**

当目标副本节点短暂宕机时，Coordinator 会为该节点保存 Hint：

- Coordinator 将写入数据以 Hint 形式**本地保存**，同时向其他可用副本写入
- 宕机节点恢复后，持有 Hint 的节点通过 Gossip 感知到，将 Hint **异步重放**给恢复的节点
- Hint 默认保存 **3 小时**，超时未重放的 Hint 会被丢弃，数据需通过反熵修复补齐
- **适用于短暂节点故障**，对写入性能影响小，但不保证长期数据一致性

**4、Read Repair（读取时修复同步）：**

读操作时，Coordinator 发现副本间数据不一致会触发修复：

- Coordinator 向多个副本发送 Digest 请求，比对数据摘要
- 摘要不一致时，向所有副本请求全量数据，通过时间戳（LWW）确定最新版本
- 将最新数据返回客户端，同时向**返回旧数据的副本**发起写请求进行修复
- **自动触发**，覆盖日常读写负载下的不一致场景，但冷数据无法覆盖

**5、Anti-Entropy Repair（全量修复同步）：**

使用 Merkle Tree 主动比对副本数据，修复所有差异：

- 节点为指定 Token Range 构建 Merkle Tree，比较各副本的树结构定位不一致数据
- 找到不一致的数据块后，按时间戳（LWW）更新过时副本
- 需要手动触发（`nodetool repair`）或定时调度执行
- **兜底修复所有数据**，包括冷数据和 Hint 超时丢失的数据，是最终一致性的保障

**6、Streaming（流式传输）：**

节点加入、退出或负载均衡时的数据同步机制：

- **新节点加入**：Token Range 的 owner 发生变化，原 owner 将属于新节点的数据通过 Streaming 传输给新节点
- **节点退役**：数据从退役节点 Streaming 到新的 owner 节点
- **数据迁移**：Token 重分配时触发 Streaming
- Streaming 是**全量数据传输**，基于 SSTable 文件级别复制，效率高于逐行同步

**7、各机制的协作关系：**

- **写入时实时复制**是主要同步方式，保证数据在多个副本独立写入
- **Hinted Handoff** 是写入路径的容错机制，处理短暂故障
- **Read Repair** 是读取路径的修复机制，逐步收敛不一致
- **Anti-Entropy Repair** 是兜底机制，确保长期一致性
- **Gossip** 提供节点状态感知，是其他机制的基础
- **Streaming** 处理结构性变化（节点加入/退出），保证数据分布正确

面试追问点：

- Coordinator 如何选择副本节点？根据 **Snitch** 提供的网络拓扑信息，优先选择同机架、同数据中心的节点，减少网络延迟
- 写入时某个副本失败怎么办？Coordinator 会尝试重试，如果重试仍失败且满足一致性级别则返回成功，未成功的副本通过 Hint 或反熵修复补齐
- Streaming 和 Hint 的区别？Streaming 是**全量数据迁移**（节点加入/退出），Hint 是**增量写入暂存**（短暂宕机）
- Gossip 能否同步用户数据？**不能**，Gossip 只同步节点状态元数据，用户数据通过写入复制、Read Repair、Anti-Entropy 机制同步
