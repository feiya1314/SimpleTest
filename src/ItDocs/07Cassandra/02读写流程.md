# 1. CS如何完成读请求的

**协调器节点选择**：

- 客户端连接集群中的任意节点并发起请求，该节点即成为**协调器**，作为客户端与副本之间的代理
- 协调器由客户端驱动根据**负载均衡策略**选择，常见策略有 `DCAwareRoundRobinPolicy`（就近数据中心轮询）和 `TokenAwarePolicy`（感知Token直接路由到数据所在节点）
- 驱动会根据**网络距离算法**选择"最近"的节点作为协调器；并且一旦驱动获取集群拓扑后，可能将协调器切换到更近的节点

**驱动连接机制**：

- 驱动配置中设置一个或多个**连接点（contact points）**，按列表顺序依次尝试连接
- 只需连上其中一个节点即可获取集群全部节点的**元数据（metadata）**，无需连接所有连接点
- 驱动根据元数据创建到各节点的连接池；最佳实践是将**响应最快的节点**设为连接点

**读请求执行流程**：

- 协调器使用 **自定义TCP通信协议** 传输二进制数据，向副本节点发送三类请求：**直接请求**、**摘要请求**、**后台读修复请求**
- 协调器首先向"最近"的一个副本节点发送**直接请求**，获取完整的行数据
- 接着向一致性级别指定数量的副本节点发送**摘要请求**，这些节点返回数据的**摘要（digest）**，协调器在内存中比较这些摘要是否一致
- 如果摘要不一致，协调器以**时间戳最新**的副本数据为准，将该数据作为最终结果返回客户端
- 协调器继续向剩余的所有副本节点发送摘要请求，若发现有过期数据，则在后台发起**读修复请求**，确保所有副本更新到最新数据

**快速读保护（Speculative Retry）**：

- 当最初选择的副本节点宕机或响应超时时，协调器不会一直等待
- 若表配置了 `speculative_retry` 属性，协调器会在超时后将读请求发送到另一个副本节点进行重试，避免尾部延迟过高

举个例子：

- 10节点集群，3副本，数据分布在N1、N2、N3，协调器为N4
- N4向 **最近副本N1** 发送直接请求；收到N1的数据后，为满足一致性级别，向N2发送摘要请求
- N4对比N1和N2的数据时间戳，返回最新的版本给客户端
- N4继续向剩余副本N3发送摘要请求；若发现三者中存在过期数据，后台发起读修复请求

底层数据加载**整体读路径（按顺序）详细见04底层原理Q5**：

Memtable检查 → Row Cache检查（可选） → Bloom Filter过滤 → Partition Key Cache检查（可选） → Partition Summary定位 → Partition Index精确定位 → Compression Offset Map映射磁盘位置 → SSTable数据读取 → 多版本合并返回

# 2. CS如何完成写请求的

写入路径分为五个阶段：**Coordinator分发** → **Commit Log持久化** → **Memtable写入** → **Flush生成SSTable** → **后台Compaction**。全程顺序IO，无原地更新。 **详细见04底层原理Q6**

**协调器路由**：

- 客户端连接任意节点（协调器），协调器根据 **分区键（Partition Key）** 计算出该行数据所属的Token范围，确定所有**副本节点**
- 协调器向所有副本节点并行发送写请求，无论这些副本是否可用，写请求都会被转发给所有副本节点
- 客户端的**写一致性级别（Write Consistency Level）** 决定了协调器需要等待多少个副本节点**确认写入成功**后，才能向客户端返回成功

**副本节点写入流程**：

- 每个副本节点收到写请求后，先**追加写入 Commit Log**（磁盘顺序写，保证持久化）
- 再将数据**写入 MemTable**（内存中的可变数据结构，一般使用跳表或红黑树）—— 只要这两步完成，即视为该节点写入成功，即可向协调器返回确认
- MemTable 写满后，后台异步**flush 为 SSTable**（不可变的磁盘文件），该过程不影响写入性能

**一致性级别与异常处理**：

- 不同写一致性级别决定了至少需要多少副本确认：`ANY`（至少写入一个副本或 Hint 即可）、`ONE`、`QUORUM`（多数派，即 `replication_factor / 2 + 1`）、`ALL`（全部副本）
- 如果协调器无法联系到**足够多的副本来满足一致性级别**，直接抛出 `UnavailableException`，不执行任何写入
- 如果有足够副本可用，但在超时时间内未收到足够确认，抛出 `TimeoutException`
- 注意：即便抛出异常，部分副本**可能已经写入成功**，Cassandra 不保证回滚写操作，因此已写入的数据最终会通过修复机制达成一致

**写失败的补偿机制**：

- **Hinted Handoff（提示移交）**：若某个副本宕机，协调器在本地保存一条 Hint（提示信息），记录需要写入的数据和目标节点；目标节点恢复后，协调器将 Hint 重放给它，保证不丢失写入。一致性级别为 `ANY` 时，协调器保存 Hint 即可认为写入成功
- **Read Repair（读修复）**：读取时若发现副本间数据不一致，以最新时间戳为准修复过期副本（详见读请求流程）
- **Anti-Entropy Repair（反熵修复）**：通过 `nodetool repair` 命令发起 Merkle Tree 对比，定期修复集群中的所有不一致数据；这是唯一能完全修复所有差异的机制

**多数据中心写入**：

- 使用 `NetworkTopologyStrategy` 在多个数据中心间进行副本放置
- 协调器将写请求发送到**所有数据中心**的所有副本节点
- 写一致性级别可针对不同数据中心单独配置，例如 `EACH_QUORUM` 要求**每个数据中心**的多数派副本确认后才返回成功
- 跨数据中心延迟较高时，建议使用 `LOCAL_QUORUM` 保证本地数据中心强一致，同时依赖最终一致性和修复机制实现跨数据中心同步

# 3. 负载均衡策略

**概念与作用**：

- 负载均衡策略运行在**客户端驱动**中，而非服务端组件，它决定了：驱动与哪些节点建立连接、每次查询选择哪个节点作为协调器、以及故障切换时使用哪个节点
- 初始化后**不可动态更改**，只能在运行时查看当前策略状态

**主机距离（Host Distance）**：

- 驱动对每个节点计算一个**距离**，决定如何处理与该节点的连接
- `LOCAL`：与驱动配置所在数据中心相同的节点，驱动保持完整的连接池，是首选的协调器
- `REMOTE`：其他数据中心的节点，驱动只维持少量连接（连接池大小可配置），仅当本地节点不可用时才使用
- `IGNORE`：驱动完全忽略这些节点，不尝试建立任何连接

**查询计划（Query Plan）**：

- 每次执行查询时，驱动请求策略计算一个查询计划，本质是一个**主机迭代器**
- 驱动按计划中的主机顺序依次尝试：先用第一个主机作为协调器，若失败则按重试策略依次尝试后续主机
- 查询计划的特征：**每次请求不同**以均衡各节点负载；**只包含能处理该查询的主机**（即 LOCAL 和 REMOTE，排除 IGNORE）；**优先使用 LOCAL** 节点

**常见策略**：

- **RoundRobinPolicy**：最简单的轮询策略，在所有 LOCAL 节点间均匀分发请求。**仅适用于单数据中心**，多数据中心下不会跨 DC 查询
- **DCAwareRoundRobinPolicy**：多数据中心下的基础策略，优先轮询**本地数据中心**的全部节点，仅当所有本地节点都失败时，才尝试远程数据中心中**可配置数量**的节点
- **TokenAwarePolicy**：作为**装饰器**包装上述策略。优先将请求路由到**持有该数据对应 Token 范围的副本节点**（即数据所在节点），避免了协调器转发带来的额外网络跳转（称 TokenAware 路由）。当副本节点不可用时，退化为被包装策略的行为
- **默认组合**：现代驱动默认使用 `TokenAwarePolicy(DCAwareRoundRobinPolicy)`，既利用 Token 感知减少网络跳数，又通过 DC 感知保证跨数据中心容灾

# **4. 列存储是什么？与MySQL行存储有什么区别和优缺点**

Cassandra采用的存储模型是**宽列存储(Wide-Column Store)**，也称为**列族存储(Column-Family Storage)**，基于LSM-Tree的SSTable文件组织数据，与MySQL InnoDB的B+树行存储有本质区别。

**什么是宽列存储**：

- 数据按**行键(Row Key)**组织，但**同一行的列不固定**，每行可拥有不同的列集合
- 逻辑上表现为行，物理上同一行内**列按列名排序后连续存储**，空列不占用任何空间
- 区别于**列式存储**（如ClickHouse，同一列的所有值连续存储用于分析聚合），Cassandra是面向行的宽列模型，核心优化目标是**按主键的高效读写**

**与MySQL行存储的结构区别**：

- **存储结构**：MySQL InnoDB使用B+树聚簇索引，同一行所有列的数据连续存储在数据页中；Cassandra **使用Memtable + SSTable** ，写入先追加到内存Memtable再刷盘为SSTable，数据按列名排序存储在SSTable的连续区域
- **Schema灵活性**：MySQL需预先定义所有列及其类型，空列也占用固定存储位；Cassandra的列可以**动态添加**，每行独立，空列不占空间，修改Schema无需锁表
- **数据组织**：MySQL按聚簇索引顺序存储整行数据；Cassandra按**分区键**将数据分布到不同节点，分区内按**聚类列**排序，列按名称排序集中存储

**宽列存储的优点**：

- 支持**超宽表**，单行可包含数百万列，适合时序指标、监控打点等高基数场景
- Schema灵活，修改列定义无需ALTER TABLE锁表，上线速度快
- **稀疏存储高效**，大量空列不消耗存储空间和IO
- 写入性能极高，append-only LSM-Tree无随机IO和原地更新
- 读取时可仅select需要的列，避免读取整行降低IO量

**宽列存储的缺点**：

- 不支持跨行事务，仅单行级原子性，复杂操作需应用层补偿
- 不支持Join、子查询等复杂查询，必须按查询模式反推表设计（Query-First Design）
- 范围查询**必须指定完整分区键**，否则为跨分区低效查询
- 默认最终一致性，读后写一致性需调节一致性级别实现
- 若一行包含大量列但只需几列，未select \* 则仍可能因宽行扫描产生IO放大

**MySQL行存储的优点**：

- 完整ACID事务，强一致性，适合资金、订单等核心业务
- 查询能力强大，支持Join、子查询、聚合、复杂过滤条件
- 整行连续存储，按主键读取单行时一次IO即可获取完整行数据
- 生态成熟，工具链完善，运维经验丰富

**MySQL行存储的缺点**：

- Schema固定，ALTER TABLE大表耗时久，可能阻塞读写
- 空列浪费存储空间，每个列无论是否有值都占用位置
- 写入性能受限，B+树的原地更新和随机IO在高并发时存在瓶颈
- 单表扩展依赖垂直扩容，水平分库分表引入复杂度

# 5. 介绍下索引和MySQL索引有什么区别

**Cassandra的索引类型**：

- **主键索引（Primary Key Index）**：基于分区键的哈希自动建立的索引，用于**数据路由定位**——确定数据分布在哪个节点、哪个分区。解决分布式环境下"去哪找数据"的问题
- **二级索引（Secondary Index / 2i）**：对非主键列手动创建索引，每个节点**本地独立维护**自己的索引。适合**低基数**列（如状态字段），高基数列（如时间戳）会导致性能灾难——查询需要广播到所有节点
- **SASI（SSTable Attached Secondary Index）**：改进版二级索引，索引附着在每个 SSTable 上，支持**全文搜索和范围查询**，比旧版 2i 性能更优
- **SAI（Storage Attached Index）**：Cassandra 4.0+ 引入的最新索引，**每个 SSTable 独立建索引**，不再依赖本地 Hash Map，查询性能接近主键扫描，且支持更丰富的过滤条件

**MySQL的索引类型**：

- **聚簇索引（Clustered Index）**：B+Tree结构，叶子节点存储**整行数据**。主键即聚簇索引，解决"按主键快速获取完整行"的问题
- **二级索引（Secondary Index）**：B+Tree，叶子节点存储**主键值**，通过回表查询获取完整行。解决"按非主键列快速定位主键"的问题
- **复合索引、唯一索引、全文索引**等变体，分别解决前缀匹配、唯一性约束、文本搜索等问题

**核心区别**：

- **解决的问题不同**：Cassandra 主键索引解决**数据分布路由**（数据在哪个节点）；MySQL 索引解决**数据快速查找**（数据在哪个磁盘页）
- **底层结构不同**：Cassandra 使用一致性哈希 + 局部索引（每个节点独立），本质是**分布式索引**；MySQL 使用 B+Tree 全局索引，是**集中式索引**
- **查询约束**：Cassandra 高效查询**必须指定分区键**，二级索引无法替代主键索引；MySQL 任何索引列都可以作为查询入口，支持最左前缀匹配、范围查询、排序
- **二级索引能力差异**：Cassandra 二级索引**功能有限**，高基数列查询效率极低（全节点扫描），仅适合作辅助过滤；MySQL 二级索引**成熟高效**，覆盖索引可避免回表，是日常查询的核心手段
- **写放大**：Cassandra 的 LSM-Tree 结构天然对写友好，索引维护是 append-only 的；MySQL B+Tree 的索引更新涉及随机 IO 和页分裂，高并发写入压力更大

# 6. 列存储具体是什么样的，需要表结构吗

Cassandra **需要表结构**，通过 CQL 预先定义键空间和表的 schema，包括列名和类型。但它不是传统"无模式"数据库——Schema 定义了列的基本框架，但**行与行之间可以有不同的列集合**，且 Schema 变更（增删列）无需锁表，属于**灵活模式（Flexible Schema）**。

**物理存储模型**：

- 数据按 **分区键哈希** 被路由到对应节点
- 节点上，**每个分区内的数据连续存储在 SSTable** 中，按 **聚类列** 排序
- 同一行内，**列按列名排序后连续存储**，空列不占任何物理空间
- 每个单元格本质是 `(column_name, value, timestamp, ttl)` 的元组，Cassandra 用时间戳做冲突协调

**示例**：定义一个用户行为表

```sql
CREATE TABLE user_actions (
    user_id   UUID,
    day       text,
    ts        timestamp,
    action    text,
    url       text,
    duration  int,
    PRIMARY KEY ((user_id, day), ts)
) WITH CLUSTERING ORDER BY (ts DESC);
```

- **分区键**：`(user_id, day)` —— 决定数据存储在哪个节点，同一个用户同一天的数据在同一个分区
- **聚类列**：`ts` —— 分区内按时间戳降序排列
- **普通列**：`action`、`url`、`duration`

**写入以下数据**：

| user_id | day | ts | action | url | duration |
| --- | --- | --- | --- | --- | --- |
| u1 | 2024-01-01 | 10:00 | click | /a | 5 |
| u1 | 2024-01-01 | 09:00 | view | /b | null |
| u1 | 2024-01-02 | 11:00 | click | /c | 3 |

**物理存储形态**（以 `user_id=u1, day=2024-01-01` 分区为例）：

```
Partition Key: (u1, 2024-01-01)
→ SSTable 中连续存储，按 ts 降序排列

Row 1: ts=10:00
  action:click  url:/a  duration:5

Row 2: ts=09:00
  action:view   url:/b          (duration 不存在，不占空间)
```

- 分区以 `(u1, 2024-01-01)` 为单元整体落在一个节点上
- 该分区内两行按 `ts` 降序连续存放
- `duration` 在第二行为 null，**物理上完全不存储**，不消耗磁盘和 IO
- `user_id=u1, day=2024-01-02` 由于分区键不同，可能落在**另一个节点**或同一节点的不同分区

**宽列存储的核心含义**：

- 不是说列式存储（如 ClickHouse 按列聚合），而是**一行可以有大量列**，且列是动态的——通过聚类列和多值列实现类似"宽表"的效果
- 对于时序场景，同一分区可插入百万行（按聚类列排序），非常适合**时序打点、事件溯源、日志汇聚**等场景

# 7. 如何理解列存储和行存储的底层区别

**底层存储单元不同**：

- **行存储**（MySQL InnoDB）：数据以**行**为原子单元进行存储和IO。一行所有列的数据作为一个连续记录写入数据页。读一行时无论 select 几列，物理上都是读取整行数据（聚簇索引的完整记录）
- Cassandra 的列存储：底层以**单元格**为原子存储单元，每个单元格 `(column_name, value, timestamp, ttl)` 独立存在。逻辑上的"行"只是一个**共享同一主键的单元格集合**，物理上这些单元格在 SSTable 中按**列名排序**独立存储，并非连续的一条记录

**数据组织方式区别**：

- 行存储在磁盘上是**行连续**的：`[row1_col1|row1_col2|row1_col3] [row2_col1|row2_col2|row2_col3]`，每行固定宽度（即使有空列也占位）
- Cassandra 在 SSTable 内是**列连续**的：`partition_key + 聚类列 + [col1_val@ts] [col2_val@ts] [col3_val@ts]`，行的各列分散存储，按列名排序后形成连续区域，空列不占任何字节
- 换句话说，行存储的表是**横向扩展**（一行接一行），Cassandra 的列存储是**纵向扩展**（一个分区内所有行的所有列，按列名聚合排序）

**写操作对比**：

- 行存储更新一行中某一列，需要**读数据页 -> 修改 -> 写回**整行，即使只改一列也涉及整行的随机IO
- Cassandra 写某一列时，**仅追加该单元格的新版本** `(列名, 新值, 新时间戳)` 到 MemTable，与同行的其他列完全无关

**读操作对比**：

- 行存储 select 单列时，物理上仍需要将整行加载到内存再过滤，存在**IO放大**
- Cassandra 在 SSTable 中按列名排序，select 指定列时只需扫描该列名对应的连续区域（称为**列修剪**），跳过其他列，读取时直接**按需读取指定列的内容**

**Schema 的约束力不同**：

- 行存储的 Schema 是**刚性**的：列的数量、顺序、类型一旦定义，所有行都必须遵守，空列仍占用固定偏移量
- Cassandra 的 Schema 是**柔性**的：表定义只是确定了**列名的范围**，实际每条记录只存储有值的列；不同行可以有**完全不相同的列集合**，增加新列等同于写入新单元格，无需修改 Schema

**直观理解**（存同一份数据）：

```
行存储（MySQL）:
┌─────────┬──────┬───────┬──────┐
│ user_id │ name │ phone │ age  │
├─────────┼──────┼───────┼──────┤
│   u1    │ tom  │ 12345 │  25  │  ← 连续整行
│   u2    │ jerry│ 67890 │ null │  ← age 空也占位
└─────────┴──────┴───────┴──────┘

Cassandra 列存储（SSTable 内）:
Partition: u1
  name:tom @ts1,  phone:12345 @ts2,  age:25 @ts3
Partition: u2
  name:jerry @ts4,  phone:67890 @ts5
                  ↑ age 不存在，无存储开销
```

- 行存储每一行物理上绑在一起，适合**整行读写**、**复杂查询（Join/聚合）**
- Cassandra 按单元格独立存储，天然适合**高并发写入单个字段**、**稀疏数据**、**灵活 Schema** 的场景，但跨单元格的"行"概念是逻辑组装出的，不支持跨行事务和 Join

# 8. CS的特点和适用场景

Cassandra是面向行的宽列模型， **核心优化目标是按主键的高效读写**

**核心特点**：

- **去中心化 P2P 架构**：所有节点地位平等，无单点故障，任意节点均可作为协调器
- **线性可扩展**：增加节点即可线性提升吞吐和容量，无需停机重新分片
- **最终一致 + 可调一致性**：默认 AP 系统，但可通过 CL 级别调节到强一致
- **写入性能极高**：基于 LSM-Tree 的 append-only 写入，无随机 IO 和行锁，适合高并发写密集场景
- **多数据中心原生支持**：NetworkTopologyStrategy 配合 Snitch，天然支持跨地域部署
- **灵活 Schema**：宽列存储，可动态加列，空列不占空间，适合稀疏数据
- **自愈能力**：Hinted Handoff、Read Repair、Anti-Entropy Repair 自动修复不一致
- **不完全适用场景**： **不擅长复杂 Join、聚合分析、跨分区范围查询** ，这些需求应配合 Spark 或 Presto

**适用场景**：

- **写密集型应用**：Cassandra 最大的优势场景，写入 QPS 远超传统数据库。例如用户行为日志、埋点数据、IoT 传感器上报、**Netflix 用户访问痕迹和日志**、**eBay 的 Like/Want/Own 等社交信号（400M+ 写/天）**
- **用户画像与个性化**：低延迟查询用户标签、偏好设置，支持 PB 级画像存储。**饿了么将其用于用户画像和历史订单存储**
- **时序数据与物联网**：传感器设备持续上报，天然按时间戳聚类，单分区可存海量时间点。**华为用于运动健康数据、应用统计**
- **消息与 Inbox**：Cassandra 最早由 Facebook 开发用于消息存储，适合高并发写入和按时间线拉取。**Instagram 用于 inbox 和 newsfeed**
- **风险控制与反欺诈**：写多读少，实时写入访问记录和事件，配合规则引擎做实时判断。**Netflix 用于推荐风控，eBay 用于反作弊和欺诈检测**
- **大容量产品目录与收藏夹**：产品分类、收藏列表等场景，Cassandra 的集合类型（Set/List/Map）天然适合存储用户选中项。**eBay 存储 200TB+ 商品社交信号数据**
- **大数据平台的数据服务层**：Cassandra 作为接口层提供毫秒级点查，背后由 Spark/Flink 写入分析结果。**饿了么通过 Cassandra 为自助化 SQL 平台自动生成 HTTP/SOA 接口**

# **7. CS比MySQL适合什么场景，能替代MySQL吗**

Cassandra和MySQL的**适用场景完全不同**，不是替代关系而是互补关系。**Cassandra不能替代MySQL**。

**Cassandra的适用场景（写多读少、海量写入）**：

- **时序与日志场景**：用户行为日志、IoT设备上报、监控指标采集，写入量每秒百万级，Cassandra LSM-Tree引擎全程顺序IO，写入远超MySQL B+Tree
- **写多读少且无强事务需求**：Cassandra写入是内存 + 顺序IO，几乎无锁，**写入吞吐可水平扩展**；MySQL B+Tree写入涉及B+Tree分裂、随机IO和行锁，写入成为瓶颈
- **多DC部署、跨地域容灾**：Cassandra原生支持跨DC复制，写请求自动分发到多DC，无需中间件；MySQL跨DC需要主从复制或中间件层
- **Schema灵活、动态列**：Cassandra写时无需提前定义列，同一表不同行列集合不同，适合字段不确定的数据写入；MySQL必须先ALTER TABLE加列

**MySQL的适用场景（读多写少、复杂查询、强事务）**：

- **核心交易系统**：订单、支付、账户余额，需要**ACID事务**和行级强一致性，Cassandra不支持跨分区事务，LWT性能差且受限
- **复杂关联查询**：多表JOIN、子查询、GROUP BY、窗口函数，Cassandra不支持JOIN，只能在应用层做
- **读多写少场景**：MySQL B+Tree在静态数据上的**点查和范围查延迟极低**；Cassandra读路径需要合并多个SSTable，Bloom Filter + Index多步查找，读延迟比MySQL高且不稳定
- **数据量可控（单表千万\~亿级）**：MySQL单表性能优秀，运维成熟；Cassandra更适合百TB到PB级

**关键对比**：

- **写入**：Cassandra >> MySQL（LSM-Tree顺序IO vs B+Tree随机IO + 行锁）
- **点查**：MySQL > Cassandra（B+Tree深度3-4层，一次IO定位；Cassandra需合并多个SSTable、经过Bloom Filter + Index多步）
- **复杂查询**：MySQL >> Cassandra（MySQL支持JOIN/子查询，Cassandra不支持）
- **事务**：MySQL >> Cassandra（MySQL完整ACID，Cassandra只有行级原子性）
- **水平扩展**：Cassandra >> MySQL（Cassandra加节点自动rebalance；MySQL需分库分表中间件）
- **多DC**：Cassandra原生支持，MySQL需额外方案

**结论**：

- Cassandra是**写优化的分布式键值存储**，适合海量写入、水平扩展、多DC、无强事务需求
- MySQL是**读优化的关系型数据库**，适合强事务、复杂查询、存量数据高效读取
- 不能也不应替代MySQL，实际架构中常**两者并用**——Cassandra存海量时序/日志数据，MySQL存核心交易和关系型数据

# 9. CS对主键的读写很高效?

**精准路由，零广播**：

- Cassandra 对分区键做 **Murmur3 哈希**，直接计算出该数据所属的 Token 范围，确定数据所在的节点。查询只需发送到目标节点，无需向集群广播或使用中心路由表
- 这是最高效的部分——**单次网络请求即可定位到正确节点**，不存在扫描全部节点的开销

**Bloom Filter 快速跳过**：

- 每个 SSTable 都附带一个 **Bloom Filter**，用于快速判断该 SSTable 中是否包含目标分区键
- Bloom Filter 在内存中，查询时先检查 Bloom Filter，如果返回"不存在"，则直接跳过该 SSTable，**避免执行无效的磁盘 IO**。多数场景下只需扫描 1-2 个 SSTable

**分区内连续存储 + 排序**：

- 同一个分区的所有行在 SSTable 中**连续存储**，且按聚类列排好序
- 按分区键查询时，在 SSTable 中定位到分区起始位置后就是**顺序扫描**，不产生随机磁盘 IO
- 按聚类列的范围查询也因预排序而高效，只需读取连续的一段数据

**局部索引（Partition Index）**：

- 每个 SSTable 维护一个**分区索引**，记录每个分区键到 SSTable 内偏移位置的映射
- 索引保存在内存中，定位分区时通过索引直接跳到目标偏移量，**无需遍历整个 SSTable**

**写入：append-only 无随机 IO**：

- 按主键写入时，数据先追加到 Commit Log（顺序写），再写入内存 MemTable
- MemTable 按分区键和聚类列在内存中排序，flush 时以顺序 IO 写入 SSTable
- **整个过程不存在随机磁盘 IO**，即使主键分布散乱，写入吞吐依然稳定

**对比 MySQL 的 B+Tree**：

- MySQL 按主键查询虽然也很快（B+Tree 聚簇索引），但数据集中在一个节点，扩展受限于单机磁盘 IO
- 当数据量超过单机容量时，MySQL 需要分库分表，跨分片查询复杂度剧增
- Cassandra 的主键路由天然是分布式的，**扩展节点后主键查询效率几乎不变**，因为每个节点只负责其分区范围，路由仍旧是一次哈希计算

# **8. CS为什么对写性能好**

Cassandra写性能好的核心原因在于**LSM-Tree架构 + 无SQL引擎开销 + 全程顺序IO + 写后异步持久化**四个方面。

**1. 无SQL引擎和事务锁开销**：

- RDBMS处理每个写请求需要SQL解析、语义优化、执行计划生成、行锁管理、事务日志、MVCC版本维护等一系列操作，**每条SQL都有一整套开销链条**
- Cassandra没有SQL解析层（CQL很薄，本质是key-value API），**无事务、无行锁、无MVCC版本管理**。写操作只是简单追加，不涉及任何加锁和冲突检测（冲突通过时间戳LWW解决，无需锁）

**2. LSM-Tree：全程顺序IO，无随机写入**：

- 写入路径：写Commit Log（**顺序追加**）→ 写Memtable（**纯内存**）→ Flush生成SSTable（**批量顺序写盘**）
- 对比MySQL InnoDB B+Tree：更新一行需要**随机IO**定位数据页、写Undo/Redo Log、可能触发**页分裂**和**索引随机写**
- Cassandra**全程无原地更新、无随机IO**，所有磁盘操作都是**串行顺序追加**，这是写性能远超MySQL的根本原因

**3. 写后快速返回（异步持久化）**：

- 写请求写入Memtable后即认为"写入完成"，无需等待SSTable刷盘。客户端在**Commit Log + Memtable完成后就可返回**（取决于Consistency Level），整个路径在毫秒级
- Commit Log采用**periodic fsync**（默认每10秒），批量刷盘，大幅减少磁盘IO次数。代价是节点宕机可能丢失最后10秒内的数据
- Flush到SSTable在后台异步执行，不阻塞前台写入（除非Memtable积压超过`memtable_cleanup_threshold`）

**4. 无锁并发**：

- SSTable**不可变**，读取无锁，写入直接追加新文件，不存在读写冲突
- 每行数据自带**时间戳**，冲突时最后写入者胜出（LWW），无需分布式锁或行级锁
- 同一分区下多个Row的写入互不影响，因为是不同的聚类列值

**5. SSD友好，抑制写放大**：

- LSM-Tree的**批量顺序写入**非常适合SSD特性。SSD随机写性能差且有写放大（Write Amplification），Cassandra将随机写转化为顺序写，**大幅降低SSD的写放大因子**，延长SSD寿命
- 虽然后台Compaction也有写放大，但相比B+Tree的随机写放大，LSM-Tree的写放大更可控且对SSD更友好

**6. 多级缓存补偿读性能**：

- Cassandra设计上**牺牲部分读性能换取最大写入能力**，但通过多级缓存显著改善了读路径：Bloom Filter排除不相关SSTable、Key Cache跳过索引查找、Row Cache缓存整行、Summary内存索引快速定位。因此**基于RowKey的点查性能并不差**
- 读时合并多个SSTable + Memtable的结果，按时间戳返回最新版本

**关键权衡**：

- **写放大**：Compaction将多个小SSTable合并为大SSTable会产生写放大，LeveledCompactionStrategy写放大更高但读性能更好
- **读路径复杂**：读请求需要检查Memtable + 多个SSTable，经过Bloom Filter → Summary → Index → 解压多步，延迟比B+Tree高且不稳定
- **但不影响Cassandra作为写优化存储的定位**——它追求的是**海量写入吞吐**而非单次写入低延迟，后者是MySQL的强项
