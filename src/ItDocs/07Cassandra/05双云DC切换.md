# 2. 集群拆分

**适用场景**：业务持续增长，单集群承载多业务线，资源相互干扰；或者需要将部分业务迁移到新集群进行资源隔离、降低运维风险。

**整体流程（7步）**：

- 业务要拆分的keyspace，**搭建新集群并复制表结构**（只建表，无数据）
- 业务集成SDK插件，通过**拦截器方式拦截写请求**，将增量数据的主键信息写入DMQ
- 对老集群的业务keyspace**创建快照**，使用**sstableloader工具**将存量数据批量导入到新集群
- 部署**双写服务（sds sync）**，消费DMQ中的主键消息，根据主键到老集群读取完整记录，写入新集群
- **检查两个集群数据一致性**，确保DMQ无堆积
- 部署**全量数据检查修复工具**，扫描对比两集群全量数据，保证最终一致
- 业务去掉插件或关闭双写，**更改配置滚动重启**，正式接入新集群

**增量同步机制详解**：

**插件层（生产者）**：

- 以**拦截器**的方式埋点在业务服务中，拦截写入请求，记录`（接口名、表名、主键）`作为消息写入DMQ
- 每个分区对应一个DMQ分区，保证同一主键的消息路由到同一分区，维持有序性

**双写服务（消费者）**：

- 从DMQ消费消息，根据表名和主键到**老集群查询**完整数据：
  - 如果是**删除操作**（老集群查不到，新集群有）→ 从新集群删除该记录
  - 如果是**写入/更新操作** → 对比新老集群数据，不一致则用老集群数据覆盖新集群
- 本质是一个**增量数据补偿同步**机制，保证新集群最终追上老集群的增量变更

**存量导入**：

- 使用Cassandra自带的 **sstableloader工具** 将快照SSTable直接导入新集群，比逐条写入快得多
- 快照创建期间的数据变更通过DMQ增量同步补偿

**问题1：两边数据不一致**

- **现象**：双写服务运行期间，发现新老集群部分数据不一致
- **根因**：双写服务连接到了**旧集群的备用DC**（非主DC），数据在主DC写入后还未通过跨DC复制同步到备用DC时，双写服务去查，结果查不到，导致新集群缺少该记录
- **解决方案**：确保双写服务**强制连接到旧集群的主DC**进行查询，避免跨DC复制延迟导致的数据缺失

**问题2：DMQ某个分区消息持续堆积，消费极慢**

- **背景**：30个分区，一亿五千万数据
- **现象**：仅**一个分区**堆积严重，其他29个分区消费正常
- **排查分析**：
  - 一般堆积原因：消费慢 或 发送端数据量大
  - 本次只有**单个分区**堆积，且非业务高峰期 → **排除发送端问题**
  - 检查消费端日志，发现大量**重试日志**，消费耗时剧增
  - 进一步发现大量 `keyspace not exist` 异常——新集群中没有对应keyspace，该keyspace属于**业务另一个不需要拆分的集群**
- **解决方案**：在插件配置中将该keyspace加入**黑名单**，拦截器不对该keyspace写DMQ消息，消除无效消息和重试开销
- **经验教训**：插件配置需精确限定待拆分keyspace的白名单/黑名单，避免误拦截无关keyspace的操作导致无效消息污染DMQ

**面试追问点**：

- **为什么用DMQ而不是直接双写？** 解耦生产者与消费者，便于异步削峰填谷；同时主键消息量远小于全量数据，DMQ吞吐压力小
- **双写期间业务如何保证读一致？** 读请求仍指向老集群，新集群只作为被动同步目标，切换前不对外提供读服务
- **如何验证两集群数据一致？** 全量数据检查修复工具在低峰期逐分区扫描比对，使用checksum对比，不一致则用老集群数据覆盖
- **回滚方案**：如新集群异常，保留插件和双写服务，业务配置切回老集群即可快速回滚，主键消息不丢失
