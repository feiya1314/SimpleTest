# 1. Cassandra是什么？

Cassandra是**Apache基金会开发的分布式NoSQL数据库系统**，设计目标是处理大规模数据的高吞吐量写入和读取，同时保证高可用性和线性可扩展性。它最初由Facebook开发，后开源捐赠给Apache基金会，广泛应用于需要处理海量数据的互联网应用，如消息系统、物联网、推荐系统等。Cassandra采用**去中心化的主从架构**，没有单点故障，所有节点地位平等，通过一致性哈希算法实现数据分布。


Cassandra的 **核心概念** 包括：

- **节点(Node)**：集群中的单个服务器实例，每个节点存储部分数据，所有节点协同工作形成完整数据集
- **集群(Cluster)**：由多个节点组成的Cassandra服务器集合，是Cassandra部署的基本单位
- **数据中心(DataCenter)**：节点的逻辑分组，通常对应一个物理数据中心或可用区， **用于数据冗余和故障隔离**
- **键空间(Keyspace)**：数据库的顶层命名空间，类似于关系数据库的数据库，定义了数据复制策略
- **表(Table)**：键空间下的数据存储单元，由行和列组成，但不需要预先定义严格的表结构
- **分区键(Partition Key)**：决定数据分布到哪个节点的主键，通过哈希函数计算节点位置
- **副本(Replication)**：同一数据在多个节点上存储，提供数据冗余和高可用性

Cassandra集群中的 **节点类型** 包括：

- **普通节点**：存储数据并参与查询处理的基本节点
- **协调器节点**：接收客户端请求并协调查询到相关节点的节点，每个节点都可以充当协调器
- **种子节点**：用于引导新节点加入集群的节点，在集群启动时需要至少一个种子节点
- **虚拟节点(VNode)**：将物理节点划分为多个虚拟分区，使数据分布更均匀，减少节点添加/删除时的数据迁移量

Cassandra的数据模型基于**宽列存储(Wide Column Store)**，主要特点包括：

- **无模式设计**：同一表中的行可以有不同的列，不需要预先定义所有列
- **二维表结构**：数据按行键(Row Key)和列族(Column Family)组织
- **分区存储**：数据通过分区键哈希后分布到不同节点
- **静态列和动态列**：静态列在同一行中所有列族共享，动态列可以灵活添加
- **聚类列(Clustering Column)**：控制行内数据的排序方式，支持复杂查询

# 2. 一致性级别有哪些？

Cassandra提供可调的一致性级别，主要包括：

- **ONE**：只读写一个副本，延迟最低但一致性最弱
- **QUORUM**：读写多数副本（(副本因子/2)+1），平衡一致性和性能
- **ALL**：读写所有副本，一致性最强但延迟最高
- **LOCAL_QUORUM**：只考虑本地数据中心的多数副本
- **EACH_QUORUM**：每个数据中心都需要多数副本确认
- **LOCAL_ONE**：只读写本地数据中心的一个副本
- **SERIAL**：用于轻量级事务（LWT）的串行化一致性

# 3. 复制策略是什么？

Cassandra支持两种复制策略：

- **SimpleStrategy**：用于单数据中心部署，将副本均匀分布在环上，适合开发和测试环境
- **NetworkTopologyStrategy**：用于多数据中心部署，可为每个数据中心配置不同的副本数，适合生产环境
- **副本因子(Replication Factor)**：定义每个数据中心中数据的副本数量，副本因子=3表示每个数据中心有3份数据
- **数据放置算法**：通过一致性哈希和机架感知策略决定副本的具体放置位置

# 4. 如何保证高可用性？

Cassandra通过以下机制保证高可用性：

- **去中心化架构**：所有节点地位平等，没有主从之分，任何节点故障不影响整体服务
- **数据复制**：每个数据项在多个节点上存储副本，单个节点故障时数据仍可访问
- **无单点故障**：集群中没有中心节点或协调器，请求可路由到任何可用节点
- **自动故障检测**：通过Gossip协议检测节点状态，自动将请求路由到健康节点
- **可调一致性**：根据业务需求选择适当的一致性级别，在可用性和一致性间权衡

# 5. 与传统关系数据库的主要区别是什么？

Cassandra与传统RDBMS的主要区别包括：

- **数据模型**：Cassandra是宽列存储，RDBMS是行存储，Cassandra无固定模式
- **扩展方式**：Cassandra通过增加节点水平扩展，RDBMS通常垂直扩展
- **事务支持**：Cassandra支持轻量级事务但不支持复杂事务，RDBMS支持完整ACID事务
- **查询语言**：Cassandra使用CQL（类SQL），但功能有限，RDBMS使用完整SQL
- **一致性模型**：Cassandra采用最终一致性，RDBMS采用强一致性
- **设计目标**：Cassandra优化写入性能和高可用，RDBMS优化复杂查询和数据完整性
