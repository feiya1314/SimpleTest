# 1. Cassandra是什么？介绍其核心架构特点

Cassandra是**Apache基金会开发的分布式NoSQL数据库**，最初由Facebook开源，设计目标是大规模数据的高吞吐写入和高可用，同时具备**线性可扩展能力**。

整体架构特点：

- **去中心化的P2P架构**：所有节点地位完全平等，没有主从之分，不存在单点故障。任何节点都可以充当**协调器(Coordinator)**接收客户端请求，根据一致性级别将请求转发到目标副本节点并汇总结果
- **基于一致性哈希的数据分布**：数据通过分区键的哈希值分布在环上，节点负责特定Token范围的数据
- **可调一致性**：CAP理论中属于AP系统，但允许根据业务需求灵活调节一致性级别
- **最终一致性**：默认采用最终一致性模型，但通过调节一致性级别可以做到强一致

核心概念包括：

- **节点(Node)**：运行Cassandra实例的服务器，每个节点负责部分数据
- **集群(Cluster)**：多个节点组成的集合，对外表现为统一的数据库系统
- **数据中心(DataCenter)**：节点的逻辑分组，通常对应物理数据中心或云可用区，用于容灾隔离
- **键空间(Keyspace)**：顶层命名空间，类似于关系数据库中的数据库，定义复制策略和副本因子
- **表(Table)**：键空间下的数据存储单元，由行和列组成，但不需要预先定义严格的列结构，同一表的不同行可以拥有不同列
- **分区键(Partition Key)**：决定数据分布到哪个节点的主键，通过哈希计算确定目标节点
- **聚类列(Clustering Column)**：控制同一分区内数据的排序顺序，支持多维排序查询
- **副本(Replica)**：同一数据在多个节点上的拷贝，提供冗余和高可用
- **虚拟节点(VNode)**：将物理节点划分为多个虚拟分区，使数据分布更均匀

Cassandra集群中的 **节点类型** 包括：

- **普通节点**：存储数据并参与查询处理的基本节点
- **协调器节点**：接收客户端请求并协调查询到相关节点的节点， **每个节点都可以充当协调器**
- **种子节点**：用于引导新节点加入集群的节点，在 **集群启动时需要至少一个种子节点**
- **虚拟节点(VNode)**：将物理节点划分为多个虚拟分区，使数据分布更均匀，减少节点添加/删除时的数据迁移量

# 2. Cassandra无中心化是如何实现的

Cassandra的无中心化架构通过以下机制协同实现：

**P2P对等架构**：

- 集群中**所有节点地位完全平等**，不存在主节点或从节点
- 每个节点都可以接收客户端请求并充当协调器，不存在单点故障
- 节点增减无需重启集群或重新配置，自动感知与加入

**Gossip协议**：

- 每个节点**每秒**与集群中**至多3个**随机节点交换状态信息
- 传播内容包括：节点存活状态、负载信息、Schema变更、Token范围等
- 使用 **Phi Accrual Failure Detector** 基于历史心跳间隔统计计算故障概率，比固定超时更准确
- 新节点启动时通过种子节点(Seed Node)加入集群，通过Gossip逐步发现整个集群

**一致性哈希**：

- 数据通过 **Murmur3Partitioner** 将分区键哈希到 `-2^63` 到 `2^63-1` 的环上
- 每个节点负责环上的一段或多段Token范围，无需中心路由表
- 新增或移除节点时仅迁移相邻节点的数据，影响范围最小化
- 虚拟节点(VNode)进一步打散数据分布，使负载在各节点间均匀分摊

**协调器(Coordinator)**：

- 客户端**随机选择任意节点**发起请求，该节点自动成为协调器
- 协调器根据一致性级别将请求转发到目标副本节点，汇总结果后返回客户端
- 协调器不持有特殊状态，故障时客户端切换到其他节点即可

**副本策略 + Snitch**：

- 数据自动维持多个副本（生产环境通常RF=3），副本分布在不同的物理节点上
- **NetworkTopologyStrategy** 结合 **Snitch** 的拓扑感知能力，确保副本跨机架/数据中心分布
- 读取时Snitch帮助协调器选择**延迟最低的副本**返回数据，实现智能路由

这些机制组合在一起，使得Cassandra集群不存在中心组件，任意节点故障不影响整体可用性，真正做到**去中心化的水平扩展**。

# 3. 一致性哈希和虚拟节点

Cassandra使用**一致性哈希**算法决定数据分布：

- 整个哈希环范围是 **-2^63 到 2^63-1**
- 每个 **节点负责环上的一段连续Token范围**
- 数据写入时，根据分区键计算哈希值，定位到环上负责该Token范围的节点
- 新增或移除节点时**只需迁移相邻节点的数据**，而非全量重新分布

虚拟节点(VNode)的改进：

- 每个物理节点被划分为**多个虚拟节点**（默认256个），每个VNode独立分配Token
- **物理节点负责一组VNode** ，天然解决了异构节点负载不均的问题
- 节点增删时，数据迁移在**集群范围内均匀分摊**，而非仅影响相邻节点，减少热点压力
- 支持对不同性能的节点分配不同数量的VNode，实现按权重分布

Token的计算方式：

- 使用**Murmur3Partitioner**（默认）进行哈希，速度快且分布均匀
- 也可以选择RandomPartitioner或ByteOrderedPartitioner，后者支持范围查询但容易产生热点

# 4. 集群如何扩容？数据如何同步

**新增节点流程**：

新节点启动后通过配置的种子节点加入集群，进入 **Bootstrap（引导）** 状态：

- **Token分配**：启用虚拟节点(VNode)时，新节点**自动计算**需要负责的一组Token范围，从环上已有节点中"借取"部分数据范围；未启用VNode时需手动分配Token
- **数据流式传输**：新节点向负责相同Token范围的现有节点发起数据请求，源节点将对应的SSTable文件**流式传输(Streaming)**到新节点，基于Netty实现高效数据传输
- **加入完成**：数据同步完成后，新节点状态变为 **Normal**，开始对外提供服务；整个过程对客户端**透明无感知**

**数据同步机制**：

- **Streaming（流式传输）**：Bootstrap期间的核心同步方式，源节点将SSTable文件按Token范围切分后直接传输到目标节点，效率最高
- **Read Repair（读修复）**：Bootstrap完成后新节点可能缺少部分刚写入的增量数据，读请求时协调器对比各副本版本自动修复
- **Hinted Handoff（提示移交）**：Bootstrap期间若部分源节点宕机，协调器会为其存储Hint，待恢复后回放
- **nodetool repair（反熵修复）**：推荐在节点加入完成后**手动执行一次全量修复**，确保所有副本数据一致

**扩容注意事项**：

- 扩容速度受限于**网络带宽和磁盘IO**，流式传输会占用集群资源，建议低峰期操作
- **建议逐个节点**加入，等待数据同步完成后再加入下一个，避免多节点同时Bootstrap造成网络拥堵
- 新节点加入后不会自动触发整个集群的Rebalance，只迁移被"借取"范围的对应数据，VNode使数据迁移在集群范围均匀分摊
- 扩容**无需重启现有集群**，对线上业务无影响

# 5. 集群如何下线节点？数据和Hint怎么处理

Cassandra节点下线分为**正常下线**和**强制移除**两种方式，处理逻辑不同。

**正常下线（nodetool decommission）**：

- 针对**存活的节点**执行，节点将自己的Token范围重新分配给环上其他节点
- 节点将负责的数据通过 **Streaming** 流式传输到其他副本节点，确保复制因子不降低
- 完成后节点自动离开集群，状态从环中移除

**强制移除（nodetool removenode）**：

- 针对**宕机或不可达的节点**，需要指定该节点的Host ID
- 不涉及数据传输（目标节点已不可用），由其他副本节点通过 **Read Repair** 和 **nodetool repair** 后续补齐数据
- 如果宕机节点持有唯一副本（即RF不足），强制移除会导致数据永久丢失
- 执行后协调器不再将请求路由到该节点

**数据处理**：

- decommission 时会**主动迁移数据**到负责新Token范围的节点，保证副本数不变
- 下线完成后剩余节点通过 Gossip 感知环变化，自动更新Token路由表
- 下线操作前建议先执行 **nodetool drain** 刷入Memtable，防止数据丢失
- **RF > 1** 是安全的，单副本场景下下线节点会造成数据不可恢复

**Hint处理**：

- **存储在本节点的Hint**：decommission 前会先**回放(Replay)**所有未完成的Hint，确保已暂存数据写入目标节点后再下线；通过 `nodetool drain` 可以加速该过程
- **其他节点指向本节点的Hint**：Cassandra 自动检测到节点已下线后，会**删除**这些Hint（`system.hints` 或磁盘Hint文件），不再尝试回放
- 强制移除时，其他节点不再向该节点写入新Hint，已有Hint自动失效删除

# 6. 节点长时间宕机未恢复怎么办？数据怎么处理

**问题分析**：

- Hinted Handoff 默认窗口 **3小时**（`max_hint_window_in_ms`），超过窗口后协调器不再为该节点存储Hint，**新写入的数据该节点缺失**
- 该节点上的数据逐渐变旧，如果该节点是某些分区的**唯一副本**（RF=1场景），数据永久丢失
- 读写时协调器跳过宕机节点，集群整体可用性不受影响，但**实际副本数降低**

**处理策略**：

预期节点会恢复：

- 恢复后立即执行 **nodetool repair -full** 全量反熵修复，使用 Merkle Tree 比对并同步缺失数据
- 也可先执行 **nodetool repair -pr**（仅修复主范围）快速恢复关键数据，再逐步执行全量修复
- 修复期间网络和磁盘IO会上升，建议低峰期执行

节点无法恢复（物理损坏/永久下线）：

- 执行 **nodetool removenode** 从环中移除该节点，Token范围重新分配给其他节点
- 移除后必须执行 **nodetool repair** ，因为移除本身不触发数据迁移，需要修复使剩余节点补齐副本满足复制因子
- 如果宕机节点持有的部分数据在其他节点没有完整副本（RF=2场景，宕机节点与另一节点各持一份），需确保剩余节点通过修复补齐

**数据一致性保障**：

- **RF >= 3** 的场景容忍度高，单个节点宕机不会造成数据丢失，另外两个副本通过Read Repair和repair可补齐
- **RF = 2** 的场景存在风险，一个节点宕机后只剩单副本，须尽快修复
- **RF = 1** 是生产环境禁用配置，节点宕机即丢失该节点上的全部数据

**预防措施**：

- 配置 **监控告警**（节点存活、复制因子缺口、Hint窗口等指标）
- 合理设置 RF（生产环境至少3）和一致性级别（LOCAL_QUORUM）
- 定期执行 **nodetool repair**（建议每周），缩短数据不一致窗口
- 多数据中心部署提供跨DC容灾能力

# 7. 节点宕机后恢复怎么保证数据一致性

**节点恢复流程**：

节点启动后通过种子节点重新加入集群：

- Gossip 检测到节点重新在线，向**所有其他节点**广播该节点的存活状态
- 该节点重新加入一致性哈希环，协调器开始将读写请求重新路由到该节点
- 如果宕机时间在 **Hinted Handoff 窗口内**（默认3小时），协调器将存储的Hint逐条回放给恢复节点，补回宕机期间缺失的写入数据
- 如果**超出Hint窗口**，协调器已不再存储Hint，恢复节点缺少宕机期间的全部写入数据

**数据一致性保障**：

Hinted Handoff 回放（窗口期内）：

- 协调器重放Hint到恢复节点，每条Hint包含完整的数据和时间戳，恢复节点按时间戳写入
- 回放顺序不保证严格时序，最终通过**时间戳（LWW）**确定最新值
- 回放完成前，恢复节点上的数据是**不完整**的，但不会对外提供旧数据（协调器会对读请求做Digest对比）

Read Repair（读修复）：

- 恢复节点参与读请求时，协调器对比多个副本的Digest（数据摘要）
- 发现不一致时，协调器将**最新版本的数据**写入过时副本
- 读修复是**按需修复**，只修复被读取的分区，未被读取的分区仍可能不一致

nodetool repair（反熵修复）：

- 恢复后**必须手动执行**全量修复，这是保证数据最终一致的根本手段
- 使用 Merkle Tree 比对恢复节点与其他副本节点的数据，精确找出差异并同步
- 推荐先执行 **nodetool repair -pr**（仅主范围）快速恢复核心数据，再执行全量修复
- 修复期间**不应立即降低一致性级别**，建议先用LOCAL_QUORUM读写

**恢复期间的注意事项**：

- 恢复节点上**旧数据不会自动删除**，通过修复用新数据覆盖或标记为Tombstone
- 恢复后立即降低一致性级别（如从QUORUM降到ONE）可能导致读取到恢复节点上的旧数据
- 建议恢复后按顺序：先确认节点在线 → Hint回放监控完成 → 执行全量repair → 恢复正常业务
- 多数据中心场景下，跨DC的恢复流程相同，但Hint回放只在同一数据中心内进行

# 8. 复制策略有哪些

Cassandra支持两种复制策略：

- **SimpleStrategy**：仅用于单数据中心测试环境。将所有副本分布在环上的连续节点上，不考虑机架和数据中心拓扑，生产环境禁止使用
- **NetworkTopologyStrategy**：生产环境的标准选择。为每个数据中心分别指定副本因子，结合Snitch的拓扑信息，确保副本分布在不同的机架/数据中心，实现容灾

复制因子(Replication Factor)配置要点：

- 生产环境通常设为 **3**，允许容忍2个节点同时故障
- 写入和读取的一致性级别不能超过复制因子，否则操作无法完成
- 不同数据中心可以设置不同的复制因子，例如主数据中心设3，灾备数据中心设2

副本放置策略（NetworkTopologyStrategy下）：

- **第一个副本**放在分区键确定的节点上
- **后续副本**按顺时针方向放置在下一节点，且必须跨机架放置
- 如果有多数据中心，副本在各个数据中心独立分布

# 9. Snitch是什么？有哪些类型？

Snitch负责**感知网络拓扑结构**，决定节点间的网络距离，直接影响副本放置和请求路由：

- **SimpleSnitch**：默认配置，适用于单数据中心，所有节点在同一机架
- **GossipingPropertyFileSnitch**：生产环境最常用，通过配置文件指定数据中心和机架，信息通过Gossip传播，无需逐节点配置
- **Ec2Snitch**：自动识别AWS EC2地域和可用区，适合云部署
- **GoogleCloudSnitch**：适配GCP可用区
- **DynamicEndPointSnitch**：监控实际读写延迟，在运行时动态调整路由优先级，优先选择响应最快的副本

Snitch的关键作用：

- **请求路由**：协调器节点优先选择同一数据中心内延迟最低的副本
- **副本放置**：NetworkTopologyStrategy依赖Snitch确保副本跨机架/数据中心
- **故障切换**：远程数据中心副本仅在本地副本不可用时才被选择

# 10. 一致性级别有哪些？如何选择？

**读一致性级别**：

- **ONE**：从最近的一个副本读取，延迟最低，可能读到脏数据
- **QUORUM**：从大多数副本（N/2+1）读取并返回最新版本，平衡一致性和性能
- **ALL**：读取所有副本，一致性最强但任一节点故障会导致操作失败
- **LOCAL_QUORUM**：仅读取本地数据中心的大多数副本，避免跨DC延迟
- **EACH_QUORUM**：每个数据中心都需要大多数副本确认，适用于跨DC强一致场景
- **SERIAL/LOCAL_SERIAL**：用于轻量级事务(LWT)，实现线性一致性

**写一致性级别**：

- 规则与读一致类似，确认写入的副本数达到指定级别即返回客户端

**一致性选择原则**：

- **读+写一致性级别 > 复制因子**时实现强一致（例如 RF=3，R+W>3，如 QUORUM+QUORUM）
- 日常业务推荐 **LOCAL_QUORUM**，兼顾一致性和可用性
- 日志型或非关键数据可以用 **ONE** 追求极致性能
- LWT场景使用 **SERIAL** 配合Paxos协议实现原子条件更新

# 11. 与传统关系数据库的主要区别？

- **数据模型**：Cassandra是宽列存储，支持灵活的模式，同一表内行可拥有不同列；RDBMS是严格的行存储，模式固定
- **扩展方式**：Cassandra支持**线性水平扩展**，增加节点即可提升容量和吞吐；RDBMS主要靠垂直扩展
- **一致性模型**：Cassandra默认最终一致性，通过调节实现可调一致性；RDBMS提供ACID强一致性
- **事务支持**：Cassandra仅支持行级原子性和轻量级事务(Paxos)；RDBMS支持完整事务、外键、Join
- **查询语言**：CQL语法类似SQL但**不支持Join、子查询、聚合能力有限**；需按查询模式设计表（Query-First Design）
- **写入性能**：Cassandra**写入远优于RDBMS**，append-only的LSM-Tree结构，无随机IO和行锁竞争
- **适用场景**：Cassandra适合**写多读少、海量数据、多数据中心部署**的互联网场景；RDBMS适合复杂查询、事务一致性要求高的传统业务

# 12. 列存储是什么？与MySQL行存储有什么区别和优缺点

Cassandra采用的存储模型是**宽列存储(Wide-Column Store)**，也称为**列族存储(Column-Family Storage)**，基于LSM-Tree的SSTable文件组织数据，与MySQL InnoDB的B+树行存储有本质区别。

**什么是宽列存储**：

- 数据按**行键(Row Key)**组织，但**同一行的列不固定**，每行可拥有不同的列集合
- 逻辑上表现为行，物理上同一行内**列按列名排序后连续存储**，空列不占用任何空间
- 区别于**列式存储**（如ClickHouse，同一列的所有值连续存储用于分析聚合），Cassandra是面向行的宽列模型，核心优化目标是**按主键的高效读写**

**与MySQL行存储的结构区别**：

- **存储结构**：MySQL InnoDB使用B+树聚簇索引，同一行所有列的数据连续存储在数据页中；Cassandra **使用Memtable + SSTable** ，写入先追加到内存Memtable再刷盘为SSTable，数据按列名排序存储在SSTable的连续区域
- **Schema灵活性**：MySQL需预先定义所有列及其类型，空列也占用固定存储位；Cassandra的列可以**动态添加**，每行独立，空列不占空间，修改Schema无需锁表
- **数据组织**：MySQL按聚簇索引顺序存储整行数据；Cassandra按**分区键**将数据分布到不同节点，分区内按**聚类列**排序，列按名称排序集中存储

**宽列存储的优点**：

- 支持**超宽表**，单行可包含数百万列，适合时序指标、监控打点等高基数场景
- Schema灵活，修改列定义无需ALTER TABLE锁表，上线速度快
- **稀疏存储高效**，大量空列不消耗存储空间和IO
- 写入性能极高，append-only LSM-Tree无随机IO和原地更新
- 读取时可仅select需要的列，避免读取整行降低IO量

**宽列存储的缺点**：

- 不支持跨行事务，仅单行级原子性，复杂操作需应用层补偿
- 不支持Join、子查询等复杂查询，必须按查询模式反推表设计（Query-First Design）
- 范围查询**必须指定完整分区键**，否则为跨分区低效查询
- 默认最终一致性，读后写一致性需调节一致性级别实现
- 若一行包含大量列但只需几列，未select \* 则仍可能因宽行扫描产生IO放大

**MySQL行存储的优点**：

- 完整ACID事务，强一致性，适合资金、订单等核心业务
- 查询能力强大，支持Join、子查询、聚合、复杂过滤条件
- 整行连续存储，按主键读取单行时一次IO即可获取完整行数据
- 生态成熟，工具链完善，运维经验丰富

**MySQL行存储的缺点**：

- Schema固定，ALTER TABLE大表耗时久，可能阻塞读写
- 空列浪费存储空间，每个列无论是否有值都占用位置
- 写入性能受限，B+树的原地更新和随机IO在高并发时存在瓶颈
- 单表扩展依赖垂直扩容，水平分库分表引入复杂度

# 13. 集群如何保证高可用？

Cassandra通过多层机制保障高可用：

**故障检测与恢复**：

- **Phi Accrual Failure Detector**：基于Gossip的历史心跳间隔统计，动态计算节点故障概率，比固定超时更准确
- 检测到节点故障后，协调器自动将请求路由到其他副本节点

**Hinted Handoff（提示移交）**：

- 目标节点宕机时，协调器节点**本地保存写操作的Hint**（包含数据和目标节点信息）
- 默认保存 **3小时**（由max_hint_window_in_ms控制）
- 待目标节点恢复后，协调器将Hint回放给该节点，确保数据不丢失
- 适合短时间故障恢复，但不作为长期容错方案

**Read Repair（读修复）**：

- 读取时协调器会对比多个副本的数据版本（基于时间戳），发现不一致时**在后台异步修复旧版本副本**
- **Read Repair Chance**（默认0.1）控制读请求触发修复的概率
- 全范围读修复通过`nodetool repair`定期执行，修复Merkle Tree检测到的数据差异

**Anti-Entropy（反熵修复）**：

- 使用Merkle Tree（哈希树）比对不同节点上的数据
- 通过`nodetool repair`命令定期执行，是确保最终一致性的根本手段
- 推荐每周执行一次，避免数据长期不一致

# 14. CS的快照机制和恢复过程

**快照机制**：

`nodetool snapshot` 是Cassandra的**备份工具**，通过创建SSTable文件的**硬链接(Hard Link)**实现快速备份：

- 执行快照时Cassandra在表数据目录下创建 `snapshots/<快照名称>/` 子目录，内部是到当前SSTable文件的硬链接
- 硬链接只占用inode，**不复制实际数据**，因此快照创建速度极快（毫秒级），不消耗额外磁盘IO
- 当原SSTable被Cassandra的Compaction删除后，硬链接仍然保留数据，**快照中的数据独立于原始文件**
- 快照默认**不包含Memtable中的内存数据**，执行快照前建议先执行 `nodetool flush` 将Memtable刷盘为SSTable，确保快照包含最新数据
- 快照存储在 `<data_dir>/<keyspace>/<table>/snapshots/<snapshot_name>/` 路径下，集群中**每个节点各自执行**，不存在全局快照

**恢复过程**：

整节点恢复：

- 停止目标节点服务（`nodetool stopdaemon` 或直接 kill 进程）
- 清空数据目录下对应的 SSTable 文件（保留 `snapshots/` 目录）
- 将快照目录下的所有文件**复制回**数据目录的对应位置
- 重启节点，节点启动时从commit log回放未提交数据
- 最后执行 `nodetool repair` 确保与其他副本数据一致

单表恢复（无需停节点）：

- 使用 `nodetool refresh` 命令加载新的SSTable文件到指定表中
- Cassandra会自动检测并加载放置在数据目录下的新SSTable
- 恢复完成后执行 `nodetool repair` 同步差异数据

增量备份（Incremental Backup）：

- 在 `cassandra.yaml` 中设置 `incremental_backups: true` 启用
- 每次Flush时，SSTable的硬链接同步创建到 `backups/` 子目录
- 恢复时需要组合使用**全量快照 + 增量备份**一起恢复

**快照管理**：

- 使用 `nodetool listsnapshots` 查看当前所有快照及其占用的磁盘空间
- 使用 `nodetool clearsnapshot -t <快照名称>` 清理过期快照释放磁盘空间
- 快照不是增量备份，每次都是全量SSTable文件的硬链接，长期保留需注意磁盘占用
- 建议定期清理不再需要的快照，避免磁盘写满影响Compaction
