# 1. Cassandra是什么？介绍其核心架构特点

Cassandra是**Apache基金会开发的分布式NoSQL数据库**，最初由Facebook开源，设计目标是大规模数据的高吞吐写入和高可用，同时具备**线性可扩展能力**。

整体架构特点：

- **去中心化的P2P架构**：所有节点地位完全平等，没有主从之分，不存在单点故障。任何节点都可以充当**协调器(Coordinator)**接收客户端请求，根据一致性级别将请求转发到目标副本节点并汇总结果
- **基于一致性哈希的数据分布**：数据通过分区键的哈希值分布在环上，节点负责特定Token范围的数据
- **可调一致性**：CAP理论中属于AP系统，但允许根据业务需求灵活调节一致性级别
- **最终一致性**：默认采用最终一致性模型，但通过调节一致性级别可以做到强一致

核心概念包括：

- **节点(Node)**：运行Cassandra实例的服务器，每个节点负责部分数据
- **集群(Cluster)**：多个节点组成的集合，对外表现为统一的数据库系统
- **数据中心(DataCenter)**：节点的逻辑分组，通常对应物理数据中心或云可用区，用于容灾隔离
- **键空间(Keyspace)**：顶层命名空间，类似于关系数据库中的数据库，定义复制策略和副本因子
- **表(Table)**：键空间下的数据存储单元，由行和列组成，但不需要预先定义严格的列结构，同一表的不同行可以拥有不同列
- **分区键(Partition Key)**：决定数据分布到哪个节点的主键，通过哈希计算确定目标节点
- **聚类列(Clustering Column)**：控制同一分区内数据的排序顺序，支持多维排序查询
- **副本(Replica)**：同一数据在多个节点上的拷贝，提供冗余和高可用
- **虚拟节点(VNode)**：将物理节点划分为多个虚拟分区，使数据分布更均匀

Cassandra集群中的 **节点类型** 包括：

- **普通节点**：存储数据并参与查询处理的基本节点
- **协调器节点**：接收客户端请求并协调查询到相关节点的节点， **每个节点都可以充当协调器**
- **种子节点**：用于引导新节点加入集群的节点，在 **集群启动时需要至少一个种子节点**
- **虚拟节点(VNode)**：将物理节点划分为多个虚拟分区，使数据分布更均匀，减少节点添加/删除时的数据迁移量

# 2. 一致性哈希和虚拟节点

Cassandra使用**一致性哈希**算法决定数据分布：

- 整个哈希环范围是 **-2^63 到 2^63-1**
- 每个节点负责环上的一段连续Token范围
- 数据写入时，根据分区键计算哈希值，定位到环上负责该Token范围的节点
- 新增或移除节点时**只需迁移相邻节点的数据**，而非全量重新分布

虚拟节点(VNode)的改进：

- 每个物理节点被划分为**多个虚拟节点**（默认256个），每个VNode独立分配Token
- 物理节点负责一组VNode，天然解决了异构节点负载不均的问题
- 节点增删时，数据迁移在**集群范围内均匀分摊**，而非仅影响相邻节点，减少热点压力
- 支持对不同性能的节点分配不同数量的VNode，实现按权重分布

Token的计算方式：

- 使用**Murmur3Partitioner**（默认）进行哈希，速度快且分布均匀
- 也可以选择RandomPartitioner或ByteOrderedPartitioner，后者支持范围查询但容易产生热点

# 3. 复制策略有哪些

Cassandra支持两种复制策略：

- **SimpleStrategy**：仅用于单数据中心测试环境。将所有副本分布在环上的连续节点上，不考虑机架和数据中心拓扑，生产环境禁止使用
- **NetworkTopologyStrategy**：生产环境的标准选择。为每个数据中心分别指定副本因子，结合Snitch的拓扑信息，确保副本分布在不同的机架/数据中心，实现容灾

复制因子(Replication Factor)配置要点：

- 生产环境通常设为 **3**，允许容忍2个节点同时故障
- 写入和读取的一致性级别不能超过复制因子，否则操作无法完成
- 不同数据中心可以设置不同的复制因子，例如主数据中心设3，灾备数据中心设2

副本放置策略（NetworkTopologyStrategy下）：

- **第一个副本**放在分区键确定的节点上
- **后续副本**按顺时针方向放置在下一节点，且必须跨机架放置
- 如果有多数据中心，副本在各个数据中心独立分布

# 4. Snitch是什么？有哪些类型？

Snitch负责**感知网络拓扑结构**，决定节点间的网络距离，直接影响副本放置和请求路由：

- **SimpleSnitch**：默认配置，适用于单数据中心，所有节点在同一机架
- **GossipingPropertyFileSnitch**：生产环境最常用，通过配置文件指定数据中心和机架，信息通过Gossip传播，无需逐节点配置
- **Ec2Snitch**：自动识别AWS EC2地域和可用区，适合云部署
- **GoogleCloudSnitch**：适配GCP可用区
- **DynamicEndPointSnitch**：监控实际读写延迟，在运行时动态调整路由优先级，优先选择响应最快的副本

Snitch的关键作用：

- **请求路由**：协调器节点优先选择同一数据中心内延迟最低的副本
- **副本放置**：NetworkTopologyStrategy依赖Snitch确保副本跨机架/数据中心
- **故障切换**：远程数据中心副本仅在本地副本不可用时才被选择

# 5. 一致性级别有哪些？如何选择？

**读一致性级别**：

- **ONE**：从最近的一个副本读取，延迟最低，可能读到脏数据
- **QUORUM**：从大多数副本（N/2+1）读取并返回最新版本，平衡一致性和性能
- **ALL**：读取所有副本，一致性最强但任一节点故障会导致操作失败
- **LOCAL_QUORUM**：仅读取本地数据中心的大多数副本，避免跨DC延迟
- **EACH_QUORUM**：每个数据中心都需要大多数副本确认，适用于跨DC强一致场景
- **SERIAL/LOCAL_SERIAL**：用于轻量级事务(LWT)，实现线性一致性

**写一致性级别**：

- 规则与读一致类似，确认写入的副本数达到指定级别即返回客户端

**一致性选择原则**：

- **读+写一致性级别 > 复制因子**时实现强一致（例如 RF=3，R+W>3，如 QUORUM+QUORUM）
- 日常业务推荐 **LOCAL_QUORUM**，兼顾一致性和可用性
- 日志型或非关键数据可以用 **ONE** 追求极致性能
- LWT场景使用 **SERIAL** 配合Paxos协议实现原子条件更新

# 6. 与传统关系数据库的主要区别？

- **数据模型**：Cassandra是宽列存储，支持灵活的模式，同一表内行可拥有不同列；RDBMS是严格的行存储，模式固定
- **扩展方式**：Cassandra支持**线性水平扩展**，增加节点即可提升容量和吞吐；RDBMS主要靠垂直扩展
- **一致性模型**：Cassandra默认最终一致性，通过调节实现可调一致性；RDBMS提供ACID强一致性
- **事务支持**：Cassandra仅支持行级原子性和轻量级事务(Paxos)；RDBMS支持完整事务、外键、Join
- **查询语言**：CQL语法类似SQL但**不支持Join、子查询、聚合能力有限**；需按查询模式设计表（Query-First Design）
- **写入性能**：Cassandra**写入远优于RDBMS**，append-only的LSM-Tree结构，无随机IO和行锁竞争
- **适用场景**：Cassandra适合**写多读少、海量数据、多数据中心部署**的互联网场景；RDBMS适合复杂查询、事务一致性要求高的传统业务

# 7. 集群如何保证高可用？

Cassandra通过多层机制保障高可用：

**故障检测与恢复**：

- **Phi Accrual Failure Detector**：基于Gossip的历史心跳间隔统计，动态计算节点故障概率，比固定超时更准确
- 检测到节点故障后，协调器自动将请求路由到其他副本节点

**Hinted Handoff（提示移交）**：

- 目标节点宕机时，协调器节点**本地保存写操作的Hint**（包含数据和目标节点信息）
- 默认保存 **3小时**（由max_hint_window_in_ms控制）
- 待目标节点恢复后，协调器将Hint回放给该节点，确保数据不丢失
- 适合短时间故障恢复，但不作为长期容错方案

**Read Repair（读修复）**：

- 读取时协调器会对比多个副本的数据版本（基于时间戳），发现不一致时**在后台异步修复旧版本副本**
- **Read Repair Chance**（默认0.1）控制读请求触发修复的概率
- 全范围读修复通过`nodetool repair`定期执行，修复Merkle Tree检测到的数据差异

**Anti-Entropy（反熵修复）**：

- 使用Merkle Tree（哈希树）比对不同节点上的数据
- 通过`nodetool repair`命令定期执行，是确保最终一致性的根本手段
- 推荐每周执行一次，避免数据长期不一致

# 8. 节点如何加入和移除集群？

**节点加入(Bootstrapping)**：

- 新节点配置种子节点列表，启动后通过Gossip发现集群
- **自动获取Token**：新节点向种子节点请求Token范围，接管已有节点的部分数据
- **流式传输(Stream)**：数据从邻近节点流式传输到新节点，期间不影响集群服务
- 加入完成后新节点正常参与读写

**节点移除(Decommission)**：

- 使用 `nodetool decommission` 安全退出，节点将**负责的数据均匀迁移**到其他节点
- 使用 `nodetool removenode` 强制移除故障节点，数据通过Hinted Handoff和修复机制补充
- 数据迁移通过增量流式传输完成，对集群性能影响可控
- **建议逐个操作**，避免多个节点同时变更导致数据分布失衡

**Token范围变更**的注意事项：

- 虚拟节点(VNode)机制下节点增删的数据迁移被**全集群分摊**，影响远小于无VNode方案
- 变更期间建议降低一致性级别要求（如降为ONE），避免等待额外副本确认
