# **1. 读写时如何找到shard**

ES 通过**路由算法**定位文档所在的 shard，核心公式为：

**shard = hash(document_id) % number_of_primary_shards**

- 客户端发送读写请求到协调节点
- 协调节点根据文档 ID 计算 hash 值，对**主分片数量取模**，得到目标主分片编号
- 请求被转发到对应的主分片执行

写入时的 shard 定位

- 写请求通过路由算法定位到**主分片**
- 主分片写入成功后，将操作复制到该分片组的所有副本分片

读取时的 shard 定位

- 读请求同样通过路由算法定位到**主分片或副本分片**（随机轮询选择）
- 一个搜索请求会遍历**所有分片**，每个分片独立执行查询后汇总（因为无指定\_id路由id）
- 没有路由依据，无法预判数据在哪一个分片，执行**分片广播策略**：协调节点把查询下发给**该索引全部主 / 副本分片**，所有分片独立检索，最后协调节点合并结果。

**主分片数量的约束**

- 主分片数量在**索引创建时确定**，之后不可修改
- 这是因为路由算法依赖主分片数量取模， **修改后会导致已有数据的路由失效，数据无法找到**
- 如果需要增加分片，只能通过**新建索引并 reindex** 数据来实现

# **2. 多字段查询原理**

每个字段都有一个倒排索引，多字段查询是怎么实现的，特别是排序分页

**多字段查询的整体流程**：

当查询多个字段时（如 `multi_match` 或 `bool` 查询），ES 会分别在每个字段的倒排索引中查找，然后合并结果。整体分为 Query 阶段和 Fetch 阶段。

**Query 阶段的多字段处理**：

- **逐字段查找**：ES 在每个字段的倒排索引中独立查找匹配的文档 ID 列表。例如查询 `{"multi_match": {"query": "Elasticsearch", "fields": ["name", "content"]}}`，会分别在 name 和 content 的倒排索引中查找。
- **收集评分信息**：每个字段返回匹配文档的 docID 和该字段的 BM25 评分。
- **合并字段评分**：对于同一文档在多个字段的匹配，ES 根据配置的方式合并得分：
  - `best_fields`（默认）：取最高分字段的得分
  - `most_fields`：所有字段得分相加
  - `cross_fields`：把多个字段视为一个大字段，合并倒排列表后统一计算
  - `phrase` / `bool`：按短语查询或布尔逻辑分别计算后合并

**Fetch 阶段的全局排序**：

- 各分片在 Query 阶段返回的是 **本地 Top N**（按 \_score 或排序字段），不是全量结果。
- 协调节点收集各分片的本地 Top N，进行 **全局归并排序**，得到全局 Top N。
- 如果需要文档详情（如 \_source 字段），再根据全局排序后的 docID 列表去各分片 Fetch 完整文档。

**排序分页的实现原理**：

ES 支持两种排序方式，实现机制不同：

**按 \_score 评分排序（默认）**：

- Query 阶段各分片返回本地 Top K 的 docID 和 \_score。
- 协调节点对所有分片结果做 **堆排序（Top K 堆）**，维护全局 Top N。
- 使用 `from + size` 参数控制分页：协调节点需要收集 `from + size` 条结果，在内存中全局排序后，截取对应分页。

**按自定义字段排序（如 timestamp）**：

- 排序字段必须是 **doc_values**（列式存储）或 **fielddata**，不依赖倒排索引。
- Query 阶段各分片返回本地 Top K 的 docID 和排序字段值。
- 协调节点归并排序，但 **不返回文档内容**，只维护排序后的 docID 列表。
- Fetch 阶段根据最终排序的 docID 列表去各分片取 `_source`，返回给客户端。

**深度分页问题**：

- `from + size` 方式，`from` 越大，协调节点需要在内存中维护的结果集越大，性能急剧下降。
- **search_after**：基于上一页最后一条记录的排序值，作为下一页的起始点，避免深度遍历。适合实时搜索滚动加载场景。
- **scroll API**：在各分片上维护一个快照游标，一次性获取所有结果。适合批量导出，但不适合实时分页。

**为什么倒排索引不直接参与排序**：

- 倒排索引存储的是 **Term → 文档列表** 的映射，适合搜索匹配，但不适合按字段值排序。
- 排序依赖 **doc_values**（磁盘列式存储，内存映射）或 **fielddata**（内存中构建，开销大）。
- *score 评分虽然基于倒排索引的 TF/IDF 计算，但排序时也需要读取 doc*values 中的评分值进行归并。

**面试追问点**：

- **multi_match 的 best_fields 和 most_fields 有什么区别** ？→ best_fields 取匹配度最高的字段得分，most_fields 把所有字段得分相加；best_fields 适合精确匹配，most_fields 适合多字段都需匹配的场景。
- **为什么 from + size 默认不能超过 10000** ？→ `index.max_result_window` 默认 10000，防止深度分页导致内存溢出；超过时应使用 search_after 或 scroll。
- **search_after 能实现跳页吗？** → 不能，search_after 是基于上一页末尾的游标，只能顺序翻页；跳页需要重新查询或用其他方案。
- **多字段排序的优先级怎么定？** → 按 sort 数组顺序，第一个字段为主排序，相同时用第二个字段，以此类推；每个字段可指定升序/降序。

# **3. 多个倒排索引如何联合查询**

当查询包含多个条件（如 `age=18 AND gender=女`）时，ES 需要分别从每个字段的倒排索引中获取 Posting List，然后将多个列表合并得到同时满足所有条件的文档。**核心问题是：两个 Posting List 都是磁盘上的有序整数列表，如何高效地取交集？**

**单字段倒排索引的查询过程**：

- 以 `age=18 AND gender=女` 为例，先在 age 字段的倒排索引中查找：Term Index（内存）→ 定位 Term Dictionary 位置 → 找到 Term "18" → 获取 Posting List（包含所有 age=18 的文档 ID 列表）。
- 同样在 gender 字段的倒排索引中查找 Term "女"，获取对应的 Posting List。
- 两个 Posting List 分别是有序的文档 ID 数组，例如：age=18 → [2, 5, 7, 10, 15]，gender=女 → [1, 5, 8, 10, 12]。

**两种合并 Posting List 的方式**：

**方式一：跳表（Skip List）遍历合并**：

- 两个 Posting List 都是有序的，用双指针同时遍历两个列表，互相跳过不匹配的文档 ID，类似于归并排序中的合并步骤。
- 跳表在 Posting List 上层建立了索引，可以一次性跳过大量不匹配的 ID，不需要逐个比较。
- **适用于两个 Posting List 都在磁盘上的场景**，直接顺序读取磁盘，不需要额外内存。

**方式二：位图（Bitset）按位 AND 运算**：

- 将每个 Posting List 转换为 bitset（位图），文档 ID 对应的 bit 置为 1。
- 对两个 bitset 做 AND 位运算，结果中 bit 为 1 的位置就是同时满足两个条件的文档 ID。
- **适用于 filter 已经缓存在内存中的场景**，ES 的 filter cache 就是缓存每个条件对应的 bitset，合并时直接做位运算，不需要重新读磁盘。

**ES 的实际选择策略**：

- **filter 缓存命中**：条件对应的 bitset 已在内存中，直接用 Bitset AND 运算合并，速度极快。
- **filter 缓存未命中**：从磁盘读取 Posting List，用 Skip List 双指针遍历合并，避免将整个列表加载到内存。
- ES 会自动根据缓存状态选择最优策略，开发者不需要手动干预。

**MySQL 的处理方式对比**：

- MySQL 的联合索引（如 `(age, gender)`）是将多个字段组合成一个 B+ 树索引，查询时走联合索引逐层匹配。
- 如果没有联合索引，MySQL 只能选择其中一个单字段索引，另一个条件在遍历过程中逐行过滤，不会真正"联合使用"两个索引。
- PostgreSQL 从 8.4 版本开始支持 Bitmap Index Scan，可以同时使用两个索引并用 bitset 合并，原理与 ES 类似。

**实际场景举例**：

- 电商搜索"华为手机，价格 2000-3000，评分 > 4.5"：ES 在 brand、price、rating 三个字段的倒排索引中分别查找，获取三个 Posting List，然后通过 Skip List 或 Bitset 依次取交集，毫秒级返回结果。
- 如果用 MySQL：需要建立 `(brand, price, rating)` 联合索引才能高效查询，但联合索引的列顺序固定，换一种查询条件组合（如只要 price 和 rating）就可能无法利用索引。ES 的每个字段独立建索引，任意组合都高效。

**面试追问点**：

- Skip List 和 Bitset 哪种更快？→ Bitset AND 运算更快（位运算 CPU 原生支持），但前提是 bitset 已在内存中。如果需要从磁盘加载，Skip List 的顺序读可能更优。
- 为什么 MySQL 不直接联合使用两个索引？→ MySQL 的 B+ 树索引是独立的，同时走两个索引需要随机 IO 交叉读取，优化器通常选择一个索引 + 回表过滤，效率更高。PostgreSQL 的 Bitmap Index Scan 是个例外。
- ES 的 filter cache 有什么限制？→ 基于 Segment 粒度缓存，Segment 被合并后缓存失效重新构建；占用内存有上限（默认为堆的 10%）。

# **4. 如何实现全文检索**

ES 全文检索的核心流程是：**写入时通过分析器（Analyzer）将文档文本拆分为词项（Term）建立倒排索引，查询时同样对查询文本分词，再通过倒排索引查找匹配文档，最后按相关性评分排序返回**。

**写入流程**：

- 文档写入时，ES 对每个 text 类型字段执行分析器处理。
- 分析器由三部分组成：**Character Filter**（字符过滤，如去除 HTML 标签）、**Tokenizer**（分词器，将文本拆分为词项）、**Token Filter**（词项过滤，如小写化、停用词去除、同义词添加、词干提取）。
- 分词后的词项写入倒排索引的 Term Dictionary，同时记录文档 ID、词频（TF）、位置信息（Position）。
- **位置信息是短语查询和高亮的关键**，如果不需要 Phrase Query 可以关闭 positions 存储以节省空间。

**查询流程**：

- 用户提交查询后，ES 对查询文本同样执行分析器分词，得到查询词项。
- 根据查询类型（Match / Match Phrase / Term 等）在倒排索引中查找匹配的文档 ID 列表。
- **Query 阶段**：各分片本地执行查询，返回匹配的文档 ID 和评分（\_score）。
- **Fetch 阶段**：协调节点汇总各分片结果，全局排序后取 top N，再向相关分片请求文档详情，最终返回。

**常见全文检索查询类型**：

- **Match Query**：对查询文本分词后，逐个词项在倒排索引中查找，取并集（OR）或交集（AND），是全文检索最常用的查询。
- **Match Phrase Query**：不仅要求文档包含所有查询词项，还要求词项在文档中 **按顺序相邻出现**（通过 positions 信息判断），适合精确短语匹配。
- **Match Phrase Prefix Query**：短语前缀查询，最后一个词项做前缀匹配，用于搜索联想/自动补全场景。
- **Multi Match Query**：同时对多个字段执行 Match，常用于跨字段搜索（如 title + content）。
- **Term Query**：不对查询文本分词，直接在倒排索引中精确匹配词项，**通常用于 keyword 类型字段**。

**相关性评分（BM25）**：

- ES 默认使用 **BM25 算法** 计算文档与查询的相关性得分，是 TF-IDF 的改进版本。
- **TF（词频）**：词项在文档中出现的次数越多，得分越高，但 BM25 对 TF 有饱和限制，避免长文档因词频过高而占优。
- **IDF（逆文档频率）**：词项在所有文档中出现的次数越少（越稀有），权重越高。例如"的"出现很多次但区分度低，"Elasticsearch"出现少但区分度高。
- **字段长度惩罚**：BM25 会惩罚字段过长的文档，因为长文档天然更容易包含更多词项。
- 评分结果存储在 `_score` 字段，查询默认按 `_score` 降序排列。

**高亮显示**：

- ES 的高亮功能依赖倒排索引中存储的 **offset（偏移量）** 信息。
- 查询时 ES 根据匹配词项的位置信息，在原文本中包裹 `<em>` 等高亮标签。
- 支持 `highlight` 参数配置高亮方式（plain/fast-vector-highlighter）、标签、片段大小等。

**全文检索的性能优化**：

- **控制字段存储**：不需要搜索的字段设置 `index: false`，不需要短语查询的字段关闭 `positions` 存储。
- **使用合适的分词器**：中文场景推荐 IK 分词器（ik_max_word 细粒度分词、ik_smart 粗粒度分词）。
- **利用 Route 和 Shard 设计**：合理设置分片数，避免查询时遍历过多分片。
- **Rescore 和 Track Scores**：如果只需要 top N 的精确排序，可以用 rescore 对 top N 重新评分，减少全局排序开销。

**面试追问点**：

- Match Query 和 Term Query 的区别？→ Match Query 会分词再查，Term Query 不分词直接精确匹配；text 字段用 Match，keyword 字段用 Term。
- 为什么 Match Phrase 比 Match 严格？→ Match Phrase 要求词项按顺序相邻出现，Match 只要求包含即可，顺序和位置不限。
- BM25 和 TF-IDF 的区别？→ BM25 对 TF 有饱和限制（词频到一定程度后增益递减）、增加了字段长度惩罚，实际效果更合理。
- 全文检索能支持聚合吗？→ 不建议对 text 字段做聚合，因为分词后词项太碎；聚合应该用 keyword 字段或开启 fielddata（但内存开销大，推荐用 keyword + doc_values）。

# **5. 介绍下 ES 的近实时问题**

ES 的搜索是**近实时（Near Real-Time，NRT）** 的，即文档写入后**默认需要等待约 1 秒才能被搜索到**，不是写入即可见。这是由 ES 的 refresh 机制决定的。

**为什么是近实时而不是实时**

- 数据写入后首先进入**内存 Index Buffer**，此时数据不可查询
- 默认每 **1 秒执行一次 refresh** 操作，将 Index Buffer 中的数据写入新的 Lucene Segment（位于文件系统缓存中）
- 只有 refresh 生成 Segment 后，数据才变为**可查询状态，此时Segment未落盘，需要等tranlog到512M或者30分钟定时**
- 因此从写入到可搜索存在**最多 1 秒的延迟**，这就是近实时的含义

**refresh 的本质**

- refresh 不是将数据写入磁盘，而是将内存数据写入**文件系统缓存**中的 Segment
- 文件系统缓存中的 Segment 可以被搜索，但尚未持久化到磁盘
- 真正的持久化由后续的 **flush** 操作完成（默认 30 分钟或 Translog 满 512MB 时触发）
- 这种设计在**写入性能**和**数据可见性**之间取得平衡：频繁写磁盘影响性能，完全不持久化则有丢失风险

![](assets/image-20260620-162626-014.png)

**如何调整近实时程度**

- **增加 refresh 间隔**（如 30 秒或更长）：减少 Segment 数量，提升写入性能，但延迟更长。适合写入密集型场景，如日志采集
- **减少 refresh 间隔**（如 100ms）：降低搜索延迟，但会产生大量小 Segment，增加 merge 压力，影响写入性能
- **手动 refresh**：写入后调用 `POST /index/_refresh` 立即触发 refresh，数据立即可查询
- **refresh=wait_for**：写入请求等待下一次 refresh 完成后才返回，确保写入后立即可查询，但会**降低写入吞吐量**
- **refresh=false**：写入后不等待 refresh，直接返回，数据仍需等待自动 refresh 才可查询

**近实时与实时系统的区别**

- 实时系统（如 MySQL）：写入事务提交后立即可查询，延迟在毫秒级
- ES 近实时：默认 1 秒延迟，是性能和实时性的折中
- 如果业务要求强实时一致性，ES 不是最佳选择，应考虑关系型数据库或消息队列

**生产环境注意事项**

- **日志场景**：可适当增加 refresh 间隔（如 30 秒），因为日志对实时性要求不高，更关注写入吞吐量
- **搜索场景**：保持默认 1 秒或适当减少，确保用户体验
- **数据导入场景**：批量导入期间可临时禁用 refresh（设置较大的 refresh 间隔），导入完成后再恢复，提升导入速度
- **监控 refresh 频率**：refresh 过于频繁会导致 Segment 碎片化，增加 merge 开销和内存占用

**面试追问点**

- **近实时的 1 秒延迟能否消除？** → 不能完全消除，但可以通过 `refresh=wait_for` 或手动 refresh 让写入后立即可查询，代价是降低写入性能
- **refresh 和 flush 的区别是什么？** → refresh 将内存数据写入文件系统缓存（可查询但未持久化），flush 将数据持久化到磁盘并清空 Translog
- **为什么不用 0 秒 refresh？** → 频繁 refresh 会产生大量小 Segment，导致 merge 开销增大、搜索性能下降、内存占用增加
- **refresh 失败会丢数据吗？** → 不会，未 refresh 的数据仍在内存 Index Buffer 和 Translog 中，进程崩溃后可从 Translog 恢复

# 6. 介绍下ES查询原理

ES的查询分为两个阶段：**Query阶段（取ID+评分）** 和 **Fetch阶段（取文档详情）**。整体流程是：协调节点将请求广播到所有分片，各分片在本地执行查询返回Top N的文档ID和评分，协调节点全局排序后取最终的Top N，再回各分片取文档详情返回客户端。

**Query阶段详细流程**：

- 协调节点将搜索请求**广播到索引的所有分片**（默认行为），每个分片独立执行查询。
- 每个分片在本地遍历自己的Segment，根据查询条件匹配文档，**计算每条匹配文档的\_score评分**，按评分降序取本地Top N（由size参数决定），返回文档ID和评分给协调节点。
- 例如搜索条件要取5条数据，索引有5个分片，每个分片返回本地Top 5，协调节点收到25条结果后**全局归并排序**，取前5条作为最终结果。
- 这里有个常见误解：**不是每个分片返回"相同数量"的文档**，而是每个分片返回自己的本地Top N。如果某个分片数据量不足size条，就返回实际数量。协调节点收到的总数可能少于 `分片数 × size`。

**Fetch阶段详细流程**：

- Query阶段只返回文档ID和评分，**不返回文档内容**（\_source）。
- 协调节点根据全局排序后的Top N文档ID列表，**回各分片请求完整的文档数据**。
- 各分片根据文档ID从\_source中取出完整文档返回给协调节点，最终返回给客户端。

**为什么分两个阶段**：

- **减少网络传输**：Query阶段只传文档ID和评分（几十字节），不传完整文档（可能几KB），大幅减少网络开销。
- **全局排序需要**：协调节点需要看到所有分片的结果才能做全局排序，如果直接传完整文档，内存和网络开销都会很大。

**DFS查询（全局IDF）**：

- 默认的`query_then_fetch`模式下，**每个分片独立计算IDF**，当各分片的数据分布不均匀时，同样的词在不同分片的IDF可能不同，导致评分不准确。
- 使用`dfs_query_then_fetch`时，ES会先执行一个DFS阶段，**收集所有分片的词频信息计算全局IDF**，然后再执行查询。适用于数据量较小、对评分精度要求高的场景。
- 大数据量场景下各分片数据趋于均匀，默认模式的评分误差可以接受，不需要开启DFS。

**分片数量对查询性能的影响**：

- **分片越多，查询并行度越高**，但同时协调节点的归并开销也越大。
- **分片数量决定了ES能承载的最大数据量**，单个分片建议控制在10-50GB。
- **分片过少**：单分片数据量大，查询时单分片执行时间长，无法充分利用并行。
- **分片过多**：每个查询要广播到所有分片，协调节点需要归并大量结果，FST等数据结构在每个分片上都要加载，内存开销增大。
- 实际应用中需要在**并行能力和协调开销之间取平衡**。

**权重得分计算的特点**：

- **BM25评分在查询时实时计算**，每搜索一次都会根据搜索条件重新计算一次，对搜索性能影响较大。
- 评分依赖倒排索引中的词频（TF）和逆文档频率（IDF），这些信息在Segment生成时就已记录。
- **filter查询不计算评分**，性能比must（计分查询）好很多，因为跳过了评分计算且结果可缓存。

**Lucene中最重要的数据结构及其作用**：

- **FST（有限状态转换器）**：保存Term字典的前缀信息，缓存在内存中，支持单Term、Term范围、Term前缀和通配符查询。FST比Trie树更紧凑，**共享前缀和后缀**，大幅降低内存占用。
- **倒排链（Posting List）**：保存每个Term对应的docId列表，采用**跳表（Skip List）**结构支持快速跳跃。交集运算时通过跳表有效跳过无效文档。
- **BKD-Tree（动态磁盘优化BSP树）**：保存多维空间点的数据结构，用于**数值类型和空间点**的快速查找。通过BKD-Tree找到的docId集合是无序的，需要转成有序数组或BitSet后再与其他结果合并。
- **DocValues**：基于docId的**列式存储**，用于排序和聚合。列式存储只读取目标字段，不需要加载整行数据，大幅减少IO。

**不同查询类型与数据结构的对应关系**：

- **单Term查询**（如`term`查询）：从FST定位Term Dictionary位置，找到Term后读取倒排链，获取有序docId列表。
- **字符串范围/前缀/通配符查询**：从FST中获取符合条件的所有Term，再逐个查找倒排链，合并结果。
- **数字范围查询**：通过BKD-Tree找到符合条件的docId集合（无序），转成有序数组或BitSet后合并。
- **多条件交集查询**（AND）：多个倒排链求交集，用跳表跳过不匹配的doc。在内存中可用RoaringBitmap做位运算，速度更快。
- **多条件并集查询**（OR）：少量倒排链用最小堆逐个合并；大量倒排链直接合并成有序数组或构造BitSet。

**倒排链合并的三种策略**：

- **策略一：最小堆 + 跳表**：多个有序倒排链的队首构成最小堆，逐个出堆生成有序并集。适合**倒排链数量少**（N小）的场景。
- **策略二：合并成有序数组**：直接把多个倒排链合并成一个有序docId数组。适合**倒排链数量多**（N大）但文档量适中的场景。
- **策略三：构造BitSet**：当doc数量超过一定阈值时，用BitSet（每个docId只需1个bit）存储比有序数组更省内存，且交集/并集运算通过位运算非常高效。适合**文档量非常大**的场景。阈值估算：32位docId在BitSet中只需1bit，当BitSet的总大小（由segment内doc总数决定）比有序数组更划算时自动切换。

**面试追问点**：

- Query阶段返回多少条数据？→ 每个分片返回size条（本地Top N），不是全量。协调节点收到分片数×size条后全局排序取前size条。
- 为什么默认不能深度分页？→ from+size越大，协调节点需要在内存中维护的结果集越大，性能急剧下降。超过10000条应使用search_after或scroll。
- filter为什么比must快？→ filter不计算BM25评分，且结果可以被node query cache缓存，命中缓存后直接复用BitSet，不需要重新查询。
- 聚合在哪个阶段执行？→ 聚合的**收集**在各分片的Lucene层面完成（遍历匹配文档收集聚合值），**合并**在协调节点的ES层面完成（如求和、求平均）。

# **7. ES有哪些优化**

ES在查询、索引、缓存、存储等多个层面做了优化，核心思路是**减少不必要的计算和IO，尽量复用已有结果**。

**查询执行顺序优化**：

- Lucene在执行多个查询子句时，**会先估算每个子句的代价（基于匹配文档数），从代价最小的子句开始执行**。
- 例如查询"标题包含Java AND 内容包含分布式 AND 价格>100"，如果"标题包含Java"只匹配1000条，而"内容包含分布式"匹配10万条，Lucene会先执行标题条件，从1000条结果上再迭代价格条件，而不是先扫描10万条。
- 这种优化对bool查询的must和filter子句都有效，**大幅减少需要处理的文档数量**。

**结果排序优化（堆排序）**：

- Lucene**不会对所有命中的doc进行全量排序**，而是构造一个**最小堆（PriorityQueue）**，只保证前(Offset+Size)个doc是有序的。
- 排序性能取决于**(Offset+Size)**和命中文档数，以及读取DocValues的开销。
- 因为(Offset+Size)通常不会太大（除非深度分页），且DocValues是列式存储读取性能很高，**排序本身对性能的影响有限**。
- 真正影响性能的是命中文档数和获取DocValues的IO，不是排序算法本身。

**Filter Cache（过滤缓存）**：

- ES会将**高频执行的filter查询结果缓存到内存中**，缓存的粒度是**(filter, segment)到bitset的映射**。
- 下次执行相同filter时，直接从缓存中获取bitset，**不需要重新查询倒排索引**，大幅减少磁盘IO。
- **只缓存filter查询，不缓存must查询**，因为must需要计算BM25评分，结果依赖具体查询上下文，不适合缓存。
- 缓存使用**RoaringBitmap**压缩存储，比原始数组和普通bitset更省内存。
- 缓存key是filter条件的字面量表示，**相同filter条件在不同查询中可以复用**。

**Posting List压缩优化**：

- **Frame of Reference（FOR）**：对较长的Posting List使用差值编码+位压缩，将有序的docId列表压缩成紧凑的字节序列，减少磁盘占用和索引尺寸。
- **RoaringBitmap**：对内存中的Posting List使用RoaringBitmap存储，压缩率高且交集/并集运算支持位运算，性能优异。
- 压缩使得同样的内存和磁盘空间可以存储更多数据，**间接提升查询性能**（更多数据可以留在缓存中）。

**查询代价估算与短路优化**：

- Lucene在执行交集运算时，**先扫描最短的倒排链**，再用跳表在其他倒排链上跳跃。
- 当某个子句已经找不到匹配文档时，**提前终止后续计算**（短路求值），不浪费资源。
- 例如AND查询中，如果第一个子句只匹配10个doc，后续子句只需要在这10个doc上验证，而不是扫描全量。

**BKD-Tree与倒排索引的协同优化（IndexOrDocValuesQuery）**：

- 数字范围查询通过BKD-Tree找到的docId是**无序的**，与倒排索引的有序docId合并时需要先构造BitSet，这一步可能非常耗时。
- ES通过**IndexOrDocValuesQuery**优化：当匹配文档数较少时，用倒排索引（有序docId，skipList合并）；当匹配文档数较多时，用DocValues（列式存储，直接位运算）。**自动选择最优路径**。
- 对数字类型字段，如果只做Term查询不涉及范围查询，建议建成**keyword类型**（倒排索引，跳表合并快）；如果需要范围查询，**数值类型**（BKD-Tree）性能更好。

**索引层面的优化**：

- **Segment Merge策略**：后台合并进程根据Segment大小和数量自动触发合并，减少查询时需要扫描的Segment数量。可通过`max_num_segments`控制合并后的Segment数。
- **refresh_interval调整**：写入密集场景适当增大refresh间隔（如30s），减少小Segment生成，降低Merge压力。
- **预热（Warmup）**：ES启动后首次查询较慢，因为FST、Filter Cache等需要加载到内存。通过**warmup API**或定期查询可以提前预热，避免首次查询延迟。
- **Index Sorting**：ES 7.x+支持索引写入时预排序，Segment内部按指定字段排序，查询时减少排序开销。

**路由优化**：

- 默认情况下搜索请求**广播到所有分片**，分片越多查询开销越大。
- 通过**routing参数**指定文档路由键，可以将相关文档路由到同一分片，查询时只访问特定分片而非全部。
- 例如按用户ID路由，查询某个用户的数据时只访问一个分片，**查询性能提升N倍**（N为分片数）。

**批量查询优化**：

- **Multi-Get（mget）**：一次请求获取多个文档，减少网络往返次数，且同一分片的文档会自动合并为一次分片请求。
- **Scroll/Scroll Search**：适合批量导出场景，在各分片上维护快照游标，避免深度分页的性能问题。
- **search_after**：适合实时滚动加载，基于上一页末尾的排序值作为游标，避免深度遍历。

**面试追问点**：

- **filter缓存什么时候失效？** → Segment被合并后，该Segment上的缓存自动失效，下次查询重新构建。
- 查询顺序优化是ES做的还是Lucene做的？→ **Lucene层面做的**，ES将查询传递给Lucene，Lucene内部的查询优化器决定执行顺序。
- IndexOrDocValuesQuery什么时候用倒排索引，什么时候用DocValues？→ 匹配文档数少时用倒排索引（有序docId，skipList快），匹配文档数多时用DocValues（位运算快），Lucene自动根据代价估算选择。
- 为什么数字范围查询推荐用数值类型而不是字符串？→ 数值类型走BKD-Tree，range查询性能O(logN)；字符串类型走倒排索引，range查询需要遍历所有Term，性能差很多。

# **8. ES适合什么场景,**写多读少和读多写少怎么优化

**ES 适合的场景**：

ES 的核心优势是**倒排索引 + 分布式架构**，适合需要**全文搜索、复杂查询、近实时分析**的场景：

- **全文搜索**：商品搜索、文章搜索、知识库检索等，需要对文本内容进行分词匹配和相关性排序
- **日志分析**：ELK Stack（Elasticsearch + Logstash + Kibana）是日志收集分析的标准方案，ES 适合存储和查询结构化日志数据
- **指标监控**：存储时间序列指标数据，支持多维度聚合分析
- **安全分析**：入侵检测、异常行为分析等需要快速检索和关联多维度数据的场景
- **地理位置搜索**：基于地理位置的搜索和筛选，ES 内置 BKD-Tree 支持高效的空间查询

**写多读少 vs 读多写少场景的区别**：

**写多读少场景**（如日志采集、IoT 数据、聊天记录）：

- **特点**：写入吞吐量大，查询频率低，数据时效性强
- **瓶颈**：写入性能、磁盘 IO、refresh 和 merge 开销
- **优化策略**：
  - 增大 `refresh_interval`（如 30 秒或更长），减少小 Segment 生成，降低 merge 压力
  - 减少副本数（`number_of_replicas: 0`），写入时只写主分片，提升写入速度
  - 使用 **Bulk API** 批量写入，减少网络往返开销
  - 合理设置分片数，避免单分片过大影响写入性能
  - 写入密集期可临时禁用副本，写入完成后再恢复

**读多写少场景**（如电商搜索、新闻搜索、知识库）：

- **特点**：查询频率高，写入相对较少，对查询延迟和吞吐量要求高
- **瓶颈**：查询性能、内存占用、缓存命中率
- **优化策略**：
  - 增加副本数（`number_of_replicas: 1~2`），副本分片可分担查询压力，提升读取吞吐量
  - 充分利用 **Filter Cache**，高频 filter 查询结果缓存到内存，避免重复计算
  - 优化查询语句，减少不必要的字段检索和聚合计算
  - 使用 **routing 参数**，将相关文档路由到同一分片，查询时只访问特定分片而非全部
  - 合理设置 **doc_values**，排序和聚合字段使用列式存储，减少 IO
  - 对热点数据进行 **预热**，提前加载 FST、Filter Cache 等到内存

**两种场景的共同优化点**：

- **分片设计**：单分片控制在 10-50GB，分片数过多会增加协调节点归并开销，过少则无法充分利用并行能力
- **映射优化**：不需要搜索的字段设置 `index: false`，不需要短语查询的字段关闭 `positions` 存储
- **硬件配置**：ES 是内存密集型应用，建议堆内存不超过 31GB（避免指针压缩失效），剩余内存留给文件系统缓存

**面试追问点**：

- **ES 和 MySQL 各自适合什么场景？** → MySQL 适合事务性操作、强一致性场景（如订单、支付）；ES 适合搜索、分析、日志等需要全文检索和复杂查询的场景。两者常配合使用，MySQL 做主存储，ES 做搜索引擎。
- **写多读少场景为什么增大 refresh_interval 能提升性能？** → 频繁 refresh 会产生大量小 Segment，后台 merge 开销大，磁盘 IO 高。增大 refresh 间隔减少 Segment 数量，降低 merge 压力。
- **读多写少场景增加副本为什么能提升性能？** → 副本分片是主分片的完整复制，查询时可在主分片和副本分片之间负载均衡，分担查询压力，提升整体读取吞吐量。
- **ES 适合存储结构化数据吗？** → 可以，但不如 MySQL。ES 的优势是搜索和分析，如果是纯结构化数据的 CRUD 操作，MySQL 更合适。ES 适合作为 MySQL 的补充，提供搜索能力。

# **9. 查询时怎么评分的**

ES 的评分机制是**查询时实时计算**的，核心流程是：查询文本分词后，每个词项在倒排索引中查找匹配文档，每个匹配文档针对每个词项计算基础分，再根据查询类型合并多个词项的分数，最终得到 `_score`。

**评分的整体流程**：

- **第一步：查询分词**：对查询文本执行分析器分词，得到多个词项（如 "分布式搜索" 分为 "分布式" 和 "搜索"）
- **第二步：逐词项查找**：每个词项在倒排索引中查找，获取匹配的文档 ID 列表和该词项的 IDF 值
- **第三步：逐文档计算基础分**：每个匹配文档针对每个词项，根据 BM25 公式计算一个基础分
- **第四步：合并多词项分数**：根据查询类型合并同一文档在多个词项上的得分
- **第五步：合并多字段分数**：如果查询多个字段（如 multi_match），再合并不同字段的得分
- **第六步：返回最终 \_score**：排序后返回评分最高的文档

**BM25 详细计算公式**：

BM25 是 ES 默认的评分算法，对单个词项 t，文档 d 的评分为：

**score(t, d) = IDF(t) × (tf × (k1 + 1)) / (tf + k1 × (1 - b + b × dl/avgdl))**

核心因子解释：

- **IDF（逆文档频率）**：`log(1 + (N - n + 0.5) / (n + 0.5))`，N 是总文档数，n 是包含该词项的文档数。词项越稀有，IDF 越高，区分度越大
- **TF（词频）**：词项在文档中出现的次数，BM25 对 TF 有**饱和限制**，词频到一定程度后增益递减，避免长文档因重复词项而占优
- **dl/avgdl（字段长度归一化）**：dl 是当前文档字段长度，avgdl 是所有文档的平均字段长度。长文档天然更容易匹配更多词项，BM25 通过此因子**惩罚过长的字段**
- **k1（词频饱和参数）**：默认 1.2，控制 TF 饱和的速度。k1 越大，TF 饱和越慢，高词频的增益越大
- **b（长度惩罚参数）**：默认 0.75，控制字段长度惩罚的强度。b=0 不惩罚长度，b=1 完全按长度归一化

**多词项分数合并方式**：

不同查询类型对多个词项的分数合并策略不同：

- **Match Query（默认 OR）**：取所有匹配词项中**最高分**作为文档得分。如查询 "分布式 搜索"，文档同时包含两个词时，取两个词中 BM25 分更高的那个
- **Match Query（AND 模式）**：多个词项的分数**相加**。要求文档必须包含所有词项，总分是各词项 BM25 分之和
- **Match Phrase Query**：除了 BM25 分数外，还引入**位置相近度加权**，词项在文档中位置越接近，额外加分越多
- **Bool Query**：must 和 should 子句的分数**相加**，must_not 不计分，filter 不计分
- **Dis Max Query**：取所有子查询中**最高分**，而不是求和，适合需要突出最强匹配的场景

**多字段分数合并方式（multi_match）**：

当查询多个字段时，不同方式合并字段得分：

- **best_fields（默认）**：取**最高分字段**的得分，适合精确匹配场景
- **most_fields**：所有字段得分**相加**，适合多字段都需要匹配的场景
- **cross_fields**：把多个字段视为一个大字段，合并倒排列表后统一计算 BM25，适合跨字段搜索
- **phrase / bool**：按短语或布尔逻辑分别计算后合并

**function_score 自定义评分**：

ES 提供 function_score 查询支持自定义评分逻辑：

- **weight**：给文档指定一个固定权重，乘以原始 \_score
- **script_score**：用脚本（Painless）根据字段值动态计算分数，如 `doc['price'].value * 0.1`
- **衰减函数（Decay Functions）**：根据数值、日期或地理位置距离递减评分，适合按距离/时间排序
- **random_score**：返回随机分数，适合需要随机展示的场景
- 多个 function 可组合使用，通过 `boost_mode`（相加、相乘、替换等）控制最终分数

**评分性能优化**：

- **filter 不计算评分**：filter 子句跳过 BM25 计算，且结果可缓存，性能远优于 must
- **Rescore 重评分**：只对 top N 结果重新评分，减少全局计算开销。如 first_pass 取粗排 top 100，rescore 对 100 条精排取 top 10
- **Track Scores 控制**：如果不需要 \_score（如按时间排序），设置 `track_scores: false` 避免无意义的评分计算
- **DFS Query Then Fetch**：数据量小且对精度要求高时开启，先收集全局 IDF 再查询，避免各分片 IDF 不一致导致评分偏差

**面试追问点**：

- **BM25 和 TF-IDF 的区别？** → BM25 对 TF 有饱和限制（词频到一定程度后增益递减）、增加了字段长度惩罚（dl/avgdl），实际效果更合理；TF-IDF 的 TF 无上限，长文档容易占优
- **为什么 filter 比 must 快？** → filter 不计算 BM25 评分，且结果可被 node query cache 缓存，命中后直接复用 BitSet；must 需要实时计算评分，无法缓存
- **多个 should 条件的分数怎么合并？** → 默认 OR 模式下，每个 should 条件是独立的，文档满足任一即可；分数计算取决于 minimum_should_match 设置，满足越多分数越高，但不是简单相加
- **function_score 的 boost_mode 有哪些选项？** → multiply（相乘，默认）、replace（替换原始分）、sum（相加）、avg（平均）、max（取最大）、min（取最小）

# **10. ES filter和query的不同**

ES 中 filter 和 query 是两种不同的查询方式，**核心区别是是否计算相关性评分**。filter 用于精确过滤，不计分、可缓存；query 用于全文搜索，计分、不可缓存。

**核心区别对比**：

- **评分计算**：filter 不计算 `_score`（评分为0），query 计算 `_score`（基于 BM25）
- **缓存机制**：filter 结果可被 **node query cache** 缓存（BitSet 格式），query 结果不缓存
- **性能表现**：filter 性能远优于 query，因为跳过评分计算且可复用缓存
- **使用场景**：filter 用于精确匹配、范围过滤等不需要相关性排序的场景；query 用于全文搜索、需要按相关性排序的场景

**bool 查询中的组合使用**：

ES 的 bool 查询包含四个子句，各有不同用途：

- **must**：必须匹配，计分。文档必须满足所有 must 条件，分数相加
- **should**：可选匹配，计分。文档满足任一 should 条件即可，分数根据匹配情况计算
- **must_not**：必须不匹配，不计分。文档必须不满足所有 must_not 条件，常用于排除
- **filter**：必须匹配，不计分。文档必须满足所有 filter 条件，但不计算评分

**实际应用示例**：

电商搜索 "华为手机 性价比高，价格 2000-3000，排除下架商品"：

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "华为手机 性价比高" } }
      ],
      "should": [
        { "match": { "brand": "华为" } }
      ],
      "filter": [
        { "range": { "price": { "gte": 2000, "lte": 3000 } } },
        { "term": { "status": "on_sale" } }
      ],
      "must_not": [
        { "term": { "status": "deleted" } }
      ]
    }
  }
}
```

- **must**：全文搜索 "华为手机 性价比高"，计分排序
- **should**：品牌是华为加分，不是也不排除
- **filter**：表明使用的filtercontext，价格区间和在售状态，不计分，只过滤
- **must_not**：排除已下架商品，不计分

**为什么优先使用 filter**：

- **性能优势**：filter 跳过 BM25 评分计算，减少 CPU 开销
- **缓存复用**：filter 结果被缓存为 BitSet，相同条件后续查询直接命中缓存
- **确定性过滤**：精确匹配（term、range）结果确定，不需要评分干扰
- **查询优化**：Lucene 会将 filter 条件提前执行，缩小候选集后再做 query 计分

**什么时候必须用 query**：

- **全文搜索**：需要对文本分词匹配，按相关性排序（如搜索商品标题）
- **模糊匹配**：需要近似匹配、同义词扩展等（如用户输入有错别字）
- **相关性排序**：结果需要按匹配度排序，而不是固定规则排序

**混合使用的最佳实践**：

- 能用 filter 就不用 query，除非确实需要评分排序
- 将过滤条件放在 filter 中，搜索条件放在 must/should 中
- 不需要评分的排除条件放在 must_not 中
- 需要加分但不强制的条件放在 should 中

**面试追问点**：

- **filter 和 must 的性能差距有多大？** → 相同条件下，filter 比 must 快 2-10 倍，因为 filter 可缓存且跳过评分计算；首次查询差距小，后续查询 filter 命中缓存后差距更大
- **filter 缓存的粒度是什么？** → 以 (filter条件, segment) 为粒度缓存，同一个 filter 在不同查询中可复用，但 Segment 被合并后缓存失效
- **should 必须要有一个匹配吗？** → 默认不需要，但如果设置了 minimum_should_match，则必须满足指定数量的 should 条件
- **must_not 会影响评分吗？** → 不会，must_not 只做排除，不参与评分计算

# **11. filter执行原理**

ES 的 filter 执行原理是：**在倒排索引中查找匹配文档，构建 BitSet，优先从最稀疏的 BitSet 开始合并，结果缓存到 node query cache 供后续复用**。filter 不计算评分，性能远优于 query。

**filter 执行的整体流程**：

- **第一步：倒排索引查找**：对 filter 条件（如 `term`、`range`）在倒排索引中查找，获取匹配的 document list
- **第二步：构建 BitSet**：为每个 filter 条件构建一个 BitSet（位图），文档 ID 对应的 bit 置为 1。例如 `[0, 0, 0, 1, 0, 1]` 表示文档 4 和 6 匹配
- **第三步：合并 BitSet**：遍历每个过滤条件对应的 BitSet，**优先从最稀疏的 BitSet 开始**（匹配文档最少的），逐个做 AND 位运算，快速缩小候选集
- **第四步：返回结果**：合并后的 BitSet 就是同时满足所有 filter 条件的文档集合

**为什么从最稀疏的 BitSet 开始合并**：

- 最稀疏的 BitSet 匹配文档最少，用它作为起点，后续 AND 运算能**最快缩小候选集**
- 例如：条件 A 匹配 100 个文档，条件 B 匹配 10 个文档，先用条件 B 的 BitSet 做基准，再用条件 A 的 BitSet 做 AND，只需处理 10 个 bit 的比较
- 如果先用条件 A，需要处理 100 个 bit 的比较，性能差 10 倍

**node query cache 缓存机制**：

ES 通过 **node query cache**（节点级查询缓存）缓存 filter 的 BitSet 结果：

- **缓存粒度**：以 `(filter条件, segment)` 为 key，value 是对应的 BitSet
- **缓存策略**：跟踪最近 256 个 query，**超过一定次数**（默认 12 次）的 filter 条件才会被缓存
- **小 Segment 不缓存**：对于文档数 <1000 或 <3% 总文档数的小 Segment，不缓存其 BitSet，因为缓存收益低
- **缓存命中**：相同 filter 条件在不同查询中可直接复用缓存的 BitSet，**不需要重新查询倒排索引**
- **缓存失效**：Segment 被 merge 后，该 Segment 上的缓存自动失效，下次查询重新构建

**Segment 不可变性与缓存更新**：

ES 的 Segment 是**不可变的**，一旦生成就不会被修改：

- **新增文档**：写入新的 Segment，旧 Segment 不变，旧 Segment 的 filter 缓存仍然有效
- **修改文档**：ES 采用"标记删除 + 新增"策略，旧文档在旧 Segment 中被标记删除，新文档写入新 Segment
- **删除文档**：文档在旧 Segment 中被标记删除，不会立即从缓存中移除
- 因此，**cached BitSet 不会被自动更新**，而是通过 Segment 不变性保证缓存的有效性。新 Segment 生成后，相同 filter 条件会重新构建 BitSet 并缓存

**filter 在查询执行中的位置**：

- **先 filter 后 query**：ES 会优先执行 filter 条件，**先过滤掉尽可能多的数据**，缩小候选集后再执行 query 计分
- **短路优化**：如果 filter 已经将候选集缩小到很少的文档，后续 query 的计算量大幅减少
- **布尔查询中的 filter**：在 bool 查询中，filter 子句在 must/should 之前执行，确保计分只在满足 filter 条件的文档上进行

**filter 查询的底层数据结构**：

- **RoaringBitmap**：ES 使用 RoaringBitmap 存储 BitSet，比普通 BitSet 压缩率更高，位运算效率更好
- **跳表（Skip List）**：倒排索引中的 Posting List 使用跳表结构，支持快速跳跃，加速 BitSet 构建过程
- **位运算**：多个 BitSet 的合并使用 AND 位运算，CPU 原生支持，速度极快

**实际场景示例**：

电商搜索 "华为手机，价格 2000-3000，在售状态"：

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "华为手机" } }
      ],
      "filter": [
        { "term": { "brand": "华为" } },
        { "range": { "price": { "gte": 2000, "lte": 3000 } } },
        { "term": { "status": "on_sale" } }
      ]
    }
  }
}
```

执行流程：

- **Step 1**：在 brand 字段倒排索引中查找 "华为"，构建 BitSet brand_bitset
- **Step 2**：在 price 字段 BKD-Tree 中查找 2000-3000 范围，构建 BitSet price_bitset
- **Step 3**：在 status 字段倒排索引中查找 "on_sale"，构建 BitSet status_bitset（最稀疏）
- **Step 4**：从 status_bitset 开始，依次做 AND 运算：`status_bitset AND brand_bitset AND price_bitset`
- **Step 5**：合并后的 BitSet 作为候选集，交给 must 子句执行全文搜索和评分

**面试追问点**：

- **filter 缓存的 BitSet 有多大？** → 取决于 Segment 内文档总数，每个 bit 对应一个文档。例如 100 万文档的 Segment，BitSet 约 125KB（未压缩），RoaringBitmap 压缩后更小
- **filter 条件变化后缓存会失效吗？** → 只有 filter 条件字面量完全相同才命中缓存，条件变化（如价格从 2000-3000 变成 3000-5000）是不同的缓存 key
- **为什么小 Segment 不缓存？** → 小 Segment 文档数少，filter 执行本身就很快，缓存收益低；且小 Segment 容易被 merge，缓存命中率低
- **filter 能用 doc_values 优化吗？** → 可以，对于不需要倒排索引的简单 filter（如 `exists`），ES 会使用 doc_values 做列式扫描，性能更好

# **12. ES有哪些查询类型，有什么区别，怎么使用**

ES 的查询分为四大类：**复合查询**（组合多个子查询）、**全文查询**（对文本字段分词匹配）、**Term 级别查询**（精确匹配不分词）、**Join 查询**（嵌套和父子关系查询）。

**查询类型总览**：

- **复合查询**：bool、constant_score、boosting、function_score、dis_max
- **全文查询**：match、match_phrase、match_bool_prefix、multi_match、intervals
- **Term 级别查询**：term、terms、range、exists、fuzzy、prefix、wildcard、regexp
- **Join 查询**：nested、has_child、has_parent

**复合查询**：

复合查询用于组合多个子查询，最常用的是 **bool 查询**：

- **must**：必须匹配，计分。文档必须满足所有 must 条件，分数相加
- **should**：可选匹配，计分。文档满足任一 should 条件即可，分数根据匹配情况计算
- **must_not**：必须不匹配，不计分。文档必须不满足所有 must_not 条件
- **filter**：必须匹配，不计分，可缓存。文档必须满足所有 filter 条件

**查询标题含华为 / 手机、价格 2000\~3000、未删除的商品；品牌是华为的商品排序更靠前。**
**默认行为：会返回整条文档**

可以通过

\_source.includes 指定包含字段（最常用）

\_source.excludes 排除不需要的字段

```json
{
  "query": {
    "bool": {
      "must": [{ "match": { "title": "华为手机" } }],
      "should": [{ "term": { "brand": "华为" } }],
      "filter": [{ "range": { "price": { "gte": 2000, "lte": 3000 } } }],
      "must_not": [{ "term": { "status": "deleted" } }]
    }
  }
}


1、must 强制要求：文档 title 字段必须包含「华为」或「手机」（match 分词查询），这条会参与打分，标题匹配度越高，搜索排名越靠前。
2、should 加分项：文档 brand 精确等于「华为」更好；，不满足也能搜到，但相关性分数更低，排序靠后；只加分，不做硬性过滤。
3、filter 硬性过滤：价格必须 ≥2000 且 ≤3000；不满足价格区间的文档直接丢弃。无打分，ES 会对 filter 条件缓存，提升重复查询性能；
4、must_not 排除项：文档 status 不能是 deleted（已删除商品）；不打分，只要命中该条件直接过滤掉。
```

其他复合查询：

- **constant_score**：包装 filter 查询，所有匹配文档返回固定分数（boost 值），适合不需要相关性排序的过滤
- **boosting**：返回匹配 positive 的文档，但降低同时匹配 negative 的文档分数，适合降权而非排除
- **function_score**：自定义评分逻辑，支持 weight、script_score、衰减函数等，适合按业务规则调整排序
- **dis_max**：取所有子查询中最高分，而不是求和，适合需要突出最强匹配的场景

**全文查询**：

全文查询会对查询文本**分词**，在倒排索引中匹配：

- **match**：最常用的全文查询，对查询文本分词后逐个词项匹配，支持 OR（默认）和 AND 模式

  ```json
  { "query": { "match": { "title": { "query": "华为手机", "operator": "and" } } } }
  ```
- **match_phrase**：短语匹配，要求词项在文档中**按顺序相邻出现**，适合精确短语搜索

  ```json
  { "query": { "match_phrase": { "title": "华为手机" } } }
  ```
- **match_bool_prefix**：将查询文本分词后，前面的词项做 term 匹配，最后一个词项做 prefix 匹配，适合搜索联想

  ```json
  { "query": { "match_bool_prefix": { "title": "华为手" } } }
  ```
- **multi_match**：同时对多个字段执行 match，支持跨字段搜索，通过 type 参数控制评分合并方式

  ```json
  { "query": { "multi_match": { "query": "华为手机", "fields": ["title^3", "content"] } } }
  ```
- **intervals**：间隔查询，对匹配项的顺序和邻近性进行细粒度控制，适合复杂的位置匹配场景

**Term 级别查询**：

Term 级别查询**不对查询文本分词**，直接精确匹配：

- **term**：精确匹配单个词项，**避免用于 text 字段**（text 字段会被分词，term 无法匹配）

  ```json
  { "query": { "term": { "brand.keyword": "华为" } } }
  ```
- **terms**：精确匹配多个词项，满足任一即可

  ```json
  { "query": { "terms": { "brand.keyword": ["华为", "小米", "苹果"] } } }
  ```
- **range**：范围查询，支持数值、日期、字符串，用 gte、gt、lte、lt 控制边界

  ```json
  { "query": { "range": { "price": { "gte": 2000, "lte": 3000 } } } }
  ```
- **exists**：判断字段是否存在（非 null、非空数组），常配合 bool must_not 使用

  ```json
  { "query": { "exists": { "field": "title" } } }
  ```
- **fuzzy**：模糊查询，基于 Levenshtein 编辑距离匹配相似词，适合容错搜索

  ```json
  { "query": { "fuzzy": { "title": { "value": "华为手", "fuzziness": "AUTO" } } } }
  ```
- **prefix**：前缀查询，匹配以指定字符串开头的词项，适合自动补全

  ```json
  { "query": { "prefix": { "title.keyword": "华为" } } }
  ```
- **wildcard**：通配符查询，支持 `*`（任意字符）和 `?`（单个字符），性能较差

  ```json
  { "query": { "wildcard": { "title.keyword": "*手机*" } } }
  ```
- **regexp**：正则表达式查询，功能最强但性能最差，慎用

  ```json
  { "query": { "regexp": { "title.keyword": "华.*手.*" } } }
  ```

**Join 查询**：

用于处理嵌套对象和父子关系：

- **nested**：查询嵌套对象，因为 ES 会将对象数组扁平化，nested 查询能保持对象内部字段的关联性

  ```json
  {
    "query": {
      "nested": {
        "path": "comments",
        "query": {
          "bool": {
            "must": [
              { "match": { "comments.name": "张三" } },
              { "range": { "comments.stars": { "gte": 4 } } }
            ]
          }
        }
      }
    }
  }
  ```
- **has_child** / **has_parent**：查询包含特定子文档或父文档的文档，用于父子关系场景

**查询类型选择指南**：

- **全文搜索**：用 match（分词匹配）、match_phrase（短语匹配）、multi_match（多字段）
- **精确过滤**：用 term/terms（精确匹配）、range（范围）、exists（存在判断）
- **容错搜索**：用 fuzzy（编辑距离）、prefix（前缀）、wildcard（通配符）
- **组合条件**：用 bool 查询的 must/should/filter/must_not 组合
- **自定义排序**：用 function_score 调整评分逻辑
- **嵌套对象**：用 nested 查询保持对象关联性

**text 字段 vs keyword 字段的查询选择**：

- **text 字段**：分词存储，用 match/match_phrase 查询，支持全文搜索
- **keyword 字段**：不分词存储，用 term/terms/prefix 查询，适合精确匹配和聚合
- **最佳实践**：同时存储 text 和 keyword 两种字段（如 `title` 和 `title.keyword`），分别用于搜索和精确匹配

**面试追问点**：

- **term 和 match 的区别？** → term 不分词直接精确匹配，match 会分词后匹配；term 适合 keyword 字段，match 适合 text 字段
- **match_phrase 和 match 的区别？** → match_phrase 要求词项按顺序相邻出现，match 只要求包含即可；match_phrase 适合精确短语搜索
- **wildcard 和 regexp 的性能差异？** → wildcard 只支持 `*` 和 `?`，性能较好；regexp 支持完整正则，性能差很多，大数据量慎用
- **nested 查询为什么性能差？** → nested 查询需要对嵌套对象做独立查询再合并，涉及额外的存储和查询开销，应尽量减少嵌套层级

# **13. match和term的区别**

match 和 term 是 ES 最常用的两种查询，**核心区别是是否对查询文本分词**。match 会分词后匹配，适合全文搜索；term 不分词直接精确匹配，适合结构化数据过滤。

**核心区别对比**：

- **分词处理**：match 对查询文本**执行分析器分词**，term **不分词**直接查询
- **匹配方式**：match 逐个词项在倒排索引中匹配，term 精确匹配整个查询值
- **适用字段**：match 适合 **text 字段**（已分词存储），term 适合 **keyword 字段**（不分词存储）
- **评分计算**：两者都计算 `_score`（基于 BM25），但 match 的评分受分词结果影响

**查询流程对比**：

**match 查询流程**：

- 对查询文本执行分析器分词，得到多个词项。如 "华为手机" 分词为 ["华为", "手机"]
- 每个词项在倒排索引中查找，获取匹配的文档 ID 列表
- 默认 OR 模式：任一词项匹配即可，取最高分；AND 模式：所有词项必须匹配
- 合并多个词项的评分，返回结果

**term 查询流程**：

- 不分词，直接将查询值作为完整词项在倒排索引中查找
- 精确匹配，必须字段值与查询值完全一致（包括大小写、空格）
- 返回匹配的文档和评分

**实际应用示例**：

假设索引中有两条文档：

```json
{ "title": "华为手机", "brand": "华为" }
{ "title": "小米手机", "brand": "小米" }
```

**示例1：match 查询 title 字段（text 类型）**

```json
{ "query": { "match": { "title": "华为" } } }
```

- "华为" 分词后为 ["华为"]，在 title 倒排索引中查找
- 匹配文档1（"华为手机" 包含 "华为"），返回结果

**示例2：term 查询 title 字段（text 类型）**

```json
{ "query": { "term": { "title": "华为" } } }
```

- 不分词，直接查找 "华为" 这个完整词项
- **无法匹配**，因为 title 字段存储的是分词后的 ["华为", "手机"]，不是完整的 "华为"
- **结论：term 不能用于 text 字段**

**示例3：term 查询 brand 字段（keyword 类型）**

```json
{ "query": { "term": { "brand": "华为" } } }
```

- 不分词，直接查找 "华为" 这个完整词项
- 匹配文档1（brand 值完全等于 "华为"），返回结果

**示例4：match 查询 brand 字段（keyword 类型）**

```json
{ "query": { "match": { "brand": "华为" } } }
```

- "华为" 分词后为 ["华为"]，在 brand 倒排索引中查找
- 匹配文档1，但 keyword 字段不分词存储，实际效果与 term 相同

**性能差异**：

- **term 性能更好**：跳过分词处理，直接精确匹配，CPU 开销小
- **match 性能稍差**：需要执行分析器分词，再逐个词项查找，但分词结果可缓存
- **term 适合过滤**：精确匹配场景用 term + filter，可缓存且不计分，性能最优
- **match 适合搜索**：全文搜索场景必须用 match，因为需要分词匹配和相关性排序

**常见错误和注意事项**：

- **错误1：对 text 字段用 term 查询** → 无法匹配，因为 text 字段存储的是分词后的词项，不是完整原文
- **错误2：对 keyword 字段用 match 查询** → 能匹配但没必要，match 的分词对 keyword 无效，直接用 term 更高效
- **正确做法**：text 字段用 match，keyword 字段用 term；同时存储两种字段（如 `title` + `title.keyword`）分别用于搜索和精确匹配
- **大小写敏感**：term 查询大小写敏感，"Huawei" 和 "huawei" 是不同的词项；match 会经过分析器处理，通常会小写化

**字段类型选择建议**：

- **text 字段**：需要全文搜索的字段（如商品标题、文章内容），用 match 查询
- **keyword 字段**：需要精确匹配和聚合的字段（如品牌、状态、ID），用 term 查询
- **数值/日期字段**：用 range 查询，不需要分词
- **最佳实践**：同时存储 text 和 keyword，如 `{"name": "华为手机", "name.keyword": "华为手机"}`

**面试追问点**：

- **为什么 term 不能用于 text 字段？** → text 字段在写入时会分词存储，如 "华为手机" 存储为 ["华为", "手机"]，term 查询 "华为" 无法匹配这个分词结果，因为倒排索引中没有 "华为手机" 这个完整词项
- **match 查询 "华为手机" 会匹配哪些文档？** → OR 模式下，包含 "华为" 或 "手机" 的文档都会匹配；AND 模式下，必须同时包含 "华为" 和 "手机"
- **term 查询需要忽略大小写怎么办？** → 在 mapping 中为 keyword 字段设置 normalizer，或在写入时统一处理为小写
- **term 和 match 都计算评分吗？** → 都计算 \_score，但通常用 term + filter 组合，跳过评分计算且可缓存

# **14. match_phrase和match的区别**

match_phrase 和 match 都是全文查询，**核心区别是 match_phrase 要求词项按顺序相邻出现，match 只要求包含即可**。match_phrase 适合精确短语搜索，match 适合宽泛的全文搜索。

**核心区别对比**：

- **匹配模式**：match 任一词项匹配即可（OR 模式），match_phrase 所有词项必须**按顺序相邻出现**
- **位置要求**：match 不关心词项位置，match_phrase 要求词项在文档中**连续且顺序一致**
- **评分方式**：match 取最高分或求和，match_phrase 额外考虑**词项位置相近度**
- **使用场景**：match 适合宽泛搜索，match_phrase 适合精确短语匹配

**查询流程对比**：

**match 查询流程**：

- 对查询文本分词，得到多个词项
- 逐个词项在倒排索引中查找，任一匹配即可
- OR 模式：取最高分词项的得分；AND 模式：所有词项必须匹配，分数相加
- 不关心词项在文档中的位置关系

**match_phrase 查询流程**：

- 对查询文本分词，得到多个词项
- 所有词项必须在文档中**按顺序相邻出现**
- 利用倒排索引中存储的 **positions 信息**判断词项位置
- 额外计算**位置相近度加分**，词项位置越接近，得分越高

**实际应用示例**：

假设索引中有两条文档：

```json
{ "title": "华为手机性价比高" }
{ "title": "手机华为也不错" }
```

**示例1：match 查询 "华为手机"**

```json
{ "query": { "match": { "title": "华为手机" } } }
```

- 分词为 ["华为", "手机"]
- 文档1：包含 "华为" 和 "手机"，匹配
- 文档2：包含 "华为" 和 "手机"，匹配
- **两条文档都返回**，因为 match 只要求包含，不关心顺序

**示例2：match_phrase 查询 "华为手机"**

```json
{ "query": { "match_phrase": { "title": "华为手机" } } }
```

- 分词为 ["华为", "手机"]
- 文档1："华为" 在位置0，"手机" 在位置1，按顺序相邻，匹配
- 文档2："手机" 在位置0，"华为" 在位置2，不按顺序，不匹配
- **只有文档1返回**，因为 match_phrase 要求按顺序相邻

**slop 参数：允许词项间隔**：

match_phrase 支持 slop 参数，允许词项之间有一定的间隔：

- **slop=0（默认）**：词项必须紧邻，中间不能有其他词
- **slop=1**：词项之间允许间隔1个词
- **slop=N**：词项之间允许间隔N个词

```json
{
  "query": {
    "match_phrase": {
      "title": {
        "query": "华为手机",
        "slop": 1
      }
    }
  }
}
```

- slop=1 时，"华为不错手机" 也能匹配，因为 "华为" 和 "手机" 之间只隔了1个词

**性能差异**：

- **match_phrase 性能稍差**：需要额外读取 positions 信息判断词项位置，计算量更大
- **match 性能更好**：只关心词项是否匹配，不需要位置计算
- **slop 越大性能越差**：slop 越大，需要检查的位置组合越多，性能下降
- **优化建议**：如果不需要精确短语匹配，优先用 match；如果需要，尽量减小 slop 值

**使用场景选择**：

- **用 match 的场景**：
  - 宽泛的全文搜索，如搜索商品标题包含 "华为" 或 "手机"
  - 不关心词项顺序，只要包含关键词即可
  - 需要最大召回率的搜索
- **用 match_phrase 的场景**：
  - 精确短语搜索，如搜索 "华为手机" 必须连续出现
  - 搜索日志中的特定错误信息，如 "OutOfMemoryError"
  - 需要高精度的搜索场景
- **用 match_phrase + slop 的场景**：
  - 允许词项有一定间隔的搜索，如 "华为 手机"（中间有空格）
  - 搜索文章中 "华为" 和 "手机" 相邻出现的段落

**常见错误和注意事项**：

- **错误1：match_phrase 返回结果太少** → 因为要求严格相邻，可以尝试增加 slop 值
- **错误2：match_phrase 性能差** → positions 信息会占用存储空间，如果不需要短语查询，可以在 mapping 中关闭 positions 存储
- **注意：positions 存储** → match_phrase 依赖倒排索引中的 positions 信息，如果关闭了 positions 存储，match_phrase 无法使用

**面试追问点**：

- **match_phrase 为什么能判断词项位置？** → 因为倒排索引中存储了 positions 信息，记录了每个词项在文档中的位置，match_phrase 利用这些信息判断词项是否按顺序相邻
- **slop 和 operator 有什么区别？** → slop 控制词项之间的间隔距离，operator 控制词项之间的逻辑关系（OR/AND）；slop 是 match_phrase 特有的参数
- **match_phrase 评分怎么计算？** → 基于 BM25 评分 + 位置相近度加分，词项位置越接近，额外加分越多
- **什么时候用 match_phrase 而不是 match？** → 当搜索结果要求精确短语匹配时，如搜索文章标题、日志错误信息等场景

# **15. 最外层只能是query吗，可以是filter吗**

ES 查询最外层**只能是 query**，不能直接是 filter。但可以通过 **constant_score 查询** 或 **bool 查询的 filter 子句**实现纯过滤效果，跳过评分计算且可缓存。

**ES 查询的顶层结构**：

ES 的 Search API 中，最外层**必须是 query**，这是 ES 的语法规范：

```json
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "active" } }
      ]
    }
  }
}
```

**为什么不能直接用 filter 作为最外层**：

- ES 的查询 DSL 设计中，**query 是顶层入口**，filter 只是 query 内部的子句
- query context 和 filter context 是两种不同的执行上下文，需要通过查询类型来区分
- 直接写 `"filter": {...}` 会报语法错误

**实现纯过滤的两种方式**：

**方式一：constant_score 查询**：

```json
{
  "query": {
    "constant_score": {
      "filter": {
        "term": { "status": "active" }
      },
      "boost": 1.0
    }
  }
}
把内部过滤条件匹配到的所有文档统一设置固定打分值，不会根据文本相关性计算动态 _score，不会有高低区分，排序默认按文档内部顺序。
会返回匹配文档的完整原始数据（_source 完整字段）
```

- constant_score 包装一个 filter 查询，所有匹配文档返回固定分数（boost 值）
- **不计算 BM25 评分**，性能优于 bool + must
- 适合只需要过滤，不需要相关性排序的场景

**方式二：bool 查询的 filter 子句**：

```json
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "active" } },
        { "range": { "price": { "gte": 2000 } } }
      ]
    }
  }
}
```

- bool 查询中只有 filter 子句，没有 must/should
- 所有匹配文档返回 `_score: 0`，不计算评分
- filter 结果可被 node query cache 缓存，性能最优

**query context vs filter context**：

- **query context**：计算 `_score`，结果按评分排序，不可缓存。包含 must、should 子句
- **filter context**：不计算 `_score`，结果固定，可缓存。包含 filter、must_not 子句

**实际应用示例**：

**场景1：只需要过滤，不需要排序**

```json
// 方式1：constant_score
{
  "query": {
    "constant_score": {
      "filter": { "term": { "status": "active" } }
    }
  }
}

// 方式2：bool + filter
{
  "query": {
    "bool": {
      "filter": { "term": { "status": "active" } }
    }
  }
}
```

**场景2：过滤 + 全文搜索**

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "华为手机" } }
      ],
      "filter": [
        { "term": { "status": "active" } },
        { "range": { "price": { "gte": 2000, "lte": 3000 } } }
      ]
    }
  }
}
```

- must 子句：全文搜索，计算评分
- filter 子句：精确过滤，不计分，可缓存

**性能对比**：

- **query（must/should）**：计算 BM25 评分，不可缓存，性能较差
- **filter**：不计算评分，可缓存，性能最优
- **constant_score**：性能介于两者之间，比 query 好，比纯 filter 稍差（因为需要计算 boost）

**使用建议**：

- **纯过滤场景**：用 constant_score 或 bool + filter，不要用 must
- **过滤 + 搜索**：用 bool 查询，搜索条件放 must/should，过滤条件放 filter
- **需要排序**：必须用 query（must/should），因为需要计算 \_score
- **不需要排序**：优先用 filter，性能更好

**面试追问点**：

- **为什么最外层必须是 query？** → ES 的查询 DSL 设计规范，query 是顶层入口，filter 只是内部子句；直接写 filter 会报语法错误
- **constant_score 和 bool + filter 的区别？** → 功能相同，constant_score 语法更简洁，适合单个 filter；bool 更灵活，支持多个 filter 条件
- **filter 子句返回的 \_score 是多少？** → 返回 0，因为 filter 不计算评分
- **filter 缓存是全局的吗？** → 是节点级别的缓存，所有索引的 filter 结果都缓存在 node query cache 中

# **16. 如何用ES实现联想词功能**

ES 实现联想词有两种方式：**Suggest 词建议器**（completion、phrase、term）和**实时查询**（prefix、wildcard）。推荐方案是 completion suggester 做前缀匹配，零匹配时降级到 phrase/term suggester。

**ES Suggest 词建议器**：

ES 提供四种词建议器，精准度和性能各不同：

**1. Completion Suggester（推荐）**：

- **原理**：基于 FST（有限状态转换器）做前缀匹配，性能最好，适合实时自动补全
- **字段类型**：mapping 中必须定义为 `completion` 类型
- **使用场景**：用户输入时实时联想，每输入一个字符就触发查询

```json
// mapping 定义
PUT products
{
  "mappings": {
    "properties": {
      "title_suggest": {
        "type": "completion"
      }
    }
  }
}

// 写入数据
PUT products/_doc/1
{
  "title_suggest": {
    "input": ["华为手机", "Huawei Phone"],
    "weight": 100
  }
}

// 查询
GET products/_search
{
  "suggest": {
    "title-suggest": {
      "prefix": "华",
      "completion": {
        "field": "title_suggest",
        "size": 10
      }
    }
  }
}
```

- **input**：可匹配的输入文本，支持多个
- **weight**：权重，影响建议结果排序
- **fuzzy**：支持模糊匹配，容错输入

**2. Phrase Suggester**：

- **原理**：在 term 基础上考虑词项间的相邻关系和词频，做短语级补全
- **使用场景**：用户输入较完整时，推荐完整的短语建议

```json
GET _search
{
  "suggest": {
    "title-suggest": {
      "phrase": {
        "field": "title",
        "query": "华为手",
        "size": 3,
        "direct_generator": [
          {
            "field": "title",
            "suggest_mode": "popular"
          }
        ]
      }
    }
  }
}
```

- **direct_generator**：指定从哪个字段生成建议
- **suggest_mode**：`missing`（只补全不存在的）、`popular`（推荐热门的）、`popular`（返回所有）

**3. Term Suggester**：

- **原理**：基于编辑距离算法，对单个词项做模糊建议
- **使用场景**：用户输入有拼写错误时，推荐正确的词项

```json
GET _search
{
  "suggest": {
    "title-suggest": {
      "term": {
        "field": "title",
        "query": "华为手",
        "suggest_mode": "popular"
      }
    }
  }
}
```

**4. Context Suggester**：

- **原理**：基于 completion suggester 扩展，支持按上下文过滤（如分类、地理位置）
- **使用场景**：需要按分类筛选建议结果

```json
// mapping 定义
PUT products
{
  "mappings": {
    "properties": {
      "title_suggest": {
        "type": "completion",
        "contexts": [
          {
            "name": "category",
            "type": "category"
          }
        ]
      }
    }
  }
}

// 查询时指定上下文
GET products/_search
{
  "suggest": {
    "title-suggest": {
      "prefix": "华",
      "completion": {
        "field": "title_suggest",
        "contexts": {
          "category": "手机"
        }
      }
    }
  }
}
```

**实时查询方式**：

除了 Suggest，也可以用普通查询实现联想词：

**方式1：Prefix 前缀查询**：

```json
{
  "query": {
    "prefix": {
      "title.keyword": "华为"
    }
  }
}
```

- 直接用 prefix 查询，匹配以 "华为" 开头的文档
- 性能取决于索引数据量，数据量大时性能差

**方式2：Match Phrase + Slop**：

```json
{
  "query": {
    "match_phrase": {
      "title": {
        "query": "华为",
        "slop": 0
      }
    }
  }
}
```

- 用 match_phrase 做短语前缀匹配
- 需要分词，性能一般

**实现方案对比**：

- **精准度**：Completion > Phrase > Term
- **召回率**：Term > Phrase > Completion
- **性能**：Completion >> Phrase > Term > 实时查询
- **推荐方案**：Completion Suggester 做前缀匹配，零匹配时降级到 Phrase/term suggester

**实际应用场景**：

用户输入 "华为手" 时的联想过程：

1. **第一步**：用 Completion Suggester 前缀匹配，返回 "华为手机"、"华为手表" 等建议
2. **第二步**：如果零匹配，用 Phrase Suggester 做短语补全，返回 "华为手机壳" 等
3. **第三步**：如果仍零匹配，用 Term Suggester 做拼写纠正，返回 "华为 手机" 等
4. **第四步**：如果需要更宽泛的匹配，用 Prefix 查询

**性能优化**：

- **Completion Suggester**：基于 FST 内存结构，性能最好，建议热数据全量加载到内存
- **Phrase/Term Suggester**：基于倒排索引搜索，数据量大时性能差，建议控制索引数据量
- **缓存策略**：高频查询的建议结果可以缓存到 Redis，减少 ES 查询压力
- **分词器选择**：中文场景用 IK 分词器，细粒度分词提高联想准确度

**面试追问点**：

- **Completion Suggester 为什么快？** → 基于 FST（有限状态转换器）做前缀匹配，FST 常驻内存，查询时不需要读磁盘，性能接近 O(1)
- **Phrase 和 Term Suggester 的区别？** → Term 基于编辑距离做单个词项纠正，Phrase 考虑词项间的相邻关系和词频，Phrase 精准度更高但性能更差
- **如何处理拼写错误？** → 用 Term Suggester 的 fuzzy 参数，设置 fuzziness 允许的编辑距离（通常为1-2）
- **联想词需要实时吗？** → 用户输入时需要实时响应，建议用 Completion Suggester；如果允许延迟，可以用普通查询

# **17. multi-fields怎么使用**

multi-fields 是 ES 的**多字段映射机制**，允许同一个字段同时以多种类型存储，实现**一个字段同时支持全文搜索和精确匹配/聚合**。最常用的组合是 text + keyword。

**为什么需要 multi-fields**：

ES 中 text 和 keyword 是两种互斥的字段类型：

- **text 字段**：分词存储，支持全文搜索（match 查询），但不支持精确匹配、排序、聚合
- **keyword 字段**：不分词存储，支持精确匹配、排序、聚合，但不支持全文搜索

实际业务中，同一个字段往往需要同时支持搜索和聚合。例如商品标题 "华为手机"：

- 搜索时需要分词匹配："华为" 或 "手机" 都能搜到
- 聚合时需要完整值："华为手机" 作为一个整体统计

使用 multi-fields 可以**一次写入，同时支持两种查询**。

**如何定义 multi-fields**：

```json
PUT products
{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "ik_max_word",
        "fields": {
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      }
    }
  }
}
```

- **title**：主字段，text 类型，支持全文搜索
- **title.keyword**：子字段，keyword 类型，支持精确匹配和聚合
- **ignore_above**：超过 256 字符的值不索引，避免异常

**写入和查询示例**：

```json
// 写入文档
PUT products/_doc/1
{
  "title": "华为手机"
}

// 全文搜索（用主字段 title）
GET products/_search
{
  "query": {
    "match": { "title": "华为" }
  }
}

// 精确匹配（用子字段 title.keyword）
GET products/_search
{
  "query": {
    "term": { "title.keyword": "华为手机" }
  }
}

// 聚合统计（用子字段 title.keyword）
GET products/_search
{
  "size": 0,
  "aggs": {
    "top_titles": {
      "terms": {
        "field": "title.keyword",
        "size": 10
      }
    }
  }
}
```

**常见使用场景**：

- **商品搜索**：title 字段需要全文搜索，同时需要按品牌聚合统计
- **日志分析**：message 字段需要全文搜索，同时需要按日志级别聚合
- **用户管理**：name 字段需要模糊搜索，同时需要精确匹配和排序

**multi-fields 的其他组合**：

除了 text + keyword，还可以有其他组合：

```json
{
  "mappings": {
    "properties": {
      "price": {
        "type": "float",
        "fields": {
          "int": {
            "type": "integer"
          }
        }
      }
    }
  }
}
```

- **price**：float 类型，支持范围查询
- **price.int**：integer 类型，支持整数聚合

**注意事项**：

- **存储开销**：每个子字段都会额外存储一份数据，增加索引体积
- **写入开销**：写入时需要同时处理多个字段类型，增加写入延迟
- **字段命名**：子字段用 `父字段.子字段` 格式命名，如 `title.keyword`
- **mapping 更新**：子字段一旦定义不能修改，需要重建索引才能调整

**性能优化**：

- **只定义必要的子字段**：不要为所有字段都添加 keyword 子字段，按需定义
- **ignore_above 参数**：对长文本设置 ignore_above，避免异常值占用资源
- **关闭不需要的功能**：如果不需要搜索，可以关闭 text 字段的 `index: false`

**面试追问点**：

- **multi-fields 和独立字段的区别？** → multi-fields 是同一个字段的多种存储形式，写入一次；独立字段需要写入多次，占用更多存储和写入开销
- **为什么 text 字段不能聚合？** → text 字段分词后词项太碎（如 "华为手机" 分为 ["华为", "手机"]），聚合结果无意义；keyword 字段存储完整值，适合聚合
- **multi-fields 会影响写入性能吗？** → 会，每个子字段都需要额外处理，但比定义独立字段的开销小，因为数据源相同
- **如何选择 text 和 keyword 的分析器？** → text 字段用分词器（如 IK），keyword 字段不需要分析器，直接存储原文

# **18. 聚合查询怎么用**

ES 的聚合查询用于**统计分析和数据洞察**，类似 SQL 的 GROUP BY + 聚合函数。聚合分为三类：**指标聚合**（计算统计值）、**桶聚合**（分组）、**管道聚合**（对聚合结果再聚合）。

**聚合的基本语法**：

```json
{
  "size": 0,
  "aggs": {
    "聚合名称": {
      "聚合类型": {
        "字段": "字段名"
      }
    }
  }
}
```

- `size: 0`：不返回文档，只返回聚合结果，提高性能
- `aggs`：聚合查询的关键字
- **聚合名称**：自定义名称，用于引用聚合结果
- **聚合类型**：指标聚合、桶聚合、管道聚合

**指标聚合（Metrics Aggregations）**：

对数值字段计算统计值，类似 SQL 的 SUM、AVG、MAX 等：

- **sum**：求和
- **avg**：平均值
- **min / max**：最小值 / 最大值
- **count**：文档数量
- **cardinality**：基数统计（去重后的数量，类似 DISTINCT COUNT）
- **stats**：一次性返回 count、min、max、avg、sum
- **extended_stats**：扩展统计，增加方差、标准差等

```json
GET products/_search
{
  "size": 0,
  "aggs": {
    "price_stats": {
      "stats": {
        "field": "price"
      }
    },
    "brand_count": {
      "cardinality": {
        "field": "brand.keyword"
      }
    }
  }
}
```

**桶聚合（Bucket Aggregations）**：

将文档分组，类似 SQL 的 GROUP BY，每个桶是一个分组：

- **terms**：按字段值分组，最常用的桶聚合
- **range**：按数值范围分组
- **date_range**：按日期范围分组
- **histogram**：按固定间隔分组
- **date_histogram**：按固定时间间隔分组（天、月、年等）
- **filters**：按多个过滤条件分组
- **significant_terms**：显著性词项，发现异常分组

```json
GET products/_search
{
  "size": 0,
  "aggs": {
    "by_brand": {
      "terms": {
        "field": "brand.keyword",
        "size": 10
      }
    },
    "price_ranges": {
      "range": {
        "field": "price",
        "ranges": [
          { "to": 1000 },
          { "from": 1000, "to": 3000 },
          { "from": 3000 }
        ]
      }
    }
  }
}
```

**管道聚合（Pipeline Aggregations）**：

对聚合结果再聚合，类似 SQL 的子查询：

- **sum_bucket**：对桶的值求和
- **avg_bucket**：对桶的值求平均
- **max_bucket / min_bucket**：取最大 / 最小值的桶
- **derivative**：计算导数（变化率）
- **cumulative_sum**：累计求和
- **moving_avg**：移动平均

```json
GET products/_search
{
  "size": 0,
  "aggs": {
    "by_month": {
      "date_histogram": {
        "field": "create_time",
        "interval": "month"
      },
      "aggs": {
        "monthly_sales": {
          "sum": { "field": "price" }
        }
      }
    },
    "total_sales": {
      "sum_bucket": {
        "buckets_path": "by_month>monthly_sales"
      }
    }
  }
}
```

**聚合嵌套**：

桶聚合内部可以嵌套指标聚合或子桶聚合，实现多维度分析：

```json
GET products/_search
{
  "size": 0,
  "aggs": {
    "by_brand": {
      "terms": { "field": "brand.keyword" },
      "aggs": {
        "avg_price": {
          "avg": { "field": "price" }
        },
        "by_category": {
          "terms": { "field": "category.keyword" },
          "aggs": {
            "max_price": {
              "max": { "field": "price" }
            }
          }
        }
      }
    }
  }
}
```

- 第一层：按品牌分组
- 第二层：每个品牌内计算平均价格，同时按类别再分组
- 第三层：每个类别内计算最高价格

**聚合与查询条件结合**：

聚合默认在**所有文档**上执行，可以通过以下方式限定范围：

- **query + aggs**：只对满足 query 条件的文档聚合
- **filter + aggs**：在聚合内部用 filter 缩小范围
- **post_filter**：只影响文档展示，不影响聚合结果

```json
// 只对在售商品聚合
GET products/_search
{
  "size": 0,
  "query": {
    "term": { "status.keyword": "on_sale" }
  },
  "aggs": {
    "by_brand": {
      "terms": { "field": "brand.keyword" }
    }
  }
}
```

**聚合查询原理**：

- **收集阶段**：各分片在本地执行聚合，收集局部聚合结果
- **合并阶段**：协调节点汇总各分片结果，做全局聚合
- **DocValues 支撑**：聚合依赖 DocValues 列式存储，只读取目标字段，性能高

**性能优化**：

- **设置 size: 0**：不需要文档时，不返回文档，减少网络传输
- **控制桶数量**：terms 聚合的 size 参数限制返回桶数，避免返回过多桶
- **使用 filter 缩小范围**：先用 query/filter 缩小聚合范围，减少计算量
- **keyword 字段聚合**：聚合必须用 keyword 字段，text 字段聚合无意义
- **避免高基数字段聚合**：对唯一值很多的字段（如用户ID）做 terms 聚合，会消耗大量内存

**实际应用场景**：

- **电商统计**：按品牌分组统计商品数量、平均价格、最高价格
- **日志分析**：按时间间隔统计日志量，发现流量高峰
- **用户分析**：按地域、年龄段分组统计用户数量
- **销售报表**：按月统计销售额，计算环比增长

**面试追问点**：

- **聚合查询的执行流程？** → 各分片本地聚合（收集阶段）→ 协调节点合并结果（合并阶段），依赖 DocValues 列式存储
- **text 字段能做聚合吗？** → 不能，text 字段分词后词项太碎，聚合结果无意义；必须用 keyword 字段
- **terms 聚合返回的桶是精确的吗？** → 不一定，大数据量下各分片本地统计后合并，可能存在误差；需要用 shard_size 参数提高精度
- **聚合结果能排序吗？** → 可以，terms 聚合支持 order 参数，按桶的文档数、子聚合值等排序
