# 1.ES 的搜索读过程是怎样的

ES 最强大的能力是做**全文检索**，其读（搜索）过程分为 Query Phase 和 Fetch Phase 两个阶段：

- 客户端发送搜索请求到一个 **coordinate node**（协调节点）
- 协调节点将搜索请求转发到该索引所有分片对应的 **primary shard 或 replica shard**，每个分片本身是一个 Lucene 实例，会进行分词并基于**倒排索引**（term → term dictionary → posting list → doc id）执行查询
- **Query Phase**：每个分片将搜索结果（doc id 列表）返回给协调节点，由协调节点进行数据合并、排序、分页，产出最终结果
- **Fetch Phase**：协调节点根据 doc id 去各个节点拉取实际的 document 数据，最终返回给客户端（根据 id 定位到具体 shard 涉及路由策略）

**读请求可以从 primary shard 或 replica shard 读取** ，采用**随机轮询算法**分配负载

搜索过程的几个关键点

- 一个搜索会遍历索引下的**所有分片**，每个分片独立执行搜索并返回相同数量的文档 ID。例如搜索条件要查 5 条数据，有 5 个分片，则每个分片返回 5 条共 25 条，排序后取前 5 条
- **查询和计算权重得分在 Lucene 完成，聚合操作在 ES 中实现**
- 分片数量影响搜索性能，也决定了 ES 能承载的最大数据量，需要在二者之间平衡
- 文档权重得分每次搜索都会根据搜索条件重新计算，对性能影响较大

# 2. 介绍下 ES 的写入流程

1、数据先写入 **内存buffer ，**同时将数据写入**translog日志文件（5s刷新时**也是 **先写入os cache** **）（5s刷新或者每次刷新）**

2、每隔一秒**refresh**操作写入 segment ，同时清空 1的内存buffer

3、segment 是 **OS Cache** （定时或者tranlog到512M flush后落盘）

4、flush触发后，清空translog，Merge segment, 写一个commit point，标识所有新的segment file，然后打开segment file供搜索使用，同时删除旧的segment file。


ES 在创建、更新、删除 document 时会修改其 `_version` 版本号，写入流程涉及**内存**和**磁盘**两个层面：

数据写入内存和磁盘

- 数据首先写入 **Index buffer**（内存）和 **Transaction log**（磁盘），即使内存数据丢失，也可从 Transaction log 恢复， **此时新写入的数据还不可查**
- 默认 **1s 一次的 refresh** 操作将 Index buffer 中的数据写入 segments（内存），此时数据变为**可查询**状态，**此时Segment未落盘，需要等tranlog到512M或者30分钟定时**
- 默认 **30 分钟一次的 flush** 操作将 segments 写入磁盘，同时清空 Transaction log；若 Transaction log 满（默认 **512MB**），也会触发 flush
- **merge** 操作定期合并 segment，减少文件数量提升查询性能

路由和主副分片写入

- 每个索引操作首先通过路由（基于文档 ID）解析到一个副本组，再转发到当前的**主分片**
- 主分片负责验证数据格式并转发到副本，副本由主分片**异步复制**
- ES 维护一个**同步副本列表**（replica set），所有操作必须复制到该列表中的每个副本，主分片确认所有副本成功后才向客户端确认完成

主分片写入流程

- **验证**传入操作，结构无效时拒绝（如字段格式与 mapping 不匹配）
- **本地执行**操作（索引或删除文档），验证字段内容，字段值过长等场景会拒绝
- 将操作**并行转发**到同步副本列表中的每个副本分片
- 所有副本分片都执行成功并响应后，主分片向客户端**确认请求成功**

副本分片的作用

- **高可用**：一个副本分片是主分片的拷贝，数量可动态调整，主分片节点挂掉时副本可晋升为新主分片
- **提升读性能**：增加副本分片可分担读请求负载
- **故障转移**：主分片和副本分片不会在同一个节点上，确保节点故障时数据不丢失
- **写入并发**：文档的写操作必须在主分片完成后才能复制到副本，ES 通过**乐观锁**（`_version` 版本号递增）控制并发写冲突
- 默认情况下 ES 为每个主分片创建**一个副本**

协调节点的作用

- 一个节点默认同时承担 master、data、coordinator 三个角色
- 协调节点专门负责**接收请求、转发请求到其他节点、汇总返回数据**，不存储数据
- 将 `node.master` 和 `node.data` 都配置为 false，该节点仅担任协调节点
- 客户端采用**轮询方式**选择配置的节点作为协调节点，也可自定义节点选择器
- 生产环境建议部署**独立的协调节点**，避免数据节点承担过多协调开销

# 3. 先写lucene还是translog

**不是先写Lucene后写Translog**，而是**同时写入内存和Translog**，Lucene Segment是后续通过refresh操作生成的。具体写入顺序如下：

**写入流程详细步骤**：

- **第一步：写入内存Index Buffer**：文档数据首先写入内存中的Index Buffer（这是Lucene的内存结构），此时数据在内存中，尚未持久化。
- **第二步：同时写入Translog**：几乎在写入内存的同时，数据也会写入磁盘上的Translog（事务日志），确保即使进程崩溃，数据也能从Translog恢复。
- **第三步：refresh生成Lucene Segment**：默认每1秒执行一次refresh操作，将内存Index Buffer中的数据写入新的Lucene Segment（位于文件系统缓存中），此时数据变为**可查询**状态。
- **第四步：flush持久化到磁盘**：默认每30分钟或Translog满（默认512MB）时执行flush操作，将文件系统缓存中的Segment持久化到磁盘，同时清空Translog。

**关键点解析**：

- **Translog的作用是保证持久性**：即使内存中的Index Buffer丢失，也可以通过Translog恢复数据，类似数据库的WAL（Write-Ahead Logging）机制。
- **Lucene Segment是近实时可查询的**：通过refresh操作生成Segment后，数据才对搜索可见，这就是ES的**近实时（Near Real-Time）搜索特性**。
- **更新和删除操作**：更新文档时，ES会将旧文档标记为删除，然后写入新文档；删除操作只是给文档打上删除标记，真正的物理删除在Segment Merge时发生。
- **Segment的不可变性**：Lucene Segment一旦生成就是不可变的，这带来了无需加锁、可高效缓存、支持压缩等优势。

**生产环境注意事项**：

- **调整refresh间隔**：写入密集型场景可以适当增加refresh间隔（如30秒），减少Segment数量，提升写入性能。
- **监控Translog大小**：Translog过大会导致flush时间变长，需要合理设置`index.translog.flush_threshold_size`参数。
- **理解近实时搜索**：写入后默认1秒才可查询，如果需要立即查询，可以使用`refresh=wait_for`参数，但会影响写入性能。
- **避免频繁更新**：Lucene的不可变性意味着更新操作实际上是删除+插入，频繁更新会产生大量小Segment，增加merge压力。

可以总结为：

- **ES写入是同时写入内存Index Buffer和磁盘Translog，不是先写Lucene后写Translog**。
- **Lucene Segment通过refresh操作从内存生成，数据才变为可查询状态**。
- **Translog保证持久性，类似数据库的WAL机制，进程崩溃后可恢复数据**。
- **Segment的不可变性带来性能优势，但更新操作是逻辑删除+新写入**。
- **生产环境需要根据场景调整refresh间隔和Translog配置**。

# 4. 为什么要写 Translog

Transaction log（事务日志）的核心作用是**保证数据持久性**，类似数据库的 **WAL（Write-Ahead Logging）机制**：

- 数据首先写入内存的 Index buffer，如果进程崩溃或节点宕机，内存中的数据会丢失
- Transaction log 同时写入磁盘，即使内存数据丢失，也可以从 Transaction log 中**恢复未持久化的数据**
- 默认每 30 分钟执行一次 flush 操作将数据持久化到磁盘，这期间如果发生故障，Transaction log 是**唯一的数据安全保障**

没有 Transaction log 的风险

- 写入 Index buffer 后到 flush 执行前，数据仅存在于内存中，节点故障将导致数据丢失
- refresh 操作只将数据写入文件系统缓存（segments），并非真正持久化到磁盘，仍有丢失风险

Transaction log 的具体保障

- 每次写入操作都会**同步写入** Transaction log，确保操作记录不丢失
- Transaction log 满（默认 512MB）或达到 30 分钟间隔时触发 flush，将 segments 持久化并**清空 Transaction log**
- 节点重启后，ES 会**重放 Transaction log** 中未 flush 的操作，恢复数据到最新状态

# **5. 数据变化时Segment的变化**

**不是每条数据一个 Segment，也不是每次写入都生成新 Segment**。ES 的写入流程是：文档先写入内存缓冲区，等 refresh 触发时才把缓冲区中的多条文档一起刷写成一个新的 Segment。

**写入流程详解**：

- **第一步：写入内存缓冲区**。文档到达分片后，先写入 in-memory buffer，同时写入 translog（用于宕机恢复）。
- **第二步：refresh 刷写 Segment**。默认每 1 秒（`index.refresh_interval`），ES 把内存缓冲区中的所有文档一起刷写成一个新的 Segment，此时数据对搜索可见。**此时Segment未落盘，需要等tranlog到512M或者30分钟定时。**
  **一个 Segment 包含多条文档，不是一条文档一个 Segment**。
- **第三步：translog 持续记录**。在 refresh 之前如果宕机，内存缓冲区的数据会丢失，translog 用于恢复这部分数据。

**更新文档的处理**：

- 更新操作的本质是 **新增一条新文档 + 标记旧文档为删除**。
- 新文档写入内存缓冲区，下次 refresh 时生成新 Segment。
- 旧文档所在的老 Segment 不会被修改，而是在删除位图（.del 文件）中标记为已删除。
- 查询时，ES 会过滤掉被标记删除的文档，所以用户看到的是更新后的数据。

**老 Segment 怎么办**：

- 老 Segment 中被标记删除的文档 **不会立即物理清除**，而是等 **Segment Merge** 时处理。
- 后台 Merge 进程会把多个小 Segment 合并成大 Segment，合并时 **跳过被标记删除的文档**，新生成的大 Segment 不包含这些旧数据。
- Merge 完成后，老 Segment 文件会被删除，空间释放。

**为什么这样设计**：

- **避免随机写**：Segment 是不可变文件，追加写新 Segment 比修改老 Segment 效率高得多。
- **批量刷写**：攒多条文档一起生成 Segment，比每条文档单独生成一个 Segment 的 IO 效率高很多。
- **延迟清理**：标记删除 + 延迟 Merge，把随机删除操作转化为顺序合并操作，减少 IO 压力。

**面试追问点**：

- refresh 太频繁会有什么问题？→ 生成大量小 Segment，增加 Merge 压力和查询时的 Segment 遍历成本，降低性能。
- translog 和 Segment 的关系？→ translog 是写入日志，保证宕机不丢数据；Segment 是搜索索引，数据从缓冲区刷写而来。translog 在 flush 时清空。
- 为什么更新不直接修改老 Segment？→ Segment 是不可变的，修改需要重写整个文件，代价远高于追加写新 Segment + 标记删除。

# 6. 读写时如何找到shard

ES 通过**路由算法**定位文档所在的 shard，核心公式为：

**shard = hash(document_id) % number_of_primary_shards**

- 客户端发送读写请求到协调节点
- 协调节点根据文档 ID 计算 hash 值，对**主分片数量取模**，得到目标主分片编号
- 请求被转发到对应的主分片执行

写入时的 shard 定位

- 写请求通过路由算法定位到**主分片**
- 主分片写入成功后，将操作复制到该分片组的所有副本分片

**读取时的 shard 定位**

- 读请求同样通过路由算法定位到**主分片或副本分片**（随机轮询选择）
- 一个搜索请求会遍历**所有分片**，每个分片独立执行查询后汇总（因为无指定\_id路由id）
- 没有路由依据，无法预判数据在哪一个分片，执行**分片广播策略**：协调节点把查询下发给**该索引全部主 / 副本分片**，所有分片独立检索，最后协调节点合并结果。

**主分片数量的约束**

- 主分片数量在**索引创建时确定**，之后不可修改
- 这是 **因为路由算法依赖主分片数量取模，修改后会导致已有数据的路由失效，数据无法找到**
- 如果需要增加分片，只能通过**新建索引并 reindex** 数据来实现

# **7. Translog 何时写入，何时写入磁盘，Translog 写入成功后意味着数据已经成功了吗，此时 segments 写入磁盘了吗**

**Translog写入时机**：

- 每次**index/delete/update**写操作时，数据**同步写入translog**（和写入buffer是同一操作）。
- translog写入是**顺序追加写**，性能很高，不会成为写入瓶颈。
- translog保证了**写操作的可恢复性**，即使buffer中的数据还没refresh到segment，宕机重启后也能从translog中恢复。

**Translog刷盘时机**：

- 默认translog也是**先写入os cache**，**每隔5秒**由后台线程fsync到磁盘。
- 这意味着即使translog写入"成功"（返回客户端），数据可能还在os cache中，**最多可能丢失5秒的translog数据**。
- 可以将`index.translog.durability`设置为`request`，实现**每次写操作都同步fsync到磁盘**， **数据零丢失** 但性能下降明显。

**Translog写入成功≠数据已持久化**：

- 默认模式下，translog返回成功**不等于数据已落盘**，数据还在os cache中。
- 如果此时宕机，os cache中的translog数据会丢失，**最多丢失5秒写操作**。
- 要实现数据真正持久化，需要配置`durability=request`或等待自动flush。

**此时Segment的状态**：

- Translog写入成功时，**segment数据不在磁盘上**，还在os cache中。
- segment需要通过refresh操作从buffer写入os cache，默认 **30 分钟一次，**再通过flush操作从os cache fsync到磁盘。
- 写入流程：**buffer → refresh → os cache（可搜索） → flush → 磁盘（持久化）**。

**数据恢复的两个来源**：

- **translog恢复**：宕机重启后，ES加载最近一次flush后的segment，然后**重放translog**中未flush的写操作，恢复到宕机前状态。
- **flush保证**：flush后translog被清空，此时segment已持久化到磁盘，不需要translog保底。

**面试追问点**：

- 配置`durability=request`后性能下降多少？→ 每次写操作都要等待磁盘IO完成，写入吞吐量可能下降10-100倍，视磁盘性能而定。
- 为什么translog不每次都fsync？→ 磁盘IO延迟高（毫秒级），每次都fsync会严重影响写入吞吐量，5秒一次是性能和可靠性的平衡点。
- 5秒数据丢失的场景如何处理？→ 可用消息队列（如Kafka）做前置缓冲，配合ES的异步写入策略，或配置同步刷盘。

# **8. Translog 何时清理**

Translog在**flush操作完成后被清理**，清空translog是flush的最后一个步骤。

**触发flush的条件**：

- **translog文件大小达到阈值**：默认`index.translog.flush_threshold_size`为**512MB**，超过后自动触发flush。
- **定时触发**：默认**每隔30分钟**自动执行一次flush，由`index.translog.sync_interval`控制。
- **手动触发**：可通过`POST /_flush`或`POST /<index>/_flush`手动执行。

**flush清理流程**：

- 执行refresh：将buffer中剩余数据写入os cache。
- 写入**commit point**：记录当前所有segment的元数据信息。
- 将os cache中的segment数据**fsync到磁盘**：保证segment持久化。
- **清空translog文件**：删除或截断translog，因为数据已安全落盘，不再需要translog保底。

**清理后的状态**：

- translog文件被**截断为空**或**删除重建**。
- 所有segment数据已持久化到磁盘，**宕机重启后可直接加载segment**，无需重放translog。
- 下一次写操作开始时，translog重新从头记录。

**为什么不立即清理**：

- translog需要**保留到flush完成**，因为flush期间如果宕机，还需要translog恢复。
- 只有当segment**确认fsync到磁盘**后，translog才有资格被清理。

**面试追问点**：

- translog可以配置不清空吗？→ 不可以，translog的设计就是为了在flush后清空，否则会无限增长。
- 手动flush的场景？→ 索引大量数据后、宕机恢复前、索引生命周期结束时，手动确保数据持久化。
- flush和refresh的区别？→ refresh是buffer→os cache（可搜索，1秒），flush是os cache→磁盘（持久化，30分钟）。

# **9. Segment 何时落磁盘**

Segment通过**flush操作**从os cache持久化到磁盘，默认**每30分钟**或translog达到**512MB**时触发。

**Segment的生命周期**：

- **refresh阶段**：buffer中的数据写入os cache，**生成新的segment文件**，此时segment在内存中可搜索，但**未落盘**。
- **flush阶段**：os cache中的segment数据**fsync到磁盘**，此时segment**真正持久化**，断电不丢失。

**落盘触发条件**：

- **translog文件大小达到阈值**：默认`index.translog.flush_threshold_size`为**512MB**，超过后自动触发。
- **定时触发**：默认**每隔30分钟**自动执行一次flush。
- **手动触发**：通过`POST /<index>/_flush`手动执行，确保数据落盘。

**落盘流程**：

- 执行refresh：将buffer中剩余数据写入os cache。
- 写入**commit point**：记录当前所有segment的元数据，作为持久化的检查点。
- 将os cache中的segment数据**fsync到磁盘**：调用系统调用确保数据写入物理存储。
- 清空translog：数据已落盘，translog不再需要。

**落盘前后的差异**：

- **落盘前**：segment在os cache中，断电后丢失，需要通过translog恢复。
- **落盘后**：segment在磁盘上，断电后可直接加载，**无需translog恢复**。

**面试追问点**：

- refresh生成的segment在os cache中，如果内存不足被换出怎么办？→ 操作系统会将冷数据换出到磁盘的swap区域，对ES透明，但可能影响查询性能。
- flush期间segment可搜索吗？→ 可以，flush期间旧segment正常提供搜索，flush完成后才切换到新segment，**不阻塞查询**。
- merge后的segment什么时候落盘？→ merge生成的新segment同样需要通过flush操作才落盘，merge本身只在os cache中操作。
