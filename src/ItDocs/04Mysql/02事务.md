### 1. 什么是事务？事务的四个特性是什么？

**原理分析**

事务是数据库操作的**最小工作单元**，一组逻辑操作要么全部成功，要么全部失败回滚。

**ACID四个特性：**

| 特性 | 说明 | 面试重点 |
| --- | --- | --- |
| **A**tomic 原子性 | 事务中的操作要么全部提交，要么全部回滚 | 通过 **Undo Log** 实现 |
| **C**onsistency 一致性 | 事务执行前后，数据库状态保持一致 | 由应用层保证 |
| **I**solation 隔离性 | 并发事务之间相互隔离，不互相干扰 | 通过 **锁 + MVCC** 实现 |
| **D**uration 持久性 | 事务提交后，数据永久保存 | 通过 **Redo Log** 实现 |

**关系：**

- **原子性**是基础，保证操作不可分割
- **隔离性**是手段，通过锁机制控制并发
- **持久性**是保障，通过日志确保数据不丢失
- **一致性**是目标，原子性、隔离性、持久性共同保证一致性

> 原子性和一致性的区别是什么？

原子性关注的是"操作"层面的不可分割，一致性关注的是"状态"层面的正确性。原子性没做好会导致中间状态泄露，一致性没做好会导致业务数据错误。

### 2. MySQL事务的隔离级别有哪些？分别存在什么问题？

**原理分析**

SQL标准定义了4个隔离级别：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
| --- | --- | --- | --- |
| 读未提交 Read Uncommitted | ✓ 允许 | ✓ 允许 | ✓ 允许 |
| 读已提交 Read Committed | ✗ 不允许 | ✓ 允许 | ✓ 允许 |
| 可重复读 Repeatable Read | ✗ 不允许 | ✗ 不允许 | ✗ 不允许（InnoDB） |
| 串行化 Serializable | ✗ 不允许 | ✗ 不允许 | ✗ 不允许 |

**三种异常现象：**

1. **脏读**：读取到其他事务未提交的数据
2. **不可重复读**：同一个事务中，两次读取同一行数据结果不同
3. **幻读**：同一个事务中，两次查询返回的记录数不同（新增或删除）

**隔离级别实现原理：**

- 读未提交：不做任何隔离
- 读已提交/可重复读：通过 **MVCC** 实现
- 串行化：通过 **锁** 实现，读写都加锁

> MySQL默认隔离级别是什么？

MySQL默认是 **Repeatable Read**（可重复读）。Oracle默认是 **Read Committed**。

> InnoDB在RR级别下如何解决幻读？

InnoDB通过 **MVCC + Next-Key Lock** 解决幻读：

- 快照读：使用MVCC，每次读取事务开始时的快照
- 当前读：使用Next-Key Lock，锁定记录间隙

### 3. 事务的原子性和持久性是如何保证的？

**原理分析**

**Redo Log（重做日志）保证持久性：**

```
事务提交 → 写Redo Log → 刷盘 → 返回成功
```

- 采用 **WAL（Write-Ahead Logging）** 技术
- 先写日志再写数据，保证异常时可通过日志恢复
- Redo Log 顺序写入，性能高
- 通过 `innodb_flush_log_at_trx_commit` 控制刷盘策略

**刷盘策略：**

| 参数值 | 行为 | 安全性 |
| --- | --- | --- |
| 0 | 每秒刷盘，不等待 | 低，可能丢失1秒数据 |
| 1 | 每次提交刷盘 | 高，性能低 |
| 2 | 每次提交写系统缓存，由系统刷盘 | 中，宕机可能丢失数据 |

**Undo Log（回滚日志）保证原子性：**

- 记录数据的**反向操作**
- UPDATE生成UPDATE的反向记录
- INSERT生成DELETE记录
- DELETE生成INSERT记录
- 事务回滚时执行反向操作

> Redo Log和Binlog有什么区别？

| 对比项 | Redo Log | Binlog |
| --- | --- | --- |
| 位置 | InnoDB特有 | MySQL Server层 |
| 作用 | 保证事务持久性 | 主从复制、数据恢复 |
| 内容 | 物理修改（哪个页改了什么） | 逻辑SQL（Row/Statement/Mixed） |
| 写入方式 | 循环写，会覆盖 | 追加写，不覆盖 |
| 刷盘时机 | 事务提交时 | 由sync_binlog控制 |

### 4. MVCC是什么？是如何实现的？

**原理分析**

**MVCC（Multi-Version Concurrency Control）多版本并发控制：**

每个事务看到的数据版本不同，通过版本链实现读写不阻塞。

**InnoDB行记录结构：**

```sql
DB_ROW_ID（隐藏列） | DB_TRX_ID | DB_ROLL_PTR | 实际数据
```

- **DB_TRX_ID**：最近修改的事务ID
- **DB_ROLL_PTR**：指向Undo Log的指针，形成版本链

**Read View（读视图）：**

```sql
struct ReadView {
    int m_ids[];        //活跃事务ID列表
    int m_low_limit_id; //最小活跃事务ID
    int m_high_limit_id;//创建ReadView时最大事务ID
    int m_creator_trx_id; //当前事务ID
}
```

**快照读判断规则：**

| 数据行事务ID | 判断结果 |
| --- | --- |
| `trx_id < m_low_limit_id` | 已提交，数据可见 |
| `trx_id >= m_high_limit_id` | 未来事务，数据不可见 |
| `trx_id in m_ids` | 活跃事务，数据不可见 |
| 其他情况 | 可见 |

**版本链遍历：**

从当前行沿着 `DB_ROLL_PTR` 找到Undo Log中的历史版本，逐个判断是否对当前事务可见。

> 读已提交和可重复读在MVCC上的区别？

区别在于**创建ReadView的时机**：

- **Read Committed**：每次读取都创建新的ReadView
- **Repeatable Read**：事务开始时创建ReadView，后续都用同一个

> RC级别下能避免脏读，为什么还会出现不可重复读？

RC每次读都重新生成ReadView，如果其他事务在两次读取之间提交并修改了数据，第二次读取就会拿到新数据，导致不可重复读。

### 5. Next-Key Lock是什么？解决了什么问题？

**原理分析**

**Next-Key Lock = Record Lock + Gap Lock：**

- **Record Lock**：锁定索引记录本身
- **Gap Lock**：锁定索引之间的间隙

```
记录: [A]----[B]----[C]----[D]
Gap:   (-∞,A) (A,B) (B,C) (C,D) (D,+∞)
```

**锁定规则：**

1. **唯一索引等值查询**：`Record Lock`，只锁定当前记录
2. **唯一索引范围查询**：先 `Record Lock`，再 `Gap Lock`
3. **非唯一索引等值查询**：`Next-Key Lock`，锁定记录 + 前后间隙
4. **非唯一索引范围查询**：`Next-Key Lock`，锁定范围 + 间隙

**解决幻读：**

```sql
-- 假设有数据
id: 1, 3, 5

-- 事务A执行（当前读）
SELECT * FROM t WHERE id > 2 FOR UPDATE;
-- 锁定 (2,+∞) 范围，阻止其他事务插入 id > 2 的记录
```

**Gap Lock的死锁风险：**

```sql
-- 事务A
SELECT * FROM t WHERE id = 3 FOR UPDATE;  -- 锁定 (1,5) 间隙

-- 事务B
SELECT * FROM t WHERE id = 4 FOR UPDATE;  -- 锁定 (3,5) 间隙

-- 事务A
INSERT INTO t VALUES (4, ...);  -- 等待事务B释放锁

-- 事务B
INSERT INTO t VALUES (3, ...);  -- 等待事务A释放锁
-- 死锁！
```

> 什么情况下Gap Lock会退化为Record Lock？

在唯一索引的**等值查询**时，Gap Lock会退化为Record Lock，因为唯一性保证不会有其他事务插入相同值的记录。

### 6. 事务产生死锁的原因是什么？如何避免？

**原理分析**

**死锁产生的条件：**

1. **互斥**：资源只能被一个事务持有
2. **持有并等待**：持有资源的同时等待其他资源
3. **不可抢占**：资源不能被强制释放
4. **循环等待**：事务之间形成循环依赖

**InnoDB处理死锁策略：**

1. **检测**：通过事务wait-for graph检测死锁
2. **回滚**：回滚undo量最小的事务
3. **超时**：等待超时的事务会被回滚（`innodb_lock_wait_timeout`）

**典型死锁场景：**

```sql
-- 场景1：不同表顺序不一致
-- 事务A
UPDATE orders SET status = 1 WHERE id = 1;
UPDATE products SET stock = stock - 1 WHERE id = 100;

-- 事务B
UPDATE products SET stock = stock - 1 WHERE id = 100;  -- 先更新products
UPDATE orders SET status = 1 WHERE id = 1;            -- 后更新orders
-- 如果A和B几乎同时执行，可能死锁

-- 场景2：间隙锁 + 唯一索引
-- 事务A
INSERT INTO t VALUES (5, 'a');  -- 锁定 (3,7) 间隙

-- 事务B
INSERT INTO t VALUES (6, 'b');  -- 锁定 (5,7) 间隙

-- 事务A
INSERT INTO t VALUES (6, 'c');  -- 等待B

-- 事务B
INSERT INTO t VALUES (5, 'd'); -- 死锁！
```

**避免死锁的方法：**

| 方法 | 说明 |
| --- | --- |
| 固定顺序访问表 | 所有事务按相同顺序访问资源 |
| 降低隔离级别 | RC级别下间隙锁不生效 |
| 减少锁定时间 | 尽量快速完成事务，减少持有锁的时间 |
| 分批处理 | 大批量操作分组提交，减少锁定范围 |
| 添加合理索引 | 减少锁定的记录数量 |

> 死锁和锁等待超时有什么区别？

死锁是**循环依赖**导致，InnoDB会主动检测并回滚一个事务解决。锁等待超时是因为获取不到锁达到阈值后被动回滚，不一定是死锁，可能是锁竞争激烈。

### 7. MySQL如何保证分布式事务的一致性？

**原理分析**

**本地事务的局限：**

本地事务只能保证单节点内的一致性，跨库、跨服务场景下无法使用。

**分布式事务解决方案：**

**1. XA协议（两阶段提交）：**

```
第一阶段（Prepare）：
  - coordinator向所有节点发送Prepare
    - 各节点执行事务但不提交
    - 各节点写Prepare日志锁定资源

第二阶段（Commit/Rollback）：
  - 所有节点Prepare成功后，发送Commit
    - 任一节点失败，发送Rollback
```

问题：

- 单点故障：协调者挂了
- 同步阻塞：所有资源被锁定
- 数据不一致：部分节点Commit失败

**2. TCC（Try-Confirm-Cancel）：**

```
Try：预留资源，锁定资源
Confirm：确认执行，真正提交
Cancel：取消执行，释放资源
```

问题：

- 业务侵入性强
- 多次网络调用
- 需要处理幂等性和空回滚

**3. Saga模式：**

将长事务拆分为多个本地事务，每个事务都有对应的补偿操作。

```
T1 → T2 → T3 → T4
 ↓    ↓    ↓    ↓
C1   C2   C3   C4  （补偿操作）
```

**4. 消息最终一致性：**

```
1. 消息队列 + 本地消息表
2. 本地事务执行成功，消息发送成功
3. 消费方重试直到成功
4. 保证最终一致性，不保证强一致性
```

> 如何选择分布式事务方案？

- **强一致性要求高**：选XA，但接受性能损失
- **性能优先，允许最终一致**：选消息队列 + 本地消息表
- **跨服务调用多，业务复杂度高**：选Saga或TCC
