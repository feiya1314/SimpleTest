# 1. MySQL、MongoDB、PG、Hive、Cassandra、ES 这些数据库有什么区别和特点？

**核心回答：**
这6种数据库定位完全不同：MySQL 是**关系型事务库**，PG 是**企业级关系型增强库**，MongoDB 是**文档型 NoSQL**，ES 是**分布式搜索引擎**，Cassandra 是**列式高写入 NoSQL**，Hive 是**离线数仓**，不是数据库。

**核心区别：**

1. **MySQL**：经典关系型，行存储，支持事务 ACID，强表结构强约束，生态最成熟，适合高并发联机事务
2. **PG**：也是关系型，但比 MySQL 更强，支持复杂 SQL、窗口函数、JSON/数组、地理信息、全文检索，事务更强、MVCC 更优秀
3. **MongoDB**：文档型 NoSQL，类 JSON BSON 存储，无固定表结构，字段可随意增减，弱事务（新版支持事务），内嵌文档减少联表
4. **ES**：分布式搜索引擎，底层倒排索引，擅长全文检索、模糊查询、分词、聚合统计，写入近实时，不适合做强事务数据库
5. **Cassandra**：分布式列式 NoSQL，无中心去中心化，超高写入性能、横向无限扩容，查询只适合按主键精准查询和范围查询
6. **Hive**：数据仓库不是数据库，基于 Hadoop 把 SQL 转成 MapReduce/Spark 离线任务，不支持实时读写，只做离线批量分析

**适用场景：**

| 数据库 | 最大优势 | 最佳存储数据 | 典型场景 |
| --- | --- | --- | --- |
| MySQL | 事务、高并发、生态成熟 | 结构化固定业务数据 | 订单、支付、用户核心业务 |
| PG | 强 SQL、JSON、地理信息、复杂查询 | 半结构化、地理、复杂业务 | 风控、GIS、中台、复杂报表 |
| MongoDB | 结构灵活、字段随意变 | 动态结构、文档、画像 | 社区内容、用户标签、设备属性 |
| ES | 全文检索、分词、聚合 | 日志、文章、文本数据 | 搜索服务、日志分析、多维筛选 |
| Cassandra | 超高写入、高可用、扩容强 | 时序、设备打点、监控 | 物联网、时序指标、海量流水 |
| Hive | 海量离线大数据分析 | 历史海量日志、行为数据 | 大数据离线报表、用户画像、统计分析 |

**一句话总结：**
核心业务要事务用 MySQL，复杂查询/GIS/JSON 用 PG，结构老变用 MongoDB，搜文章查日志用 ES，物联网高并发写入用 Cassandra，海量离线分析用 Hive。

# 2. MySQL 适合什么场景？适合存储什么数据？

**核心回答：**
MySQL 是互联网**核心业务首选**，主打**事务、高并发、生态成熟**，适合结构化固定字段数据。

**底层原理：**

1. 经典关系型数据库，行存储，支持事务 ACID
2. 强表结构、强约束、支持联表查询
3. 生态最成熟、运维简单、中间件配套多（分库分表、主从）
4. InnoDB 针对简单热点请求做了极致精简，适合高并发联机事务

**适用场景：**

1. 互联网业务核心业务库
2. 需要事务、强一致性、联表查询
3. 高并发读写、主从备份、分库分表

**适合存储：**
结构化固定字段数据：用户、订单、商品、支付、权限、配置

**避坑指南：**
MySQL 不适合海量日志、非结构化不定字段、超大批量离线分析、模糊全文检索

**一句话总结：**
核心业务、要事务 → MySQL。

# 3. PostgreSQL 和 MySQL 有什么区别？怎么选？

**核心回答：**
MySQL 追求**简单高并发、业务首选**；PG 追求**功能强、复杂查询、数据分析、兼容性强**。技术上 PG 能干 MySQL 所有事还多一堆高级能力，但工程和生态上绝大多数公司不会全量替代。

**核心区别：**

1. **标准 SQL**：PG 更完整，窗口函数、CTE、递归查询、复杂子查询更强
2. **JSON 支持**：PG 原生支持 JSON/JSONB，既可以关系表又能当 MongoDB 用
3. **地理信息**：PG 原生 GIS 支持，做地图点位距离碾压 MySQL
4. **全文检索**：PG 内置，简单场景不用搭 ES
5. **事务 MVCC**：PG 比 MySQL 更稳健，丢数据概率更低
6. **数据类型**：PG 支持数组、自定义类型、触发器、存储过程更强大
7. **分区表并行查询**：PG 更优秀

**一句话总结：**
高并发选 MySQL，复杂查询/GIS 选 PG。

# 4. PG 能替代 MySQL 吗？

**核心回答：**
技术上 PG 完全能替代 MySQL；但工程和生态上，绝大多数公司不会全量替代，只能部分替代。不是 PG 不行，是**生态、习惯、运维、人才、中间件**卡住了。

**底层原理：**

1. **生态 & 人才壁垒（最大原因）**：国内 Java/后端开发只会 MySQL，懂 PG 的人少；运维、DBA 大多只会 MySQL 主从、调优、故障排查；面试、招聘、团队习惯全是 MySQL
2. **中间件配套差距**：MySQL 分库分表有 Sharding-JDBC、MyCat，读写分离、binlog 生态无敌，各种云服务商默认 MySQL 模板；PG 的分库分表、同步、canal 类订阅、运维工具生态弱很多
3. **轻量化 & 资源消耗**：MySQL 更轻，低配机器也能跑很稳；PG 更吃内存、配置调优更复杂，新手容易玩崩
4. **历史存量太大**：老系统全是 MySQL，迁移成本极高，语法小差异、函数差异、排序、大小写、时间函数都要改

**适用场景：**

**完全可以用 PG 替代 MySQL 的场景：**

1. 新项目从零开始，没人绑死 MySQL
2. 复杂查询、报表、后台管理、风控系统
3. 需要 JSON 动态字段，又不想上 MongoDB
4. 地图、LBS、点位距离、地理范围业务
5. 对数据可靠性、不丢数据、事务严谨要求高
6. 中小型业务，不需要重度分库分表、超大规模并发

**不建议用 PG 替代 MySQL 的场景：**

1. 超高并发互联网核心业务（秒杀、订单、支付、百万 QPS），MySQL 生态、运维方案、踩坑经验太成熟，PG 扛得住但没人敢赌
2. 需要大量 Binlog 订阅、数据同步、中间件依赖
3. 小团队、无专业 DBA，PG 调优、故障排查比 MySQL 难
4. 老旧系统迁移，代码大量依赖 MySQL 特有函数、语法

**实战选型：**
核心交易、高并发业务：继续用 MySQL；

后台、风控、报表、中台、GIS、动态结构业务：直接上 PG。

混合架构，各干各的，不用谁替代谁。

**落地选型建议**

1. 电商订单、支付、超高简单 QPS → 用 MySQL 更稳妥
2. 后台系统、风控、报表、中台、复杂 SQL、大表统计 → 直接 PG
3. 字段不定、要 JSON 又不想上 Mongo → PG 性能完胜
4. 小团队不想搞分库分表、想单库扛更大数据量 → PG 更合适

**一句话总结：**
技术能替，工程难替；核心交易留 MySQL，复杂业务上 PG，混合架构最实际。

# 5. PG 的性能比 MySQL 强吗？

**核心回答：**
不是全场景碾压，分场景：复杂查询、多表关联、大结果集、聚合分析、并发读写 PG 性能远强于 MySQL；简单主键单条增删改、超高 QPS 简单 CRUD MySQL 略占优势或持平。

**底层原理：**

PG 复杂查询吊打 MySQL 的原因：

1. **查询优化器不是一个级别**：
   PG 优化器是企业级，支持：
   - 更优的多表 Join 计划
   - 智能子查询、CTE、递归查询
   - 并行查询（大表自动并行扫表）
   - 更优秀的成本估算

   MySQL 优化器很弱：
   - 多表 Join 经常选错驱动表
   - 子查询容易被强行转成关联，性能雪崩
   - 大表没法并行查询

     结论：3 张表以上联表、分组统计、子查询，PG 轻松几倍碾压 MySQL
2. **MVCC 机制 PG 更先进**：
   PG：
   - 基于版本链无回滚段，读写冲突极小
   - 高并发下更少锁等待

   MySQL InnoDB：
   - 用 undo log + 事务隔离实现
   - 高并发热点更新容易锁等待、事务排队
     结论：高并发读写混合场景 PG 更稳、延迟更平滑
3. **索引能力更强**：
   PG：
   - 支持复合索引、GiST、GIN、BRIN 多种索引
   - 模糊查询、数组、JSONB、地理索引原生极强

   MySQL：
   - 只有普通 B+树
   - 全文索引弱

MySQL 简单 CRUD 略强的原因：

1. MySQL 更轻、线程模型简单、开销小
2. InnoDB 针对简单热点请求做了极致精简
3. PG 架构更重，单次简单请求略高一点点开销
4. 但差距不大，不是量级差距，只是微优

**适用场景：**

| 场景 | 性能谁更强 | 原因 |
| --- | --- | --- |
| 单表主键简单 CRUD | MySQL 略强 | 更轻量、开销小 |
| 多表联表、复杂 SQL | PG 吊打 | 优化器、执行计划更强 |
| 大表千万/亿级查询统计 | PG 强 | 并行查询、分区、优化器优秀 |
| 高并发读写混合 | PG 强 | MVCC 更优、锁竞争小 |
| 批量导入海量数据 | PG 强 | Copy 协议、批量写入优化 |
| GIS、JSONB、数组查询 | PG 碾压 | 原生索引和类型支持 |

**避坑指南：**
写入性能对比：

- 普通单机写入两者差不多；
- 大批量批量导入 PG 更快（copy 协议、并行导入）；
- 热点行高频更新 MySQL 稍微好一点，
- PG 高并发下隔离更稳健不会抖动雪崩。
- 大数据量千万级亿级表 PG 完胜，分区表更成熟更稳定，大表统计分页排序分组碾压 MySQL，
- MySQL 亿级大表很容易查询慢、统计卡死、优化器失效。
- 高并发多连接读写混合 PG 更强，MySQL 连接多了线程切换开销大、事务隔离锁竞争变严重，
- PG 进程模型成熟更不容易抖动崩库延迟飙升。

**一句话总结：**
简单 CRUD MySQL 略优，复杂查询/大数据量/高并发 PG 完胜，选型看业务特征。

# 7. MongoDB 适合什么场景？适合存储什么数据？

**核心回答：**
MongoDB 是**文档型 NoSQL**，核心优势是**结构灵活、字段随意变**，适合 schema 经常变、不想建固定表结构的场景。

**底层原理：**

1. 类 JSON BSON 存储，无固定表结构，字段可随意增减
2. 弱事务（新版支持事务），适合 schema 经常变
3. 内嵌文档、数组，能做一对多内嵌，减少联表

**适用场景：**

1. 需求频繁变更、不敢建固定表结构
2. 内容社区、CMS、小程序动态数据
3. 物联网设备属性、用户标签画像
4. 不想设计复杂表关联，用文档内嵌搞定

**适合存储：**
结构不固定、字段经常变、半结构化数据：用户画像、行为日志、内容文章、评论、配置模板、设备上报

**避坑指南：**
MongoDB 不适合强事务、强一致性、金融支付核心账务

**一句话总结：**
结构老变、文档型数据 → MongoDB。

# 8. Elasticsearch 适合什么场景？适合存储什么数据？

**核心回答：**
ES 是**分布式搜索引擎**，核心优势是**全文检索、模糊查询、分词、聚合统计**，不是数据库，不适合做核心业务存储。

**底层原理：**

1. 底层倒排索引，擅长全文检索、模糊查询、分词、聚合统计
2. 写入近实时，聚合、分组、过滤非常强
3. 不适合做强事务数据库

**适用场景：**

1. 全文检索：商城搜索、文章检索、知识库
2. 日志分析：ELK 收集系统日志、业务日志
3. 多维筛选、标签筛选、复杂条件聚合
4. 海量数据快速模糊查询

**适合存储：**
日志、文章、商品标题、文档、APP 行为轨迹、系统日志

**避坑指南：**
ES 不适合做主库存核心业务、强事务、高频率更新修改。常见架构：MySQL 存数据，ES 同步数据做检索。

**一句话总结：**
搜文章、搜商品、查日志 → ES。

# 9. Cassandra 适合什么场景？适合存储什么数据？

**核心回答：**
Cassandra 是**分布式列式 NoSQL**，核心优势是**超高写入性能、横向无限扩容、高可用容错极强**，但查询能力弱，只适合按主键精准查询和范围查询。

**底层原理：**

1. 无中心、去中心化架构
2. 超高写入性能、横向无限扩容
3. 高可用、容错极强，适合海量时序数据
4. 查询只适合按主键精准查询、范围查询，不适合复杂 SQL

**适用场景：**

1. 物联网海量设备高频上报
2. 监控时序数据、风控流水、行为打点
3. 要求写入极高、不怎么改、按主键查

**适合存储：**
海量时序、设备打点、监控指标、轨迹、物联网上报

**避坑指南：**
Cassandra 不适合复杂条件筛选、联表、事务、随意条件查询

**一句话总结：**
物联网高并发时序写入 → Cassandra。

# 10. Hive 适合什么场景？适合存储什么数据？

**核心回答：**
Hive 是**数据仓库**，不是数据库，核心能力是**海量离线大数据分析**，不支持实时读写。

**底层原理：**

1. 基于 Hadoop，把 SQL 转成 MapReduce/Spark 离线任务
2. 不支持实时读写，只做离线批量分析
3. 存超大历史海量数据

**适用场景：**

1. 大数据离线分析、报表、用户画像建模
2. 日/月统计、行为分析、运营报表
3. 海量冷数据归档、历史数据查询

**适合存储：**
海量历史日志、业务流水、用户行为、离线数仓分层数据

**避坑指南：**
Hive 不适合实时查询、线上业务读写、高并发接口

**一句话总结：**
海量历史数据离线分析 → Hive。

# 11. OBS 对象存储是怎么存储的？

**核心回答：**
OBS 是**扁平结构、对象为单位、分布式存储、高可靠冗余**，只有桶（Bucket）和对象（Object）两层，没有目录树。

**底层原理：**

逻辑结构（两层）：

1. **桶 Bucket**：全局唯一的容器，相当于"顶级文件夹"，有地域、存储类别（标准/低频/归档）、权限
2. **对象 Object**：每个上传的文件都变成一个对象，由三部分组成：
   - **Key（键名）**：唯一名字，如 `user/001/avatar.jpg`（用 `/` 模拟文件夹）
   - **Metadata（元数据）**：系统元数据（大小、类型、最后修改时间、ETag）+ 用户自定义元数据（如 `author=zhangsan`）
   - **Data（数据）**：文件的二进制内容

物理存储（分布式 + 多副本/纠删码）：

1. **底层架构三层**：接入层（RESTful API，鉴权路由）→ 元数据层（分布式数据库，记录桶+Key→物理位置）→ 数据层（多个服务器节点 OSD，存真实数据）
2. **数据写入流程**：上传文件 → 切分数据块 → 按哈希算法分散到不同节点（不同机架/机房）→ 冗余保护（二选一：多副本写2\~3份到不同节点简单可靠；纠删码EC数据+校验块占用更少空间恢复快）
3. 效果：11个9持久性（年丢失概率极低），跨 AZ 高可用

访问过程：

1. 客户端请求：`GET /桶名/对象Key`
2. 元数据集群：查哈希表 → 找到数据所在节点
3. 数据节点：返回数据块 → 客户端拼接成文件
4. 全程无目录遍历，直接定位，支持高并发

**核心区别：**

和文件存储、块存储的区别：

1. **文件存储（NAS）**：有目录树，路径深、并发差、扩展有限
2. **块存储（EBS）**：按块读写，需格式化，适合单服务器
3. **对象存储（OBS）**：扁平、无目录、分布式、无限扩展，适合海量非结构化数据（图片/视频/备份）

**一句话总结：**
OBS 扁平无目录、分布式冗余、直接哈希定位，适合海量非结构化数据存储。
