# 1. MySQL 的查询流程是怎样的？

**核心回答：** MySQL 查询流程分为五个核心步骤：**连接器→查询缓存→分析器→优化器→执行器**，最终由存储引擎返回结果。

**原理分析：**

1. **连接器**：建立客户端与 MySQL Server 的连接，校验用户权限。
2. **查询缓存**：检查该 SQL 是否已有缓存结果（ **MySQL 8.0 已移除此功能，因命中率过低** ）。
3. **分析器**：进行词法分析（提取关键字、表名、字段名）和语法分析（校验 SQL 语法）。
4. **优化器**：生成执行计划，选择最优索引和表连接顺序。
5. **执行器**：调用存储引擎 API 获取数据并返回结果。

**延展思考：**
从性能角度看，查询缓存虽然理论上可加速重复查询，但在高并发场景中弊大于利：缓存失效频繁、维护成本高，因此 MySQL 8.0 直接移除了该特性。实际生产中建议使用 Redis 等外部缓存替代。

# 2. 详细介绍下查询流程各步骤的执行内容

**核心回答：** 各步骤依次负责：**权限校验→缓存查询→SQL解析→执行计划生成→存储引擎调用**。

**原理分析：**

1. **连接器**：验证用户身份与权限，无权限直接返回错误。
2. **查询缓存**：以 SQL 文本为 Key、结果集为 Value 存储于内存，命中则直接返回。
3. **分析器**：
   - **词法分析**：提取 SELECT、表名、字段名、WHERE 条件。
   - **语法分析**：校验 SQL 是否符合 MySQL 语法规范。
4. **优化器**：评估多种执行计划的成本（如索引选择、表连接顺序），选择成本最低的方案。
5. **执行器**：根据执行计划调用 InnoDB 存储引擎接口，获取数据并返回客户端。

**延展思考：**
SQL 执行顺序与书写顺序不同，实际顺序为：**FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT**。理解这个顺序对分析慢查询和优化 SQL 至关重要。

以实际 SQL 为例：
`SELECT dept,COUNT(phone) AS num FROM User WHERE age < 25 GROUP BY dept HAVING num >= 3 ORDER BY num DESC,dept ASC LIMIT 0,2;`

当 MySQL 没有命中缓存时，执行流程如下：

1. **FROM User**：负责把数据库的表文件加载到内存中。
2. **WHERE age < 25**：会把表中数据进行过滤，取出符合条件的记录行，生成一张临时表。

![](assets/image-20260517-133949-959.png)

1. **GROUP BY dept**：会把上图的临时表分成若干临时表。

![](assets/image-20260517-134004-961.png)

查询的结果只有部门2和部门3才有符合条件的值，生成如上两图的临时表。接着执行 SELECT 后面的字段，SELECT 后面可以是表字段也可以是聚合函数。

1. **SELECT dept,COUNT(phone)**：这里 SELECT 的情况与是否存在 GROUP BY 有关。若是不存在，MySQL 直接按照内存中整列读取；若是存在则分别 SELECT 临时表的数据。

![](assets/image-20260517-134016-362.png)

1. **HAVING num >= 3**：过滤员工数小于等于3的部门。对于 WHERE 和 HAVING 都是进行过滤，那么这两者有什么不同呢？

- **第一点**：WHERE 后面只能对表字段进行过滤，不能使用聚合函数，而 HAVING 可以过滤表字段也可以使用聚合函数进行过滤。
- **第二点**：WHERE 是对执行 from User 操作后，加载表数据到内存后，WHERE 是对原生表的字段进行过滤，而 HAVING 是对 SELECT 后的字段进行过滤，也就是 WHERE 不能使用别名进行过滤。因为执行 WHERE 的时候，还没有 SELECT，还没有给字段赋予别名。

1. **ORDER BY num DESC,dept ASC**：执行排序操作，按员工数降序、部门升序排序。
2. **LIMIT 0,2**：取得前两个数据。

**优化器**

MySQL使用基于成本的查询优化器。它会尝试预测一个查询使用某种执行计划时的成本，并选择其中 成本最少的一个。 

**执行执行器** 

由优化器生成得执行计划，交由执行器进行执行，执行器调用存储引擎得接口，存储引擎获取数据并返回，结束整个查询得过程。

# 3. MySQL 更新数据的过程是怎样的？

**核心回答：** 更新流程涉及**Undo Log 生成→数据修改→Redo Log 写预提交→Binlog 写入→两阶段提交**的核心机制。

**原理分析：**

1. **连接验证及解析**：客户端与 MySQL Server 建立连接，发送语句给 MySQL Server，接收到后如果是查询语句会先去查询缓存中看，有的话就直接返回了（新版本的 MySQL 已经废除了查询缓存，命中率太低了），如果是缓存没有或者是非查询语句，会创建一个解析树，然后进行优化（解析器知道语句是要执行什么，会评估使用各种索引的代价，然后去使用索引，以及调节表的连接顺序），然后调用 InnoDB 引擎的接口来执行语句。
2. **写 undo log**：InnoDB 引擎首先开启事务，获得一个事务 ID（是递增的），根据执行的语句生成一个反向的语句（如果是 INSERT 会生成一条 DELETE 语句，如果 UPDATE 语句就会生成一个 UPDATE 成旧数据的语句），用于提交失败后回滚。将这条反向语句写入 undo log，得到回滚指针，并且更新这个数据行的回滚指针和事务 ID。事务提交后，Undo log 并不能立马被删除，而是放入待清理的链表，由 purge 线程判断是否有其他事务在使用 undo 段中表的上一个事务之前的版本信息，决定是否可以清理 undo log 的日志空间。简单的说就是看之前的事务是否提交成功，这个事务及之前的事务都提交成功了，这部分 undo log 才能删除。
3. **从索引中查找数据**：根据索引去 B+树中找到这一行数据（如果是普通索引，查到不符合条件的索引，会把所有数据查找出来；唯一性索引查到第一个数据就可以了）。
4. **更新数据**：判断数据页是否在内存中？
   - **数据页在内存中**：
     - 普通索引：直接更新内存中的数据页。
     - 唯一性索引：判断更新后是否会数据冲突（不能破坏索引的唯一性），不会的话就更新内存中的数据页。
   - **数据页不在内存中**：
     - 普通索引：将对数据页的更新操作记录到 change buffer，暂时不更新到磁盘。change buffer 会在空闲时异步更新到磁盘。
     - 唯一性索引：因为需要保证更新后的唯一性，所以不能延迟更新，必须把数据页从磁盘加载到内存，然后判断更新后是否会数据冲突，不会的话就更新数据页。
5. **写 redo log（prepare 状态）**：将对数据页的更改写入到 redo log，此时 redo log 中这条事务的状态为 prepare 状态。
6. **写 binlog（同时将 redo log 设置为 commit 状态）**：通知 MySQL server 已经更新操作写入到 redo log 了，随时可以提交，将执行的 SQL 写入到 binlog 日志，将 redo log 中这条事务的状态改成 commit 状态，事务提交成功。

**延展思考：**
两阶段提交是保证主从数据一致性的核心机制： **Prepare 阶段确保 Redo Log 已落盘** ， **Commit 阶段确保 Binlog 已同步** 。若崩溃发生在 Prepare 阶段后、Commit 阶段前，恢复时会检查 Binlog 是否完整来决定提交或回滚。生产环境中需配置 `innodb_flush_log_at_trx_commit` 和 `sync_binlog` 参数来平衡性能与可靠性。
