# 1. 数据库三范式是什么，怎么理解

三范式是关系型数据库设计的基本规范，逐级递进，核心目的是**减少数据冗余、消除插入/更新/删除异常**。

**第一范式（1NF）：字段不可再分**，每个字段必须是原子值，不能在一个字段里存多个值。比如"地址"字段如果同时存省、市、区，就违反1NF，应该拆成独立字段。

**第二范式（2NF）：非主键字段必须完全依赖主键，不能只依赖主键的一部分**。这主要针对联合主键的场景，比如订单明细表用(订单ID, 商品ID)做联合主键，如果"商品名称"只依赖商品ID，就属于部分依赖，应该拆到商品表中。

**第三范式（3NF）：非主键字段必须直接依赖主键，不能通过其他非主键字段间接依赖**，即消除传递依赖。比如员工表中有部门ID和部门名称，部门名称依赖部门ID，部门ID依赖员工ID，这就是传递依赖，应该把部门信息拆到单独的部门表。

实际项目中**经常会做反范式设计**，用适当的冗余换取查询性能，比如订单表里冗余商品名称避免JOIN，但前提是先满足范式再根据性能需要适度反范式。

# 2. MySQL开发有哪些规范和注意点

从索引、SQL写法、表设计三个维度来说：

**索引规范：**

- 单表索引不超过5个，单个索引字段不超过5个
- 联合索引遵循最左前缀，区分度高的字段放前面
- 字符串索引用前缀索引，长度不超过10字符
- 索引命名用idx\_字段名/uniq\_字段名

**SQL写法规范：**

- 禁止SELECT \*，只取需要的列
- WHERE条件类型必须与字段类型一致，避免隐式转换导致索引失效
- 避免在索引列上做运算和函数操作
- OR改写为IN()，IN值不超过200个且禁止子查询
- UNION不去重时用UNION ALL
- 用预编译语句防止SQL注入
- 大SQL拆成小SQL，一条SQL只能用一颗CPU核，简单SQL缓存命中率更高且减少锁表时间

**表设计规范：**

- 表必须有主键，推荐自增ID或雪花算法，禁止UUID
- 禁止使用外键，高并发下外键极易死锁
- 禁止使用存储过程和触发器，业务逻辑放应用层
- 字段尽量NOT NULL，NULL值影响索引和统计
- 使用DECIMAL存金额，禁止FLOAT/DOUBLE

# 3. 如何设计一个索引

**索引不是越多越好，能不加的尽量不加**，索引越多写操作越慢。设计步骤分四步：

**第一步，先看业务查询。** 找到最常用、最慢、过滤性最强的查询，比如SELECT \* FROM user WHERE age=? AND city=? AND create_time>?，这是设计索引的唯一依据。

**第二步，选择索引字段：**

- WHERE条件里的字段优先
- **区分度高的字段优先**（性别/状态不适合单独建索引，区分度公式count(distinct col)/count(\*)一般要求0.1以上）
- 小字段优先（int < varchar < text）
- 表必须有主键，推荐使用独立于业务的AUTO_INCREMENT列或全局ID生成器做主键，禁止使用多字段做联合主键
- 不使用UUID/MD5/HASH等函数生成的无规则值做主键
- 避免在索引里放函数和运算（[索引失效场景](./02%E7%B4%A2%E5%BC%95.md)第22题）
- **联合索引优于多个单值索引**
- 避免冗余和重复索引，例如创建了联合索引（A,B），又创建了A索引

**第三步，联合索引设计遵循最左匹配原则。** 核心规则是**等值查询放前面，范围查询放最后**。比如WHERE age=? AND city=? AND create_time > ?，最优索引是INDEX idx_age_city_time(age, city, create_time)。MySQL会一直向右匹配直到遇到范围查询(>、<、between、like)就停止匹配，所以范围列放最后。等值条件的顺序可以任意调整，优化器会自动优化。

**第四步，控制索引数量：**

- 单表索引不超过5个
- 联合索引尽量覆盖多个查询减少索引个数
- 已有(a)索引要加(a,b)索引时，直接修改原索引即可，不要新建

一句话总结：**数量要少、区分度要高、联合索引最左前缀、索引列不做运算**。

# 4. 如何设计主键

主键设计的核心原则：**自增、短小、独立于业务**。

**必须要有主键**，InnoDB没有显式主键时会自动选择唯一非空索引或隐式生成ROW_ID，性能更差。**禁止使用多字段联合主键**，增加索引体积。

**主键要尽量短**，因为所有普通索引的叶子节点都存储主键值，主键越长普通索引越庞大。**推荐使用AUTO_INCREMENT或雪花算法**生成趋势递增的主键，禁止使用UUID、MD5等无规则值。

**主键要独立于业务** ，避免业务变更导致主键调整。有分库分表需求时用雪花算法生成全局唯一ID，单机场景用AUTO_INCREMENT即可。

# 5. 主键使用自增ID还是UUID

**强烈推荐自增ID，UUID在生产环境下问题很多。**

**存储空间：** UUID是36字符的字符串，自增ID是8字节BIGINT，**UUID存储空间是自增ID的4-5倍**，而且所有二级索引都存主键值，空间膨胀严重。

**插入性能：** 自增ID顺序追加到B+树尾部，几乎无页分裂；**UUID随机无序，每次插入都可能触发页分裂**，大量数据移动导致性能骤降。实测在百万级数据量下，UUID插入速度可能只有自增ID的一半甚至更低。

**查询性能：** 自增主键数据物理连续，范围查询和分页查询走顺序IO；UUID数据离散分布，**大量随机IO，查询效率低**。

**可读性：** UUID无业务含义，调试和排查问题时不如自增ID直观。

UUID唯一的优势是全局唯一不依赖数据库生成，适合分库分表场景。但这种场景用雪花算法更好，**既有全局唯一性又能保证趋势递增**，兼顾了两者的优点。

# 6. 为什么主键最好递增

**递增主键让数据按顺序追加到B+树页尾，几乎不触发页分裂，插入性能最优；乱序主键会频繁页分裂，导致数据碎片化和性能骤降。**

InnoDB是聚簇索引，数据行按主键顺序物理存储在B+树叶子节点中，主键有序的核心原因：

- **减少页分裂、页合并**——有序主键自增写入时，数据永远追加在索引最后一页，不会往中间插入数据，大幅减少索引页分裂与合并操作，提升写入性能
- **索引结构更紧凑，查询更快**——有序主键索引排列整齐，磁盘读写顺序连续，磁盘顺序IO效率远高于随机IO，范围查询、分页查询速度更快
- **索引填充率高，节省磁盘空间**——有序插入让索引页面填充饱满，碎片少，存储空间利用率更高
- **主键索引查询效率稳定**——自增有序主键分布均匀，B+树层级稳定，查询耗时可控

乱序主键插入时，新数据要插入到已有页的中间位置，页满时触发**页分裂**——将一页拆成两页，大量数据移动，产生半满的碎片页，导致：

- 占用更多磁盘空间
- 数据存储不连续，后续全表扫描变慢
- 磁盘随机IO增多

另外，**普通索引叶子节点存储主键值而非物理地址**，主键越长普通索引越庞大，所以主键既要递增又要尽量短。推荐使用AUTO_INCREMENT或雪花算法，禁止UUID和MD5。

# 7. 如何优化SQL和索引

从SQL写法、索引使用、分页优化三个层面来说：

**SQL写法优化：**

- 禁止SELECT \*，只取需要的列避免无谓IO
- 大SQL拆成小SQL，一条SQL只能用一个CPU核，简单SQL缓存命中率更高且减少锁表时间
- 减少MySQL端运算，避免md5()、order by rand()等
- WHERE中OR改IN()且值不超过200个
- 不去重用UNION ALL代替UNION
- 用预编译语句防SQL注入
- WHERE条件类型必须与字段类型一致，避免隐式转换。比如code是varchar类型，code=10001不会走索引，code='10001'才会走索引

**索引使用优化：**

- 利用覆盖索引避免回表，利用索引下推减少回表次数
- WHERE中非等值条件（IN、BETWEEN、<、>）会导致联合索引后续字段失效 （[索引失效场景](./02%E7%B4%A2%E5%BC%95.md)第22题）
- 避免在索引列上做运算和函数操作，比如id+1>10001不走索引，id>10000才走

**分页优化：** 传统LIMIT 10000,10会扫描10010条再丢弃前10000条，偏移量越大越慢。**四种优化方式：**

- 游标分页：WHERE id>=上次最大id LIMIT 11，适合连续翻页场景
- 子查询优化：SELECT \* FROM table WHERE id>=(SELECT id FROM table LIMIT 10000,1) LIMIT 10，子查询走覆盖索引只查id不回表
- INNER JOIN优化：SELECT \* FROM table INNER JOIN (SELECT id FROM table LIMIT 10000,10) t USING(id)，先通过覆盖索引取出id再关联查询
- 程序端分批：先SELECT id FROM table LIMIT 10000,10取id，再SELECT \* FROM table WHERE id IN(...)

# 8. 什么是回表

**回表是指通过二级索引找到主键值后，再用主键到聚簇索引中查询完整行数据的过程。**

InnoDB的聚簇索引叶子节点存储完整的行数据，二级索引叶子节点只存储索引列值和主键值。当查询的列不完全在二级索引中时，就需要回表：先在二级索引B+树上找到对应的主键值，再拿主键值去聚簇索引B+树上查找完整行数据。

比如索引(name)，执行SELECT \* FROM user WHERE name='张三'，先在name索引树上找到主键id=5，再到聚簇索引上用id=5查到整行数据，这就是一次回表。**回表的代价是额外的随机IO**，每回表一次就是一次B+树从根到叶的查找，所以回表次数多时性能会很差。

**减少回表的方式就是覆盖索引和索引下推**，让查询在索引层面就能完成，避免回聚簇索引查数据。

# 9. 什么是覆盖索引

**覆盖索引是指查询所需的全部列都包含在索引中，SQL无需回表即可获取完整结果。**

比如索引(name, age)，执行SELECT name, age FROM user WHERE name='张三'，name和age都在索引中，直接从索引返回数据，不需要回聚簇索引。如果查SELECT name, age, sex FROM user WHERE name='张三'，sex不在索引中就必须回表。

**性能收益：避免了回表的额外IO，理论上可减少一半的磁盘访问**。覆盖索引是典型的空间换时间，但索引列过多会导致索引体积膨胀、写入开销增大，建议高频查询的SELECT列和WHERE列控制在3-4个以内。

常见利用覆盖索引的场景：

- COUNT统计时在count字段上加索引
- 列查询升级为联合索引避免回表
- 分页查询只查索引列再回表取数据

# 10. 什么是索引下推（ICP）

**索引下推（Index Condition Pushdown）是MySQL 5.6引入的优化，将WHERE条件的过滤下推到存储引擎层，在索引遍历过程中提前过滤不满足条件的记录，减少回表次数。**

比如索引(name, age)，查询WHERE name='张三' AND age>20。**没有ICP时**，存储引擎只按name条件找到所有'张三'的主键，全部回表后在Server层再过滤age>20。**有ICP时**，存储引擎在遍历索引时直接利用age>20过滤，只返回满足条件的主键回表，大幅减少回表次数。

ICP的前提条件：

- **只能用于二级索引**，聚簇索引不需要回表所以没意义
- 只能下推索引中包含的列，非索引列的过滤仍在Server层完成
- 适用于SELECT/UPDATE/DELETE语句

典型场景是(city, status)组合索引查某城市已激活用户，ICP可在索引层直接完成status过滤，不用把该城市所有用户都回表再过滤，实测可降低30%-50%查询耗时。

# 11. 什么是深度分页，怎么优化

**深度分页是指LIMIT偏移量很大时的分页查询，偏移量越大性能越差。**

MySQL的LIMIT m,n工作原理是先读取前m+n条记录，再丢弃前m条取后n条。比如SELECT \* FROM test WHERE val=4 LIMIT 300000,5，需要先通过val索引找到所有val=4的主键，回聚簇索引读300005条数据，再丢掉前300000条，**浪费了大量回表IO**。

**优化核心思路：减少回表次数，让偏移量只作用在索引上，不作用在聚簇索引上。**

四种优化方式：

- **游标分页**——WHERE id>=上次最大id LIMIT 11，适合连续翻页场景
- **子查询优化**——SELECT \* FROM table WHERE id>=(SELECT id FROM table LIMIT 100000,1) LIMIT 10，子查询走覆盖索引只查id不回表
- **INNER JOIN优化**——SELECT \* FROM table INNER JOIN (SELECT id FROM table LIMIT 100000,10) t USING(id)，先通过覆盖索引取出10条id再关联查询
- **程序端分批**——先SELECT id FROM table LIMIT 100000,10取id，再SELECT \* FROM table WHERE id IN(...)

# 12. 出现全表扫描的原因可能有哪些

全表扫描就是引擎放弃索引遍历整张表的每一条记录。常见原因：

**索引缺失或未使用：**

- 没有在WHERE和ORDER BY涉及的列上建索引
- 联合索引没有使用最左前缀列
- 复合索引必须用到第一个字段否则索引不生效

**索引失效：**

- WHERE中对字段做NULL值判断
- 使用!=或<>操作符
- 使用OR连接条件
- IN和NOT IN值过多
- 在索引列上做表达式运算如num/2=100应改为num=200
- 在索引列上使用函数如substring(name,1,3)='abc'
- LIKE以通配符开头如'%abc'

**隐式转换：** 字段类型和传入值类型不一致，比如varchar字段用整数值查询会导致隐式转换不走索引。

**索引选择性差：** 索引列大量数据重复时优化器可能放弃索引走全表扫描，比如性别字段male/female各一半，建索引也没用。用count(distinct col)/count(\*)计算区分度，低于0.1的列不适合建索引。

**其他原因：**

- 使用参数变量时优化器无法在编译时确定访问计划可能走全表
- 数据量小时优化器判断全表扫描比走索引更快
- 统计信息不准确导致优化器选错执行计划，可用ANALYZE TABLE更新

# 13. 如何验证索引是否失效

**用EXPLAIN查看执行计划，重点关注type和key两列。**

**type列判断访问类型：** const/eq_ref/ref/range说明索引生效；index是全索引扫描；ALL是全表扫描说明索引失效。业务SQL至少要达到range或ref级别，杜绝ALL。

**key列判断实际使用的索引：** 有值说明用到了索引；NULL说明没用到索引走了全表扫描。

**key_len列判断联合索引用了几列：** 值越小索引利用率越高，可以判断联合索引是否完全生效还是只用了部分列。

**Extra列补充判断：**

- Using index表示用了覆盖索引
- Using index condition表示用了索引下推
- Using where表示Server层额外过滤
- Using filesort表示额外排序
- Using temporary表示用了临时表

# 14. 如何分析SQL，执行计划中各项含义

**EXPLAIN是分析SQL执行计划的核心工具，各列含义如下：**

**id：** 查询中SELECT子句的执行顺序，id越大越先执行，id相同从上往下执行。

**select_type：** 查询类型：

- SIMPLE：简单查询无子查询无UNION
- PRIMARY：主查询
- SUBQUERY：SELECT或WHERE中的子查询
- DERIVED：FROM中的子查询（衍生表）
- UNION：UNION后面的查询
- UNION RESULT：UNION的结果

**type：** 访问类型，**性能从好到差：system > const > eq_ref > ref > range > index > ALL**

- const：主键/唯一索引等值匹配只查一次
- eq_ref：唯一索引关联每次匹配一行
- ref：普通二级索引等值匹配可能多行
- range：索引范围扫描如BETWEEN、<、>
- index：全索引扫描
- ALL：全表扫描

**业务SQL至少达到range/ref级别。**

**possible_keys：** 可能用到的索引，不一定实际使用。

**key：** 实际使用的索引，NULL表示没用到。

**key_len：** 索引使用的字节数，用于判断联合索引用了几列，越短越好代表索引利用率高。

**ref：** 与索引比较的列或常量。

**rows：** 估算需要扫描的行数。

**filtered：** 返回结果占扫描行数的百分比，越高过滤效率越好。

**Extra：**

- Using index：覆盖索引
- Using index condition：索引下推
- Using where：Server层过滤
- Using temporary：用了临时表常见于GROUP BY
- Using filesort：额外排序无法用索引完成
- Impossible where：条件永远不成立

# 15. MySQL的两种排序方式是什么

**MySQL有双路排序和单路排序两种方式，核心区别是排序时取多少数据进sort buffer。**

**双路排序（MySQL 4.1之前）：** 第一次扫描取出排序字段和行指针，排序后第二次回表取出SELECT需要的列。优点是sort buffer占用小，缺点是**两次IO且第二次是随机IO**，效率低。数据量大时会分block排序写入磁盘临时文件再合并。

**单路排序（MySQL 4.1之后默认）：** 一次性取出排序字段和SELECT需要的所有列进sort buffer排序，排序后直接返回结果无需第二次回表。优点是**只有一次IO**，缺点是sort buffer占用更大，如果SELECT的列很多，sort buffer装不下就要用磁盘临时文件，反而更慢。

**优化要点：**

- 避免SELECT \*减少sort buffer占用
- 适当增大sort_buffer_size
- 利用索引的有序性避免filesort，比如在(type, id)上建联合索引可以让WHERE type=5 ORDER BY id直接走索引不额外排序
- 多表排序时MySQL会先创建临时表再filesort，尽量避免

EXPLAIN中出现Using filesort就说明没走索引排序，需要额外排序操作，常见于WHERE和ORDER BY作用在不同列上。

# 16. count(\*)和count(1)有什么区别

**在InnoDB下count(\*)和count(1)没有区别，执行效果和性能完全一样**，都统计所有行数包括NULL值。

count(列名)则不同，**只统计该列非NULL的行数**，如果列中有NULL值结果会少。执行时InnoDB会选择最小的二级索引来遍历计数，因为二级索引比聚簇索引体积小扫描更快。

count(1)和count(*)的误区：早期MyISAM引擎下count(*)很快因为有存储行数，但InnoDB因为MVCC不同事务可能看到不同行数所以必须遍历计数。**两者在InnoDB下等价，优化器会做相同处理**，不存在count(1)比count(\*)快的说法。

性能排序：count(\*) ≈ count(1) > count(主键) > count(普通列)，因为普通列还要判断NULL值。

# 17. MySQL如何快速清空大表

**用TRUNCATE而不是DELETE，TRUNCATE是DDL语句不记录事务日志，速度极快。**

DELETE是DML语句逐行删除并记录binlog和redo log，大表DELETE会长时间持有锁、产生大量日志、事务回滚段暴涨，可能导致数据库卡死。

TRUNCATE直接释放数据页重建表结构，**不经过事务不记录行级日志**，瞬间完成。但TRUNCATE不能回滚、不能带WHERE条件、会重置AUTO_INCREMENT。

如果需要保留表结构但清除数据，TRUNCATE是最优选择。**线上清空大表还可以用RENAME+新建空表的方式：**

- 先RENAME旧表为临时表名
- 再CREATE同名新表
- 异步DROP旧表

这样避免长时间锁表影响线上业务。

# 18. drop和truncate的区别，为什么truncate清空更快

**DROP是删表（结构和数据全删），TRUNCATE是清数据（保留表结构），DELETE是逐行删数据。**

**TRUNCATE比DELETE快的原因：**

- TRUNCATE是DDL直接释放数据页重建表，不记录行级binlog
- 不触发触发器
- 不经过事务
- DELETE是DML逐行删除，每行都记录binlog和redo log，还要处理索引更新和触发器，大表极慢

**TRUNCATE比DROP更快吗？** 不是，DROP也是DDL直接删除表文件，速度也很快。但两者功能不同：DROP整张表都没了包括结构，TRUNCATE表还在只是数据清空。**如果要清数据保留表结构用TRUNCATE，如果表都不要了用DROP**，两者都是DDL速度都很快。

核心区别总结：

- **TRUNCATE**：不可回滚、不可WHERE、重置AUTO_INCREMENT、不触发触发器
- **DELETE**：可回滚、可WHERE、不重置AUTO_INCREMENT、触发触发器
- **DROP**：整表删除不可回滚

# 19. drop和rename哪个更快

**两者都是DDL操作，执行速度都很快，但场景完全不同。**

DROP是删除表文件释放空间，RENAME是修改表名只是修改数据字典中的元数据指针，**RENAME几乎瞬间完成**因为不涉及数据移动。

线上大表操作的经典方案就是用RENAME做无缝切换：

- 先CREATE新表
- 数据导入新表后，RENAME旧表为备份名、RENAME新表为正式表名，整个过程几乎无锁
- 之后再异步DROP旧表，DROP大表可能涉及文件IO耗时，但已不影响线上业务

所以不是谁更快的问题，**RENAME是零停机换表的利器，DROP是清理废弃表的手段**，线上场景通常组合使用：RENAME切换+异步DROP。

# 20. 批量向MySQL导入千万级数据，如何优化

核心思路是**减少IO次数、减少索引维护压力、减少事务开销**，三管齐下。

**合并SQL语句：** 一条INSERT插入多行数据代替多条单行INSERT。合并后binlog日志量减少，降低日志刷盘的数据量和频率，同时减少SQL解析次数和网络传输IO。比如INSERT INTO t VALUES(1,'a'),(2,'b'),(3,'c')比三条单行INSERT效率高得多。

**在事务中批量提交：** 不要每条INSERT都提交一次事务。MySQL每次INSERT内部都会创建事务再提交，批量用START TRANSACTION包裹后只在最后COMMIT一次，大幅减少事务创建开销。但**事务不能太大**，超过innodb_log_buffer_size会把数据刷盘导致效率下降。

**数据有序插入：** 插入记录按主键顺序排列。无序插入会导致B+树频繁分裂合并，消耗大量计算资源且产生大量磁盘IO；有序插入直接追加到索引末尾，定位效率高且索引调整最小。

**三种方式组合效果最佳：**

- 合并数据+事务在小数据量时提升明显
- 数据量超过千万时，如果数据无序，性能会急剧下降，因为超过innodb_buffer容量后每次索引定位都需要磁盘IO
- **合并数据+事务+有序数据**三者结合表现最好，有序数据让索引定位高效，避免频繁磁盘读写

**注意两点限制：**

- SQL语句有长度限制，由max_allowed_packet控制（默认1M），合并插入不能超限
- 事务大小要合理控制，事务过大影响执行效率，超过innodb_log_buffer_size会触发刷盘

# 21. ORDER BY如何优化，性能提升10倍的思路是什么

ORDER BY优化的核心是**让数据本身有序，跳过排序步骤**。

**ORDER BY的执行流程分两步：** 第一步根据WHERE条件走二级索引查找，回表聚簇索引取出需要的字段；第二步对原始数据按ORDER BY字段排序。排序优先在内存中进行，每个线程分配sort_buffer，数据量太大则借助磁盘临时文件。

**所以优化的关键就是避免第二步排序，两种手段：**

**联合索引优化——让索引天然有序：** 比如查询SELECT name,age,city FROM user WHERE city='北京' ORDER BY age LIMIT 5，如果只有city单列索引，找到所有city='北京'的记录后必须对age排序。**建立(city,age)联合索引后**，索引本身按city排序、相同city内按age排序，MySQL直接按索引顺序读取就是有序的，**完全跳过排序步骤**，性能提升可达10倍。

**参数优化：** 调整max_length_for_sort_data参数控制排序算法选择。当SELECT的列总长度不超过该值时用单路排序（一次性取出所有列进sort_buffer），超过则用双路排序（只取排序字段和主键，排序后再回表）。单路排序IO少但占内存大，双路排序IO多但内存占用小，根据场景调参。

实际优化中**联合索引是根本手段**，参数调优是辅助手段。EXPLAIN中出现Using filesort说明没走索引排序需要额外排序操作，优化目标就是消除Using filesort。

# 22. 百万级数据去重，该用DISTINCT还是GROUP BY

**推荐用GROUP BY，语义更清晰、灵活性更高。**

GROUP BY相比DISTINCT有三个优势：

- **语义更明确**——GROUP BY本身就是分组操作，去重只是分组的附带效果，代码意图一目了然
- **支持更复杂的处理**——GROUP BY可以配合HAVING对分组结果过滤，配合COUNT/SUM等聚合函数做运算，DISTINCT做不到
- **灵活性更高**——DISTINCT会对SELECT的所有字段去重，无法单独对某些字段去重而保留其他字段的不同值；GROUP BY可以精确指定分组字段，其他字段用聚合函数灵活处理

底层执行层面，MySQL优化器在处理DISTINCT和GROUP BY时**走的是相同的执行计划**，都会用到索引或临时表，性能上没有本质差异。所以选择依据不在性能而在功能需求——**简单去重两者都行，但凡有一点复杂逻辑就选GROUP BY**。

# 23. InnoDB单表建议不超过2000万行，为什么

这个数字来源于**B+树的层级分析**，2千万行大概是一个3层B+树能舒适容纳的数据量上限。

InnoDB一个页默认16KB，B+树每个非叶子节点存储的主键+指针大约占12字节，一个页约能存1170个指针。叶子节点每行数据假设约1KB，一个页存15行左右。**3层B+树：1170×1170×15≈2000万行**。

超过2000万行后B+树可能从3层变成4层，意味着：

- **走聚簇索引查询从3次IO变成4次IO**
- **走二级索引查询从6次IO变成8次IO**（二级索引查到主键后还要回聚簇索引，路径翻倍）

每次多一层索引就是多一次磁盘IO，性能下降明显。

当然以上分析是简化模型，实际情况中**根节点常驻内存、中间节点大概率也在buffer pool中**，所以3层B+树实际可能只需要1-2次磁盘IO。2000万是推荐值不是硬限制，最终取决于硬件条件和优化措施，但超过这个量级确实要考虑分库分表或归档方案。

# 24. 产生临时表的原因有哪些，如何解决

**EXPLAIN中出现Using temporary说明MySQL创建了临时表来处理查询，临时表可能存在内存或磁盘上，磁盘临时表性能很差。**

常见产生临时表的原因：

- **GROUP BY的列没有索引**——分组操作需要临时表来聚合，给GROUP BY列加索引可避免
- **GROUP BY和ORDER BY作用在不同列上**——无法用同一索引同时满足分组和排序，MySQL必须建临时表
- **UNION查询**——合并结果集需要临时表，不需要去重时用UNION ALL代替UNION
- **DISTINCT查询**——去重操作需要临时表，能改写为GROUP BY时优先GROUP BY
- **子查询结果作为派生表**——FROM中的子查询会生成临时表，尽量改写为JOIN
- **INSERT...SELECT同表**——MySQL为避免数据冲突会使用临时表

**解决临时表问题的思路：**

- 给GROUP BY和ORDER BY涉及的列建合适的联合索引，让分组和排序走索引而非临时表
- 避免GROUP BY和ORDER BY作用在不同列上，或用联合索引同时覆盖两者
- 用UNION ALL代替UNION，避免去重产生的临时表
- 子查询改写为JOIN，减少派生表
- 适当增大tmp_table_size和max_heap_table_size，让更多临时表留在内存而非磁盘
- 如果必须用临时表，确保数据量小且走内存临时表，避免磁盘临时表
