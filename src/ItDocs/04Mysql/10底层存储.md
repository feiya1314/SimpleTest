# 1、InnoDb逻辑存储结构

InnoDB 逻辑结构从上到下是：**表空间 → 段 → 区 → 页 → 行**

1. **表空间（Tablespace）**：最大的逻辑单位，所有数据都存在表空间里。
   - **分类**：
     - **系统表空间**：`ibdata1`（存储数据字典、undo、doublewrite 等）
     - **独立表空间**：`.ibd` 文件（`innodb_file_per_table=1`，默认开启），一个表一个 `.ibd` 文件
     - **临时表空间**：存储临时表、临时数据
     - **undo 表空间**：存储 undo log
   - **作用**：管理所有表、索引、数据、日志的存储容器。
2. **段（Segment）**：表空间的下一级，**一个索引 = 一个段**。
   - **常见段**：
     - **数据段**：聚簇索引（叶子节点）
     - **索引段**：二级索引（非叶子节点）
     - **回滚段**：存储 undo log
   - **一句话**：段 = 索引的存储容器。
3. **区（Extent）**：固定大小 = **1MB**，一个区 = 连续 **64 个页**（因为一页 16KB，16KB×64=1MB）。
   - **特点**：
     - 连续物理空间
     - 减少磁盘随机 I/O
     - 段不够空间时，一次申请 1\~多个区
   - **作用**：保证数据连续存储，提升顺序读写性能。
4. **页（Page）**：**InnoDB 最小的 I/O 单位**！！！默认大小 **16KB**。
   - **常见页类型**：
     - 数据页（index page）—— 存行数据
     - undo 页
     - 系统页
     - 事务数据页
     - 位图页
   - **关键点**：
     - MySQL 读写数据以页为单位
     - 即使只取 1 行数据，也要加载整页 16KB 到 Buffer Pool
     - 页是 InnoDB 最核心的结构
5. **行（Row）**：最小的逻辑单位，存储真实的行数据。
   - **一行包含**：
     - 真实字段数据
     - 隐藏字段：`trx_id`、`roll_pointer`
     - 行头信息

**三、最核心的关系**

- 1 个表空间 = N 个段
- 1 个段 = N 个区
- 1 个区 = 64 个连续页（1MB）
- 1 个页 = 16KB = N 行数据

**四、一句话总结**

InnoDB 逻辑结构：表空间包含段，段包含区，区包含页，页包含行；页是最小 I/O 单位，区是连续空间，段对应索引，表空间是最大容器。

# 2. 数据页是什么?

**核心回答：**

数据页是 **InnoDB 管理数据的最小单位**，默认 **16KB**，磁盘读写、Buffer Pool 缓存、B+ 树节点都以页为单位，一次最少读写一整页。

**数据页的内部结构（自上而下 7 部分）**

- **File Header（文件头，38 字节）**：页号、页类型、上一页/下一页页号（叶子双向链表靠它）、校验和、LSN
- **Page Header（页头，56 字节）**：记录数、槽数量、堆顶指针、Free 链表指针、最大事务 ID、所属索引 ID
- **Infimum + Supremum（最小最大记录，26 字节）**：两条虚拟边界记录，比页内任何主键都小/都大，作为页内单链表的头尾
- **User Records（用户记录区）**：真实行数据，按主键顺序通过单向链表串联
- **Free Space（空闲空间）**：尚未使用的部分，新插入记录从这里申请空间
- **Page Directory（页目录 / 槽 Slot）**：把记录分组（每组 4~8 条），每组最大记录的指针存入槽，用来在页内做二分查找
- **File Trailer（文件尾，8 字节）**：校验和 + LSN，与 File Header 对照，校验页是否完整，防止刷盘时写一半导致页损坏

**页内查找一行数据的过程**

- 通过 Page Directory 的槽数组做 **二分查找**，定位到目标记录所在的分组
- 在分组内沿单链表 **顺序遍历**，找到具体记录
- 这样既不用全页扫描，也不用每条记录都建索引，是空间和时间的折中

**为什么要以页为单位**

- **减少磁盘 I/O**：一次读 16KB，比一行一行读少几个数量级的 I/O 次数
- **对齐操作系统/磁盘块**：贴合文件系统块大小（通常 4KB 的整数倍），避免跨块读写
- **利用局部性原理**：取一行后，同页相邻行大概率紧接着用到，缓存命中率高
- **B+ 树天然契合**：每个 B+ 树节点正好就是一页，磁盘 I/O 次数 ≈ 树高

**页之间的关系**

- 同一层叶子页通过 File Header 中的 prev/next 串成 **双向链表**，支持范围扫描和顺序读
- 父页通过 **键 + 子页号** 指向子页，构成 B+ 树的层级结构

**一句话总结：**

数据页是 InnoDB 最小的 I/O 单位，默认 16KB，由文件头、页头、最小最大记录、用户记录、空闲空间、页目录、文件尾 7 部分组成，页内用槽做二分查找，页之间通过双向链表串联。

# 3. B+树是怎么存储的？

**核心回答：**

B+树按**页（Page）**为单位在磁盘上存储，每个节点对应一页，非叶子节点只存索引键，叶子节点存完整数据行，叶子之间用双向链表串联。

InnoDB 最小磁盘读写单位 = 数据页，默认固定 16KB，MySQL 不会单行读写数据，**一次只读写一整页**。

**存放内容**

- 叶子页：真实表数据、索引行
- 非叶子页：**索引键 + 子页地址（B + 树分支，**子节点指针
  ），每个节点对应一页，节点存储的是索引的值，**非叶子节点的键值，理解成「子节点的区间上限」**， 是「区间分界点」

**底层原理：**

1. InnoDB **每页默认16KB** ， **B+树的每个节点就是一页**
2. 非叶子节点（索引页）：**只存索引键值**+**子节点页号指针**，不存数据，子节点指针指向的是子节点，**子节点的值都是在当前指针所在区间值范围之间**，**指针越小、索引键越短，一个节点能存的键值对就越多 → 扇出越大 → 树越矮 → 查询越快。**
3. 叶子节点（数据页）：聚簇索引存完整数据行，二级索引存索引列+主键值
4. 同一层的叶子节点通过双向链表串联，支持范围查询和顺序扫描
5. 根节点常驻内存，查询时只需对非根节点做磁盘IO
6. 三层B+树（假设扇出1000） **可索引约10亿条记录** ，仅需3次IO

**B + 树层级：根层 (1 层)→中间层 (2 层)→叶子层 (3 层)扇出 = 单个节点最大子节点数**

- 1 层根节点：最多分出 1000 个子节点
- 2 层中间节点：总计 (1000 \\times 1000 = 1000000) 个节点
- 3 层叶子节点：总计 (1000 \\times 1000 \\times 1000 = \\boldsymbol{10亿}) 条数据

公式：总容量 = 扇出 ^ 树层数

指针越小、索引键越短，一个节点能存的键值对就越多 → 扇出越大 → 树越矮 → 查询越快。


**为什么要这么设计？**

- 非叶子节点不存数据：能让节点变得很小，单个节点能存更多的键值，扇出更大，树的高度更低，磁盘 IO 更少
- 叶子节点存有序数据：聚簇索引能直接拿到数据；二级索引能快速定位主键，再通过主键索引拿数据，兼顾了查找效率和数据一致性

**一句话总结：**

B+树以页为存储单位，非叶子只存键，叶子存数据，叶子链表串联，三层可索引10亿数据。

# 4. 树什么时候分支（页分裂）？

MySQL InnoDB 的 B + 树 “分支”（页分裂）核心触发条件：数据页（16KB）空间满了，放不下新记录，才会分裂。下面把条件、时机、过程讲清楚。

**一、分裂的本质**

InnoDB 用 16KB 的页（Page） 作为 B + 树节点：

- 叶子页：存完整行数据（聚簇索引）或索引列 + 主键（二级索引）
- 内部页（非叶子）：只存索引键 + 子页指针，不存数据

**分裂 = 页满 → 开新页 → 数据均分 → 父页加索引 → 递归向上**

**二、什么时候分裂（触发场景）**

1\. 插入新记录（最常见）

- 页已达到填充阈值（默认约 93.75%，16KB 用满 15KB 左右）
- 插入后记录数 / 空间超出页容量 → 分裂

2\. 更新导致记录变大

- 比如 VARCHAR 从短字符串更新为长字符串
- 原页剩余空间不够 → 分裂

3\. 无序插入（随机主键 / UUID）

- 自增主键：总是追加到最后一页，极少分裂（顺序写）
- 随机主键 / UUID：中间位置频繁插入，页快速填满且分裂更频繁、碎片多

**三、分裂的具体时机（判定规则）**

1. 叶子页：插入 / 更新后，空间不足（无法分配新记录）
2. 内部页：插入子页指针 + 索引键后，键数量 > 阶数 - 1（InnoDB 阶数由页大小决定，16KB 页通常允许几百个键）
3. 根节点：根页满了也会分裂，此时树高 + 1（B + 树长高仅发生在根分裂）

**四、分裂过程（极简版）**

1. 目标页满，分配新空白页
2. 找中间分裂点，原页留前半，后半移新页
3. 新记录插入原页或新页
4. 新页的最小键 + 页指针插入父页
5. 父页若满，重复 1–4，递归向上
6. 根分裂 → 新根产生，树高 + 1

**五、举例（直观理解）**

假设叶子页最多放 4 行： **[10,20,30,40]**

- **插入 50 → 满了，分裂：**
  - 原页：[10,20]
  - 新页：[30,40,50]
- **父页加键 30 指向新页**
- 父页若满，继续分裂父页

**六、关键结论**

- 不是按 “多少行” 固定分裂，而是按 “页空间是否满”
- 自增主键：分裂极少、性能好
- 随机主键：分裂频繁、碎片多、性能差
- 分裂是 B + 树维持平衡的必要机制，但会带来 I/O 开销

# 5. 什么是扇出？

**核心回答：**

**扇出是指B+树每个节点能容纳的子节点数量**，扇出越大，树越矮，查询IO越少。

**底层原理：**

1. 扇出 = 一个节点中索引键的数量，每个键对应一个子节点指针
2. 扇出取决于节点大小（一页16KB）和每个索引条目的大小
3. 扇出越大，每层能索引的数据量越多，整棵树的层数越少
4. 指针少（扇出小）的情况下要保存大量数据，只能增加树的高度，导致IO操作变多，查询性能变低

**一句话总结：**

扇出就是每个节点的分支数，扇出越大树越矮IO越少，扇出越小树越高查询越慢。


# 6. B+树在磁盘存储中的应用

**核心回答：**

B+ 树之所以是数据库索引的事实标准，是因为它的 **节点大小可以和磁盘页对齐**，每次磁盘 I/O 正好读一个节点，把磁盘随机读的代价压到接近树的高度。

**底层原理：**

- **磁盘 I/O 不按字节交换**：主存和磁盘之间是按扇区/块为单位（一个扇区 512 字节），文件系统层面通常是 **4KB、8KB、16KB、64KB**
- **InnoDB 把页大小定为 16KB**，B+ 树的每个节点正好对应一页，节点大小 = 一次 I/O 的物理单位
- **节点对齐磁盘块**的好处：
  - 不会出现「读半个节点又读半个节点」的情况
  - 一次 I/O 上来就是完整节点，节点内的查找全在内存里完成
  - 磁盘 I/O 次数严格等于 **B+ 树高度**

**为什么是 16KB 而不是 4KB 或 64KB**

- **太小（如 4KB）**：单节点能存的键变少，扇出降低，树变高，I/O 次数变多
- **太大（如 64KB）**：单次 I/O 时间变长，且页内大部分数据用不上，浪费 Buffer Pool 内存
- **16KB**：在扇出和单次 I/O 成本之间取得平衡，三层 B+ 树即可索引约 10 亿数据

**为什么不用其他树结构**

- **二叉树/红黑树/AVL**：树高 ≈ log₂(n)，10 亿数据要 30 层，每层一次 I/O 磁盘扛不住
- **B 树**：每个节点都存数据，节点变大，扇出变小，树变高，且不支持高效范围扫描
- **B+ 树**：非叶子只存键 + 子节点指针，扇出极大；叶子链表串联，范围扫描、顺序读极快

**B+ 树磁盘存储的关键特性**

- **节点大小固定**（16KB），与磁盘页对齐，I/O 次数 = 树高
- **非叶子节点不存数据**，单节点能放更多键，扇出大、树矮
- **叶子节点用双向链表串联**，叶子页物理上尽量连续分配（区/Extent），配合预读减少随机 I/O
- **根节点常驻 Buffer Pool**，几乎无 I/O；中间层也大概率缓存命中
- **真正落到磁盘的 I/O 通常只有 1~2 次**（叶子页），三层树查 10 亿数据也只要几次磁盘 I/O

**一句话总结：**

B+ 树通过把节点大小对齐磁盘页（16KB），让一次 I/O 正好读一个节点；非叶子只存键提升扇出降低树高，叶子链表串联支持范围扫描，整棵树的查询代价基本等于树的高度。

# 7. InnoDB 和 MyISAM 有什么区别和特点？

**核心回答：**

InnoDB 是 **MySQL 5.5 之后的默认存储引擎**，支持事务、行锁、外键、MVCC、崩溃恢复，面向 OLTP 场景；MyISAM 不支持事务、只有表锁，面向只读或读多写少场景，**MySQL 8.0 后系统表已全部改为 InnoDB，MyISAM 基本被淘汰**。

**核心区别**

- **事务支持**
  - InnoDB：支持 ACID 事务，依赖 redo log + undo log
  - MyISAM：不支持事务，写操作不可回滚
- **锁粒度**
  - InnoDB：支持 **行锁 + 表锁**，并发写性能好
  - MyISAM：**仅表锁**，一写阻塞所有读写
- **MVCC**
  - InnoDB：支持，读写不阻塞，并发读高
  - MyISAM：不支持
- **外键**
  - InnoDB：支持外键约束
  - MyISAM：不支持
- **崩溃恢复**
  - InnoDB：支持，通过 redo log 重放未刷盘脏页
  - MyISAM：不支持，宕机后数据可能损坏，需要手动 REPAIR TABLE
- **索引结构**
  - InnoDB：**聚簇索引**，数据和主键索引存在同一棵 B+ 树（叶子节点 = 完整行数据）；二级索引叶子存主键值，需要回表
  - MyISAM：**非聚簇索引**，数据和索引分离，索引叶子节点存的是数据行的物理地址（行号/偏移），无回表概念
- **count(\*)**
  - InnoDB：需要扫描统计（无 where 时优化器会选最小索引）
  - MyISAM：维护行数计数器，**O(1) 直接返回**（仅在无 where 时）
- **存储文件**
  - InnoDB：`.ibd`（数据 + 索引）+ `.frm`（表结构，8.0 后并入数据字典）
  - MyISAM：`.MYD`（数据）+ `.MYI`（索引）+ `.frm`，**索引和数据物理分离**
- **缓存**
  - InnoDB：Buffer Pool 缓存数据页 + 索引页
  - MyISAM：Key Buffer **只缓存索引**，数据靠操作系统 Page Cache
- **全文索引**
  - InnoDB：5.6 之后才支持
  - MyISAM：一直支持

**InnoDB 特点**

- **聚簇索引**：主键决定数据物理排列，主键查询非常快，**主键尽量自增**避免页分裂
- **行级锁 + MVCC**：高并发下读写互不阻塞
- **崩溃恢复**：依赖 WAL（redo log），断电也能恢复到一致状态
- **支持外键**：能在数据库层做强一致性约束
- 适合：**OLTP、高并发写、对一致性要求高的业务系统**

**MyISAM 特点**

- **表锁**：写并发差，但纯读场景吞吐高
- **索引数据分离**：索引文件小，缓存命中率高
- **count(\*) 快**：维护行计数器，全表 count 是 O(1)
- **不支持事务和崩溃恢复**：宕机数据可能损坏
- 适合：**只读静态表、归档数据、日志类只追加场景**

**怎么选**

- 默认 **InnoDB**（事务、并发、崩溃恢复都需要）
- 历史只读表/字典表才考虑 MyISAM
- 现在生产环境基本一律用 InnoDB

**一句话总结：**

InnoDB 支持事务、行锁、外键、MVCC、崩溃恢复，是聚簇索引，适合 OLTP；MyISAM 不支持事务、只有表锁，是非聚簇索引，count(\*) 快但宕机会损坏数据，现在基本只用 InnoDB。

# 8. 介绍下 MySQL 的 WAL、LSN、Checkpoint 和作用

**核心回答：**

WAL 是「**先写日志再写数据**」的设计思想，LSN 是 redo log 的全局递增版本号，Checkpoint 是把脏页刷盘并推进 redo log 起点的过程，三者配合实现了 **InnoDB 的高性能写入 + 崩溃恢复**。

**1. WAL（Write-Ahead Logging，预写日志）**

- **含义**：修改数据前，必须先把对应的 **redo log 写入磁盘**，数据页可以延后再异步刷盘
- **解决什么问题**：
  - 数据页是 16KB **随机写**，太慢
  - redo log 是 **顺序追加写**，非常快
  - 用顺序 I/O 的日志保证持久性，把随机 I/O 的脏页延后批量刷
- **流程**：
  - 事务修改 Buffer Pool 中的数据页 → 页变脏
  - 同时把这次修改写入 redo log buffer
  - **事务提交时把 redo log 刷盘（fsync）**，此时事务可视为持久化成功
  - 脏页之后由后台线程异步刷盘
- **好处**：
  - 提升性能：写日志比写数据页快几个数量级
  - 保证持久性：宕机后用 redo log 重做未刷盘的脏页
  - 减少随机 I/O：脏页可合并、批量刷

**2. LSN（Log Sequence Number，日志序列号）**

- **含义**：redo log 中每条日志的 **全局单调递增编号**，本质是 8 字节整数，标识日志在 redo log 中的字节偏移
- **作用**：
  - 标记 redo log 写入位置
  - 每个数据页头部记录了这页对应的最新 LSN（**page_lsn**）
  - 崩溃恢复时，通过对比页 LSN 和日志 LSN 判断哪些日志要重做
  - 是 Checkpoint 推进的依据
- **几个关键 LSN**：
  - **redo_lsn**：当前 redo log 写到了哪
  - **flushed_lsn**：redo log 已经刷盘到的位置
  - **checkpoint_lsn**：检查点位置，之前的 redo log 对应的脏页都已刷盘，可以释放
  - **page_lsn**：每个数据页头记录的最近一次修改对应的 LSN

**3. Checkpoint（检查点）**

- **含义**：把 Buffer Pool 中的脏页 **刷到数据文件**，并推进 checkpoint_lsn 的过程
- **为什么需要**：
  - redo log 是 **固定大小的环形文件**，写满后会循环覆盖
  - 必须保证「被覆盖的日志对应的脏页已经刷盘」，否则数据会丢
  - 崩溃恢复要从 checkpoint_lsn 开始重放，太老恢复时间会很长
- **触发时机**：
  - redo log 快写满（强制 Sharp Checkpoint）
  - 脏页比例过高（`innodb_max_dirty_pages_pct` 默认 75%）
  - 后台线程定期触发（Fuzzy Checkpoint）
  - 数据库正常关闭（Sharp Checkpoint，全部脏页刷盘）
- **类型**：
  - **Sharp Checkpoint**：把所有脏页一次性刷盘，关库时使用，会卡顿
  - **Fuzzy Checkpoint**：在线运行时使用，每次只刷一部分脏页，平滑推进
- **过程**：
  - 选择一批旧脏页（按 LSN 升序）
  - 把这些脏页写入数据文件（`.ibd`）
  - 推进 checkpoint_lsn 到这些脏页中最小 LSN 的位置
  - checkpoint_lsn 之前的 redo log 空间可以被复用

**三者的关系**

- **WAL 是设计原则**：先日志后数据
- **LSN 是度量单位**：标记日志和数据页的版本进度
- **Checkpoint 是机制**：推动脏页落盘、释放 redo log 空间、缩短崩溃恢复时间

**崩溃恢复流程（串起来看）**

- 启动时读取 **checkpoint_lsn**
- 从 checkpoint_lsn 开始扫描 redo log
- 对每条日志找到对应数据页，比较 page_lsn 和日志 LSN：
  - 页 LSN ≥ 日志 LSN：说明已刷盘，跳过
  - 页 LSN < 日志 LSN：重放这条日志
- redo 重放完后，再用 undo log 回滚未提交事务

**一句话总结：**

WAL 是「先写日志后写数据页」的核心思想，LSN 是日志和数据页的版本号，Checkpoint 通过把脏页刷盘并推进 LSN 起点来释放 redo log 空间、缩短崩溃恢复时间，三者一起保证了 InnoDB 又快又可靠。
