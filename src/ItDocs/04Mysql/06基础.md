# 1. 介绍下几种 Join？

常见的 Join 类型有三种：
- **left join**：保留左表所有数据，右表没有匹配的行设为 null
- **right join**：保留右表所有数据，左表没有匹配的行设为 null
- **inner join**：取左右表的交集，只保留两表都匹配的行

MySQL 在执行 Join 时有三种算法：

**Simple Nested-Loop Join（SNL）**，最基础的嵌套循环连接，本质就是一个双层 for 循环，外层表逐行与内层表所有行进行比较，匹配次数 = 外层表行数 × 内层表行数，性能最差，MySQL 实际上不会选择这个算法。

**Index Nested-Loop Join（INL）**，当被驱动表的 Join 字段有索引时使用。外层表的每行直接通过索引匹配内层表，不需要全表扫描，匹配次数从 外层表行数 × 内层表行数 降为 **外层表行数 × 索引树高度**，性能大幅提升。注意走的是二级索引，如果需要非索引列还要**回表**查询。

**Block Nested-Loop Join（BNL）**，当 Join 字段没有索引时使用。核心优化是引入了 **Join Buffer**，一次性缓存外层表的多条数据，然后批量与内层表匹配，减少了内层表的扫描次数。Join Buffer 默认大小 256K，如果 Buffer 足够大能放下驱动表所有数据，被驱动表只需扫描一次。

# 2. Exists 和 In 的区别？

**in** 是先执行子查询，再把结果作为条件过滤主表；**exists** 是先查主表，再对每一行执行子查询判断是否存在。

选择原则是**小表驱动大表**：
- 主表大、子表小且有索引 → 用 **in**，因为子查询结果集小，in 效率高
- 主表小、子表大且有索引 → 用 **exists**，因为外层循环次数少，每趟走索引也快

另外 exists 子查询不返回实际数据，只返回 **True 或 False**。

对于 not in 和 not exists：**not in** 内外表都会全表扫描，无法用索引；**not exists** 子查询仍能走索引，所以无论哪个表大，**not exists 都比 not in 快**。

# 3. Union 与 Union All 的区别？

- **union**：对两个结果集取并集，**去除重复行**，并按默认规则排序
- **union all**：对两个结果集取并集，**保留重复行**，不排序

从效率上看，union all 比 union 快很多，因为 union 多了一步**去重和排序**操作。所以如果确认两个结果集没有重复数据，优先使用 **union all**。

# 4. Group By 和 Distinct 的区别？

两者都能去重，但处理逻辑不同：**distinct** 是先获取结果集再去除重复记录，**group by** 是先按 KEY 分组再返回结果。

在相同语义下，两者执行效率没有明显差异，都可以利用索引。但更推荐使用 **group by**，原因：
- 语义更清晰、更灵活
- 支持更复杂的处理，比如通过 **having** 过滤分组、配合**聚合函数**做统计运算
- distinct 会对所有字段生效，复合业务场景下灵活性不如 group by

# 5. 什么是存储过程？有哪些优缺点？

存储过程是一组**预编译的 SQL 语句集合**，封装成类似方法的代码块，实现特定功能（增删改查），取一个名字后可以反复调用。

优点：
- **预编译执行**，效率比逐条执行 SQL 高
- 在服务器端运行，**减少客户端压力**
- 支持模块化设计，创建一次可多次调用，复用性强
- 一次调用替代大量 SQL 语句，**减少网络传输量**
- 可通过权限控制保障数据安全

缺点：
- **调试困难**，不像应用代码可以方便断点调试
- **可移植性差**，不同数据库的存储过程语法差异大
- 存在**重新编译问题**，表结构变更可能导致存储过程失效需要重新编译
