# 1. MySQL 有哪些锁

**核心回答：** MySQL 锁分三大类：**全局锁、表级锁、行级锁**，其中 InnoDB 重点是行锁 + 衍生的间隙锁/临键锁。表级锁又包含普通表锁（S/X）、MDL 元数据锁、意向锁（IS/IX）；行级锁包含记录锁、间隙锁、临键锁。

**底层原理：**

1. **全局锁**：锁定整个数据库实例，所有库表只读，一般用于**全库逻辑备份**。语法 `FLUSH TABLES WITH READ LOCK` 加锁，`UNLOCK TABLES` 解锁。加锁后**所有写操作阻塞**（增删改、DDL），读正常执行。长时间持有会阻塞业务写入，线上慎用。替代方案：`mysqldump --single-transaction`（InnoDB 靠 MVCC 热备，不用全局锁）。全局锁是 **Server 层**实现，不属于引擎层锁。
2. **表级锁**：粒度粗、开销小、并发低，MyISAM 默认使用，InnoDB 也支持。分三种：
   - **表共享读锁（S 锁）**：`LOCK TABLES 表名 READ`，多个会话可**同时加 S 锁**互不阻塞；但持有 S 锁时，其他会话**加不了写锁、不能写数据**。
   - **表排他写锁（X 锁）**：`LOCK TABLES 表名 WRITE`，**独占锁**，一个会话加 X 锁后，其他会话读、写全部阻塞。
   - **MDL 元数据锁**：**默认隐式加锁**，无需手动操作。保护表结构，防止**查询/修改数据时表被 DDL 篡改**。执行 SELECT/DML 自动加 **MDL 读锁**（共享兼容），执行 ALTER TABLE 等 DDL 加 **MDL 写锁**（排他）。**经典问题**：长事务持有 MDL 读锁 → DDL 拿不到写锁阻塞 → 后续所有读写排队，引发库卡死。排查用 `show processlist`。
3. **行级锁**：**仅 InnoDB 支持**，锁定表中**单行/多行**，基于索引实现（无索引会降级为表锁）。行锁只在**当前读**场景触发：`SELECT ... FOR UPDATE / LOCK IN SHARE MODE`、`UPDATE`、`DELETE`。
   - **行共享锁（S 锁）**：`SELECT * FROM t WHERE id=1 LOCK IN SHARE MODE`，多个事务可同时对同一行加 S 锁；持有 S 锁时，其他事务**不能对该行加 X 锁、不能修改数据**。
   - **行排他锁（X 锁）**：`SELECT * FROM t WHERE id=1 FOR UPDATE`、`UPDATE`、`DELETE` 自动加 X 锁。同一行**只能有一个事务持有 X 锁**，阻塞其他事务的 S 锁、X 锁、数据修改。

   行锁兼容性矩阵：

   | 当前锁\\请求锁 | S锁 | X锁 |
| --- | --- | --- |
| S锁 | 兼容 | 阻塞 |
| X锁 | 阻塞 | 阻塞 |
4. **InnoDB 特殊锁（RR 隔离级别重点）**：基于行锁延伸，**临键锁（Next-Key Lock）、间隙锁（Gap Lock）、记录锁（Record Lock）**，三者是递进关系，**解决幻读核心**。
   - **记录锁 Record Lock**：只锁**存在的索引行**，精准锁定单条记录。场景：等值查询命中**唯一索引/主键**，查询到数据时触发。阻止其他事务修改、删除该行。
   - **间隙锁 Gap Lock**：锁定**索引记录之间的空隙**（不锁记录本身）。场景：等值查询**未命中数据**、范围查询，RR 级别自动触发。核心作用：**阻止其他事务在间隙内插入新数据，防范幻读**。间隙锁**之间互相兼容**，多个事务可同时持有同一间隙的锁。
   - **临键锁 Next-Key Lock**（默认锁）：InnoDB RR 级别**默认行锁算法**，是 **记录锁 + 间隙锁** 的结合体。锁定范围：**当前记录 + 左侧间隙**，左闭右开区间。触发场景：范围查询、普通索引等值查询命中数据。彻底封堵插入入口，最大程度防止幻读。

   **三者转化规则（面试高频）**：
   1. 主键/唯一索引 + 等值查询 + 命中数据 → **降级为记录锁**
   2. 主键/唯一索引 + 等值查询 + 未命中数据 → **降级为间隙锁**
   3. 普通索引 / 范围查询 → **使用临键锁**
5. **意向锁（表级别辅助锁）**：InnoDB 为了**快速判断表锁与行锁是否冲突**，引入的表级意向锁，**不阻塞读写**，仅做标记。分为两种：**意向共享锁 IS**（事务要对表中某行加 S 锁，先在表上加 IS 锁）、**意向排他锁 IX**（事务要对表中某行加 X 锁，先在表上加 IX 锁）。意向锁之间**完全兼容**；意向锁和普通表锁互斥：表已存在 IS/IX → 其他会话无法加表写锁(X)；表已加表读锁(S)/表写锁(X) → 其他会话无法加 IS/IX。

**各类锁对比速记：**

| 锁类型 | 粒度 | 并发 | 适用引擎 | 核心场景 |
| --- | --- | --- | --- | --- |
| 全局锁 | 整实例 | 极低 | 所有引擎 | 全库备份 |
| 表级锁(S/X) | 整张表 | 低 | MyISAM/InnoDB | 临时管控、无索引场景 |
| MDL锁 | 整张表 | 中 | InnoDB | 防 DDL 篡改表结构 |
| 行锁(S/X) | 单行/多行 | 高 | InnoDB | 常规 DML、当前读 |
| 意向锁(IS/IX) | 整张表 | 不阻塞 | InnoDB | 辅助判断行锁与表锁冲突 |
| 记录锁 | 单行 | 高 | InnoDB(RR) | 主键/唯一索引等值命中 |
| 间隙锁 | 索引间隙 | 高 | InnoDB(RR) | 防幻读、等值未命中 |
| 临键锁 | 记录+间隙 | 高 | InnoDB(RR默认) | 范围查询、普通索引 |

**一句话总结：**
MySQL 锁分全局锁、表级锁（S/X/MDL/意向锁）、行级锁（记录锁/间隙锁/临键锁），InnoDB 行锁依赖索引无索引降级为表锁，RR 级别靠临键锁+间隙锁解决幻读，意向锁是标记锁仅加速冲突检测。

# 2. 各种锁是如何实现的

**核心回答：** 全局锁和表级锁（S/X/MDL）在 **Server 层**实现，靠标记位和队列管控；意向锁和行级锁在 **InnoDB 引擎层**实现，意向锁是表级标记锁，行锁依托索引存储在索引行头与内存锁链表中，间隙锁/临键锁作用于索引区间。

**底层原理：**

1. **全局锁实现**：实例内部维护一个**全局状态标识**，执行 `FLUSH TABLES WITH READ LOCK` 时将该标记置为**全局只读**。标记生效后，所有写请求（DML/DDL）检测到该状态直接进入等待队列，读请求正常执行。释放靠 `UNLOCK TABLES`、客户端断开或实例重启。**Server 层实现**，所有存储引擎共用。
2. **普通表 S/X 锁实现**：每张表在 Server 层维护**表锁结构体**，记录锁类型(S/X)、持有会话、等待队列。多会话可共存 S 锁；X 锁独占互斥所有 S/X 锁。手动 `LOCK TABLES` 加锁，`UNLOCK TABLES`、会话断开或新锁请求覆盖时释放。**Server 层实现**。
3. **MDL 元数据锁实现**：内存维护**MDL 锁队列**，按「库+表」维度管理，分 **MDL 读锁(共享)**、**MDL 写锁(排他)**。隐式加锁，无需手动：SELECT/DML 加 MDL 读锁，ALTER/DROP/CREATE 加 MDL 写锁。表上存在 MDL 读锁 → MDL 写锁排队等待；已有 MDL 写锁 → 所有读写请求排队。**MDL 读锁在事务结束（commit/rollback）才释放**，DDL 执行完立即释放 MDL 写锁。长事务不提交 → MDL 读锁不释放 → DDL 卡死是经典问题。
4. **意向锁 IS/IX 实现**：每张 InnoDB 表在内存中维护**意向锁标记位**（IS/IX）。加锁流程：事务要给某行加行 S 锁 → 先给表加 IS 锁；要给某行加行 X 锁 → 先给表加 IX 锁。IS、IX 之间**全部兼容**；表已存在 IS/IX 禁止再加表级 X 写锁；表已有表级 S/X 锁禁止再加 IS/IX。事务提交/回滚时连同行锁一起释放。**InnoDB 引擎层实现，纯标记锁，不阻塞正常读写。**
5. **行锁实现**：行锁**依托索引实现**，锁信息记录在**索引行头 + 内存锁链表**中；无索引时行锁退化为表锁。
   - **存储载体**：每条索引记录的行头信息中保存锁标记、持有事务 ID；InnoDB 内存维护**锁等待链表**，记录持有锁的事务、等待锁的事务、锁类型、锁定索引记录。
   - **加锁场景**：仅当前读触发（SELECT ... FOR UPDATE/LOCK IN SHARE MODE、UPDATE、DELETE）。
   - **加锁与冲突检测**：遍历索引找到目标记录 → 检查行头锁状态 → 无锁则加 S/X 行锁 → 已存在互斥锁则进入锁等待队列 → 超时触发 `lock_wait_timeout` 抛异常。
   - **释放时机**：**事务提交/回滚统一释放**，不会中途释放。
   - **关键细节**：InnoDB 行锁是**锁住索引而非记录**，使用主键索引时锁住主键索引；使用非主键索引时**先锁住非主键索引，再锁定主键索引**。
6. **间隙锁/临键锁实现**：本质是**行锁的范围变种**，作用于**索引区间**。锁信息记录在**索引页的元数据 + 全局锁链表**中，标记锁定的索引范围。
   - **临键锁**（RR 默认算法）：本质是 `记录锁 + 左侧间隙锁`，锁定**左闭右开**的索引区间。普通索引查询、范围查询触发。区间内不能新增、修改、删除数据，从根源阻挡幻读。
   - **降级规则**：主键/唯一索引 + 等值查询 + 命中数据 → 降级为纯记录锁；主键/唯一索引 + 等值查询 + 未命中数据 → 降级为纯间隙锁。
   - **间隙锁特殊规则**：间隙锁互相兼容，多个事务可同时锁定同一间隙；仅作用于 **RR 隔离级别**，RC 级别关闭间隙/临键锁。
   - **释放**：和行锁一致，事务结束才释放。

**分层总结：**

- **Server 层实现**：全局锁、普通表 S/X 锁、MDL 元数据锁
- **InnoDB 引擎层实现**：意向锁(IS/IX)、行 S/X 锁、记录锁、间隙锁、临键锁

**一句话总结：**
全局锁/表锁/MDL 是 Server 层标记位+队列实现；意向锁是 InnoDB 表级标记加速冲突判断；行锁依托索引存在行头与内存链表，无索引降级表锁；间隙/临键锁是 RR 级别索引区间锁，由临键锁按需降级，用于防幻读。

# 3. 一致性非锁定读和锁定读是什么？有什么区别

**核心回答：** 一致性非锁定读是**普通 SELECT，靠 MVCC 读历史版本快照，不加锁**；锁定读是**加锁读（FOR UPDATE / LOCK IN SHARE MODE），读最新数据，加行锁**。核心区别在于**是否加锁**和**读取数据版本**。

**底层原理：**

1. **一致性非锁定读（快照读）**：InnoDB 用 **MVCC 多版本机制**提供查询在某个时间点的快照。**RC 隔离级别**下，同一事务中每次一致性读都读到自己的刷新快照版本；**RR 隔离级别**下，同一事务中所有一致性读都读的是事务中第一次读到的快照。一致性读不会给访问的表加任何形式的锁，因此其他事务可以同时并发修改数据。**普通 SELECT 就是一致性非锁定读**，不加锁、读 undo log 历史版本、靠 MVCC 保证一致性。
2. **锁定读（当前读）**：在一个事务中，标准 SELECT 不加锁，但有两种锁定读方式：
   - `SELECT ... LOCK IN SHARE MODE`：给记录加**共享锁（S 锁）**，其他事务只能读不能修改，直到当前事务提交。
   - `SELECT ... FOR UPDATE`：给索引记录加**排他锁（X 锁）**，加锁情况与 UPDATE 一样，其他事务不可读写。
   - 锁定读读取的是**最新已提交数据**，不是快照版本。
3. **核心区别对比**：

   | 维度 | 一致性非锁定读 | 锁定读 |
| --- | --- | --- |
| 加锁 | 不加锁 | 加 S 锁或 X 锁 |
| 读取版本 | MVCC 历史快照 | 最新已提交数据 |
| 并发影响 | 不阻塞其他事务 | 可能阻塞其他事务 |
| 适用场景 | 普通查询 | 需保证数据实时性和一致性 |

**一句话总结：**
一致性非锁定读是普通 SELECT 不加锁读 MVCC 快照，锁定读是 FOR UPDATE/LOCK IN SHARE MODE 加锁读最新数据，前者高并发、后者保一致。

# 4. 普通读写会加锁吗？如何加锁？

**核心回答：** 普通 SELECT **不加锁**（快照读）；DML（UPDATE/DELETE/INSERT）**自动加隐式 X 锁**；DDL 自动加 **MDL 锁**。手动加锁通过 `LOCK IN SHARE MODE`（加 S 锁）和 `FOR UPDATE`（加 X 锁），只在事务内才生效。

**底层原理：**

1. **隐式锁 vs 显式锁**：
   - **隐式锁**：数据库管理系统自动为数据加锁、解锁，无需手动干预。DML 操作（INSERT/UPDATE/DELETE）InnoDB 自动给涉及数据集加排他锁（X 锁）；DDL 操作（CREATE/ALTER/DROP）自动加 MDL 锁。
   - **显式锁**：手动加锁，通过 `SELECT ... FOR UPDATE` 或 `LOCK IN SHARE MODE` 显式加锁。需注意手动加锁后若忘记提交事务，锁会一直持有不释放。
2. **手动加锁方式**：
   - `SELECT ... LOCK IN SHARE MODE`：加**共享锁（S 锁）**，其他事务可读不可更新。
   - `SELECT ... FOR UPDATE`：加**排他锁（X 锁）**，其他事务不可读写。
   - 这两种语句**只有在事务之中才能生效**，不在事务中加锁无效。
3. **是否需要手动加锁？**取决于隔离级别：
   - **RR 和 Serializable**：不需要手动加锁。RR 靠 MVCC + Next-Key Lock 自动保证一致性，其他会话无法获取事务执行中的数据；Serializable 更是所有操作串行执行。在这两个级别下手动加锁意义不大，InnoDB 已自动完成保护。
   - **RC 和 Read Uncommitted**：可能需要手动加锁。这两个级别可能出现脏读、不可重复读，必要时需通过 `FOR UPDATE / LOCK IN SHARE MODE` 手动加锁保证数据一致性。
4. **InnoDB 锁算法关键规则**：

   在默认 RR 隔离级别下，**普通 SELECT 用一致性读不加锁**；锁定读（FOR UPDATE / LOCK IN SHARE MODE）、UPDATE、DELETE 需要加锁，加什么锁视情况而定：
   - 对**唯一索引**使用**唯一检索条件**（等值查询命中）→ 只需加**记录锁（Record Lock）**，锁定索引记录即可；
   - **没有使用唯一索引**作为检索条件，或者用到**索引范围扫描** → 使用**间隙锁（Gap Lock）或临键锁（Next-Key Lock）**，阻塞其他会话向这个范围内的间隙插入数据。

   其他关键规则：
   - 不通过索引条件查询时，InnoDB 会**锁定表中所有记录**（降级为表锁），WHERE 条件字段应加索引。
   - InnoDB 通过索引实现行锁，不是锁住记录。操作两条不同记录拥有相同索引值时，也会因行锁被锁而发生等待。
   - 使用主键索引时 InnoDB 锁住主键索引；使用非主键索引时，**先锁住非主键索引，再锁定主键索引**。
   - 辅助索引不仅锁住辅助索引值所在范围，还会对其下一键值加 Gap Lock。
   - 当查询的索引是唯一索引时，InnoDB会将 **Next-Key Lock降级为Record Lock，** 即只锁住索引本身，而不是范围。
   - InnoDB 使用 Next-Key Lock 机制避免幻读问题。
   - **RR 级别**采用 Next-Key Lock 加锁，**RC 级别**仅采用 Record Lock 加锁。

**一句话总结：**
普通 SELECT 不加锁，DML 自动加隐式 X 锁，手动加锁靠 LOCK IN SHARE MODE / FOR UPDATE 且必须在事务内；RR/Serializable 不需手动加锁，RC 可能需要；无索引查询锁全表，务必确保走索引。

# 5. MVCC 是什么？用途、实现原理、优缺点是什么？

**核心回答：** MVCC 是**多版本并发控制**，不是锁，而是一种**用版本链替代加锁**的并发控制机制。核心用途是让**读操作不加锁**就能读到一致性数据，解决读写互斥问题。通过**隐藏列 + undo log 版本链 + ReadView** 实现，只在 **RC 和 RR** 隔离级别下工作。

**底层原理：**

1. **核心思想**：传统锁机制下读和写互斥，并发性能差。MVCC 让读操作通过版本链读取历史数据，**读写不冲突**，大幅提升并发性能。MVCC 和锁机制是**互补关系**：快照读靠 MVCC 不加锁，当前读靠锁机制加锁互斥。

2. **实现原理**（三个核心组件配合）：

   - **隐藏列**：每行记录两个隐藏列——**trx_id**（最近修改该行的事务 ID）和 **roll_pointer**（指向 undo log 中该行的上一个版本）。
   - **undo log 版本链**：每次修改都在 undo log 中保存旧版本，通过 roll_pointer 串联成链表，形成该行的多版本历史。
   - **ReadView（一致性视图）**：记录四个关键字段——**m_ids**（当前活跃未提交事务 ID 列表）、**min_trx_id**（活跃事务最小 ID）、**max_trx_id**（下一个将分配的事务 ID）、**creator_trx_id**（创建该 ReadView 的事务 ID）。遍历版本链，按可见性规则判断哪个版本对当前事务可见。

3. **RC 和 RR 的 ReadView 差异**：
   - **RC**：每次 SELECT 都**生成新的 ReadView**，所以能看到其他事务已提交的最新数据，存在不可重复读。
   - **RR**：事务中**第一次快照读生成 ReadView 后全程复用**，所以同一事务内多次读取结果一致，解决不可重复读。

4. **适用场景**：
   - **快照读场景**：普通 SELECT 就是 MVCC 的典型应用，不加锁、读历史版本、高并发。
   - **不适合 MVCC 的场景**：当前读（FOR UPDATE / UPDATE / DELETE）需要读到最新数据并加锁保证互斥，MVCC 无法替代锁机制。

5. **MVCC 与锁的分工**：

   | 场景 | 机制 | 是否加锁 | 读取版本 |
   | --- | --- | --- | --- |
   | 普通 SELECT（快照读） | MVCC | 不加锁 | 历史版本 |
   | FOR UPDATE / UPDATE / DELETE（当前读） | 锁机制 | 加行锁/间隙锁 | 最新数据 |

   InnoDB 隔离性 = **MVCC（管读不阻塞）+ 锁机制（管写互斥）**，两者配合才完整。

6. **优点**：
   - **读写不冲突**：读操作不加锁，写操作不阻塞读，大幅提升并发性能。
   - **解决脏读和不可重复读**：RR 级别下快照读全程复用 ReadView，保证事务内一致性。
   - **无死锁风险**：读操作不加锁，不会因读-写锁竞争产生死锁。

7. **缺点**：
   - **只能解决快照读的并发问题**，当前读仍需依赖锁机制（Next-Key Lock 等）。
   - **存在幻读风险**：RR 级别下快照读和当前读混用可能出现幻读（MVCC 读不到新插入的行，但当前读能看到），需统一用当前读解决。
   - **额外存储开销**：undo log 版本链占用空间，长事务会导致大量历史版本无法清理（purge 线程必须等最老事务结束）。
   - **不适用于串行化级别**：Serializable 下所有操作串行执行，MVCC 不生效。

**一句话总结：**
MVCC 不是锁，是靠隐藏列 + undo log 版本链 + ReadView 让读不加锁的并发控制机制，优点是读写不冲突提升并发，缺点是只管快照读、当前读仍需锁、长事务会导致 undo 膨胀。
