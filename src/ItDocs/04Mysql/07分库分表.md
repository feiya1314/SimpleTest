# 1. 为什么需要分库分表？

**核心回答：** 单库单表在**数据量、并发量、连接数**三个维度存在物理上限，分库分表是突破单机瓶颈、实现数据库水平扩展的唯一手段。

**底层原理：**

1. **单表数据量瓶颈**：MySQL 单表数据超过 2000 万行（InnoDB B+树约 3 层），查询性能急剧下降，即使有索引，B+树高度增加导致磁盘 IO 次数上升，响应时间从毫秒级退化到百毫秒级。
2. **单库并发瓶颈**：单机 MySQL 的 QPS 上限约 3000-5000（取决于硬件和 SQL 复杂度），互联网高并发场景下远不够用。连接数也有上限（max_connections 默认 151），高并发时连接耗尽直接拒绝服务。
3. **单机硬件瓶颈**：CPU、内存、磁盘 IO 均有上限，数据量持续增长后单机无法承载，必须通过分库将数据分散到多台机器。

**通俗理解：**
一张表就像一个房间，人少的时候没问题，人多了就挤、就慢。分表是把一个大房间隔成多个小房间，分库是直接多建几栋楼。本质上就是**把压力分摊到多个节点上**。

**一句话总结：**
单机扛不住了就分，分表解决数据量问题，分库解决并发量问题。

# 2. 分库分表有哪些拆分方式？

**核心回答：** 拆分方式分为**垂直拆分**和**水平拆分**两个维度，每种又分分库和分表，共四种组合：垂直分库、垂直分表、水平分库、水平分表。

**核心区别：**

1. **垂直分库**：按业务拆库，比如用户库、订单库、商品库各自独立。核心是**业务解耦**，每个库独立部署、独立扩容。
2. **垂直分表**：按列拆表，把大字段（如 TEXT、BLOB）拆到扩展表，主表只保留高频访问列。核心是**减少 IO 和内存占用**，提升查询效率。
3. **水平分库**：同一个表的数据按规则分散到多个库中，比如按 user_id % 4 分到 4 个库。核心是**突破单库并发和存储上限**。
4. **水平分表**：同一个库内把一张表拆成多张结构相同的子表，比如 order_0、order_1、order_2。核心是**突破单表数据量上限**。

**适用场景：**
垂直分库：微服务拆分时自然要做，每个服务对应独立数据库。
垂直分表：单表字段过多（超过 30 列），且有大字段低频访问时。
水平分库：单库 QPS 过万，连接数不够，需要分散并发压力。
水平分表：单表数据超 2000 万行，查询变慢，需要分散数据量。

**一句话总结：**
垂直拆是按业务/列拆，水平拆是按行数据分散；先做垂直拆解耦业务，再做水平拆突破单机瓶颈。

# 3. 什么时候该分表？什么时候该分库？

**核心回答：** **单表数据量过大导致慢查询就分表，单库并发/连接数扛不住就分库**，两者可以独立做也可以组合做。

**适用场景：**

1. **只分表不分库**：单表数据量大（超 2000 万行）但并发量不高（QPS < 3000），只需要在同一个库内拆成多张表减少单表数据量即可。
2. **只分库不分表**：并发量高（QPS 过万）但单表数据量不大，只需要把同一个表分散到多个库实例上，每个库扛一部分并发。
3. **既分库又分表**：数据量大且并发量高，最常见的情况。比如电商订单表，数据量过亿且 QPS 过万，必须分库分散并发 + 分表分散数据量。

**避坑指南：**

1. **不要过早分库分表**：分库分表引入的复杂度（跨库事务、跨库查询、数据迁移）远超想象，数据量不到瓶颈时优先考虑优化 SQL、加索引、读写分离、加缓存。
2. **不要盲目追求拆分**：拆分后很难回退，一旦分了就很难合并回来。建议先做读写分离，再考虑分库分表。
3. **拆分粒度要留余量**：分 8 个库还是 16 个库？要预估未来 3-5 年数据增长，一次拆到位，避免二次拆分（数据迁移成本极高）。

**一句话总结：**
数据多分表，并发高分库，两者都高就既分库又分表；但别过早拆，优先用缓存和读写分离兜底。

# 4. 如何选择分片键？分片策略有哪些？

**核心回答：** 分片键的选择原则是**数据分布均匀、查询尽量命中单分片、避免跨分片查询**。分片策略主要有范围分片、Hash 取模分片、一致性 Hash 分片。

**底层原理：**

1. **分片键选择原则**：
   - **高基数**：分片键的值要足够分散，避免数据倾斜（如按 status 分片只有几个值，大量数据集中在一个分片）。
   - **查询高频**：分片键要出现在大部分核心查询的 WHERE 条件中，保证查询能路由到单个分片。
   - **避免热点**：不能选自增 ID 做范围分片，否则新数据全部分到最新分片，写热点严重。
   - 常见选择：用户表选 user_id，订单表选 user_id 或 order_id，日志表选 create_time。
2. **范围分片（Range）**：按分片键的值范围分配，如 id 1-100 万在分片1，100 万-200 万在分片2。优点是范围查询友好、扩容方便；缺点是热点问题严重，新数据集中写入最新分片。
3. **Hash 取模分片**：shard = hash(key) % N，数据均匀分布。优点是数据分布均匀、无热点；缺点是扩容时需要重新计算所有数据位置（rehash），数据迁移量大。
4. **一致性 Hash 分片**：将分片节点和数据都映射到 Hash 环上，数据顺时针找最近节点。优点是扩容只需迁移少量数据；缺点是数据分布可能不均匀，需引入虚拟节点解决。

**避坑指南：**

1. **分片键一旦选定不可更改**：后续所有查询和路由都依赖分片键，改分片键等于重新做数据迁移，代价极大。
2. **非分片键查询必跨分片**：如果按 user_id 分片但按 order_time 查询，会扫描所有分片再合并，性能极差。解决方案是建立**基因法索引**或**广播表**。
3. **取模数量建议用 2 的幂次**：方便后续扩容时做翻倍扩容，减少数据迁移量。

**一句话总结：**
分片键选高基数、查询高频的字段，策略选 Hash 取模最均匀，一致性 Hash 扩容最友好，范围分片最容易出现热点。

# 5. 分库分表后分布式 ID 怎么生成？

**核心回答：** 分库分表后自增 ID 无法保证全局唯一，常用方案有**UUID、数据库号段模式、Snowflake 雪花算法、Redis 自增**。

**核心区别：**

1. **UUID**：128 位随机值，全局唯一但不有序、无业务含义、索引性能差（随机插入导致 B+树频繁页分裂），**不推荐作为主键**。
2. **数据库号段模式**：用一张表存储当前最大 ID 和步长，每次取一批号段缓存在本地（如一次取 1000 个），用完再取。美团的 Leaf-Segment 就是这个方案。优点是简单可靠、ID 有序；缺点是依赖数据库，DB 挂了就没号发。
3. **Snowflake 雪花算法**：64 位 Long 型 ID = 1 位符号 + 41 位时间戳 + 10 位机器 ID + 12 位序列号。优点是高性能、趋势递增、不依赖数据库；缺点是依赖时钟，时钟回拨会生成重复 ID。
4. **Redis 自增**：利用 Redis 的 INCR 命令生成 ID，速度快但依赖 Redis 可用性，且需要持久化防止丢号。

**实战选型：**
中小规模系统用**数据库号段模式**，简单可靠；大规模分布式系统用**Snowflake**，性能最高，但要解决时钟回拨问题（可引入 ZooKeeper 分配 workerId 或用 Leaf-Segment 兜底）。阿里、美团内部都基于 Snowflake 做了定制化改造。

**一句话总结：**
UUID 不做主键，中小系统用号段模式，大系统用雪花算法，时钟回拨是唯一坑。

# 6. 分库分表后跨库事务怎么处理？

**核心回答：** 分库后本地事务失效，必须用**分布式事务方案**解决。常见方案有 XA 两阶段提交、TCC、Saga、本地消息表 + 最终一致性。

**底层原理：**

1. **XA 两阶段提交**：数据库层面支持，协调者先 Prepare 再 Commit。优点是对业务无侵入；缺点是同步阻塞、性能差、协调者单点故障风险大。适合强一致性要求高、并发量不大的场景。
2. **TCC（Try-Confirm-Cancel）**：业务层面实现，Try 预留资源，Confirm 确认执行，Cancel 取消释放。优点是性能好、无锁；缺点是业务侵入大，每个接口要写三个方法，还要处理幂等、空回滚、悬挂问题。
3. **Saga 模式**：长事务拆成多个本地事务，每个本地事务配一个补偿操作，失败时反向补偿。优点是适合长流程业务；缺点是缺乏隔离性，中间状态可见。
4. **本地消息表 + 最终一致性**：业务操作和消息写入在同一个本地事务中，后台线程异步发送消息到 MQ，消费方幂等消费。这是互联网公司**最常用**的方案，实现简单、可靠性高。

**避坑指南：**

1. **不要用强一致性方案处理高并发业务**：XA 性能太差，电商订单场景下 TPS 会掉到百级别。
2. **优先用最终一致性**：大部分业务场景（下单、支付回调、库存扣减）都能接受短暂不一致，本地消息表是最务实的选择。
3. **幂等是前提**：无论哪种方案，消费方都必须幂等，否则重复消费会导致数据错乱。

**一句话总结：**
强一致性用 XA（性能差），业务侵入用 TCC，互联网首选本地消息表 + 最终一致性，幂等是所有方案的前提。

# 7. 分库分表后跨库分页查询怎么做？

**核心回答：** 跨库分页是分库分表后**最难处理的问题之一**，常见方案有全局汇总法、二次查询法、允许折中的业务方案。

**底层原理：**

1. **全局汇总法**：每个分片执行 ORDER BY x LIMIT offset, size，把所有分片结果汇总到应用层再排序、截取。优点是结果准确；缺点是深分页时每个分片返回大量数据，应用层合并压力大，性能极差。比如查第 1000 页，每个分片要返回前 1000 页数据再合并。
2. **二次查询法**：第一次查询各分片 LIMIT offset, size，找到所有分片返回的最小值；第二次以最小值为起点再查各分片，合并后取正确结果。减少了数据传输量，但实现复杂，且仍然无法避免多次查询。
3. **业务折中方案**：
   - **禁止深分页**：只提供上一页/下一页游标分页（基于上次最后一条记录的 ID 翻页），不提供跳页功能。这是互联网产品的通用做法。
   - **用 ES 辅助查询**：将分页查询需要的字段同步到 ES，分页查询走 ES，精确定位后拿主键回分库查详情。
   - **用汇总表/宽表**：另建一张不分片的汇总表，专门供分页查询使用，通过 binlog 同步数据。

**避坑指南：**

1. **深分页是性能杀手**：LIMIT 100000, 10 在单库就慢，跨库更慢，必须从业务上限制或用游标分页替代。
2. **不要用 count 汇总总数**：跨库 COUNT 需要扫描所有分片，大数据量下极慢。业务上可以用近似值或缓存总数。

**一句话总结：**
跨库分页无完美方案，业务上优先游标分页禁跳页，技术上用 ES 或汇总表兜底，深分页必须干掉。

# 8. 分库分表后如何做跨库 JOIN？

**核心回答：** 分库分表后跨库 JOIN 基本不可用，需要通过**冗余字段、广播表、应用层组装、ES 宽表**等方式替代。

**适用场景：**

1. **冗余字段**：在订单表中冗余用户姓名，避免 JOIN 用户表。这是最简单直接的方案，用数据冗余换查询性能，适合冗余字段不常变更的场景。
2. **广播表/小表广播**：将小表（如字典表、配置表、地区表）在每个分库中都存一份完整数据，写入时同步到所有分库。这样 JOIN 操作在单个库内即可完成。ShardingSphere 原生支持广播表。
3. **应用层组装**：先查订单表拿到 user_id 列表，再批量查用户表，在应用内存中组装结果。优点是灵活；缺点是代码复杂、多次网络调用。适合关联不频繁的场景。
4. **ES 宽表**：将需要 JOIN 查询的多张表数据合并写入 ES，形成宽表文档，直接在 ES 中查询。适合复杂查询、多维度搜索场景。

**避坑指南：**

1. **分库分表前就要梳理 JOIN 关系**：频繁 JOIN 的表尽量放在同一个库中（绑定表），避免后续改造。
2. **广播表数据量必须小**：广播表数据量大了写入同步压力大，一般建议不超过 1 万行。
3. **冗余字段要考虑一致性**：冗余字段变更时需要同步更新，可用 MQ 异步更新或 binlog 监听。

**一句话总结：**
跨库 JOIN 靠冗余字段和广播表解决小表关联，靠应用层组装和 ES 宽表解决复杂查询，设计阶段就要提前规划绑定表。

# 9. 分库分表后数据平滑迁移

**核心回答：** 分库分表数据迁移有**四种主流方案**：停机迁移、纯 Canal Binlog 同步、应用双写+Canal 兜底、纯应用双写。核心拆分为**存量处理、增量追赶、时序对齐、更新删除兼容**四个环节。选型看**是否允许停机、一致性要求、开发成本**：绝大多数业务用**纯 Canal 同步**，资金核心用**应用双写+Canal 兜底**，内部系统直接停机迁移。

**方案一：停机迁移（零双写、最简单）**

流程：业务全量停服，关闭写入接口、定时任务、MQ 消费 → 锁库/一致性快照导出老库全量 → DataX 按分片规则路由分发写入新分库 → 校验一致 → 修改数据库连接和分片配置重启服务

- 存量处理：mysqldump / Dumpling / DataX 导出停机时刻静态全量，无中途写入干扰
- 增量处理：停机期间无新增增量，不存在增量追赶问题，一次性迁移完成
- 优点：**业务代码几乎零改造**，无需双写逻辑；**数据天然 100% 一致**，无分布式一致性难题；出错极易回滚（切回老库配置重启）
- 缺点：**必须停服**，数据量越大迁移耗时越长（千万/亿级可能数小时）；电商、支付、高在线业务无法接受；大库导出导入短暂挤压服务器 IO
- 适用：后台内部系统、低访问量业务、允许凌晨长时间维护窗口

**方案二：纯 Canal Binlog 同步迁移（主流低侵入不停服方案）**

核心逻辑：应用只正常写老库，完全不改造双写代码；依靠 Canal 监听老库 binlog，同时覆盖「存量全量快照」+「T0 后所有增量变更」

**完整执行步骤**

- 开启老库一致性快照，拿到 T0 binlog 位点；
- DataX/Dumpling 导出 T0 时刻存量，按分片规则分发写入新分库；
- Canal 指定 T0 位点启动消费 binlog；
- Canal 客户端解析每条 Insert/Update/Delete，复用统一分片算法写入对应分片；
- 等待 Canal 同步延迟接近 0， **新旧数据对账一致** ；
- 灰度切读流量到新库；
- 稳定后切换写入路由只写新库，下线老库。

**关键步骤**

- 存量处理：DataX 路由分发导入分片，快照定格 T0，导入期间线上业务正常写入老库
- 增量处理：T0 后所有变更全在 binlog 中，Canal 从 T0 位点完整回放自动补齐；存量是 T0 静态底账，binlog 补齐后续所有改动，二者合并为完整数据集
- 更新/删除处理：Insert 路由分片直接插入，业务唯一键做幂等 ON DUPLICATE KEY UPDATE；Update 按分片键定位对应分片执行；Delete 匹配主键删除；每条 binlog 携带唯一位点+bizId，重复消费只执行一次

**存在问题：**

- **存在同步延迟窗口**
  - MySQL 写入→生成 binlog→Canal 拉取→解析路由→写入新分片，整条链路天然存在毫秒至秒级延迟。高并发峰值、大批量事务时延迟可能冲到十几秒。
  - 灰度切读阶段，如果用户刚提交修改，立刻读新库会读到旧数据；对强实时场景（支付余额、库存扣减、下单锁库存） 不友好。

  解决办法：
  **灰度读阶段双读校验降级：** 灰度流量同时查新库 + 老库，若新旧数据不一致 / 新库延迟过大，直接返回老库结果

  ```
        强实时接口不走灰度新库：余额、库存、支付扣减这类强一致接口，迁移全程固定读老库；等 100% 切写新库、延迟长期稳定 0 再统一切读。
        热点数据本地缓存、Redis 缓存以老库写入为准，新库只做持久存储层，不做主读数据源。
  
         **使用双写结合Canal**
  ```
- **峰值 binlog 堆积雪崩**
  - 大促、批量任务一次性产生几万条 binlog，Canal 消费速度跟不上生成速度，延迟持续走高，堆积越堆越多，恢复耗时很久。

  解决办法：
  Canal+MQ 架构抗堆积（标准抗雪崩架构）：峰值 binlog 全部堆积在 MQ 磁盘，不会阻塞 Canal 拉取；Canal 消费速度永远快于回放入库速度，不存在 Canal 自身内存打爆 OOM。

  低峰加急追赶
  白天正常限速平稳同步；凌晨低峰关闭限速、加大线程批量全力追平堆积。
  死信隔离
  失败 SQL 单独投递死信队列，不阻塞正常队列；定时重试死信，多次失败落异常表人工核对修复。
- **binlog 过期清理风险**

  若迁移周期长、expire_logs_days 配置太短，Canal 长时间宕机未重启，中间 binlog 被 MySQL 自动删除，这段时间增量彻底丢失，只能靠对账修复。

优点：
**业务代码改动极小**，无需侵入所有 DAO、MQ、定时任务做双写，改造成本最低；

```
全程不停服用户无感知；天然时序有序，先存量后增量不会逆序；

回滚简单切回读老库即可，写入逻辑从没改过
```

缺点：
**Canal 客户端必须开发分片路由消费逻辑**，需独立开发维护消费服务；

```
**同步存在毫秒\~秒级延迟**，不能用于对强实时性要求极高的资金交易；

binlog 因过期、故障丢失会出现数据缺口（可拉长 binlog 过期时间、定时对账兜底）；

DDL 变更需要手动同步双库
```

- 风险：分片算法不一致（存量导入和 Canal 消费两套代码不同）直接分错数据；大事务 binlog 批量回放瞬间压新分片；无应用层双写兜底，极端 binlog 故障只能靠离线对账修复
- 适用：绝大多数普通在线业务（商城、用户、内容、订单非资金核心链路），企业最常用标准方案


**方案三：应用双写+Canal 兜底（高可靠强一致性不停服）**

核心逻辑：两条写入链路双重保险——应用层写逻辑同时写入老库+新分库，Canal 依然监听老库 binlog 做兜底补偿

流程步骤：

步骤 1：锁定全量快照 T0，导出老库存量数据
在老库从库执行一致性快照事务：START TRANSACTION WITH CONSISTENT SNAPSHOT;
查询拿到精准数据库时间 T0、SHOW MASTER STATUS 获取 binlog File+Position 位点
使用 Dumpling/DataX 开启 snapshot 一致性模式，读取老单表全量数据
DataX 写入端配置分片路由、幂等写入INSERT ... ON DUPLICATE KEY UPDATE，按分片规则分发写入各个新分表
全量导入全程不开启应用双写、不启动 Canal 消费；仅完成 T0 静态存量落新库
步骤 2：初始化 Canal，绑定 T0 位点
停止 Canal 实例，清空原有持久化消费位点（ZK/meta.dat）
将步骤 1 拿到的 binlog 文件、position 写入 Canal 实例配置
启动 Canal，Canal 只拉取 T0 位点之后所有 binlog 事件；此时暂时不开启消费写入新库，仅拉取缓存 binlog，等待双写就绪
步骤 3：上线改造后的 DAO 双写代码，开启应用双写开关
底层 BaseMapper 统一封装写入逻辑：Insert/Update/Delete 自动执行新旧库双写
执行顺序铁律：先写老库，老库事务完全提交成功后，再处理新库写入
新库写入采用主线同步老库、异步线程池写新库，不阻塞业务响应
新库写入捕获异常失败后，实体数据推入 Redis 有序重试队列，阶梯退避重试
MQ 消费者、定时任务、后台管理操作全部强制复用封装 DAO，杜绝裸写数据库
灰度放量：先切少量测试流量验证双写成功率、RT、报错率，无异常全量打开业务流量双写
步骤 4：启动 Canal 消费服务，开启兜底同步
Canal 消费逻辑核心规则：
每条 binlog 事件解析 biz_id、分片键，用公共路由类计算目标分库分表
统一幂等 SQL：INSERT ... ON DUPLICATE KEY UPDATE，依靠 update_time 只保留最新数据
同分片键数据串行消费，保证 insert→update→delete 时序不乱
Update 无匹配数据时自动转换为插入完整新数据；物理 delete 转为软更新 deleted=1
此时形成双链路写入：
实时链路：业务双写同步写入新旧库；兜底链路：Canal 回放老库全部 binlog 补偿漏写 / 失败写入
步骤 5：等待双链路追平，执行多层数据校验
实时分钟级粗校验：新旧库按分片统计总条数、金额 sum、最大 bizId，差值超阈值立刻告警
实时抽样对账：灰度流量同时双读新旧库，单条数据不一致落异常库告警
夜间离线精准逐行对账：以老库为基准，自动修复新库缺失、不一致数据
观测指标：Canal 同步延迟稳定 < 200ms、双写失败重试队列堆积接近 0、对账差异条数持续 0，稳定观察 1\~3 天
步骤 6：灰度切读流量至新分库
梯度放大读比例：0% → 10% → 30% → 70% → 100%，每一档最少稳定运行半天
增加熔断保护：Canal 延迟 > 1s、对账出现差异、新库慢 SQL 暴涨时，自动下调读比例甚至切回全量读老库
强实时资金 / 库存接口可长期固定只读老库，不参与灰度切流
步骤 7：100% 读新库稳定运行观察期
全量读新库后持续观察 7\~15 天：
业务报错、超时、资损、对账差异全部 0
Canal 延迟、双写失败率长期平稳无波动
大流量、定时批量任务运行无异常
步骤 8：切换写入，关闭老库写入，只写新库
前置：提前部署反向同步程序（新库 binlog 同步回老库，回滚备用）
配置关闭老库写入开关：应用写入逻辑只执行新库写入，不再操作老库
短时保留双写代码骨架、重试队列、Canal 同步（做兜底缓冲 1\~2 天）
持续对账：新库为主数据源，反向同步保证老库数据同步一致
步骤 9：分阶段下线冗余组件
稳定无问题 2 天后，停止 Canal 消费同步
移除 Redis 双写重试队列逻辑
注释 / 下线 DAO 层双写分支代码，简化为单写新库逻辑
下线反向同步脚本（确认完全无回滚可能）
步骤 10：资源清理收尾
持续备份老库数据一段时间，确认业务无历史查询需求
下线老库 MySQL 实例，释放服务器、内存、磁盘资源
清理迁移配套监控、对账定时任务、迁移配置开关


可能存在的问题及解决办法：

问题 1：双写无分布式事务，存在短暂不一致窗口
风险场景
老库写入成功，新库网络 / 锁 / 连接池满写入失败，主线业务已返回成功，新库缺失数据
极端：老库事务回滚，但异步线程已把数据写入新库，新库生成脏数据
长事务导致新旧库锁同时持有，死锁、超时概率上升
解决方案
强制写入顺序：老库事务完整提交成功，才允许发起新库写入；老库失败整体回滚，绝不先行写入新库
异步柔性重试隔离主线：主线只等老库结果，新库交给独立线程池异步处理，失败进 Redis 重试队列，阶梯退避多次重试
Canal 终极兜底：就算重试队列消息丢失、服务宕机，老库 binlog 完整回放补齐缺失数据
高资损链路可选 Seata AT 分布式事务：新旧库作为两个分支事务，一阶段锁、二阶段统一提交回滚；普通业务不用，避免性能损耗
全局统一幂等 SQL+update_time 时间戳，重复写入只保留最新状态

**问题 6：应用双写 + Canal 双重写入，重复变更频繁，幂等疏漏即数据错乱
风险
同一条更新，应用写入一次、Canal 再回放一次；无幂等会造成金额翻倍、计数重复累加
解决方案
biz_id 全局唯一索引，作为幂等核心依据
全链路统一带时间戳判断的幂等 InsertOrUpdate SQL：旧时间戳数据不会覆盖最新业务数据
Canal 消费增加 Redis 短时间幂等缓存，窗口内重复 biz_id 事件直接丢弃，减少无效 SQL 压力
软删除统一替换物理 Delete，删除动作等价更新 deleted 标记，规避时序删除漏洞
问题 7：存量导入与实时更新时序冲突（A 未写入新库，业务先 Update/Delete A）
风险
业务更新 A，新库无数据 Update 无效，后续 DataX 导入 T0 旧 A，丢失中间更新
业务 Delete A，新库无数据删不动，后续导入 A 产生已删除脏数据
解决方案
全链路统一幂等 SQL：Update 场景无数据自动插入最新变更值；Delete 转为软更新标记
update_time 时间戳天然屏障：后续导入的 T0 快照时间更早，不会覆盖业务最新数据
夜间对账兜底，自动修复少量时序错乱脏数据**

**问题 10：重试队列死信堆积、消息丢失风险
风险
Redis 队列持久化配置不当、服务宕机未落地消息，新库写入失败数据丢失；多次重试失败消息无人工介入机制
解决方案
Redis 开启 RDB+AOF 持久化，队列消息落地磁盘，防止宕机丢失
阶梯退避策略，达到最大重试次数的数据转入 MySQL 死信异常表，触发电话 / 钉钉告警
定时脚本巡检死信表，支持一键批量重试修复**


方案适用边界与选型对比
适用场景：支付、账务、资金、交易订单、账户余额等对资损零容忍、强一致性核心业务；可以接受更高开发运维成本换取极致可靠性
不推荐场景：普通商品、内容、用户基础信息等低风险业务，直接使用纯 Canal 方案即可，成本更低、改造更小
可靠性层级：停机迁移 < 纯 Canal 不停服 < 应用双写 + Canal 兜底（行业最高容错标准）



**方案四：纯应用双写（无 Canal，极少使用）**

逻辑：不依赖 binlog 中间件，只靠应用双写；全量导入后直接上线双写，所有新增、更新、删除全由应用双写承载，无兜底补偿

**方案步骤：**

1、服务改造支持双写，同时写旧表和新表

2、旧表数据开始迁移

3、进行数据一致性校验，以旧表数据为准

4、数据一致后，慢慢切流量读新库

5、流量全部读新库，系统稳定后进行单写，只写新表

6、旧表下线


**适用边界**

- 数据量小：单表几十万以内；
- 并发低：TPS 几十以内，无尖峰流量；
- 业务容错高：短时间少量数据不一致无资损（如后台配置、用户标签、非资金类基础资料）；
- 团队无运维 Canal 的人力、技术储备。

**存在问题：**

**1. 双写原子性无法保证，天然存在不一致窗口**

```
老库成功、新库超时 / 异常失败：老库有新变更，新库无；

老库失败回滚、新库已经写入：新库多出脏数据；

1、数据刚好读取准备迁移到新表，还未写入新表，但是update进行，此后迁移的数据写入新表，导致新表中该记录未更新

2、数据刚好读取准备迁移到新表，还未写入新表，但是delete进行，此时新表本身就没有该记录，导致空删，后续迁移执行，导致新表中该记录未删除
```

**2.“仅靠一致性校验修复” 方案的局限性**

- 时效性差：对账一般夜间低峰跑，白天业务带着错误数据跑一整天；
- 如果切了灰度读新库，用户看到错误数据；
- 大表成本高：千万、亿级数据逐行比对，CPU、IO 开销大，迁移周期拉长；
- 无法处理事务内多表联动差异：比如下单同时插订单、订单详情，一张对一张错，单纯 count、sum 粗校验发现不了细微行差异；
- 修复脚本有风险：以老库为准覆盖新库，如果修复脚本逻辑 bug，会误删正确新库数据。

**解决办法：**

只能依靠数据一致性校验工具，可以多次扫描，保证数据一致性

- 优点：架构简单少一套 Canal 组件；实时性最高
- 致命缺点：
  - MQ 消费、定时任务、离线脚本极易漏写，一旦漏写永久丢数据；
  - 双写失败仅靠本地重试无第二修复通道，故障后修复极麻烦；
  - 亿级大存量迁移时全量导入期间业务增量完全无兜底
  - 只适合极小体量、低并发、可容忍事后对账修复的业务
- 结论：**几乎不推荐生产使用**，仅小体量测试环境临时过渡

**全量导入与实时更新的时序冲突问题：**

这是不停服迁移中最容易踩坑的环节。

场景还原：

```
记录 A 属于 T0 快照存量数据，DataX 正在分批导入还没写到新库；

同一时间业务发起 update A，双写尝试更新新库 A，但新库没有 A，update 影响 0 行；

等 DataX 把 T0 旧快照 A 导入新库后，中间这段更新丢失。
```


核心解法——**统一使用幂等 InsertOrUpdate + 时间戳版本控制**：

- 所有往新分库的写入（全量导入、应用双写、Canal 回放），一律不用单纯 INSERT/UPDATE，统一用 INSERT ... ON DUPLICATE KEY UPDATE，依靠业务唯一主键 bizId 做冲突判定：记录不存在则插入完整数据，已存在则覆盖更新
- 表统一加 update_time 时间戳字段，幂等语句配合版本判断：ON DUPLICATE KEY UPDATE 字段1=IF(VALUES(update_time) > update_time, VALUES(字段1), 字段1)，保证实时更新的新数据不会被旧快照反向覆盖

分链路处理：

1. **应用双写**：DAO 层统一封装 InsertOrUpdate 替代裸 Update，新库无 A 则直接插入更新后完整数据，后续 DataX 导入 T0 旧 A 时因时间戳更小不覆盖
2. **DataX 全量导入**：写入端开启 duplicateKey 幂等模式，自动生成 ON DUPLICATE KEY UPDATE，配合时间戳判断
3. **Canal 增量同步**：消费逻辑同样封装 InsertOrUpdate + 时间戳比对，作为第二层保险弥补应用层偶发失败；UPDATE 事件前置判空，无数据则转为 INSERT
4. **Canal 链路额外保障**：同一分片键/bizId 的 binlog 事件串行消费，保证 insert 先于 update 执行顺序不乱；按分片键哈希分区投递 MQ，同一分片消息进同一队列保证有序
5. **离线对账兜底**：每日按 bizId 对比老库新库，老库有新库无则 insert 补全，两边都有但字段不一致则以老库为准覆盖，新库有老库无则删除脏数据

完整安全时序示例：T0 快照 A(name=张三, update_time=1000) → DataX 读取 A 还没写入新库 → 业务更新 A(name=李四, update_time=2000) → 应用双写 InsertOrUpdate 新库无 A 直接插入(李四, 2000) → DataX 把旧 A(张三, 1000) 写入新库 → 唯一键冲突因 1000<2000 不覆盖 → 全程数据无丢失无旧值回灌


**离线对账兜底**：每日按 bizId 对比老库新库，老库有新库无则 insert 补全，两边都有但字段不一致则以老库为准覆盖，新库有老库无则删除脏数据


**通用约束（所有不停服方案必须遵守）：**

1. **全局唯一 ID**：废弃自增 ID，统一雪花/号段，杜绝分片主键冲突
2. **分片逻辑唯一**：存量导入、Canal 消费、应用双写共用同一分片工具类，杜绝分错
3. **幂等写入**：所有新分片写入带业务唯一键 INSERT ... ON DUPLICATE KEY UPDATE
4. **时序锚点 T0**：全量快照绑定 binlog 位点，存量底账+增量日志时序无断层
5. **多级校验**：实时抽样对账+夜间离线全量比对，差异自动修复
6. **分层开关**：双写开关、读灰度比例、老写入开关，操作顺序不可逆

**一句话总结：**
停机迁移最简单但必须停服，纯 Canal 同步是 90% 企业首选的低侵入方案，应用双写+Canal 兜底是资金核心链路的最高可靠方案，纯应用双写不推荐生产；所有不停服方案的核心难点是全量导入与实时更新的时序冲突，统一用幂等 InsertOrUpdate + 时间戳版本控制解决。

# 10. ShardingSphere 和 MyCAT 怎么选？

**核心回答：** ShardingSphere 是**当前主流的分布式数据库中间件**，社区活跃、功能完善、架构先进；MyCAT 是早期方案，社区已基本停滞，**新项目不建议使用**。

**核心区别：**

1. **架构模式**：
   - MyCAT 是 **Proxy 模式**，独立部署代理层，所有 SQL 经过 MyCAT 路由转发，多一次网络跳转。
   - ShardingSphere 支持 **Proxy 模式 + JDBC 模式**。JDBC 模式（ShardingSphere-JDBC）以 jar 包嵌入应用，无额外网络开销，性能最高；Proxy 模式（ShardingSphere-Proxy）独立部署，适合异构语言场景。
2. **社区与维护**：
   - MyCAT 社区已停更，issue 无人处理，版本停留在 1.6，bug 多且无修复。
   - ShardingSphere 是 Apache 顶级项目，社区活跃，版本持续迭代，功能不断增强。
3. **功能对比**：
   - MyCAT：支持分库分表、读写分离，但对复杂 SQL 支持差，跨库聚合、子查询容易出 bug。
   - ShardingSphere：支持分库分表、读写分离、广播表、绑定表、分布式事务（XA/Seata）、数据加密、影子库压测，SQL 兼容性好。

**适用场景：**
ShardingSphere-JDBC：Java 技术栈首选，性能最高，零网络开销，适合微服务架构。
ShardingSphere-Proxy：多语言技术栈（Go/Python/Node.js），需要统一代理入口。
MyCAT：仅维护已有老项目时使用，新项目禁止选用。

**实战选型：**
Java 技术栈无脑选 ShardingSphere-JDBC；多语言异构系统选 ShardingSphere-Proxy；MyCAT 只在维护遗留系统时使用，绝不用于新项目。

**一句话总结：**
新项目选 ShardingSphere，Java 用 JDBC 模式性能最高，多语言用 Proxy 模式，MyCAT 已淘汰别碰。

# 11. 分库分表如何不停服迁移数据？

**核心回答：** 双写迁移是**零停机迁移的实战方案**（方案对比见 #9），核心流程是**全量同步 → 增量同步 → 开启双写 → 多轮校验 → 灰度切流**，全程业务不中断、随时可回滚。

**双写迁移详细执行流程（10 步）：**

 1. **环境准备**：新分库分表环境搭建完成，网络、权限、配置就绪。 **注意观察监控，选择业务流量低的时间段进行迁移**
 2. **全量同步**：用数据迁移工具将老库存量数据全量导入新库
 3. **增量同步**：全量完成后开启增量同步，自动回溯全量开始时间的 binlog，消息幂等消费保证不丢不重
 4. **数据校验**：全量数据校验，逐表逐行比对新旧库一致性
 5. **切流测试**：预发环境采集线上流量回放，验证分库分表代码、路由逻辑、切流开关
 6. **二次校验**：增量追平后再次全量校验并订正不一致数据
 7. **开启双写**：上线双写代码，所有写操作同时写新旧库，保证新数据实时性
 8. **读灰度切流**：凌晨低峰期按 userId % x 灰度切部分读流量到新库，逐步放大比例，持续观察
 9. **只写新库**：确认稳定后切换写入到新库，关闭老库写入，完成迁移
10. **收尾清理**：系统稳定运行一段时间后，下线双写代码和迁移工具，释放老库资源

**避坑指南：**

1. **导数时不能覆盖新数据**：增量同步时用 gmt_modified 判断，老数据不覆盖新数据，防止双写期间的新修改被回滚
2. **校验必须反复进行**：一次校验不够，增量追平后要二次校验，确保数据完全一致
3. **灰度切流从小到大**：先 1% 再 10% 再 50% 再 100%，每步观察监控，出问题立即切回
4. **回滚方案随时待命**：切换期间老库不能下线，新库出问题随时切回老库
5. **双写失败做异步重试**：新库写入失败不能阻塞主流程，应记录失败日志后台重试，同时用对账脚本兜底

**一句话总结：**
双写迁移核心十步：环境准备 → 全量同步 → 增量同步 → 数据校验 → 切流测试 → 二次校验 → 开启双写 → 灰度切流 → 只写新库 → 收尾清理；关键是全量先行、双写保实时、校验保一致、灰度保安全。

# 12. MySQL 主从复制原理是什么？

**核心回答：** MySQL 主从复制基于 **binlog 日志**实现，主库将变更写入 binlog，从库通过**三个核心线程**拉取并回放日志，最终实现数据同步。

**底层原理：**

1. **主库 Binlog Dump 线程**：从库连接主库时，主库启动 Binlog Dump 线程，负责读取 binlog 事件发送给从库。发送完已有事件后进入等待，主库有新写入时通过信号唤醒继续发送。
2. **从库 I/O 线程**：I/O 线程连接主库请求 binlog，收到事件后写入本地的 **relay log（中继日志）**。I/O 线程持续监听主库发来的事件，追加到 relay log 尾部。
3. **从库 SQL 线程**：SQL 线程读取 relay log 中的事件，在从库上**串行回放**这些 SQL 语句，使从库数据与主库一致。回放完成后 relay log 被清理。

**复制流程三步走：**

1. 主库执行事务，将变更事件写入 binlog
2. 从库 I/O 线程拉取 binlog 事件，写入 relay log
3. 从库 SQL 线程读取 relay log，回放 SQL 使数据一致

**关键细节：**

- **主从复制默认是异步的**：主库写完 binlog 就返回客户端，不等待从库回放完成，因此存在主从延迟
- **从库 SQL 线程是单线程的**（MySQL 5.6 之前）：这是主从延迟的主要原因，主库并发写入但从库串行回放
- **binlog 格式**：STATEMENT（语句级）、ROW（行级）、MIXED（混合），推荐使用 **ROW 格式**，数据一致性最可靠

**核心用途：**

- 读写分离：主库写，从库承担查询，
- 提升吞吐量 数据备份、故障容灾
- 数据归档、离线分析
- 也是 Canal 监听 Binlog 的底层依赖（Canal 就是伪装成从库）

**一句话总结：**
主库写 binlog，从库 I/O 线程拉取写 relay log，SQL 线程回放，三个线程协作完成数据同步，本质是异步日志复制。

# 13. MySQL 主从复制同步方式有哪些？

**核心回答：** MySQL 主从复制有**异步复制、同步复制、半同步复制**三种方式，核心区别在于**主库写入后等待从库确认的范围**不同，安全性和性能此消彼长。

**核心区别：**

1. **异步复制（默认）**：主库写入 binlog 后立即返回客户端，不等待从库接收。优点是性能最高，主库不受从库影响；缺点是主库宕机时未同步的数据会丢失，存在数据不一致窗口。
2. **同步复制**：主库写入后等待**所有从库**确认收到事件并执行完成才返回客户端。优点是数据最安全，主从完全一致；缺点是性能最差，任何一个从库慢都会拖垮主库，实际生产几乎不用。
3. **半同步复制**：主库写入后等待**至少一个从库**确认收到事件（写入 relay log）才返回客户端。优点是在性能和安全之间取得平衡，至少保证一主一从数据一致；缺点是比异步复制多一次网络 RTT，超时后自动降级为异步复制。

**避坑指南：**

1. **半同步复制不是强一致**：从库只是确认收到事件（写入 relay log），不是确认执行完成，主库宕机时已确认的事件可能还没回放
2. **半同步超时降级**：从库网络延迟导致超时，半同步会自动降级为异步，数据安全性退回到异步水平，需监控 rpl_semi_sync_master_no_times 观察降级次数
3. **互联网公司多用异步 + 补偿**：大部分业务用异步复制追求性能，通过业务层补偿（如读写主库、缓存兜底）解决延迟问题

**实战选型：**
对数据一致性要求不高（日志、统计）用异步复制；金融场景用半同步复制；同步复制一般不用。

**一句话总结：**
异步复制性能最好但可能丢数据，同步复制最安全但性能最差，半同步折中但超时会降级，互联网常用异步 + 业务补偿。

# 14. MySQL 主从同步延迟产生原因？怎么优化？

**核心回答：** 主从延迟的根本原因是**主库并发写入、从库串行回放**，加上网络传输和从库性能差异，导致从库数据落后于主库。

**产生原因：**

1. **大事务**：主库执行一个大事务（如批量 UPDATE 百万行），从库回放同样耗时，延迟持续累积
2. **主库多线程写入 vs 从库单线程回放**：MySQL 5.6 之前从库 SQL 线程是单线程，主库并发写入速度远超从库串行回放速度
3. **从库机器性能差**：从库使用了低配机器，CPU、磁盘 IO 能力不足
4. **网络延迟**：主从之间网络不稳定或带宽不足，binlog 传输慢
5. **从库数量过多**：主库需要向多个从库发送 binlog，发送压力随从库数量增加
6. **锁冲突**：从库回放 SQL 时遇到锁等待，导致回放阻塞

**优化方案：**

 1. **拆分大事务**：将大事务拆成小事务分批执行，避免单次事务占用过多时间
 2. **开启多线程复制**：MySQL 5.7 支持基于组提交的**多线程复制**（并行回放），大幅提升从库回放速度
 3. **减少从库数量**：从库不超过 5 个，减少主库 binlog 发送压力
 4. **提升从库硬件**：从库使用与主库同级别机器，尤其磁盘 IO 不能是瓶颈
 5. **监控 IO 延迟**：用 iostat 监控从库数据盘 IO，发现单次 IO 延迟高时排查磁盘或 RAID 卡问题
 6. **排查锁问题**：通过 SHOW PROCESSLIST 和 information_schema 下的锁/事务表定位锁冲突
 7. **业务层兜底**：对实时性要求高的读操作直接走主库，或用缓存降低新数据被立即读取的概率
 8. **二次查询。** 如果从库查不到数据，则再去主库查一遍，由 API 封装即可，算是一个兜底策略，比较简单。不过等于读的压力又转移到主库身上了，如果有不法分子估计搞一下必定查不到的查询，这就难受了。
 9. **强制将写之后立马读的操作转移到主库上。** 这种属于代码写死了，比如一些写入之后立马查询的操作，就绑定在一起，写死都走主库。不推荐，太僵硬了。
10. **关键业务读写都走主库，非关键还是读写分离。** 比如上面我举例的用户注册这种，可以读写主库，这样就不会有登陆报该用户不存在的问题，这种访问量频次应该也不会很多，所以看业务适当调整此类接口。

**一句话总结：**
主从延迟根因是并发写 vs 串行回放，优化核心是开多线程复制 + 拆大事务 + 提升从库性能，业务层用读主库和缓存兜底。
