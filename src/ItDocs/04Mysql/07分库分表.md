# 1. 为什么需要分库分表？

**核心回答：** 单库单表在**数据量、并发量、连接数**三个维度存在物理上限，分库分表是突破单机瓶颈、实现数据库水平扩展的唯一手段。

**底层原理：**

1. **单表数据量瓶颈**：MySQL 单表数据超过 2000 万行（InnoDB B+树约 3 层），查询性能急剧下降，即使有索引，B+树高度增加导致磁盘 IO 次数上升，响应时间从毫秒级退化到百毫秒级。
2. **单库并发瓶颈**：单机 MySQL 的 QPS 上限约 3000-5000（取决于硬件和 SQL 复杂度），互联网高并发场景下远不够用。连接数也有上限（max_connections 默认 151），高并发时连接耗尽直接拒绝服务。
3. **单机硬件瓶颈**：CPU、内存、磁盘 IO 均有上限，数据量持续增长后单机无法承载，必须通过分库将数据分散到多台机器。

**通俗理解：**
一张表就像一个房间，人少的时候没问题，人多了就挤、就慢。分表是把一个大房间隔成多个小房间，分库是直接多建几栋楼。本质上就是**把压力分摊到多个节点上**。

**一句话总结：**
单机扛不住了就分，分表解决数据量问题，分库解决并发量问题。

# 2. 分库分表有哪些拆分方式？

**核心回答：** 拆分方式分为**垂直拆分**和**水平拆分**两个维度，每种又分分库和分表，共四种组合：垂直分库、垂直分表、水平分库、水平分表。

**核心区别：**

1. **垂直分库**：按业务拆库，比如用户库、订单库、商品库各自独立。核心是**业务解耦**，每个库独立部署、独立扩容。
2. **垂直分表**：按列拆表，把大字段（如 TEXT、BLOB）拆到扩展表，主表只保留高频访问列。核心是**减少 IO 和内存占用**，提升查询效率。
3. **水平分库**：同一个表的数据按规则分散到多个库中，比如按 user_id % 4 分到 4 个库。核心是**突破单库并发和存储上限**。
4. **水平分表**：同一个库内把一张表拆成多张结构相同的子表，比如 order_0、order_1、order_2。核心是**突破单表数据量上限**。

**适用场景：**
垂直分库：微服务拆分时自然要做，每个服务对应独立数据库。
垂直分表：单表字段过多（超过 30 列），且有大字段低频访问时。
水平分库：单库 QPS 过万，连接数不够，需要分散并发压力。
水平分表：单表数据超 2000 万行，查询变慢，需要分散数据量。

**一句话总结：**
垂直拆是按业务/列拆，水平拆是按行数据分散；先做垂直拆解耦业务，再做水平拆突破单机瓶颈。

# 3. 什么时候该分表？什么时候该分库？

**核心回答：** **单表数据量过大导致慢查询就分表，单库并发/连接数扛不住就分库**，两者可以独立做也可以组合做。

**适用场景：**

1. **只分表不分库**：单表数据量大（超 2000 万行）但并发量不高（QPS < 3000），只需要在同一个库内拆成多张表减少单表数据量即可。
2. **只分库不分表**：并发量高（QPS 过万）但单表数据量不大，只需要把同一个表分散到多个库实例上，每个库扛一部分并发。
3. **既分库又分表**：数据量大且并发量高，最常见的情况。比如电商订单表，数据量过亿且 QPS 过万，必须分库分散并发 + 分表分散数据量。

**避坑指南：**

1. **不要过早分库分表**：分库分表引入的复杂度（跨库事务、跨库查询、数据迁移）远超想象，数据量不到瓶颈时优先考虑优化 SQL、加索引、读写分离、加缓存。
2. **不要盲目追求拆分**：拆分后很难回退，一旦分了就很难合并回来。建议先做读写分离，再考虑分库分表。
3. **拆分粒度要留余量**：分 8 个库还是 16 个库？要预估未来 3-5 年数据增长，一次拆到位，避免二次拆分（数据迁移成本极高）。

**一句话总结：**
数据多分表，并发高分库，两者都高就既分库又分表；但别过早拆，优先用缓存和读写分离兜底。

# 4. 如何选择分片键？分片策略有哪些？

**核心回答：** 分片键的选择原则是**数据分布均匀、查询尽量命中单分片、避免跨分片查询**。分片策略主要有范围分片、Hash 取模分片、一致性 Hash 分片。

**底层原理：**

1. **分片键选择原则**：
   - **高基数**：分片键的值要足够分散，避免数据倾斜（如按 status 分片只有几个值，大量数据集中在一个分片）。
   - **查询高频**：分片键要出现在大部分核心查询的 WHERE 条件中，保证查询能路由到单个分片。
   - **避免热点**：不能选自增 ID 做范围分片，否则新数据全部分到最新分片，写热点严重。
   - 常见选择：用户表选 user_id，订单表选 user_id 或 order_id，日志表选 create_time。

2. **范围分片（Range）**：按分片键的值范围分配，如 id 1-100 万在分片1，100 万-200 万在分片2。优点是范围查询友好、扩容方便；缺点是热点问题严重，新数据集中写入最新分片。

3. **Hash 取模分片**：shard = hash(key) % N，数据均匀分布。优点是数据分布均匀、无热点；缺点是扩容时需要重新计算所有数据位置（rehash），数据迁移量大。

4. **一致性 Hash 分片**：将分片节点和数据都映射到 Hash 环上，数据顺时针找最近节点。优点是扩容只需迁移少量数据；缺点是数据分布可能不均匀，需引入虚拟节点解决。

**避坑指南：**

1. **分片键一旦选定不可更改**：后续所有查询和路由都依赖分片键，改分片键等于重新做数据迁移，代价极大。
2. **非分片键查询必跨分片**：如果按 user_id 分片但按 order_time 查询，会扫描所有分片再合并，性能极差。解决方案是建立**基因法索引**或**广播表**。
3. **取模数量建议用 2 的幂次**：方便后续扩容时做翻倍扩容，减少数据迁移量。

**一句话总结：**
分片键选高基数、查询高频的字段，策略选 Hash 取模最均匀，一致性 Hash 扩容最友好，范围分片最容易出现热点。

# 5. 分库分表后分布式 ID 怎么生成？

**核心回答：** 分库分表后自增 ID 无法保证全局唯一，常用方案有**UUID、数据库号段模式、Snowflake 雪花算法、Redis 自增**。

**核心区别：**

1. **UUID**：128 位随机值，全局唯一但不有序、无业务含义、索引性能差（随机插入导致 B+树频繁页分裂），**不推荐作为主键**。
2. **数据库号段模式**：用一张表存储当前最大 ID 和步长，每次取一批号段缓存在本地（如一次取 1000 个），用完再取。美团的 Leaf-Segment 就是这个方案。优点是简单可靠、ID 有序；缺点是依赖数据库，DB 挂了就没号发。
3. **Snowflake 雪花算法**：64 位 Long 型 ID = 1 位符号 + 41 位时间戳 + 10 位机器 ID + 12 位序列号。优点是高性能、趋势递增、不依赖数据库；缺点是依赖时钟，时钟回拨会生成重复 ID。
4. **Redis 自增**：利用 Redis 的 INCR 命令生成 ID，速度快但依赖 Redis 可用性，且需要持久化防止丢号。

**实战选型：**
中小规模系统用**数据库号段模式**，简单可靠；大规模分布式系统用**Snowflake**，性能最高，但要解决时钟回拨问题（可引入 ZooKeeper 分配 workerId 或用 Leaf-Segment 兜底）。阿里、美团内部都基于 Snowflake 做了定制化改造。

**一句话总结：**
UUID 不做主键，中小系统用号段模式，大系统用雪花算法，时钟回拨是唯一坑。
# 6. 分库分表后跨库事务怎么处理？

**核心回答：** 分库后本地事务失效，必须用**分布式事务方案**解决。常见方案有 XA 两阶段提交、TCC、Saga、本地消息表 + 最终一致性。

**底层原理：**

1. **XA 两阶段提交**：数据库层面支持，协调者先 Prepare 再 Commit。优点是对业务无侵入；缺点是同步阻塞、性能差、协调者单点故障风险大。适合强一致性要求高、并发量不大的场景。
2. **TCC（Try-Confirm-Cancel）**：业务层面实现，Try 预留资源，Confirm 确认执行，Cancel 取消释放。优点是性能好、无锁；缺点是业务侵入大，每个接口要写三个方法，还要处理幂等、空回滚、悬挂问题。
3. **Saga 模式**：长事务拆成多个本地事务，每个本地事务配一个补偿操作，失败时反向补偿。优点是适合长流程业务；缺点是缺乏隔离性，中间状态可见。
4. **本地消息表 + 最终一致性**：业务操作和消息写入在同一个本地事务中，后台线程异步发送消息到 MQ，消费方幂等消费。这是互联网公司**最常用**的方案，实现简单、可靠性高。

**避坑指南：**

1. **不要用强一致性方案处理高并发业务**：XA 性能太差，电商订单场景下 TPS 会掉到百级别。
2. **优先用最终一致性**：大部分业务场景（下单、支付回调、库存扣减）都能接受短暂不一致，本地消息表是最务实的选择。
3. **幂等是前提**：无论哪种方案，消费方都必须幂等，否则重复消费会导致数据错乱。

**一句话总结：**
强一致性用 XA（性能差），业务侵入用 TCC，互联网首选本地消息表 + 最终一致性，幂等是所有方案的前提。

# 7. 分库分表后跨库分页查询怎么做？

**核心回答：** 跨库分页是分库分表后**最难处理的问题之一**，常见方案有全局汇总法、二次查询法、允许折中的业务方案。

**底层原理：**

1. **全局汇总法**：每个分片执行 ORDER BY x LIMIT offset, size，把所有分片结果汇总到应用层再排序、截取。优点是结果准确；缺点是深分页时每个分片返回大量数据，应用层合并压力大，性能极差。比如查第 1000 页，每个分片要返回前 1000 页数据再合并。
2. **二次查询法**：第一次查询各分片 LIMIT offset, size，找到所有分片返回的最小值；第二次以最小值为起点再查各分片，合并后取正确结果。减少了数据传输量，但实现复杂，且仍然无法避免多次查询。
3. **业务折中方案**：
   - **禁止深分页**：只提供上一页/下一页游标分页（基于上次最后一条记录的 ID 翻页），不提供跳页功能。这是互联网产品的通用做法。
   - **用 ES 辅助查询**：将分页查询需要的字段同步到 ES，分页查询走 ES，精确定位后拿主键回分库查详情。
   - **用汇总表/宽表**：另建一张不分片的汇总表，专门供分页查询使用，通过 binlog 同步数据。

**避坑指南：**

1. **深分页是性能杀手**：LIMIT 100000, 10 在单库就慢，跨库更慢，必须从业务上限制或用游标分页替代。
2. **不要用 count 汇总总数**：跨库 COUNT 需要扫描所有分片，大数据量下极慢。业务上可以用近似值或缓存总数。

**一句话总结：**
跨库分页无完美方案，业务上优先游标分页禁跳页，技术上用 ES 或汇总表兜底，深分页必须干掉。

# 8. 分库分表后如何做跨库 JOIN？

**核心回答：** 分库分表后跨库 JOIN 基本不可用，需要通过**冗余字段、广播表、应用层组装、ES 宽表**等方式替代。

**适用场景：**

1. **冗余字段**：在订单表中冗余用户姓名，避免 JOIN 用户表。这是最简单直接的方案，用数据冗余换查询性能，适合冗余字段不常变更的场景。
2. **广播表/小表广播**：将小表（如字典表、配置表、地区表）在每个分库中都存一份完整数据，写入时同步到所有分库。这样 JOIN 操作在单个库内即可完成。ShardingSphere 原生支持广播表。
3. **应用层组装**：先查订单表拿到 user_id 列表，再批量查用户表，在应用内存中组装结果。优点是灵活；缺点是代码复杂、多次网络调用。适合关联不频繁的场景。
4. **ES 宽表**：将需要 JOIN 查询的多张表数据合并写入 ES，形成宽表文档，直接在 ES 中查询。适合复杂查询、多维度搜索场景。

**避坑指南：**

1. **分库分表前就要梳理 JOIN 关系**：频繁 JOIN 的表尽量放在同一个库中（绑定表），避免后续改造。
2. **广播表数据量必须小**：广播表数据量大了写入同步压力大，一般建议不超过 1 万行。
3. **冗余字段要考虑一致性**：冗余字段变更时需要同步更新，可用 MQ 异步更新或 binlog 监听。

**一句话总结：**
跨库 JOIN 靠冗余字段和广播表解决小表关联，靠应用层组装和 ES 宽表解决复杂查询，设计阶段就要提前规划绑定表。

# 9. 分库分表后数据迁移如何平滑进行？

**核心回答：** 平滑迁移的核心思路是**双写 + 增量同步 + 校验 + 切换**，全程不停服、不丢数据。

**底层原理：**

1. **双写阶段**：应用层同时写旧库和新库（或新分片），保证新数据两边都有。旧库仍然是主库，新库先写但不对外服务。
2. **增量同步阶段**：用 Canal 等工具订阅旧库 binlog，将历史存量数据同步到新分片，追平数据差异。存量同步完成后，新库数据基本追平。
3. **数据校验阶段**：对比旧库和新库的数据一致性，逐表逐行校验（可用自研脚本或开源工具如 DataX），发现不一致则修复。
4. **灰度切流阶段**：将部分读流量切到新库验证，逐步增大比例（10% 到 50% 到 100%），确认无问题后写入也切到新库，关闭旧库双写。

**避坑指南：**

1. **双写要保证顺序**：先写旧库再写新库，若新库写入失败要做补偿（记录失败日志，后台重试）。
2. **迁移期间 DDL 变更要禁止**：迁移过程中禁止表结构变更，否则同步会出错。
3. **回滚方案必须提前准备**：切流后如果新库出问题，要能快速切回旧库，旧库在切换完成前不能下线。
4. **校验是关键**：不校验就切换等于裸奔，数据不一致上了线就是生产事故。

**一句话总结：**
双写保数据不丢，binlog 追平历史数据，校验保一致性，灰度切流保安全，全程可回滚。

# 10. ShardingSphere 和 MyCAT 怎么选？

**核心回答：** ShardingSphere 是**当前主流的分布式数据库中间件**，社区活跃、功能完善、架构先进；MyCAT 是早期方案，社区已基本停滞，**新项目不建议使用**。

**核心区别：**

1. **架构模式**：
   - MyCAT 是 **Proxy 模式**，独立部署代理层，所有 SQL 经过 MyCAT 路由转发，多一次网络跳转。
   - ShardingSphere 支持 **Proxy 模式 + JDBC 模式**。JDBC 模式（ShardingSphere-JDBC）以 jar 包嵌入应用，无额外网络开销，性能最高；Proxy 模式（ShardingSphere-Proxy）独立部署，适合异构语言场景。
2. **社区与维护**：
   - MyCAT 社区已停更，issue 无人处理，版本停留在 1.6，bug 多且无修复。
   - ShardingSphere 是 Apache 顶级项目，社区活跃，版本持续迭代，功能不断增强。
3. **功能对比**：
   - MyCAT：支持分库分表、读写分离，但对复杂 SQL 支持差，跨库聚合、子查询容易出 bug。
   - ShardingSphere：支持分库分表、读写分离、广播表、绑定表、分布式事务（XA/Seata）、数据加密、影子库压测，SQL 兼容性好。

**适用场景：**
ShardingSphere-JDBC：Java 技术栈首选，性能最高，零网络开销，适合微服务架构。
ShardingSphere-Proxy：多语言技术栈（Go/Python/Node.js），需要统一代理入口。
MyCAT：仅维护已有老项目时使用，新项目禁止选用。

**实战选型：**
Java 技术栈无脑选 ShardingSphere-JDBC；多语言异构系统选 ShardingSphere-Proxy；MyCAT 只在维护遗留系统时使用，绝不用于新项目。

**一句话总结：**
新项目选 ShardingSphere，Java 用 JDBC 模式性能最高，多语言用 Proxy 模式，MyCAT 已淘汰别碰。
