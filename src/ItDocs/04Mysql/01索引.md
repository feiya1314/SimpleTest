# 1. MySQL索引数据结构

MySQL索引使用什么数据结构？为什么不用二叉树/红黑树/跳表？

**原理分析**

**MySQL索引结构演进：**

| 数据结构 | 问题 |
| --- | --- |
| 二叉树 | 可能退化为链表，查找复杂度O(log n)→O(n) |
| 红黑树 | 树深度可能很大，IO次数多 |
| B树 | 解决了深度问题，但所有数据都存储在叶子节点 |
| **B+树** | 非叶子节点只存储索引，IO次数更少 |

**B+树特点：**

1. **非叶子节点不存储数据**：每个节点可容纳更多索引
2. **叶子节点链表连接**：支持范围查询
3. **IO次数优化**：树深度更浅

```
B树：每个节点都存储数据和指针
[data|ptr] [data|ptr] [data|ptr] ...

B+树：
        [index]
      /        \
[index]        [index]      ← 非叶子节点只有索引
   ↓             ↓
[data] ←→ [data] ←→ [data] ←→ ...  ← 叶子节点链表
```

> 为什么B+树比B树更适合数据库索引？

1. **磁盘友好**：每个节点一个page，B+树存储更多索引，深度更浅
2. **范围查询高效**：叶子节点链表支持快速范围扫描
3. **查询稳定性**：所有查询都到叶子节点，复杂度固定

# 2. 聚簇索引与非聚簇索引

聚簇索引和非聚簇索引有什么区别？InnoDB和MyISAM的索引实现有何不同？

**原理分析**

**聚簇索引（Clustered Index）：**

- 数据行存储在索引的叶子节点中
- 主键索引就是聚簇索引
- 每张表只有一个聚簇索引

**非聚簇索引（Secondary Index）：**

- 叶子节点存储主键值，需要回表
- 所有非主键索引都是非聚簇索引

**InnoDB vs MyISAM：**

| 特性 | InnoDB | MyISAM |
| --- | --- | --- |
| 索引类型 | 聚簇索引 | 非聚簇索引 |
| 主键 | 必须有，建议自增 | 可以没有 |
| 数据存储 | 主键索引叶子节点存储完整行 | 索引叶子节点存储行号 |
| 回表 | 非主键索引需要回表 | 直接获取行号，无需回表 |

**InnoDB聚簇索引示例：**

```sql
-- 创建表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(64),
    amount DECIMAL(10,2),
    created_at DATETIME,
    INDEX idx_order_no (order_no)
);

-- 主键id自动创建聚簇索引
-- order_no创建非聚簇索引
-- 通过order_no查询需要回表：
-- 1. 在idx_order_no索引树查找order_no
-- 2. 获取主键id
-- 3. 在主键索引树查找完整行
```

> 为什么推荐使用自增主键？

1. **插入效率高**：顺序写入，页分裂少
2. **索引紧凑**：占用空间小
3. **减少碎片**：避免随机插入导致的页分裂

# 3. 联合索引与最左前缀原则

什么是联合索引？为什么遵循最左前缀原则？

**原理分析**

**联合索引结构：**

```sql
CREATE INDEX idx_name_age ON users(name, age, city);
```

```
联合索引B+树：
                    [name|age|city|主键]
                    /                \
               [a|20|beijing]        [b|25|shanghai]
               /            \        ...
         [a|18|beijing]   [a|25|shanghai]
         /        \        ...
    [a|15|gz]  [a|16|cd]  [a|30|cd]
    ...
```

**最左前缀原则：**

| 查询条件 | 是否使用索引 | 原因 |
| --- | --- | --- |
| `WHERE name = 'a'` | ✓ | 命中最左前缀 |
| `WHERE name = 'a' AND age = 20` | ✓ | 命中索引 |
| `WHERE name = 'a' AND age = 20 AND city = 'beijing'` | ✓ | 命中全部 |
| `WHERE age = 20` | ✗ | 未使用最左列 |
| `WHERE city = 'beijing'` | ✗ | 未使用最左列 |

**索引失效场景：**

```sql
-- 索引 idx(name, age, city)
-- 以下情况会导致索引失效或部分失效

-- 1. 使用函数或运算
WHERE YEAR(created_at) = 2024

-- 2. 前导列使用范围查询，后面的列无法使用
WHERE name = 'a' AND age > 20

-- 3. 使用LIKE前缀通配
WHERE name LIKE '%a%'

-- 4. 类型不匹配
WHERE name = 123  -- name是varchar，传入int
```

> 为什么范围查询后面的列无法使用索引？

范围查询（如`> 20`）会定位到一个节点，后面的列虽然还在同一条路径上，但索引树无法继续精确匹配。

### 4. 索引下推与查询优化

MySQL 5.6引入的索引下推是什么？它如何优化查询？

**原理分析**

**索引下推（Index Condition Pushdown, ICP）：**

在索引遍历过程中，对索引中包含的字段先做过滤，减少回表次数。

**示例：**

```sql
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(20),
    age INT,
    city VARCHAR(20),
    INDEX idx_name_age_city (name, age, city)
);

-- 查询
SELECT * FROM user WHERE name = 'zhang' AND age > 18 AND city LIKE 'bei%';
```

**无ICP流程（MySQL 5.5）：**

1. 根据`name`在索引树定位到叶子节点
2. 逐个回表获取完整行
3. 在Server层过滤`age > 18 AND city LIKE 'bei%'`

**有ICP流程（MySQL 5.6+）：**

1. 根据`name`在索引树定位到叶子节点
2. **在索引树中先用age过滤**
3. **再判断city条件**
4. 只回表获取符合条件的行

> ICP的适用条件？

1. 需访问表的所有字段
2. 条件是索引列的一部分（从最左列开始）
3. 用于InnoDB和MyISAM引擎
4. 不能用于全文索引

### 5. 覆盖索引与回表查询

什么是覆盖索引？如何利用覆盖索引避免回表？

**原理分析**

**回表查询：**

非聚簇索引叶子节点只存储主键值，需要根据主键回主键索引获取完整数据。

```sql
-- 表结构
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(64),
    amount DECIMAL(10,2),
    INDEX idx_order_no (order_no)
);

-- 查询
SELECT id, order_no FROM orders WHERE order_no = 'A001';
-- 只需要id和order_no，不需要回表
-- idx_order_no索引树叶子节点已包含所需字段
```

**覆盖索引：**

查询的所有字段都包含在索引中，无需回表。

```sql
-- 创建覆盖索引
CREATE INDEX idx_order_cover ON orders(order_no, amount);

-- 查询
SELECT order_no, amount FROM orders WHERE order_no = 'A001';
-- 使用覆盖索引，无需回表
```

**性能对比：**

```
普通索引回表：
索引查找 → 回表读取完整行 → 返回结果

覆盖索引：
索引查找 → 直接返回结果
```

> 为什么有时候宁愿回表也不用覆盖索引？

1. 覆盖索引会让索引变大，占用更多磁盘空间
2. 写入数据时需要维护更多索引，插入性能下降
3. 查询字段较多时，覆盖索引不现实

### 6. 索引失效的场景分析

哪些情况会导致索引失效？如何排查？

**原理分析**

**常见索引失效场景：**

**1. 条件中使用函数或运算**

```sql
-- 索引失效
WHERE YEAR(created_at) = 2024
WHERE age + 1 = 20

-- 正确写法
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'
WHERE age = 19
```

**2. 类型隐式转换**

```sql
-- phone是VARCHAR类型，传入数字
WHERE phone = 13012345678  -- phone = '13012345678' 发生转换
WHERE phone = '13012345678'  -- 正确
```

**3. 使用LIKE前置通配**

```sql
-- 索引失效
WHERE name LIKE '%zhang'

-- 索引生效
WHERE name LIKE 'zhang%'
```

**4. OR连接不同字段**

```sql
-- 假设有idx_name和idx_age
WHERE name = 'zhang' OR age = 18  -- 索引失效
-- 改为UNION
WHERE name = 'zhang'
UNION ALL
WHERE age = 18
```

**5. 索引列参与计算**

```sql
WHERE id + 1 = 100  -- 失效
WHERE id = 99  -- 生效
```

**排查方法：**

```sql
EXPLAIN SELECT * FROM orders WHERE order_no = 'A001';
-- 查看type列：ref=使用索引，ALL=全表扫描
-- 查看key列：实际使用的索引
-- 查看Extra列：Using index=覆盖索引，Using filesort=需排序
```

> 如何强制使用某个索引？

```sql
SELECT * FROM orders USE INDEX (idx_order_no) WHERE order_no = 'A001';
SELECT * FROM orders FORCE INDEX (idx_order_no) WHERE order_no = 'A001';
-- 不推荐：优化器通常比人更了解数据分布
```

### 7. 索引建立原则与优化

如何合理地建立索引？有哪些原则？

**原理分析**

**索引建立原则：**

**1. 区分度高的列优先**

```sql
-- 性别（只有男/女）区分度低，不适合建索引
-- 订单号区分度高，适合建索引

SELECT COUNT(DISTINCT col) / COUNT(*) AS selectivity;
-- 区分度越接近1越好
```

**2. 尽是小而美**

```sql
-- 用INT代替BIGINT
-- 用SMALLINT代替INT（如果足够）
-- 用VARCHAR(20)代替VARCHAR(255)
```

**3. 避免重复索引**

```sql
-- 冗余
INDEX idx_a (a)
INDEX idx_a_b (a, b)  -- a已在前面的索引中

-- 正确
INDEX idx_a_b (a, b)  -- 覆盖单个a查询
```

**4. 考虑查询类型**

```sql
-- 等值查询适合B树索引
WHERE status = 1

-- 范围查询适合B树索引
WHERE id > 100 AND id < 200

-- 全文搜索需要全文索引
WHERE MATCH(content) AGAINST('keyword')

-- 地理位置查询需要空间索引
ST_Contains(location, point)
```

**5. 考虑写入性能**

每个索引都会增加写操作的开销：

- 插入时需要更新所有索引
- 更新时如果索引列被修改，需要更新索引
- 删除时需要删除所有索引中的记录
