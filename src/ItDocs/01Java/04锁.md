# 锁

### 1. AQS的核心原理与实现

AQS的核心原理是什么？它是如何实现锁的？

**原理分析**

**AQS（AbstractQueuedSynchronizer）核心组件：**

1. **state**：同步状态变量，int类型
2. **Node**：队列节点，存储等待线程
3. **CLH队列**：双向FIFO队列

**状态管理：**

```java
private volatile int state;  // 同步状态

protected final int getState() {
    return state;
}

protected final void compareAndSetState(int expect, int update) {
    return unsafe.compareAndSwapInt(this, stateOffset, expect, update);
}
```

**Node结构：**

```java
static final class Node {
    volatile int waitStatus;  // 等待状态
    volatile Node prev;       // 前驱节点
    volatile Node next;       // 后继节点
    volatile Thread thread;   // 等待线程
}
```

**获取锁流程（acquire）：**

```java
public final void acquire(int arg) {
    if (!tryAcquire(arg)) {
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg);
    }
}
```

1. `tryAcquire(arg)` - 尝试获取锁（子类实现）
2. 失败则`addWaiter`创建Node加入队列
3. `acquireQueued`自旋尝试获取锁：
   - 如果是队首且获取成功 → 出队并返回
   - 否则**park**阻塞等待前驱节点唤醒

**锁的类型区分：**

- **独占锁**：`tryAcquire`成功返回true
- **共享锁**：`tryAcquireShared`返回剩余资源数，>0表示成功

> AQS如何实现公平锁和非公平锁？

**非公平锁**：直接尝试CAS获取，不检查队列
**公平锁**：检查`hasQueuedPredecessors()`是否有前驱等待者

### 2. ReentrantLock的公平锁与非公平锁

ReentrantLock的公平锁和非公平锁有什么区别？各自的实现原理？

**原理分析**

**区别：**

| 特性 | 公平锁 | 非公平锁 |
|-----|-------|---------|
| 等待队列 | FIFO，先到先得 | 允许插队 |
| 吞吐量 | 较低 | 较高 |
| 实现 | hasQueuedPredecessors() | 直接CAS |

**非公平锁优势：**

1. **减少线程切换开销**
2. **减少阻塞/唤醒开销**
3. **让持有锁的线程快速释放**

**实现细节：**

```java
// 非公平获取
final boolean nonfairTryAcquire(int acquires) {
    Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    }
    // 可重入
    if (current == getExclusiveOwnerThread()) {
        int nextc = c + acquires;
        setState(nextc);
        return true;
    }
    return false;
}
```

> 什么场景下使用公平锁？

1. 需要严格FIFO顺序
2. 避免线程饥饿
3. 资源分配需要确定性

但由于公平锁吞吐量低，一般场景建议使用非公平锁。

### 3. ReadWriteLock读写锁与写锁饥饿问题

ReadWriteLock的读锁和写锁有什么区别？读写锁的写锁饥饿问题如何解决？

**原理分析**

**读写锁特点：**

- **读锁**：共享锁，多个线程可同时持有
- **写锁**：独占锁，只有一个线程持有
- **读写互斥**：读和写不能同时进行

**写锁饥饿问题：**

当大量读线程持有锁时，写线程可能长期无法获取锁：

```
读线程1 → 持有读锁
读线程2 → 持有读锁
...
写线程 → 等待（无法获取，写锁饥饿）
```

**解决方案：**

1. **JDK8写锁优先模式（StampedLock）**

```java
StampedLock lock = new StampedLock();

// 乐观读
long stamp = lock.tryOptimisticRead();
if (!lock.validate(stamp)) {
    stamp = lock.readLock();
    try { } finally { lock.unlockRead(stamp); }
}

// 写模式
long stamp = lock.writeLock();
try { } finally { lock.unlockWrite(stamp); }
```

2. **使用公平锁**

```java
ReadWriteLock lock = new ReentrantReadWriteLock(true);
```

> StampedLock与ReentrantReadWriteLock的区别？

| 特性 | ReentrantReadWriteLock | StampedLock |
|-----|----------------------|-------------|
| 乐观读 | 不支持 | 支持 |
| 升级 | 不支持 | 支持 |
| 公平性 | 支持 | 不支持 |
| 性能 | 较好 | 更好 |

### 4. Condition的await与signal原理

Condition的await和signal原理是什么？它与Object的wait/notify有什么区别？

**原理分析**

**Condition核心原理：**

```java
// ReentrantLock.newCondition() 返回ConditionObject
public class ConditionObject implements Condition {
    private transient Node firstWaiter;
    private transient Node lastWaiter;
}
```

**await流程：**

```java
public final void await() throws InterruptedException {
    Node node = addConditionWaiter();
    int savedState = fullyRelease(node);
    while (!isOnSyncQueue(node)) {
        LockSupport.park(this);
    }
    acquireQueued(node, savedState);
}
```

**signal流程：**

```java
public final void signal() {
    Node first = firstWaiter;
    if (first != null) doSignal(first);
}

private void doSignal(Node first) {
    do {
        firstWaiter = first.nextWaiter;
        if (firstWaiter == null) lastWaiter = null;
    } while (!transferForSignal(first) && (first = firstWaiter) != null);
}
```

**与Object.wait/notify的区别：**

| 特性 | Object.wait/notify | Condition.await/signal |
|-----|---------------------|----------------------|
| 绑定 | 与synchronized绑定 | 与ReentrantLock绑定 |
| 数量 | 单一waitSet | 多个Condition |
| 超时等待 | wait(long) | **await(long)** |
| 公平性 | 不支持 | 支持 |

> 为什么Condition.signalAll比signal更好？

signal只唤醒一个等待线程，如果被唤醒的线程无法满足条件，会导致其他线程继续等待。

### 5. StampedLock的乐观读模式

StampedLock的乐观读是什么？它是如何工作的？

**原理分析**

**StampedLock的三种模式：**

1. **写模式**：独占锁
2. **乐观读模式**：不获取锁，尝试读取
3. **读模式**：共享锁

**乐观读流程：**

```java
public class Point {
    private final StampedLock sl = new StampedLock();

    double distanceFromOrigin() {
        long stamp = sl.tryOptimisticRead();
        double result = x * x + y * y;
        if (!sl.validate(stamp)) {
            stamp = sl.readLock();
            try { result = x * x + y * y; }
            finally { sl.unlockRead(stamp); }
        }
        return result;
    }
}
```

**validate检查原理：**

```java
public boolean validate(long stamp) {
    return (stamp & SBITS) == (getState() & SBITS);
}
```

> 为什么StampedLock性能更好？

1. **乐观读不需要加锁**，没有线程阻塞
2. **读多写少场景下**，避免了大量读锁竞争
3. **验证失败时才升级为读锁**，开销较小

适用场景：读多写少的场景，如配置信息、缓存等。

### 6. 锁的公平性与性能权衡

为什么非公平锁性能更好？什么场景需要用公平锁？

**原理分析**

**非公平锁性能好的原因：**

1. **减少上下文切换**：当前持有锁的线程释放后，可以直接继续执行

```
非公平：线程A释放 → 线程A继续 → 线程B等待
公平：线程A释放 → 线程B获取 → 线程A重新竞争
```

2. **减少阻塞开销**：非公平锁让线程保持运行状态
3. **提高吞吐量**：允许插队让锁的利用率更高

**公平锁的适用场景：**

1. **需要确定性**：如FIFO顺序处理
2. **防止饥饿**
3. **资源分配需要公平**

> 如何在代码中避免锁饥饿？

1. 使用公平锁
2. 实现加权轮询
3. 使用**StampedLock**代替ReentrantReadWriteLock

### 7. 信号量Semaphore的原理与应用

Semaphore的原理是什么？它可以用于哪些场景？

**原理分析**

**Semaphore核心：**

- 维护一个计数器（permits）
- acquire()：消耗一个许可
- release()：归还一个许可

**实现原理（AQS）：**

```java
abstract static class Sync extends AbstractQueuedSynchronizer {
    final int tryAcquireShared(int reduces) {
        for (;;) {
            int available = getState();
            int remaining = available - reduces;
            if (remaining < 0 || compareAndSetState(available, remaining))
                return remaining;
        }
    }

    protected final boolean tryReleaseShared(int releases) {
        for (;;) {
            int current = getState();
            int next = current + releases;
            if (compareAndSetState(current, next))
                return true;
        }
    }
}
```

**应用场景（限流）：**

```java
class Pool {
    private final Semaphore permits = new Semaphore(10);

    public void execute(Runnable task) throws InterruptedException {
        permits.acquire();
        try { task.run(); }
        finally { permits.release(); }
    }
}
```

> Semaphore与限流算法的关系？

Semaphore是"令牌桶"的简化实现：
- 令牌桶：按固定速率生成令牌
- Semaphore：直接限制并发数，无法精确控制速率

### 8. 锁的升级与降级

锁是否可以降级？ReentrantLock支持锁降级吗？

**原理分析**

**synchronized锁升级（不可逆）：**

```
偏向锁 → 轻量级锁 → 重量级锁（单向）
```

**ReentrantLock锁降级：**

ReentrantLock支持锁降级，但需要小心处理：

```java
lock.unlock();
lock.lock();  // 重新获取写锁
```

**StampedLock支持降级：**

```java
long stamp = lock.writeLock();
// 写操作
long readStamp = lock.tryConvertToReadLock(stamp);
if (readStamp != 0) {
    lock.unlockRead(readStamp);
} else {
    lock.unlockWrite(stamp);
}
```

> 为什么不支持锁降级？

主要考虑：
1. 降级可能导致写锁被意外获取
2. 实现复杂性增加
3. 实际需求较少

StampedLock支持降级是因为它使用了**戳记机制**，更容易实现。

### 9. 可重入锁的可重入次数限制

ReentrantLock的重入次数有上限吗？如果达到上限会怎样？

**原理分析**

**重入次数上限：**

```java
// ReentrantLock使用int类型存储state
private volatile int state;

// state最大值为Integer.MAX_VALUE
// 约为21亿次重入
```

**计算方式：**

```java
protected final boolean tryAcquire(int acquires) {
    Thread current = Thread.currentThread();
    int c = getState();
    int nextc = c + acquires;
    if (nextc < 0)  // overflow
        throw new Error("Maximum lock count exceeded");
}
```

**实际影响：**

```java
public class Test {
    public synchronized void method() {
        method();  // 递归调用
    }
    // 约21亿次递归后会抛出Error
}
```

实际应用中几乎不会达到这个限制。

> 为什么设置Integer.MAX_VALUE上限？

int类型是最自然的同步状态存储方式，使用其他类型（如long）会增加复杂性，且21亿次重入在现实中不可能达到。

### 10. 死锁的检测与预防

Java中如何检测和预防死锁？

**原理分析**

**死锁条件（Coffman）：**

1. **互斥条件**：资源独占
2. **占有并等待**：持有资源同时等待其他资源
3. **不可抢占**：资源不能被强制夺取
4. **循环等待**：形成资源等待环

**检测工具：**

```bash
# JStack检测死锁
jstack <pid>
```

**预防策略：**

1. **固定加锁顺序**：

```java
void methodA() {
    int hash1 = System.identityHashCode(lock1);
    int hash2 = System.identityHashCode(lock2);
    if (hash1 < hash2) {
        synchronized (lock1) {
            synchronized (lock2) { }
        }
    } else {
        synchronized (lock2) {
            synchronized (lock1) { }
        }
    }
}
```

2. **tryLock超时获取**：

```java
if (!lock1.tryLock(1000, TimeUnit.MILLISECONDS)) return;
try {
    if (!lock2.tryLock(1000, TimeUnit.MILLISECONDS)) return;
} finally { lock2.unlock(); }
finally { lock1.unlock(); }
```

3. **减少持有锁时间**：

```java
synchronized (lock) {
    unlock();
    externalCall();
    lock();
}
```

> 活锁与死锁的区别？

- **死锁**：线程无法继续执行（相互等待）
- **活锁**：线程在不断执行但无法前进（总是互相让步）

解决活锁：增加随机等待或回退机制。
