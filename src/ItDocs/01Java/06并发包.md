# 并发包

### 1. CountDownLatch与CyclicBarrier的区别与应用

CountDownLatch和CyclicBarrier有什么区别？各自的适用场景是什么？

**原理分析**

**区别对比：**

| 特性 | CountDownLatch | CyclicBarrier |
|-----|---------------|----------------|
| 作用 | 主线程等待多个线程完成 | 多个线程相互等待 |
| 计数重置 | **不可重置**（一次性） | **可重置**（循环使用） |
| 内部机制 | 减计数 | 阻塞等待+重置 |
| 线程参与 | 不参与计数过程 | 参与等待 |

**CountDownLatch使用场景：**

```java
CountDownLatch startSignal = new CountDownLatch(1);
CountDownLatch doneSignal = new CountDownLatch(3);

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        try {
            startSignal.await();  // 等待开始信号
            System.out.println("玩家开始游戏");
            System.out.println("玩家完成");
        } finally {
            doneSignal.countDown();
        }
    }).start();
}

Thread.sleep(1000);
startSignal.countDown();  // 发出开始信号
doneSignal.await();  // 等待所有玩家完成
System.out.println("游戏结束");
```

**CyclicBarrier使用场景：**

```java
CyclicBarrier barrier = new CyclicBarrier(4);

for (int i = 0; i < 4; i++) {
    new Thread(() -> {
        Arrays.sort(arr, start, end);
        try {
            barrier.await();  // 等待其他线程
        } catch (Exception e) {}
    }).start();
}
```

> CyclicBarrier为什么可以循环使用？

因为内部有**generation**对象，每次await完成后会重置generation，准备下一轮使用。

### 2. Semaphore的原理与限流场景

Semaphore的原理是什么？它可以用于哪些场景？

**原理分析**

**核心概念：**

- **permits**：许可证数量，相当于资源数
- **acquire()**：获取许可证，资源-1
- **release()**：归还许可证，资源+1

**实现原理（AQS）：**

```java
abstract static class Sync extends AbstractQueuedSynchronizer {
    // 非公平获取
    final int nonfairTryAcquireShared(int acquires) {
        for (;;) {
            int available = getState();
            int remaining = available - acquires;
            if (remaining < 0 || compareAndSetState(available, remaining))
                return remaining;
        }
    }

    // 释放
    protected final boolean tryReleaseShared(int releases) {
        for (;;) {
            int current = getState();
            int next = current + releases;
            if (compareAndSetState(current, next))
                return true;
        }
    }
}
```

**应用场景（限流）：**

```java
class ConnectionPool {
    private final Semaphore permits = new Semaphore(10);

    public Connection getConnection() throws InterruptedException {
        permits.acquire();
        return getAvailableConnection();
    }

    public void releaseConnection(Connection conn) {
        releaseConnection(conn);
        permits.release();
    }
}
```

> Semaphore如何实现公平与非公平？

```java
// 公平模式
Semaphore fairSemaphore = new Semaphore(10, true);
```

公平模式会检查队列是否有等待更久的线程，非公平模式直接尝试CAS获取。

### 3. CompletableFuture的异步编程用法

CompletableFuture是什么？它可以用于哪些异步编程场景？

**原理分析**

**创建CompletableFuture：**

```java
// 1. supplyAsync: 有返回值
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> "result");

// 2. runAsync: 无返回值
CompletableFuture<Void> future = CompletableFuture.runAsync(() -> {});

// 3. 自定义线程池
ExecutorService executor = Executors.newFixedThreadPool(10);
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> "result", executor);
```

**组合操作：**

```java
// thenApply: 转换结果
CompletableFuture<Integer> f1 = CompletableFuture.supplyAsync(() -> "100")
    .thenApply(Integer::parseInt);

// thenCompose: 扁平化（用于异步依赖）
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> "hello")
    .thenCompose(s -> CompletableFuture.supplyAsync(() -> s + " world"));

// thenCombine: 组合两个独立任务
CompletableFuture<Integer> f3 = CompletableFuture.supplyAsync(() -> 1)
    .thenCombine(CompletableFuture.supplyAsync(() -> 2), Integer::sum);

// allOf: 等待所有完成
CompletableFuture<Void> all = CompletableFuture.allOf(f1, f2, f3);

// anyOf: 任意一个完成
CompletableFuture<Object> any = CompletableFuture.anyOf(f1, f2, f3);
```

**异常处理：**

```java
// 异常时返回默认值
CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> "value")
    .exceptionally(ex -> "default value");

// 超时处理
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> "value")
    .orTimeout(1, TimeUnit.SECONDS);
```

> CompletableFuture与Future的区别？

| 特性 | Future | CompletableFuture |
|-----|-------|-------------------|
| 获取结果 | get()阻塞 | then系列回调 |
| 组合 | 手动组合 | 内置组合方法 |
| 异常处理 | get()时抛异常 | **exceptionally**/handle |

### 4. LongAdder为什么比AtomicLong性能更好

LongAdder是什么？为什么它比AtomicLong性能更好？

**原理分析**

**AtomicLong问题：**

```java
// 多线程竞争AtomicLong
AtomicLong count = new AtomicLong();
// 线程1: CAS 0->1 (成功)
// 线程2: CAS 0->1 (失败，重试)
// 线程3: CAS 0->1 (失败，重试)
// 竞争激烈时，CAS失败重试开销大
```

**LongAdder解决思路：**

- 使用**多个计数器（cell）**，分散竞争
- 最终求和得到总数

```java
public class LongAdder extends Striped64 {
    transient volatile Cell[] cells;
    transient volatile long base;

    public void add(long x) {
        Cell[] as; long b, v; Cell a;
        // 1. 先尝试base
        if ((as = cells) != null || !casBase(b = base, b + x)) {
            // 2. cells有值或有竞争，使用cell
            if ((a = as[getAndAddKey(x)]) != null)
                a.add(x);
            // 3. cells为空，初始化
            else if (casBase(base, b + x)) {}
        }
    }

    @sun.misc.Contended
    static final class Cell {
        volatile long value;
        final boolean cas(long cmp, long val) {
            return UNSAFE.compareAndSwapLong(this, valueOffset, cmp, val);
        }
    }
}
```

**性能对比：**

| 操作 | AtomicLong | LongAdder |
|-----|-----------|-----------|
| 读 | O(1) | O(1)（取base） |
| 写 | O(1) | O(1)（分散到cell） |
| 高并发 | 竞争激烈 | 分散竞争 |

> LongAdder的sum()是准确的吗？

**不是原子操作**，可能在累加过程中有其他线程修改，结果是近似值。但对统计场景（QPS、PV等）足够。

### 5. ThreadLocalRandom与Random的区别

ThreadLocalRandom是什么？它比Random好在哪里？

**原理分析**

**Random的问题：**

```java
Random random = new Random();
// 多个线程共享同一个Random实例
// 竞争同一个seed，导致性能问题
```

**ThreadLocalRandom原理：**

- 每个线程维护**自己的Random实例**
- 无竞争，性能更好

```java
public class ThreadLocalRandom {
    private static final ThreadLocal<ThreadLocalRandom> local =
        new ThreadLocal<ThreadLocalRandom>() {
            protected ThreadLocalRandom initialValue() {
                return new ThreadLocalRandom();
            }
        };

    private long threadLocalRandomSeed;

    public int nextInt(int bound) {
        int r = mix32(nextSeed());
        return r % bound;
    }

    private long nextSeed() {
        threadLocalRandomSeed = rotateLeft(threadLocalRandomSeed + GAMMA, 21);
        return threadLocalRandomSeed;
    }
}
```

**使用：**

```java
ThreadLocalRandom random = ThreadLocalRandom.current();
random.nextInt(100);
```

> 为什么ThreadLocalRandom不需要设置种子？

每个线程初始化时基于**当前时间+内存地址**生成唯一种子。

### 6. BlockingQueue的实现原理与选型

BlockingQueue是什么？ArrayBlockingQueue和LinkedBlockingQueue有什么区别？

**原理分析**

**BlockingQueue核心方法：**

| 方法 | 抛出异常 | 阻塞 | 超时 |
|-----|---------|------|------|
| add(e) | ✓ | ✗ | ✗ |
| offer(e) | ✗ | ✗ | ✗ |
| put(e) | ✗ | ✓ | ✗ |
| offer(e, t, u) | ✗ | ✗ | ✓ |
| take() | ✗ | ✓ | ✗ |
| poll(t, u) | ✗ | ✗ | ✓ |

**ArrayBlockingQueue（数组实现）：**

```java
public class ArrayBlockingQueue<E> {
    final Object[] items;
    int takeIndex;
    int putIndex;
    int count;
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notEmpty = lock.newCondition();
    private final Condition notFull = lock.newCondition();
}
```

**LinkedBlockingQueue（链表实现）：**

```java
public class LinkedBlockingQueue<E> {
    static class Node<E> { E item; Node<E> next; }
    private transient Node<E> head;
    private transient Node<E> last;
    private final ReentrantLock putLock = new ReentrantLock();
    private final ReentrantLock takeLock = new ReentrantLock();
}
```

**区别：**

| 特性 | ArrayBlockingQueue | LinkedBlockingQueue |
|-----|-------------------|---------------------|
| 底层 | 数组（固定容量） | 链表（可动态增长） |
| 容量 | 固定 | Integer.MAX_VALUE |
| 吞吐量 | 较低 | 较高 |
| 锁 | 单一锁 | **两把锁**（take/put分离） |

> LinkedBlockingQueue为什么用两把锁？

takeLock和putLock可以**并发操作**，提高吞吐量。

### 7. ThreadPoolExecutor的执行流程与核心机制

ThreadPoolExecutor的执行流程是什么？任务提交后会经历哪些阶段？

**原理分析**

**完整执行流程：**

```java
public void execute(Runnable command) {
    int c = ctl.get();

    // 1. 检查线程数 < corePoolSize
    if (workerCountOf(c) < corePoolSize) {
        if (addWorker(command, true))
            return;
        c = ctl.get();
    }

    // 2. 检查队列并加入队列
    if (isRunning(c) && workQueue.offer(command)) {
        if (!isRunning(c) && remove(command))
            reject(command);
        else if (workerCountOf(c) == 0)
            addWorker(null, false);
    }

    // 3. 队列满，创建非核心线程
    else if (!addWorker(command, false))
        reject(command);
}
```

**Worker线程执行：**

```java
final void runWorker(Worker w) {
    Runnable task = w.firstTask;
    w.firstTask = null;
    try {
        while (task != null || (task = getTask()) != null) {
            w.lock();
            task.run();
            w.unlock();
        }
    } finally {
        processWorkerExit(w, completedAbruptly);
    }
}
```

**关键流程：**

```
任务提交
  ↓
workerCount < corePoolSize → 创建核心线程
  ↓
加入队列成功 → 检查状态
  ↓
队列已满 → 尝试创建非核心线程
  ↓
创建失败 → 拒绝
```

> 线程池如何实现keepAlive？

```java
Runnable getTask() {
    boolean timed = allowCoreThreadTimeOut || wc > corePoolSize;
    Runnable r = timed ?
        workQueue.poll(timeout, TimeUnit.NANOSECONDS) :
        workQueue.take();
    // 超时后返回null，线程退出
}
```

### 8. FutureTask的实现原理与状态管理

FutureTask是什么？它是如何实现的？

**原理分析**

**FutureTask状态：**

```java
private volatile int state;
private static final int NEW          = 0;    // 新建
private static final int COMPLETING   = 1;    // 正在完成
private static final int NORMAL       = 2;    // 正常完成
private static final int EXCEPTIONAL  = 3;    // 异常完成
private static final int CANCELLED    = 4;    // 已取消
private static final int INTERRUPTING = 5;    // 正在中断
private static final int INTERRUPTED  = 6;    // 已中断
```

**Runnable + Callable实现：**

```java
public class FutureTask<V> implements RunnableFuture<V> {
    private Callable<V> callable;
    private Object outcome;

    public void run() {
        if (state != NEW || !UNSAFE.compareAndSwapObject(...)) return;
        try {
            Callable<V> c = callable;
            if (c != null && state == NEW) {
                V result;
                boolean ran;
                try {
                    result = c.call();
                    ran = true;
                } catch (Throwable ex) {
                    result = null; ran = false;
                    setException(ex);
                }
                if (ran) set(result);
            }
        } finally { runner = null; }
    }

    protected void set(V v) {
        if (UNSAFE.compareAndSwapInt(this, stateOffset, NEW, COMPLETING)) {
            outcome = v;
            UNSAFE.putOrderedInt(this, stateOffset, NORMAL);
            finishCompletion();
        }
    }
}
```

**get阻塞逻辑：**

```java
public V get() throws InterruptedException, ExecutionException {
    int s = state;
    if (s <= COMPLETING) s = awaitDone(false, 0L);
    return report(s);
}
```

> FutureTask的cancel(true)如何中断正在执行的线程？

```java
public boolean cancel(boolean mayInterruptIfRunning) {
    if (state != NEW) return false;
    if (mayInterruptIfRunning) {
        Thread t = runner;
        if (t != null) t.interrupt();
    }
    state = INTERRUPTED;
    finishCompletion();
    return true;
}
```

### 9. 线程池拒绝策略的场景选择

线程池有哪些拒绝策略？在什么场景下需要自定义拒绝策略？

**原理分析**

**JDK内置4种策略：**

| 策略 | 行为 |
|------|------|
| AbortPolicy（默认） | 抛出RejectedExecutionException |
| CallerRunsPolicy | 由调用方线程执行 |
| DiscardPolicy | 静默丢弃任务 |
| DiscardOldestPolicy | 丢弃队列中最老的任务 |

**自定义拒绝策略：**

```java
// 场景1：记录日志 + 持久化
class LoggingRejectPolicy implements RejectedExecutionHandler {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        logger.warn("Task rejected, queue full");
        saveToDisk(r);
    }
}

// 场景2：切换到备用线程池
class FallbackRejectPolicy implements RejectedExecutionHandler {
    private final ExecutorService fallbackExecutor;
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        if (!executor.isShutdown())
            fallbackExecutor.execute(r);
    }
}
```

> 为什么CallerRunsPolicy在高并发场景危险？

高并发下，线程池已满 → CallerRunsPolicy执行 → 调用方线程被阻塞 → 新的请求继续进入 → 雪崩。

### 10. Executors工厂方法的安全隐患

Executors.newFixedThreadPool和newCachedThreadPool有什么问题？

**原理分析**

**问题分析：**

```java
// 问题1: FixedThreadPool（无界队列）
new ThreadPoolExecutor(10, 10, 0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<Runnable>())  // 无界队列

// 问题2: CachedThreadPool（无界线程数）
new ThreadPoolExecutor(0, Integer.MAX_VALUE, 60L, TimeUnit.SECONDS,
    new SynchronousQueue<Runnable>())
```

**风险：**

1. **OOM风险**：无界队列 + 任务堆积 → 内存溢出

```java
FixedThreadPool pool = Executors.newFixedThreadPool(10);
for (int i = 0; i < 1000000; i++) {
    pool.submit(() -> Thread.sleep(1000));
}
// 队列堆积 million 任务 → OOM
```

2. **线程创建过多**：CachedThreadPool在任务量大时创建大量线程

**推荐写法：**

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10, 20, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),
    new ThreadFactory() {
        @Override
        public Thread newThread(Runnable r) {
            return new Thread(r);
        }
    },
    new ThreadPoolExecutor.AbortPolicy()
);
```

> 阿里Java开发手册规定？

**不允许使用Executors创建线程池**，强制使用ThreadPoolExecutor，避免资源耗尽。
