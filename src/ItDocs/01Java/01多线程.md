# 多线程

### 1. Java线程的生命周期及状态转换

请详细描述Java线程的生命周期及状态转换过程？

**原理分析**

Java线程的生命周期分为6种状态：**NEW**、**RUNNABLE**、**BLOCKED**、**WAITING**、**TIMED_WAITING**、**TERMINATED**。

**状态转换过程：**

1. **NEW → RUNNABLE**：调用`start()`方法后，线程进入就绪队列，等待CPU调度
2. **RUNNABLE → BLOCKED**：等待获取monitor锁（如synchronized争抢）
3. **RUNNABLE → WAITING**：调用`Object.wait()`、`Thread.join()`（无超时）、`LockSupport.park()`
4. **RUNNABLE → TIMED_WAITING**：调用`Thread.sleep()`、`Object.wait(long)`、`LockSupport.parkNanos()`
5. **BLOCKED/WAITING/TIMED_WAITING → RUNNABLE**：被其他线程唤醒或超时后重新竞争锁
6. **RUNNABLE → TERMINATED**：`run()`方法执行完毕或抛出未捕获异常

**关键点：**
- **RUNNABLE**状态实际包含就绪（Ready）和运行（Running）两个子状态，由操作系统调度
- **BLOCKED**特指等待获取synchronized锁，不包括等待IO或CPU
- **WAITING**状态的线程需要被显式唤醒（如notify/ unpark）

> 为什么Java不直接提供RUNNING状态？

Java的线程状态是对JVM层面的抽象，**RUNNING**状态属于操作系统调度范畴，JVM无需关注。

**核心参数（ThreadPoolExecutor）：**

1. **corePoolSize**：核心线程数，线程池会维持此数量的线程（即使空闲）
2. **maximumPoolSize**：最大线程数，线程池允许的最大并发数
3. **keepAliveTime**：空闲线程存活时间，仅当线程数>corePoolSize时生效
4. **unit**：时间单位
5. **workQueue**：任务队列（BlockingQueue实现）
6. **threadFactory**：线程创建工厂
7. **handler**：拒绝策略处理器

**工作流程：**

```
1. 提交任务 → 线程数 < corePoolSize → 创建核心线程执行
2. 核心线程满 → 任务入队列
3. 队列满 + 线程数 < maximumPoolSize → 创建临时线程执行
4. 队列满 + 线程数 = maximumPoolSize → 触发拒绝策略
```

**拒绝策略（4种）：**

| 策略 | 行为 |
|------|------|
| AbortPolicy（默认） | 抛出**RejectedExecutionException** |
| CallerRunsPolicy | 由调用方线程执行 |
| DiscardPolicy | 静默丢弃任务 |
| DiscardOldestPolicy | 丢弃队列中最老的任务 |

> 线程池的队列选择有什么讲究？

- **LinkedBlockingQueue**：无界队列，可能导致maximumPoolSize失效（永远只用核心线程）
- **SynchronousQueue**：直接提交模式，必须创建非核心线程处理
- **ArrayBlockingQueue**：有界队列，防止资源耗尽，需合理设置队列大小和maximumPoolSize

### 3. ThreadLocal内存泄漏问题及解决方案

ThreadLocal为什么会发生内存泄漏？如何避免？

**原理分析**

**内存泄漏原因：**

ThreadLocal内部使用**ThreadLocalMap**存储键值对：
- Key是ThreadLocal对象（弱引用：**WeakReference<ThreadLocal<?>>**）
- Value是用户存入的值（强引用）

```
ThreadLocalMap.Entry extends WeakReference<ThreadLocal<?>> {
    Object value;
}
```

**泄漏场景：**

1. 线程复用场景（如线程池）：线程生命周期很长
2. ThreadLocal使用后未调用`remove()`
3. Key被GC回收后，Entry的key变为null
4. Value仍然存在，但无法访问（value != null && key == null）
5. 这些"垃圾"Entry积累导致内存泄漏

**避免方法：**
- 使用完ThreadLocal后手动调用**remove()**
- 每次使用前判断`get() == null`则`set(new Object())`
- 阿里规约：ThreadLocal变量必须手动释放

> 为什么Entry使用弱引用而不是强引用？

如果Entry持有ThreadLocal的强引用，会阻止ThreadLocal对象被GC，导致内存泄漏更严重。弱引用允许ThreadLocal被回收，但需要额外清理Value。

### 4. 线程中断机制

Java的线程中断机制是怎样的？interrupt、isInterrupted、interrupted的区别是什么？

**原理分析**

**三个方法：**

| 方法 | 作用 | 说明 |
|-----|------|------|
| **interrupt()** | 设置中断标志为true | 仅设置标志，不强制停止线程 |
| **isInterrupted()** | 判断是否被中断 | 不清除中断标志 |
| **interrupted()** | 判断并清除中断标志 | 静态方法，作用于当前线程 |

**阻塞状态响应：**

当线程处于以下状态时，调用`interrupt()`会抛出**InterruptedException**并清除中断标志：
- `Object.wait()` / `Object.wait(long)`
- `Thread.sleep()` / `Thread.sleep(long)`
- `BlockingQueue.take()` / `BlockingQueue.poll()`
- `LockSupport.park()` / `LockSupport.parkNanos()`

**正确的中断处理方式：**

```java
while (!Thread.interrupted()) {
    // 业务逻辑
}
// 退出时清理资源
```

> 为什么interrupt不直接停止线程？

强制停止会导致锁无法释放、文件无法关闭等资源泄漏问题。Java设计为**协作式中断**，让线程在合适的位置检查标志并优雅退出。

### 5. 守护线程与用户线程的区别

守护线程和用户线程有什么区别？守护线程能否守护守护线程？

**原理分析**

**区别：**

| 特性 | 用户线程 | 守护线程 |
|-----|---------|---------|
| 创建 | 默认是用户线程 | 需显式设置`setDaemon(true)` |
| JVM等待 | 会阻止JVM退出 | 不会阻止JVM退出 |
| 继承性 | 可设置为守护线程 | 可设置为守护线程 |
| 典型应用 | 主线程、业务线程 | GC线程、GC Finalizer |

**关键问题：守护线程能否守护守护线程？**

可以。守护属性具有传递性，当所有用户线程结束时，JVM会退出，此时所有守护线程都会被终止，无论是否还有子守护线程在运行。

> 为什么JVM不等守护线程执行完再退出？

守护线程通常用于辅助用户线程（如GC），如果等待守护线程完成，用户线程可能永远无法结束（GC是无限循环的）。设计上认为**守护线程不应阻止JVM退出**。

### 6. Runnable、Callable、Future的区别与配合使用

Runnable、Callable、Future的区别是什么？它们如何配合使用？

**原理分析**

**三者对比：**

| 接口 | 返回值 | 异常 | 典型实现 |
|-----|-------|------|---------|
| Runnable | void | 不允许抛出 | Thread.run() |
| Callable<V> | V | 允许抛出 | FutureTask |
| Future<V> | - | - | FutureTask、CompletableFuture |

**使用场景：**

```java
// Callable + Future
FutureTask<String> task = new FutureTask<>(new Callable<String>() {
    @Override
    public String call() throws Exception {
        return "result";
    }
});
new Thread(task).start();
String result = task.get(); // 阻塞获取结果

// 线程池方式：submit返回Future
ExecutorService executor = Executors.newSingleThreadExecutor();
Future<String> future = executor.submit(new Callable<String>() {
    @Override
    public String call() throws Exception {
        return "result";
    }
});
String result = future.get();
```

**Future核心方法：**

- `get()` - 阻塞获取结果，可抛异常
- `get(long, TimeUnit)` - 超时获取
- `cancel(boolean)` - 取消任务
- `isDone()` - 判断是否完成
- `isCancelled()` - 判断是否被取消

> FutureTask的run()方法如何处理Callable的异常？

异常会被存储到**outcome**变量中，调用`get()`时重新抛出**ExecutionException**。

### 7. 线程上下文切换的开销与CPU缓存问题

线程上下文切换的开销有哪些？CPU缓存对多线程有什么影响？

**原理分析**

**上下文切换开销：**

1. **寄存器保存/恢复**：CPU寄存器、程序计数器等
2. **缓存失效**：L1/L2/L3缓存命中率下降
3. **TLB刷新**：Translation Lookaside Buffer缓存失效
4. **调度开销**：操作系统调度器决策

**数据：**
- 一次上下文切换约 **0.5-5μs**（取决于硬件）
- 频繁切换会导致CPU利用率下降

**CPU缓存与多线程：**

```java
// 线程A和线程B可能运行在不同的CPU核心
// 每个核心有独立L1/L2缓存
// 共享L3缓存

// 缓存行伪共享问题
class Data {
    long a;
    long b; // 线程A改a，线程B改b
}
// 如果a和b在同一个缓存行
// 线程A修改a → 缓存行失效 → 线程B读取b也需要从主存加载
```

> 如何在代码中避免伪共享？

1. 使用JDK 8+的**@Contended**注解
2. 使用Disruptor等高性能队列（内部已优化）

```java
@sun.misc.Contended
class Counter {
    volatile long value;
}
```

### 8. 线程池的队列选择与饱和策略

线程池的队列是无限的吗？队列满了一定会触发拒绝策略吗？

**原理分析**

**队列类型与容量：**

| 队列 | 容量 | 特点 |
|-----|------|------|
| LinkedBlockingQueue | **无界**(Integer.MAX_VALUE) | 不会触发拒绝策略 |
| ArrayBlockingQueue | 有界 | 队列满时触发拒绝策略 |
| SynchronousQueue | 0 | 不存储任务，直接交付 |
| DelayedWorkQueue | 无界（但内部有逻辑） | ScheduledThreadPoolExecutor专用 |

**关键问题：队列满才会触发拒绝策略吗？**

不一定。触发条件是：**队列满 + 线程数达到maximumPoolSize**

```java
public void execute(Runnable command) {
    // 1. 检查线程数 < corePoolSize
    if (workerCountOf(c) < corePoolSize) {
        if (addWorker(command, true))
            return;
        c = ctl.get();
    }
    // 2. 检查队列并加入队列
    if (isRunning(c) && workQueue.offer(command)) {
        if (!isRunning(c) && remove(command))
            reject(command);
    }
    // 3. 队列满，创建非核心线程
    else if (!addWorker(command, false))
        reject(command);
}
```

> 如何设计一个"先提交先执行，超时则拒绝"的线程池？

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2, 4, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),
    Executors.defaultThreadFactory(),
    new RejectedExecutionHandler() {
        @Override
        public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
            throw new RejectedExecutionException("Task rejected");
        }
    }
);
```
