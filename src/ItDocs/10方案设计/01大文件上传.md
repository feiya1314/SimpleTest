大文件上传原理 + 服务器 / 线程池 / 连接影响（面试极简标准答案）
一、大文件上传核心原理
分片上传：前端按固定大小切分成小块，携带分片序号、文件唯一 MD5、总分片数
并发传分片：多线程并行上传单个分片
服务端暂存：分片临时落盘，不合并
合并请求：全部分片上传完成，请求后端按序号合并完整文件
断点续传：上传前校验已上传分片，跳过重复片段
二、对服务器的影响
磁盘 IO 暴涨：频繁读写分片、合并文件，占用磁盘吞吐
存储空间占用：临时分片堆积，需定时清理过期碎片
内存占用：分片解析、校验、缓存占用堆内存
带宽压力：大流量上传挤占业务带宽
安全风险：恶意超大文件攻击、磁盘打满
三、对线程池的影响
IO 密集型任务：上传阻塞线程，线程池容易被占满
并发分片过多：线程耗尽，正常业务请求排队超时
线程池参数失衡：核心线程、队列、最大线程不合理直接卡死上传服务
线程泄露：上传异常未回收线程，池性能持续下降
四、对网络连接的影响
长连接大量占用：每个分片对应 HTTP/TCP 连接，连接数飙升
端口资源耗尽：服务端文件描述符、端口上限被打满
连接超时、断连重传：网络波动产生大量重试请求
TCP 拥塞：并发上传导致丢包、延迟升高

限制单用户并发分片上传数，防止恶意压测
4. 网络连接优化
启用 HTTP 长连接复用，减少频繁建连断连开销
配置连接数、文件句柄上限，阈值告警

对象存储直传，后端仅做校验，避开应用服务器中转

二、高频追问 & 标准作答
分片为什么不能太小 / 太大？
过小：分片数量暴增，请求、连接、校验开销剧增；过大：单分片失败重传成本高，卡顿概率大。
如何实现秒传？
前端计算文件全局 MD5，请求服务端比对哈希值，文件已存在直接判定上传完成。
分片合并会不会出现文件损坏？
按分片序号有序合并，合并后二次校验整体 MD5，不一致则判定上传失败。
上传线程池和业务池混用有什么问题？
上传 IO 阻塞线程，极易占满线程池，正常接口请求无线程可用，引发服务卡顿雪崩。
大量上传导致磁盘爆满怎么处理？
设置单用户、全局存储配额；定时清理超时未合并分片；磁盘容量阈值告警拦截新上传。
TCP 连接耗尽原因与解决？
并发分片产生大量连接，耗尽句柄端口；拆分线程池、限制并发数、开启连接复用、调高系统句柄上限。
断点续传原理？
上传前查询已成功分片列表，前端仅上传缺失分片，无需从头重传。
直传 OSS 和服务中转上传区别？
直传：服务只鉴权校验，流量不走应用服务，压力最小；中转：服务转发文件，IO、带宽压力集中。
怎么防止分片重复上传？
以文件 MD5 + 分片序号做唯一键，服务端记录分片上传状态，重复请求直接忽略。
大文件上传超时怎么解决？
拆分分片降低单请求耗时；调整服务、网关超时时间；异步合并，接口快速响应。

1. 最关键：上传服务独立线程池隔离（核心方案）
不让上传任务占用业务主线程池（Tomcat 线程池）。
单独创建一个 上传专用线程池。
好处：
上传再拥堵，也不会影响登录、订单、查询等核心接口
资源彻底隔离，一码事归一码事
配置要点：
核心线程、最大线程、队列单独配置
拒绝策略用CallerRunsPolicy或丢弃旧任务，防止 OOM
IO 密集型：最大线程数可以给大一些
2. 上传接口必须异步化（非阻塞）
不要让上传在同步接口里占着线程。
方案：
使用 @Async + 独立线程池
或使用 Servlet 3.0 异步上下文
接收文件 → 丢队列 → 立刻返回客户端 “上传中”
线程只负责接收分片，不负责等待 IO 写入，快速释放。
3. 严格限流：控制单用户并发分片数
前端不能无限并发上传分片，必须限制：
单用户最多 3～5 个并发
全局总并发也做上限
防止：
连接耗尽
线程池打满
磁盘 IO 爆炸
4. 使用 NIO / 响应式模型，解放线程（高阶优化）
传统 IO：一个请求占一个线程，一直等。
NIO / WebFlux：一个线程能处理成千上万上传请求。
适合高并发上传场景，线程利用率提升 10～100 倍。
5. 文件直传对象存储（OSS/MinIO），绕开应用服务器
最彻底的方案：不让文件流量经过后端应用服务。
流程：
后端只给前端返回上传凭证
前端直接把文件传到 OSS
应用服务器不处理文件 IO
结果：
应用服务器0 磁盘 IO
0 线程阻塞
永远不会因为上传把服务搞崩
6. 超时控制 + 异常回收
上传任务强制超时
失败自动释放线程
防止死锁、线程泄露
7. 网关层限流 + 熔断
对上传接口单独限流
超过阈值直接熔断，返回 “上传繁忙”
保护后端服务
