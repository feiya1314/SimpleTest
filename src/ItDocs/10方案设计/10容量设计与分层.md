## 1. 如何进行互联网架构的容量设计？

**容量设计**的核心目标是在满足业务增长预期的前提下，以**合理的成本**保证系统在高负载下的**稳定性与可用性**。

核心步骤：

- **明确业务指标**：确定 DAU、PV、UV、写请求量、读请求量等业务数据
- **流量漏斗换算**：从用户行为出发逐层计算核心接口 QPS
  - 例：100 万 DAU，平均每人 10 次请求，核心接口占比 20% → 日均 200 万次 → 峰值 QPS ≈ 日均 / 4万 × 3 ≈ **150 QPS**（按"二八定律"估算，80% 请求集中在 20% 时间）
  - 更精确：日均请求量 / (24×3600) × 4 ~ 6 倍峰值系数
- **资源估算**：
  - **QPS**：确认单机可承载 QPS（一般 Java 服务单机 500~2000 QPS，取决于业务复杂度），计算所需机器数
  - **带宽**：平均响应体大小 × QPS × 8，确保带宽不成为瓶颈
  - **存储**：单条数据大小 × 日增量 × 存储天数，预留 2~3 倍冗余与索引空间
  - **连接数**：长连接场景估算连接池、线程池、文件句柄消耗
- **压测验证**：预估结果必须通过**压测**（单机 / 全链路）验证，确认真实承载上限
- **冗余与弹性**：按**峰值 2~3 倍冗余**部署，预留弹性扩容空间（支持动态扩缩容）

## 2. 如何估算一个系统的峰值 QPS？

常见估算方法：

- **二八定律法**：80% 请求集中在 20% 时间，峰值 QPS = (日均 PV × 0.8) / (24 × 3600 × 0.2)
- **经验倍数法**：峰值 QPS = 日均 QPS × (4~6)，简单粗暴适合早期预估
- **参考同类系统**：参考行业同类系统在相似体量下的 QPS 指标
- **全链路压测**：最准确的方式，线上压测获取真实峰值

面试更认可**从业务场景逐层推导**而非背公式：以 DAU 100 万、人均请求 20 次、核心链路 50% 流量为例，日均请求 1000 万，峰值集中在 4 小时内 → 峰值 QPS ≈ 1000万 / (4×3600) ≈ **700 QPS**。

## 3. 存储容量如何做预估？

- **结构化数据（MySQL）**：单行大小 × 日增量 × 存储天数，加上索引开销（通常 1.5~2 倍），预留 50% 余量
  - 例：单行 500 字节，日增 10 万行，存 90 天 → 500 × 10万 × 90 × 2（索引倍数）= **~900 GB**
- **缓存（Redis）**：预估缓存 key 数量和平均 value 大小，加上 Redis 自身开销（约 30%）
  - 注意设置**过期时间**，避免冷数据堆积
- **对象存储 / 文件**：按日增量 × 存储周期，关注**带宽与 IOPS**而非仅有容量
- **日志**：单条日志 ≈ 0.5~2 KB，QPS × 存盘比例 × 保留天数

## 4. 数据库连接池、线程池大小应该如何设置？

- **线程池（Tomcat）**：
  - **IO 密集型**（大量 DB/远程调用）：线程数 = CPU 核数 × (1 + IO 耗时 / CPU 耗时)，常见 2~4 倍 CPU 核数
  - **CPU 密集型**：线程数 ≈ CPU 核数 + 1
  - 不宜过大，过大导致上下文切换反而降低吞吐
- **数据库连接池（HikariCP / Druid）**：
  - 公式：连接数 = TPS / (单连接每秒可执行的查询数)
  - 经验值：**核心服务 10~30 个连接**足够，再多反而因数据库并发竞争性能下降
  - 每个服务实例**独立连接池**，避免互相影响
- **连接池不是越大越好**，过大会导致数据库端竞争加剧、锁等待上升、性能下降

## 5. 容量规划时需要注意哪些风险？

- **流量尖峰**：活动 / 秒杀 / 热点事件导致流量 10~100 倍暴涨，需要做 **限流、降级、熔断**
- **缓存穿透 / 击穿 / 雪崩**：大量请求直接打到底层存储，导致 DB 被打垮
- **慢查询拖垮连接池**：一条慢 SQL 占住连接不释放，池子被快速打满
- **连锁故障**：一个服务抖动 → 上游超时重试 → 流量放大 → 下游雪崩
- **资源依赖瓶颈**：共享数据库 / 缓存被某业务打满，波及全集群

应对策略：**冗余部署 + 限流降级 + 熔断隔离 + 容量告警 + 全链路压测**。

## 6. 高并发下如何避免带宽成为瓶颈？

- **估算带宽**：平均响应体 × QPS × 8，算出所需带宽（bit/s）
  - 例：平均响应 50 KB，QPS 10000 → 50 × 10000 × 8 = **4 Gbps**
- **优化手段**：
  - 压缩响应体（Gzip / Brotli 压缩，通常减少 70~80% 体积）
  - 精简接口返回字段，避免返回不需要的大字段
  - 静态资源走 CDN，不经过应用服务器
  - 分页、增量查询避免全量拉取
  - 协议升级（HTTP/2 多路复用减少传输开销）

## 7. 代码为什么需要分层？常见的分层架构有哪些？

**分层目的**：分离关注点、降低耦合、提高可维护性与可测试性。

常见分层模型：

- **传统三层架构（Controller → Service → DAO）**：
  - Controller：接收请求、参数校验、路由分发
  - Service：业务逻辑编排、事务管理
  - DAO / Repository：数据库访问，CRUD
  - Model：数据载体（VO / DTO / DO / PO）
- **DDD 四层架构（User Interface → Application → Domain → Infrastructure）**：
  - User Interface：接收请求、展示数据
  - Application：用例编排、事务协调，无业务逻辑
  - Domain：**核心业务逻辑**、领域模型、领域服务
  - Infrastructure：持久化、消息、缓存等技术实现
- **端口适配器架构（Hexagonal / 六边形架构）**：
  - 核心领域不依赖任何外部技术
  - 输入/输出适配器隔离外部系统变化

核心原则：**上层依赖下层，下层不依赖上层；依赖倒置让领域层保持纯净**。

## 8. 分层架构中各层的职责如何划分？

以传统三层为例：

- **Controller 层**：
  - 参数校验（格式、必填、范围）
  - 调用 Service 获取结果
  - 结果封装（VO 转换）、异常处理
  - **不含任何业务逻辑**
- **Service 层**：
  - 业务逻辑编排、流程控制
  - 事务管理（@Transactional）
  - 多个 DAO 的组合调用
  - 领域对象与 DTO/VO 的转换
- **DAO / Repository 层**：
  - 纯数据访问
  - 仅含 SQL / ORM 操作
  - **不含业务逻辑**
- **Model 层**（VO / DTO / DO / PO）：
  - VO（View Object）：前端展示所需字段
  - DTO（Data Transfer Object）：接口间传输数据
  - DO（Domain Object）：领域对象，含业务行为
  - PO（Persistent Object）：与数据库表一一对应

## 9. 分层架构有哪些常见反模式？

- **上帝类（God Class）**：一个 Service 几百行、几百个方法，集所有业务于一身 → 应拆分
- **跨层调用**：Controller 直接调用 DAO，跳过 Service → 事务缺失、逻辑分散
- **贫血模型**：DO 只有 getter/setter，业务逻辑全在 Service → 面向过程而非面向对象
- **DTO 滥用**：每层都定义完全相同的对象，过度转换 → 控制合理粒度，不要为了分层而分层
- **循环依赖**：ServiceA 调用 ServiceB，ServiceB 又调用 ServiceA → 需引入事件或中间层解耦
- **事务过大**：Service 方法中把所有操作包在一个事务里，锁范围过大 → 只对必要的写操作加事务

## 10. 微服务架构下的分层和传统分层有什么区别？

- **服务粒度不同**：传统分层在一个进程内，微服务将业务拆分为独立部署的服务
- **服务间通信**：传统分层是方法调用（内存），微服务是**RPC / HTTP 远程调用**（网络），需要考虑超时、重试、熔断、降级
- **数据存储**：传统分层共享数据库，微服务**数据隔离**（Database per Service），每个服务有自己的数据存储
- **分层延伸**：微服务增加了 **BFF 层（Backend For Frontend）**，为不同客户端定制聚合接口；增加了 **Gateway 层** 做统一路由、鉴权、限流
- **分布式事务**：传统分层可以用本地事务，微服务需 **Saga / TCC / 最终一致性**
- **复杂度转移**：微服务将单体复杂度的"大泥球"转移为网络通信、数据一致性、服务治理等分布式复杂度

## 11. DDD 分层和传统分层在实际项目中如何选择？

- **业务简单 CRUD 系统**：传统三层足够，引入 DDD 徒增复杂度
- **业务复杂、规则多变的核心域**：DDD 四层架构，领域模型封装复杂业务规则，更易于维护和演进
- **中台 / 大型业务系统**：建议**关键业务域使用 DDD**，非核心功能保持传统分层
- **团队能力**：DDD 对团队领域建模能力要求高，强行引入会导致"伪 DDD"（仍然过程式编程）
- **折中方案**：传统三层 + **领域层思维**，在 Service 中抽象领域对象和方法，降低贫血程度
