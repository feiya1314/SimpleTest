## 1. 微博用户关系（关注/粉丝）的核心数据模型是什么？

微博的用户关系是**有向图**模型，核心操作包含：

- 查询我关注了谁（**关注列表** / following）
- 查询谁关注了我（**粉丝列表** / followers）
- 判断用户A是否关注了用户B
- 判断用户A和用户B是否互粉

每条关系用二元组 `(userId, followeeId)` 表示，userId 关注 followeeId。

## 2. 微博的关系数据如何选型？为什么不用图数据库？

**选型**：使用 **MySQL** 作为主存储，**Redis** 配合缓存加速。

**为什么不用图数据库**：
- 图数据库（如 Neo4j）擅长多跳关系查询，但微博核心场景**只有一跳**（我关注的人和我的粉丝），两跳以上场景极少
- 图数据库分布式能力弱，水平扩展不如 MySQL + 中间件方案成熟
- MySQL 的 `(uid, followeeId)` 复合索引在一跳查询上性能足够好

如果不当大V被很多人关-注那只需要一跳就够了，查一下我的粉丝。

只有少数运营需要分析某些特定场景时才会用到两跳以上，这类低频场景通过离线数仓解决即可。

## 3. 粉丝关系表如何设计？如何应对超大V的千万级粉丝？

**表结构**：

- **关注表**（follow）：`(uid, followeeId, time)` — 记录谁关注了谁，按 uid 分库分表
- **粉丝表**（follower）：`(uid, followerId, time)` — 记录谁的粉丝是谁，按 uid 分库分表

关注表和粉丝表的数据通过**事务**或**异步对账**保证最终一致。

**超大V问题**：鹿晗等明星粉丝数可达千万级。查询粉丝列表时，单表扫描千万行会导致接口超时。

**解决方案**：
- 粉丝表的分片键依然是 uid，但查询时按 `followeeId`（即大V的uid）路由
- 分库分表后粉丝数据自然散列到多个分片
- 查询粉丝列表时**并发**从所有分片读取，merge 后返回
- 如果粉丝数过大（千万级），走离线提供或只展示活跃粉丝/铁粉

## 4. 如何高效判断用户A是否关注了用户B？

用户在微博上做互动（转发、评论、点赞）时，需要判断是否关注了博主。

**方案**：
- 首选 **Redis**：用 Redis Set 存储用户的关注列表，key = `follow:{uid}`，member = followeeId。时间复杂度 O(1)
- Cache 未命中时，回源 MySQL 关注表查询，MySQL 的 `(uid, followeeId)` 复合索引也能快速返回

**缓存预热**：用户登录时异步加载关注列表到 Redis，降低首次查询延迟。

## 5. 如何判断互粉关系（双向关注）？

**互粉条件**：用户A关注了用户B，且用户B也关注了用户A。

**实现方式**：两次判断即可

```
isMutual = isFollowing(A, B) && isFollowing(B, A)
```

利用 Redis 管道（pipeline）或批量查询接口，合并两次查询为一次网络开销。

**特例**：互粉数达到一定量级时，可以将互粉关系单独存储为一个集合，但增加了写操作的复杂度。大多数场景下直接用两个判断即可，不需要冗余存储。

## 6. 微博的关系数据如何分库分表？

**分片键选择**：
- 关注表按 `uid` 分片，查询我的关注列表时按 uid 路由到单个分片
- 粉丝表按 `uid`（这里的 uid 是博主的uid，即 followeeId）分片，查询我的粉丝列表时按 uid 路由到单个分片

**问题**：关注表查询"谁关注了A"时，按 uid 分片会导致跨分片查询（需要扫描所有分片）。同样，粉丝表查询"A关注了谁"也会跨分片。

**解决方案**：**数据冗余**。关注表和粉丝表互为冗余：
- 发一条关注关系时，同时写入关注表（按 uid 分片）和粉丝表（按 followeeId 分片）
- 关注表负责"我关注了谁"的查询
- 粉丝表负责"谁关注了我"的查询

这样每个查询都只需路由到**单个分片**，不需要跨分片扫描。

## 7. 关注/取关操作如何保证数据一致性？

关注表和粉丝表使用不同的分片键，写入时无法用本地事务保证一致性。

**方案**：
- **最终一致性**：先写关注表（或粉丝表），通过异步消息队列同步另一张表
- **对账系统**：定时扫描不一致数据并修复
- **补偿机制**：如果异步同步失败，通过 MQ 重试

**查询时的兜底**：读取时如果发现关注表和粉丝表数据不一致，以关注表为准（发号端），粉丝表不一致时触发异步修复。

## 8. 如何查询共同关注（我和某人都关注了谁）？

**场景**：在他人主页展示"你和他关注了同一个博主"。

**方案**：
- 从 Redis 取出我的关注列表和对方的关注列表，在应用层做**交集**运算
- Redis Set 直接提供 `SINTER` 命令求交集，时间复杂度 O(N)
- 如果关注列表较大（几千人），计算交集性能依然可控

**优化**：对高频查询的共同关注对，可以预计算结果缓存（如缓存24小时），但微博大部分场景是实时计算即可。

## 9. 关注数/粉丝数如何存储和更新？

**存储方式**：用 Redis 计数器维护，key = `fans_count:{uid}` 和 `follow_count:{uid}`。

**更新时机**：
- 关注成功 → `follow_count:{uid}` +1，`fans_count:{followeeId}` +1
- 取关成功 → 相应 -1

**数据库维度**：MySQL 中 user 表也冗余 `fans_count` 和 `follow_count` 字段，用于离线统计和兜底。

**一致性问题**：
- Redis 计数和 MySQL 计数可以不一致（最终一致即可）
- 展示以 Redis 为准（速度快），数据库计数通过定时任务同步修正
- 关注/取关操作引入防抖，防止频繁刷接口导致计数不准确

## 10. 微博的关系存储最终总结了哪些设计要点？

1. **关系存储采用 MySQL + Redis 双层架构**，MySQL 持久化，Redis 加速热点查询
2. **关注表和粉丝表的分片键不同**，通过数据冗余避免跨分片查询
3. **关注表与粉丝表之间采用最终一致性**，通过 MQ 同步 + 对账系统兜底
4. **判断是否关注走 Redis Set**，O(1) 时间复杂度，缓存未命中回源 MySQL
5. **互粉关系由两次判断组合**，不单独冗余存储
6. **千万级粉丝查询**使用并发读多个分片或走离线展示，避免单次查询超时
7. **关注/粉丝计数走 Redis 计数器**，数据库计数值通过定时任务修正
