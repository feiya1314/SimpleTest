## 1. 什么是TopK问题？常见解法有哪些？

TopK问题指在海量数据中找出出现频率最高的前K个数，或从海量数据中找出最大的前K个数（如搜索引擎热词、歌曲下载榜）。

**常见解法**：

- **全局排序**：快排后取前K个，时间复杂度 O(n log n)，大数据量不可行
- **局部淘汰**：维护一个大小为K的有序容器，遍历所有数据，每次与容器内最小元素比较并替换，O(nK)
- **分治+堆**：哈希取模分片 → 每个小文件内用HashMap统计频数 → **小顶堆**（取最大TopK）或**大顶堆**（取最小TopK）求解 → 多路归并汇总
- **快速选择（Quick Select）**：一次快排分区后只处理包含目标的一侧，平均 O(n)，最差 O(n²)

## 2. 如何使用堆解决TopK问题？时间复杂度是多少？

**原理**：求最大的TopK用**小顶堆**，求最小的TopK用**大顶堆**。

以从1亿个数中找最大的100个为例：
- 先用前100个数建一个**小顶堆**
- 遍历剩余所有数，每个数与堆顶（最小值）比较：**比堆顶大则替换并调整堆**，比堆顶小则跳过
- 遍历结束，堆中100个数就是最大的100个

**复杂度**：建堆 O(K log K)，每个元素最坏情况调整堆 O(log K)，整体 O(n log K)。当 K 远小于 n 时，接近 O(n)。

**Java实现**：直接使用 `PriorityQueue`，默认小顶堆，`Comparator.reverseOrder()` 可切换为大顶堆。

## 3. 有一个1GB文件，每行一个词（≤16字节），内存限制1MB，要求返回频数最高的100个词，怎么处理？

**分治+HashMap+小顶堆，三步走**：

**Step1（分片）**：顺序读文件，对每个词 x 计算 `hash(x) % 5000`，将结果存入对应小文件 f0~f4999。每个小文件约200KB，如有超过1MB的递归继续分片。

**Step2（统计）**：对每个小文件，用 HashMap 统计词频。遍历完该文件后，维护一个**大小为100的小顶堆**，找出该文件中频数最高的100个词，写入结果文件。

**Step3（归并）**：将所有结果文件中的 (词, 频数) 对进行多路归并（类似归并排序）或使用全局小顶堆，最终得到全局频数最高的100个词。

## 4. 海量日志中提取某天访问百度次数最多的IP，怎么处理？

**Step1**：从该天日志中过滤出访问百度的IP，写入一个大文件。

**Step2（分片）**：IP是32位（最多2³²个IP）。将大文件按 `hash(IP) % 1000` 映射为1000个小文件。

**Step3（统计）**：对每个小文件用 HashMap 统计每个IP的出现次数，找出该文件中出现次数最多的IP及其频数。

**Step4（汇总）**：在1000个候选IP中，用**小顶堆**找出全局出现次数最多的IP。

## 5. 从1亿个整数中找出最大的10000个，有哪些方法？

**方法一：小顶堆**。O(n log K)，空间 O(K)。维护大小为10000的小顶堆，遍历全部数据，比堆顶大则替换调整。

**方法二：分治+快排选择**。将1亿数据分成100份（每份100万），对每份用快排选择找出最大的10000个；汇总后从100×10000个候选数据中再用快排选择找出全局最大的10000个。每次快排选择只处理包含目标的那一侧，无需全排序。

**方法三：局部淘汰**。维护一个大小为10000的有序容器，遍历所有数据，比最小值大则插入淘汰最小值。O(nK)，数据量大时效率低。

## 6. 给定a、b两个文件各存放50亿个URL（每个64字节），内存限制4GB，找出共同的URL？

50亿×64字节 ≈ 320GB，无法一次性加载到4GB内存。

**分治方案**：
1. 遍历文件a，对每个URL计算 `hash(url) % 1000`，存入1000个小文件（a0~a999），每个约300MB
2. 同样遍历文件b，按 `hash(url) % 1000` 存入1000个小文件（b0~b999）
3. 只有对应的小文件对（ai, bi）才可能有相同URL（相同hash值保证落入同组）
4. 对每对小文件，将ai的URL加载到**HashSet**中，遍历bi检查是否存在，存在则记录为共同URL

**补充方案**：也可用**布隆过滤器**预处理a文件，遍历b时过滤器判断是否存在。但布隆过滤器有**误判率**，需权衡。精确方案用HashSet分治。

## 7. 海量手机号存在一个大文件中，内存只有512MB，怎么去重？

**方案一：Hash分片 + Set去重**
- 将大文件按 `hash(手机号) % N` 分到N个小文件
- 对每个小文件加载到内存Set中去重
- 合并去重结果

**方案二：BitMap**
- 手机号11位共约10¹¹种可能，若号码范围较集中可用BitMap标记已出现号码
- 每个bit标记一个号码，512MB可支持约40亿bit的BitMap
- 适用于号码范围可预估且集中的场景

**方案三：布隆过滤器**
- 适用于允许极小误判的场景
- 不存储原始数据，只存储哈希指纹
- 误判率可通过位数组大小和哈希函数数量控制

**方案四：外部排序去重**
- 将大文件分割排序后写入小文件
- 多路归并时逐条比较去重，类似MapReduce的reduce阶段

## 8. 大文件去重后如何排序？

**流程**：哈希分片 → 片内排序 → 多路归并

1. **分片**：读原始文件，每片用 `TreeSet` 去重并排序，写回小文件（片内有序）
2. **多路归并**：同时打开所有小文件，每次取各文件当前最小记录比较，写出全局最小者，然后该文件指针前移。循环此过程直到全部写完。

**关键点**：多路归并参考HBase、Cassandra等NoSQL的**Compaction**流程——逐层合并已排序的run文件，利用最小堆（胜者树/败者树）优化每次取最小值的复杂度。

**数据偏移问题**：哈希分片可能出现单个文件过大的情况。解决方案：设定阈值，文件太大时递归再分片，或改用**范围分区**替代哈希分区。

## 9. 百亿人排队系统如何设计？

**背景**：百亿人同时排队抢购/抢票，核心是**挡住绝大多数无效请求**，用最小成本让队列有序前进。

**架构要点**：

**1. 前端限流与排队**
- **令牌桶/漏桶限流**：API网关层限流，超出阈值的请求直接返回"排队中"
- **前端轮询**：用户提交请求后，前端定时轮询排队状态，不占服务端长连接
- **验证码/滑块**：拦截机器请求，降低有效排队人数

**2. 分布式队列**
- 请求到达后生成**排队令牌**（UUID），写入Redis Zset，score为请求时间戳
- 多个队列分片（Queue Shard），按用户ID或地域哈希到不同分片
- 每个分片独立维护一个队列，避免单点瓶颈

**3. 异步处理与消费**
- 后台Worker定时从每个队列分片取出窗口内的令牌
- 消费完后通过MQ（如Kafka/RocketMQ）发送异步处理消息
- 最终状态通过回调或前端轮询告知用户

**4. 异常处理**
- **队列积压**：动态增加Worker实例，调整消费速率
- **队列丢失**：Redis持久化+主从+哨兵，同时记日志可回溯重建
- **重复排队**：用户维度幂等性去重，同一用户只允许一个排队令牌

**关键设计原则**：**排队不等于立即处理**——排队只保证"顺序"和"公平"，真正处理能力由后端消费速率决定。成本层面，把集中式突发的流量用队列整形为平稳可控的流量。
