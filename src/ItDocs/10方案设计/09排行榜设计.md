## 1. 用Redis实现排行榜，key和value怎么设计？

Redis的 **zset（有序集合）** 天然支持按 score 排序，非常适合并发量大的排行榜。设计要点：

**key设计**：排行榜按时间段分类，key 需要与时间关联：
- **年榜**：key = `ranking:2020`、`ranking:2019`
- **月榜**：key = `ranking:202006`、`ranking:202007`
- **周榜**：通过当天处于本年的第几周确定 key，如 `ranking:25`（第25周）。Java 中 `Calendar` 获取周数时需设置周一为一周第一天。

**value设计**：zset 的 member 作为排序标识（如用户ID），score 作为排序依据（如步数/积分）。

```java
Calendar cal = Calendar.getInstance();
cal.setFirstDayOfWeek(Calendar.MONDAY);
cal.setTime(new Date());
int weekNum = cal.get(Calendar.WEEK_OF_YEAR);
```

## 2. 排行榜跨年问题怎么处理？

周榜需要查看上一周数据时，本周周数减1。但当本周是本年第一周时，上周周数为0，不合理。

**解决方案**：先把当前日期往前推7天，再获取那天所属的周数。这样跨年时自然得到上一年的最后一周周数，避免了0周问题。

## 3. zset中score相同时，按什么规则排序？

当多个 member 的 score 相同时，按照 member 的 **字典序（lexicographically）** 排列。

例如 member 为 `2020001`、`2020002`、`aa`、`bb` 时，排序结果为 `2020001` < `2020002` < `aa` < `bb`。

## 4. zset如何实现多字段复合排序？

zset 只能按 score 单字段排序，但可通过 **权重组合数值** 实现多字段排序。将排序规则按权重高低组合成一个数值，权重高的放在数值最前面。

示例：游戏闯关排行榜以 **通关次数正序**、**复活次数倒序**、**通关时间倒序** 排序：
- 通关5次 → `5`
- 复活2次（最大99次，取反得97）→ `97`
- 通关时间2020-06-09的时间戳1591632000（去掉毫秒3位），倒序用 `9999999999 - 1591632000 = 8408367999`

最终 score = `5` + `97` + `8408367999` = **5978408367999**（13位）

**注意**：score 是 double 类型，最多保存 **16位数字**。如果复合排序中包含时间戳，留给其他字段的位数只有6位，这是复合排序的精度限制。

## 5. 微信步数排行榜如何用Redis实现？

核心使用 zset 的4个命令：

- **zadd key score member**：添加用户及其步数
- **zincrby key increment member**：步数变更时增量更新
- **zrevrank key member**：获取用户排名（步数从高到低）
- **zrevrange key start end [withscores]**：获取 Top N 排行榜

例如：`zrevrange sport:ranking:20210227 0 2 withscores` 获取步数前三。

**用户维度的排行榜**：key 中加入用户标识，如 `sport:ranking:why:20210227`，不同 key 对应不同好友圈的数据。但每个用户维护一个 zset 成本较高，更优方案是 **在请求时按好友关系实时计算生成排行榜**。

**辅助字段存储**：昵称、头像、点赞数等字段用 **Hash** 结构存储。key 为 `sport:ranking:why:20210227:jay`，field 包含 nickName、headPhoto、likeNum、walkNum。

## 6. 近7天/月/年排行榜如何实现？

使用 **zunionstore**（并集）或 **zinterstore**（交集）命令聚合多个时间段的 zset。

```bash
# 最近7天总步数排行榜（并集）
zunionstore sport:ranking:why:last_seven_day 7 \
  sport:ranking:why:20210222 ... sport:ranking:why:20210228 \
  weights 1 1 1 1 1 1 1 aggregate sum
```

- 全部加总（某用户多天步数累加）→ 用 **并集**
- 每天都有数据才算（7天全勤）→ 用 **交集**

同理可推月榜、季榜、年榜，只需把时间段的 key 传入聚合命令即可。

## 7. zset命令zrange和zrevrange有什么区别？

- **zrange**：按 score **从低到高** 返回指定排名范围内的 member
- **zrevrange**：按 score **从高到低** 返回指定排名范围内的 member

步数排行榜（步数越高排名越前）应使用 `zrevrange`。获取用户排名的命令也类似：
- **zrank**：score 从低到高的排名
- **zrevrank**：score 从高到低的排名（步数排行榜用此命令）

## 8. 每日排行榜数据如何备份落库？

Redis 是内存数据库，需定时将数据落库持久化。

**实现方式**：通过定时任务在每周结束时，从 Redis 获取本周的 zset 数据，逐条写入数据库。为保证数据完整性，**在下一周的第一次同步时再备份一次上周数据**。

通过设置 **标志位** 判断上周数据是否已同步，每次更新前检查标志位，避免重复备份。

## 9. 亿级用户排行榜如何设计？

采用 **分桶（分治）** 策略，将用户按段位/积分分段拆分到多个桶中。

以王者荣耀段位为例：
1. 将用户按段位（青铜→王者）分配到不同的 zset（桶）
2. 计算全服排名：`当前段位桶内排名 + 前面所有桶的元素总数`

**关键问题**：用户分布不均匀（青铜玩家远多于王者），需通过数据分析预估各段位用户量，合理划分桶的大小。可进一步将段位细分为小段位，或按积分范围均匀分段。

## 10. 排行榜设计还有哪些注意事项？

- **马太效应**：推荐类排行榜中，头部曝光度高会导致强者恒强。需设计复杂积分公式（如掘金社区的掘力值）来平衡，避免两极分化。
- **同分排名处理**：如果要求同分同名次，需在业务层处理，zset 本身不支持同分同名。
- **过期策略**：排行榜数据需设置 TTL 或通过定时任务清理过期数据，防止 Redis 内存溢出。
- **缓存与持久化**：Redis 宕机时排行榜数据可能丢失，需结合数据库持久化做兜底方案。
