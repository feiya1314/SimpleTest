## 1. 如何防止接口被恶意刷量？从哪些层面考虑？

整套方案按**从简单到高级、从前端到后端、从单机到分布式、外网防护到架构兜底**分层防御。

**前端层面（第一道拦截，挡掉无脑刷）**：
- **按钮防重复提交**：点击后置灰、倒计时，禁止连续点击
- **频率限制**：前端 JS 控制 1 秒内只能请求 1 次
- **隐藏真实接口地址**：接口路由加密、临时签名、动态路径，防止被抓包直接刷
- **验证码/滑块/人机校验**：高风险接口（登录、下单、抽奖）接入滑块、行为验证码，机器脚本直接拦死

**外网专业防火墙层级（最外层流量拦截）**：
- **硬件防火墙**：华为、深信服等，部署在机房入口，防大流量 DDoS、SYN/UDP 洪水、端口过滤
- **WAF Web 应用防火墙**：阿里云 WAF、腾讯云 WAF 等，专门防 CC 攻击、爬虫、接口狂刷、SQL 注入、恶意请求
- **云防火墙/高防 IP**：用户请求 → 高防 IP → 清洗恶意流量 → 干净流量转发到业务服务器
- **业务层防火墙（下一代风控）**：阿里云态势感知，做账号行为、设备指纹、请求模型分析，拦截同设备多账号刷、代理 IP 池批量刷

**网关层（全局入口，最推荐）**：
- **IP 限流**：单 IP 每秒/每分钟限制请求次数，超限直接 429
- **全局限流**：整个网关总 QPS 封顶
- **接口粒度限流**：每个接口单独设置 QPS 阈值
- **黑名单封禁**：异常 IP、高频恶意 IP 直接拉黑
- **Nginx+Redis 分布式限流**：Nginx Lua 脚本读取 Redis 计数器，网关层分布式拦截

**应用层单机限流**：
- **Guava RateLimiter 令牌桶**：控制单机接口每秒请求量
- **本地计数器**：基于时间窗口计数限流

**分布式限流（微服务多实例生产必备）**：
- **Redis + 滑动时间窗口**：按 IP/用户 ID/设备 ID 统计请求次数
- **Sentinel 限流熔断**：QPS 限流、预热、集群限流、熔断降级、热点参数限流

**业务层防护**：
- 登录态校验，未登录游客严控频率
- **接口幂等 + 防重放**：唯一请求 ID、时间戳+签名，防止截包重放
- 资源互斥：一人单日限制次数
- **延迟队列削峰**：突发流量丢进 MQ 慢慢消费

**风控与黑名单体系**：
- 异常行为检测：短时间高频请求、参数高度一致、UA 异常
- 自动拉黑：触发规则自动加入 Redis 黑名单
- 灰度 & 流量隔离：新接口先小流量放量

**架构层面兜底**：
- **服务熔断降级**：流量过大时自动熔断，返回兜底数据
- 集群扩容 + 负载均衡
- **读写分离、缓存兜底**：热点接口全部走 Redis 缓存，不打数据库

## 2. 企业标准完整流量入口架构是怎样的？

从上到下：

用户/爬虫/攻击流量 → **云高防 IP / 硬件防火墙**（抗 DDoS、大流量清洗）→ **WAF Web 应用防火墙**（防 CC、爬虫、接口狂刷、注入、异常请求）→ **Nginx / 网关 Gateway**（限流、黑名单）→ **业务微服务**

恶意流量在 WAF 和高防层就直接拦死，到不了业务代码。

## 3. 网关层如何做分布式限流？和单机限流有什么区别？

**单机限流**：基于 Guava RateLimiter 或本地计数器，只在单个进程内生效。优点是无需中间件、延迟低；缺点是微服务多实例时每个实例独立计数，总 QPS 会放大 N 倍，限流不准。

**分布式限流**：
- **Redis + 滑动时间窗口**：按 IP/用户 ID/设备 ID 作为 key，Redis 自增计数 + 设置过期时间，查询单位时间内的请求次数，超限拒绝
- **Sentinel 集群限流**：阿里开源的流量控制组件，支持 QPS 限流、预热、热点参数限流，可通过控制台动态配置规则，配合 Token Server 做集群维度统一限流
- **Nginx + Redis（OpenResty）**：Nginx Lua 脚本读取 Redis 计数器，在网关层做分布式拦截，性能极高

生产环境推荐网关层用 OpenResty + Redis 或统一接入 Sentinel，业务层可以用 Redis 滑动窗口补充。

## 4. 常见的权限模型有哪些？如何设计一套权限系统？

常见模型：
- **ACL（访问控制列表）**：直接给用户关联权限，简单直接，但用户量大时管理成本高
- **RBAC（基于角色的访问控制）**：用户 → 角色 → 权限，用户不直接挂权限，而是挂角色，角色关联权限。灵活、易扩展，是**业界最主流的权限模型**
- **ABAC（基于属性的访问控制）**：通过用户属性、资源属性、环境条件动态判断权限，粒度最细，适合复杂场景但实现复杂度高

**RBAC 核心设计**：
- **用户 + 角色 + 权限**三层结构，权限不直接挂载到用户
- 权限分三层：**接口权限**（能不能访问某个 URL）、**数据权限**（能看哪些数据范围）、**菜单/按钮权限**（能看到哪些页面和操作）
- 支持角色继承（如超级管理员继承所有普通角色权限）和角色互斥（如出纳和会计不能同一人）
- 缓存策略：用户登录后加载角色权限到 Redis，接口鉴权时从缓存校验，避免每次请求查库

## 5. 接口防刷和权限设计如何协同工作？完整体系怎么搭建？

**两层分工**：
- **防刷层**（前置拦截）：保证系统不被恶意流量打垮，关注的是"能不能进来"
- **权限层**（后置管控）：保证合法用户只能访问其授权资源，关注的是"进来后能做什么"

**完整体系**：
- 入口：WAF + 高防 IP 清洗恶意流量
- 网关：IP 限流 + 接口 QPS 限流 + 黑名单封禁
- 应用层：Sentinel 限流熔断 + Redis 滑动窗口
- 认证层：JWT/OAuth2 统一认证，校验用户身份
- 鉴权层：RBAC 权限校验，拦截越权请求
- 数据层：数据权限过滤，不同角色看到不同数据范围
- 审计层：全链路日志记录，异常行为回溯
