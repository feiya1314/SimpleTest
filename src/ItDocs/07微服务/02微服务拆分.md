# 1. 微服务拆分的原则是什么？

**核心回答：** 微服务拆分的核心原则是**高内聚低耦合**：每个服务职责单一、服务间依赖清晰、数据边界明确。

**原理分析：**
1. **业务边界**：按业务流程或业务能力拆分，每个服务对应一个完整的业务功能。
2. **数据边界**：每个服务拥有独立的数据库（或Schema），服务间通过API或消息队列交互，避免直接跨库JOIN。
3. **依赖清晰**：单向依赖，避免循环依赖；核心服务不依赖非核心服务。
4. **团队边界**：按团队能力划分服务，团队对服务全生命周期负责（开发、测试、运维）。

**适用场景：**
- 业务复杂，单体应用维护困难
- 多团队并行开发，依赖冲突频繁
- 需要独立迭代某一模块

# 2. 领域驱动设计DDD如何指导服务拆分？

**核心回答：** DDD通过**领域、限界上下文、聚合根**等概念指导服务拆分，核心是识别业务边界和核心领域模型。

**原理分析：**
1. **领域（Domain）**：业务问题域，DDD将复杂业务拆分为多个子域（如订单域、用户域、支付域）。
2. **限界上下文（Bounded Context）**：每个领域有明确的边界，定义了模型的有效范围，是服务拆分的自然边界。
3. **聚合根（Aggregate Root）**：聚合内部实体的管理入口，保证聚合内数据一致性，聚合间通过ID引用。
4. **领域事件**：领域内发生的重要事件，通过消息队列跨服务传播，实现领域间协作。

**实施步骤：**
1. 识别核心业务流程，绘制业务流图
2. 识别业务实体和聚合，确定聚合根
3. 划分限界上下文，确定服务边界
4. 定义领域事件，建立服务间通信机制

**延展思考：**
DDD不是一次性设计完成的，而是需要随着业务理解深入持续迭代。前期可以先按业务流程粗粒度拆分，待业务成熟、团队稳定后，再根据需要进一步细化。

# 3. 服务拆分的粒度如何把握？

**核心回答：** 服务粒度没有绝对标准，通常遵循**两个披萨原则**（团队两个披萨能吃完），核心是保证团队能独立负责、快速迭代。

**原理分析：**
1. **过粗的问题**：一个服务承载过多业务职责，耦合度高，难以独立部署和扩展，仍然是"单体"。
2. **过细的问题**：服务数量膨胀，服务间调用链复杂，运维成本剧增，甚至出现"分布式单体"。
3. **合理的粒度**：每个服务可由2-3人负责，服务间通过API或事件协作，部署和扩展独立。

**判断标准：**
- 团队能否独立负责该服务的全生命周期
- 该服务的变更是否频繁影响其他服务
- 该服务是否有独立的扩展需求

**延展思考：**
服务拆分是渐进式的，不是第一步就拆分到位。建议先按业务模块粗粒度拆分，待业务成熟、团队稳定后，再根据需要进一步细化。

# 4. 微服务拆分策略有哪些？

**核心回答：** 常见拆分策略包括**按业务能力拆分、按数据领域拆分、按请求路径拆分**。

**原理分析：**
1. **按业务能力拆分**：最常用的方式，如订单服务、用户服务、支付服务。优势是职责清晰，劣势是可能出现跨服务调用。
2. **按数据领域拆分**：按数据的归属进行拆分，如会员数据中心、交易数据中心。适合数据量大、需要独立扩展的场景。
3. **按请求路径拆分**：根据用户请求的访问路径拆分，如商品浏览路径、交易下单路径。适合需要优化特定用户场景的场景。

**分层策略：**
- 基础层：提供底层通用能力（数据字典、配置、文件存储）
- 业务层：核心业务逻辑，按业务流程拆分
- 接入层：对外API网关，按终端类型或渠道拆分

**延展思考：**
实际拆分往往混合使用多种策略。初期按业务能力拆分，后期根据性能瓶颈和团队分工可能需要进一步调整。

# 5. 拆分过程中如何处理遗留系统？

**核心回答：** 遗留系统拆分通常采用**绞杀者模式**或**分支模式**，逐步将功能迁移到微服务。

**原理分析：**
1. **绞杀者模式（Strangler Pattern）**：在遗留系统外围构建新服务，逐步将功能从旧系统迁移到新系统，最终完全替换旧系统。
2. **分支模式**：创建与遗留系统并行的微服务，新功能在微服务中开发，老功能暂时保留在遗留系统，通过API网关统一路由。
3. **功能提取**：从遗留系统中识别可独立拆分的模块，先行拆分高频变更或高耦合模块。

**实践步骤：**
1. 梳理遗留系统的模块依赖和调用关系
2. 选择低风险、高价值的模块先行试点
3. 建立新旧系统的通信机制（API适配器）
4. 逐步将流量从旧系统切换到新系统
5. 监控切换后的稳定性和性能

**延展思考：**
遗留系统拆分的最大挑战往往不是技术，而是组织协作和业务理解。

# 6. 微服务拆分后数据如何处理？

**核心回答：** 微服务提倡**每个服务拥有独立数据**，数据共享通过API调用或消息队列实现。

**原理分析：**
1. **独立数据库**：每个服务拥有独立的数据库实例或Schema，服务间禁止直接跨库JOIN。
2. **数据共享**：需要跨服务数据时，通过API调用获取，或通过消息队列同步数据。
3. **数据聚合**：上层服务通过调用多个下游服务API，聚合数据返回给前端（如BFF模式）。
4. **分布式事务**：跨服务事务无法保证强一致性，需要使用Saga、可靠消息等最终一致性方案。

**数据同步方案：**
- **API调用**：实时性强，但增加调用延迟
- **消息同步**：异步解耦，但有延迟
- **共享数据存储**：如ES用于搜索场景，缓存用于查询场景

**延展思考：**
数据拆分是微服务拆分中最复杂的部分。在过渡期可以允许一定的数据冗余，长期来看逐步做到数据独立。
