# 1. 什么是OAuth2.0？

**核心回答：** OAuth2.0是**开放授权标准协议**，允许第三方应用在**获得用户授权**后访问用户在资源服务器上的资源，而无需获取用户用户名和密码。

**原理分析：**
1. **角色定义**：
   - **资源所有者（Resource Owner）**：用户
   - **客户端（Client）**：第三方应用
   - **授权服务器（Authorization Server）**：负责认证用户并颁发授权令牌
   - **资源服务器（Resource Server）**：存储用户资源，验证访问令牌

2. **核心流程**：用户访问第三方应用 → 第三方应用跳转到授权服务器 → 用户登录并授权 → 授权服务器颁发授权码 → 第三方应用用授权码换access_token → 第三方应用使用access_token访问资源

**适用场景：**
- 第三方登录（如微信、GitHub登录）
- 开放API授权（如微信小程序调用支付API）
- 跨系统授权（如SAAS平台访问用户在其他系统的数据）

# 2. OAuth2.0的授权模式有哪些？

**核心回答：** OAuth2.0定义了**授权码模式、简化模式、密码模式、客户端模式**四种授权模式。

**原理分析：**
1. **授权码模式（Authorization Code）**：最安全完整的流程，适合服务器端应用。
   - 步骤：请求授权码 → 用户授权 → 返回授权码 → 用授权码换token
   - 特点：token不经过浏览器，降低泄露风险

2. **简化模式（Implicit）**：token直接返回给客户端，适合纯前端应用。
   - 步骤：直接请求token → 用户授权 → token直接返回URL锚点
   - 特点：token在URL中，风险较高，已不推荐使用

3. **密码模式（Resource Owner Password Credentials）**：用户直接提供用户名密码给客户端。
   - 步骤：用户给客户端用户名密码 → 客户端直接换token
   - 特点：要求客户端完全可信，仅适用于自家产品

4. **客户端模式（Client Credentials）**：客户端以自己的名义获取访问令牌。
   - 步骤：客户端直接请求token
   - 特点：用于无用户参与的服务器间通信

**选型建议：**
- Web应用：授权码模式
- 移动应用：授权码模式 + PKCE
- 纯前端应用：授权码模式 + PKCE
- 内部服务：客户端模式或密码模式

# 3. 什么是JWT？

**核心回答：** JWT（JSON Web Token）是**基于JSON的开放标准**，用于在各方之间安全地传输声明信息，通常用于**身份认证和信息交换**。

**原理分析：**
1. **结构组成**：Header.Payload.Signature 三部分，通过"."连接
2. **Header**：声明类型和签名算法（如HS256、RS256）
3. **Payload**：存放实际传输的声明（Claims），包括iss（签发者）、exp（过期时间）、sub（主题）、roles（角色）等
4. **Signature**：对前两部分的签名，防篡改

**使用流程：**
1. 用户登录成功，服务器生成JWT并返回给客户端
2. 客户端存储JWT（通常存localStorage或Cookie）
3. 后续请求在Header中携带JWT（Authorization: Bearer <token>）
4. 服务器验证JWT签名和有效期，解析Payload获取用户信息

**调优实践：**
- 短期token：access_token有效期短（如15分钟），refresh_token有效期长（如7天）
- 验签策略：对称算法（HS256）适合单机部署，非对称算法（RS256）适合分布式

# 4. OAuth2.0和JWT是什么关系？

**核心回答：** OAuth2.0是**授权协议**，JWT是**令牌格式**。OAuth2.0可以使用JWT作为令牌载体，JWT也可以脱离OAuth2.0独立使用。

**原理分析：**
1. **OAuth2.0定义授权流程**，不关心令牌具体格式
2. **JWT是令牌的具体实现**，具有自包含、可验证的特性
3. **组合使用**：OAuth2.0的授权服务器生成JWT作为access_token，资源服务器验证JWT获取用户信息

**典型组合：**
- 授权服务器使用JWT作为access_token，包含用户身份信息
- 资源服务器验证JWT签名，解析用户角色和权限
- refresh_token用于刷新access_token，可以是简单的不透明令牌

**延展思考：**
JWT的优势是无状态验证，服务器不需要查询数据库即可验证token，但缺点是token一旦签发无法撤销（除非加入黑名单机制）。因此token有效期通常设置较短，通过refresh_token续期。

# 5. 如何设计安全的Token机制？

**核心回答：** 安全Token设计需要考虑**防窃取、防重放、防篡改、可撤销**。

**原理分析：**
1. **防窃取**：Token通过HTTPS传输，存储时加密（如敏感场景存数据库加密）
2. **防重放**：加入nonce、timestamp等随机数，服务器验证是否已使用
3. **防篡改**：使用签名算法（HMAC或非对称签名），接收方验证签名
4. **可撤销**：加入token版本号或黑名单机制，异常时可强制失效

**实现方案：**
1. **短期access_token + 长期refresh_token**：access_token短（15分钟），refresh_token长（7天），泄露影响有限
2. **Token版本号**：用户修改密码时递增版本号，验证token时检查版本号，不匹配则拒绝
3. **黑名单机制**：token泄露后加入黑名单，验证时检查黑名单（性能开销大，仅用于应急）

**调优实践：**
- 敏感操作需要二次验证（如支付、修改密码）
- 同一用户多设备登录需要支持token管理和强制下线

# 6. SSO单点登录是如何实现的？

**核心回答：** SSO（Single Sign-On）实现**一次登录，全系统通行**，核心机制是**共享登录态**。

**原理分析：**
1. **域名共享**：同父域名下的不同子域名共享Cookie
2. **分布式Session**：将Session存储在Redis等分布式存储，多个服务共享
3. **Token机制**：登录后颁发统一Token，各服务验证Token获取用户信息
4. **CAS流程**：用户访问A系统未登录 → 跳转CAS登录 → 登录成功后跳转回A系统并携带Token

**实现方案：**
- **同域名Cookie**：设置Cookie的Domain为父域名，实现跨子域名共享
- **OAuth2.0/SAML**：使用标准协议实现跨域认证
- **分布式Session**：Session存入Redis，依赖Redis的高可用

**延展思考：**
SSO的核心风险是单点故障和单点沦陷。CAS服务故障则全系统不可用，CAS泄露则所有系统被攻破。因此CAS服务需要高可用部署，登录验证需要多因素认证。

# 7. 接口鉴权有哪些方式？

**核心回答：** 常见接口鉴权方式包括**Session/Cookie、Token（JWT）、签名验证、API Key**。

**原理分析：**
1. **Session/Cookie**：服务端存储Session，客户端通过Cookie携带SessionId。适合B/S架构，但无法跨域和移动端。
2. **Token（JWT）**：服务端生成Token，客户端存储并在请求中携带。适合移动端和前后端分离架构。
3. **签名验证**：客户端用secret对请求参数签名，服务端验证签名。适合OpenAPI和服务器间调用，如微信支付签名。
4. **API Key**：为每个客户端分配唯一的API Key和Secret，类似于AWS的AK/SK。适合开放平台和第三方调用。

**选型建议：**
- Web应用：Session/Cookie
- 移动应用和前后端分离：Token (JWT)
- 开放平台和第三方合作：API Key + 签名
- 微服务间调用：mTLS（双向TLS）

# 8. 如何防止Token泄露？

**核心回答：** 防止Token泄露需要**传输安全、存储安全、使用安全**三方面措施。

**原理分析：**
1. **传输安全**：全程HTTPS，阻止中间人攻击
2. **存储安全**：
   - 不存localStorage（可被XSS攻击获取）
   - 存HttpOnly Cookie（无法被JS读取）
   - 敏感场景对Token加密存储

3. **使用安全**：
   - 短期Token降低泄露后影响时间
   - 敏感操作二次验证
   - 异常登录检测和告警

**最佳实践：**
- access_token存内存，refresh_token存HttpOnly Cookie
- 敏感操作（如支付）要求用户再次输入密码
- 监控异常登录（异地IP、异常设备）

**延展思考：**
移动端Token存储比Web端更安全，iOS Keychain和Android Keystore提供系统级安全存储。生物识别（指纹、面容）可以用于敏感操作二次验证。
