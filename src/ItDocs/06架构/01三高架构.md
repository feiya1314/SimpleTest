# 1. 什么是三高架构？

**核心回答：** 三高架构是指**高并发、高可用、高性能**三大系统设计目标的统称。高并发强调系统同时处理大量请求的能力；高可用强调系统持续稳定提供服务的能力（通常要求99.9%以上可用性）；高性能强调系统处理单个请求的速度和响应时延。

**原理分析：**
1. **高并发与高性能的关系**：高并发通常通过提升系统吞吐量实现，这需要高性能的组件支撑。单机高性能可通过多线程、异步IO、缓存等手段；分布式高性能则需解决网络开销和数据分片问题。
2. **高并发与高可用的关系**：高并发下系统更容易出现瓶颈和故障，需通过限流、熔断、降级等手段保护系统，避免级联故障影响可用性。
3. **高性能与高可用的关系**：为达到高性能目标，常引入更多组件（缓存、消息队列等），这些组件本身的高可用设计至关重要。

**延展思考：**
CAP理论是三高架构设计的理论基础。分布式系统需在一致性、可用性、分区容错性之间权衡。多数互联网业务优先保证AP，通过最终一致性实现高可用。

# 2. 高并发系统设计的核心原则

**核心回答：** 高并发系统设计的核心原则是**水平扩展、读写分离、异步处理、缓存为王**。

**原理分析：**
1. **水平扩展 vs 垂直扩展**：垂直扩展提升单机硬件配置，但存在单点瓶颈和成本指数增长问题；水平扩展通过增加服务器数量提升容量，成本线性可控，但需解决数据一致性和状态管理问题。
2. **读写分离**：写操作集中在主库，读操作分散到从库，需考虑主从同步延迟导致的读写不一致问题。
3. **异步处理**：通过消息队列将耗时操作从主请求链路剥离，实现流量削峰和系统解耦，但会引入消息丢失、重复消费等问题。
4. **多级缓存**：本地缓存（如Caffeine）响应最快但受限于单机内存，分布式缓存（如Redis）可跨节点共享但有网络开销。

**适用场景：**
- 读多写少场景适合缓存和读写分离
- 流量峰值可预期场景适合消息队列削峰
- 无状态服务适合水平扩展

# 3. 限流算法有哪些？

**核心回答：** 常见限流算法包括**计数器算法、滑动窗口算法、漏桶算法、令牌桶算法**。

**原理分析：**
1. **计数器算法**：每个周期重置计数器，实现简单但周期边界可能产生两倍峰值。
2. **滑动窗口算法**：将时间窗口划分多个格子，滑动实现更平滑的限流。
3. **漏桶算法**：请求如水流进入漏桶，以固定速率漏水，无法应对突发流量。
4. **令牌桶算法**：以固定速率向桶添加令牌，支持突发流量。Google Guava的RateLimiter基于此实现。

**调优实践：**
- 限流需分层实施：网关层→应用层→容器层
- 限流触发后返回429状态码，配合友好的错误提示
- 热点业务采用"分桶限流"，避免单一用户影响整体

# 4. 熔断和降级有什么区别？

**核心回答：** 熔断是**被动的、由下而上**的保护机制，当服务调用失败率达到阈值时直接返回降级响应；降级是**主动的、由上而下**的策略，在系统压力过大或依赖不可用时主动关闭非核心功能。

**原理分析：**
1. **熔断器三种状态**：
   - **关闭状态**：统计失败率，未达阈值时放行请求
   - **打开状态**：失败率超阈值，后续请求直接返回降级响应
   - **半开状态**：打开状态一段时间后进入，允许一个请求通过测试，成功则恢复关闭状态

2. **熔断 vs 降级**：熔断用于依赖调用，降级用于整个服务或功能单元；熔断由失败触发，降级可手动或自动触发。

**调优实践：**
- 熔断参数：失败率阈值通常50%，熔断时长通常30秒
- 分层熔断：先熔断底层数据库访问，再熔断中间层服务，最后熔断上层API

# 5. 负载均衡有哪些实现方式？

**核心回答：** 常见负载均衡实现包括**硬件负载均衡（F5）、软件负载均衡（Nginx、LVS）、DNS负载均衡、客户端负载均衡**。

**原理分析：**
1. **DNS负载均衡**：基于地理位置返回不同IP，实现就近访问，但DNS缓存导致更新不及时。
2. **四层负载均衡**：基于IP和端口转发（LVS），性能高但无法理解应用层协议。
3. **七层负载均衡**：理解HTTP协议，可根据URL、Header路由（Nginx），灵活性高但性能略差。
4. **客户端负载均衡**：consumer直接选择provider（Dubbo、Ribbon），减少一次网络跳转。

**算法分类：**
- **轮询**：依次分发，适合服务器性能相近场景
- **加权轮询**：根据服务器性能分配权重
- **最少连接**：选择当前连接数最少服务器，适合长连接场景
- **IP哈希**：相同客户端固定路由，适合会话保持场景

# 6. 高可用容灾策略有哪些？

**核心回答：** 高可用容灾策略包括**多活架构、故障转移、数据备份**。

**原理分析：**
1. **多活架构**：
   - **同城双活**：两机房同城市，网络延迟低，故障时快速切换，但无法应对城市级灾难
   - **异地多活**：多机房跨城市部署，能应对城市级灾难，但需解决数据同步延迟问题

2. **故障转移**：
   - **主备切换**：主服务故障后备服务接管，需考虑数据一致性问题
   - **健康检测**：定期检测服务状态，异常时自动触发切换

3. **数据备份**：
   - **冷备**：定期全量备份，恢复时间长
   - **热备**：实时同步备份，RPO接近零但成本高

**延展思考：**
需平衡成本和收益：核心系统采用异地多活，重要系统采用同城双活，一般系统采用主备即可。SLA 99.9%（年故障8.76小时）vs 99.99%（年故障52分钟）成本差一个数量级。

# 7. 池化技术有哪些？

**核心回答：** 池化技术通过复用资源（线程、连接、内存）提升性能。常见包括**线程池、数据库连接池、对象池、内存池**。

**原理分析：**
1. **线程池**：核心线程数、最大线程数、队列容量、拒绝策略。CPU密集型任务核心线程数=CPU核数+1，IO密集型可设为2倍核数。
2. **数据库连接池**：Druid、HikariCP（性能最优）。连接数通常设为CPU核心数的2-4倍。
3. **内存池**：Netty的PooledByteBufAllocator基于此，零拷贝技术的基础。

**调优实践：**
- 监控活跃率、空闲率、等待队列长度等指标
- 队列选择：LinkedBlockingQueue（有界）vs SynchronousQueue（无界）

# 8. 异步处理和消息队列的作用

**核心回答：** 异步处理通过将耗时操作从主链路剥离提升吞吐量。实现方式包括**多线程异步、CompletableFuture、响应式编程、消息队列**。

**原理分析：**
1. **CompletableFuture**：Java 8异步编程模型，支持链式调用、组合多个异步任务。
2. **响应式编程**：Reactor、RxJava，支持背压机制，适合流式数据处理。
3. **消息队列作用**：
   - **削峰填谷**：高峰期存储请求，消费者按能力消费
   - **系统解耦**：生产者和消费者松耦合
   - **事件驱动**：通过消息队列实现业务流程异步化

**调优实践：**
- 消息队列需关注：消息堆积、消费重复、消费顺序
- Kafka适合高吞吐，RabbitMQ适合低延迟，RocketMQ适合金融级可靠消息

# 9. 缓存架构如何设计？

**核心回答：** 缓存按访问层级分为**本地缓存、分布式缓存、多级缓存**。本地缓存延迟最低但无法跨节点共享；分布式缓存可共享但有网络开销；多级缓存结合两者优势。

**原理分析：**
1. **本地缓存**：Caffeine、Guava Cache，访问延迟纳秒级，但无法跨节点共享，更新需同步。
2. **分布式缓存**：Redis支持丰富数据结构，支持主从复制和集群模式。
3. **多级缓存**：L1（本地缓存）→L2（分布式缓存）→DB，逐级穿透。

**缓存策略：**
- **Cache-Aside**：读：先查缓存，未命中查DB并写入缓存；写：先更新DB，再删除缓存
- **Write-Behind**：写操作先更新缓存，异步批量更新数据源

**调优实践：**
- 缓存击穿：使用互斥锁或分布式锁保护并发加载
- 缓存穿透：布隆过滤器拦截不存在的数据
- 缓存雪崩：随机化过期时间，多级缓存设计

# 10. 如何保证缓存和数据库一致性？

**核心回答：** 常用方案包括**延迟双删、订阅binlog、异步更新**。

**原理分析：**
1. **延迟双删**：删缓存→更新DB→延迟再次删缓存，延迟时间需大于主从同步延迟。
2. **订阅binlog**：通过Canal订阅binlog，DB变更时异步更新缓存，延迟最低但实现复杂。
3. **异步更新**：写操作直接更新缓存，同时异步写DB，适用于一致性要求不高的场景。

**方案对比：**
- **旁路缓存模式**：写操作先更新DB再删缓存，并发写时可能出现脏缓存
- **双写模式**：同时更新DB和缓存，并发时无法保证顺序
- **订阅变更**：通过变更日志异步更新缓存

**延展思考：**
分布式系统中强一致性几乎不可能，通常采用最终一致性。需根据业务场景选择一致性级别：金融交易需强一致性，社交点赞可接受最终一致性。
