# 1. 介绍下 Redis 的事务

Redis 提供了**简单的事务机制**，但对 ACID 的支持并不完备，和 MySQL 的事务有本质区别。

**事务的基本命令：**

- **MULTI**：标记事务开始
- **EXEC**：提交并执行事务
- **DISCARD**：取消事务，清空命令队列
- **WATCH**：监视一个或多个 key，实现乐观锁

**事务执行的三个阶段：**

- **事务开始**：客户端发送 MULTI，服务端将该客户端标记为事务状态
- **命令入队**：MULTI 之后的命令不会立即执行，而是缓存到**服务端的事务队列**中，每条命令返回 QUEUED
- **事务执行**：客户端发送 EXEC，服务端**按顺序执行队列中的所有命令**，一次性返回所有结果

redis事务可以将多个命令请求打包，一次性按顺序执行多个命令，且执行期间不会中断事务而执行其他的客户端命令，将事务中的命令执行完毕才会执行其他命令

**事务的核心原理：**

- 所有命令在 EXEC 之前都不执行，只是**缓存在服务端的事务队列**中
- 收到 EXEC 后，服务器**遍历队列依次执行**所有命令，执行完毕后一次性把所有结果返回客户端
- 因为 Redis 执行命令是**单线程**的，所以事务队列中的命令是**顺序执行**的，中间**不会被其他客户端的命令打断**

**WATCH 乐观锁机制：**

- WATCH 本质是一个 **CAS（Compare And Swap）机制**
- 被监视的 key 会被记录下来，如果在 EXEC 之前这些 key 被其他客户端修改，**REDIS_DIRTY_CAS 标志**会被打开
- 服务器检测到该标志后会**拒绝执行事务**，EXEC 返回 nil，客户端可以选择重试

**Redis 事务对 ACID 的支持：**

- **原子性（部分支持）**：Redis 事务**不支持回滚**。如果命令在**入队时报错**（语法错误、命令不存在），整个事务会被拒绝执行；但如果是**运行时错误**（比如对 String 执行 LPUSH），错误命令会失败，**其他命令仍会继续执行**，已成功的命令不会回滚
- **一致性（支持）**：通过入队检查和运行时错误隔离来保证。入队错误会拒绝整个事务；运行时错误的命令不会修改数据库；服务器宕机依赖 RDB/AOF 恢复
- **隔离性（支持）**：因为 Redis 是**单线程执行命令**，事务期间不会被其他命令打断，天然具备隔离性
- **持久性（取决于持久化配置）**：
  - 纯内存模式：**不具备持久性**
  - RDB 模式：BGSAVE 异步执行，**不保证持久性**
  - AOF + appendfsync always：**具备持久性**
  - AOF + appendfsync everysec：可能丢失 1 秒数据，**不完全具备持久性**
  - AOF + appendfsync no：交给操作系统，**不具备持久性**

**Redis 不支持回滚的原因：**

- 命令入队时已经做过语法检查，运行时错误**通常是程序 bug**，在开发阶段就能发现，生产环境很少出现
- **不支持回滚可以保持 Redis 简单、快速的特性**，符合 Redis 的设计哲学

# 2. Redis 事务的注意点有哪些

- **不支持回滚**：和 MySQL 事务不一样，Redis 事务中即使某条命令运行失败，其他命令也会继续执行，已经执行的命令不会撤销
- **执行期间不会被打断**：服务端在执行事务过程中，**不会响应其他客户端的命令请求**，直到事务全部执行完毕
- **入队错误整个事务作废**：如果命令在入队时就有语法错误，整个事务会被标记为失败，EXEC 时直接拒绝执行
- **运行时错误不影响其他命令**：比如对 String 类型执行 LPUSH，这条命令会报错，但事务中**其他命令照常执行**
- **WATCH 实现乐观锁**：需要保证多个 key 一致性时，使用 WATCH + MULTI/EXEC，被监视的 key 一旦被修改，事务会失败，需要客户端**自己实现重试逻辑**
- **替代方案**：如果对原子性要求高，更推荐使用 **Lua 脚本**，Lua 脚本在 Redis 中也是单线程原子执行，且支持更复杂的逻辑判断
