# 1. 常见的缓存问题有哪些

常见的缓存问题主要围绕 **缓存没有命中、缓存集中失效、缓存数据不一致、缓存本身被无效数据占用、热点访问压力过大** 这几类展开。面试中通常重点回答缓存穿透、缓存击穿、缓存雪崩，再补充缓存并发、缓存污染、缓存一致性、热 key、大 key 和分级缓存问题。

**缓存穿透**：请求查询的数据在缓存和数据库中都不存在，导致每次请求都绕过缓存直接打到数据库。典型场景是恶意请求不存在的 id，或者业务上大量查询已删除、无效的数据。它的风险是 **缓存层完全失去保护作用，请求直接打到数据库**，如果被恶意高频请求，可能把数据库压垮。

解决方式：

- 对不存在的数据缓存空值，并设置较短过期时间，避免相同 key 反复打库。
- **使用布隆过滤器**，把可能存在的 key 提前加载进去，请求先经过布隆过滤器判断，不存在的 key 直接拦截。
- 使用白名单策略，比如提前把合法 id 加载到 bitmap 中，id 作为 bitmap 的 offset，查询前先判断是否在白名单内，不在白名单内直接拦截。
- 对入口参数做合法性校验，比如 id 范围、格式、租户权限校验，过滤明显非法请求。

注意点：

- 空值缓存适合应对短时间内针对同一个不存在 key 的重复请求，但 TTL 不能太长，避免真实数据后来写入后仍被空值挡住。
- 布隆过滤器本质是空间效率很高的概率型集合结构，不是普通哈希表；它可能误判存在，但不会误判不存在，适合拦截一定不存在的数据。
- 白名单或 bitmap 适合 id 范围相对可控的场景，如果 id 非连续、范围极大或数据变化频繁，维护成本会比较高。

**缓存击穿**：某个热点 key 过期的瞬间，大量并发请求同时发现缓存未命中，然后一起访问数据库。它可以看成缓存雪崩的一个特例，特点是 **单个热点 key 失效导致数据库瞬时压力过大**。

解决方式：

- 热点 key 设置不过期，后台异步刷新缓存。
- 使用互斥锁或分布式锁，只允许一个线程回源数据库并重建缓存，其他线程等待、短暂重试或返回旧值。
- 采用逻辑过期，在缓存值中保存过期时间，发现逻辑过期后由一个线程异步重建，其他线程先返回旧数据。
- 使用二级缓存或本地缓存，热点 key 在 Redis 失效时还能由本地缓存承接一部分流量。
- 结合热点探测和 LFU/LRU 思路识别热点数据，对热点 key 做特殊缓存策略，比如延长 TTL、异步刷新、多副本缓存。

注意点：

- 线程互斥能保护数据库，但会让部分请求等待，极端情况下会降低接口 QPS，所以更适合和逻辑过期、返回旧值一起使用。
- 锁要设置超时时间，避免死锁；重建缓存要做异常处理，避免缓存一直无法恢复。
- LRU 只能说明最近访问的数据更可能继续被访问，适合做淘汰或热点识别参考，不能单独解决击穿问题。

**缓存雪崩**：大量 key 在同一时间过期，或者 Redis 集群整体不可用，导致请求大面积回源数据库。它的特点是 **大量缓存同时失效或缓存服务不可用，数据库被整体压垮**。

解决方式：

- 给不同 key 的过期时间增加随机值，避免同一批缓存同时过期。
- 对核心热点数据设置永不过期或逻辑过期，后台刷新。
- 做缓存预热，系统上线或大促前提前加载核心数据，避免冷启动时请求全部打数据库。
- Redis 做高可用部署，比如主从、哨兵、Cluster，避免单点故障。
- 使用缓存水平切分，把不同 key 分散到多个缓存实例，单个实例故障时不至于所有流量都打到数据库。
- 服务侧增加限流、熔断、降级，Redis 故障时优先保护数据库。
- 提前做容量预估和压测，确认缓存失效或部分缓存实例故障时，数据库、连接池和应用线程池不会被瞬间打满。

注意点：

- 雪崩不只是 key 同时过期，也包括 Redis 宕机、网络故障、集群大面积不可用。
- 高可用缓存集群只能降低缓存整体不可用概率，不能替代限流、降级和容量预估。
- 水平切分可以降低单实例故障影响面，但如果没有副本和故障转移，故障分片上的流量仍然可能回源数据库。

**缓存并发问题**：缓存过期或缓存重建时，多个请求同时回源数据库并写缓存，可能造成数据库压力突增，也可能出现旧值覆盖新值、重复重建缓存等问题。它和击穿、雪崩有交集，但更强调 **并发读写缓存时的重建控制和一致性控制**。

解决方式：

- 对同一个 key 的缓存重建加互斥锁，避免多个线程同时查库和写缓存。
- 使用逻辑过期加异步刷新，减少并发请求直接回源数据库。
- 对写缓存操作增加版本号、更新时间或 CAS 判断，避免旧数据覆盖新数据。
- 对热点接口做请求合并，把同一时间对同一个 key 的多个回源请求合并成一次。

**缓存污染**：大量只访问一次或很少访问的冷数据进入缓存，把真正的热点数据挤出去，导致缓存命中率下降。典型场景是批量查询、爬虫扫描、低频报表数据写入缓存。

解决方式：

- 对低频数据不缓存，或者设置更短 TTL。
- 使用合适的淘汰策略，比如 `allkeys-lfu` 更适合保留高频访问数据，减少偶发访问数据长期占用缓存。
- 对批量任务、爬虫流量和普通在线流量做隔离，避免污染核心业务缓存。

**缓存一致性问题**：数据库和缓存中同一份数据可能短时间不一致。典型原因是更新数据库和删除缓存不是原子操作，或者并发读写导致旧值重新写入缓存。除了数据库和缓存之间的一致性，还要关注缓存主从、副本、本地缓存和 Redis 之间的数据差异。

常见方案：

- 优先使用 **先更新数据库，再删除缓存**，让下一次读请求重新加载新值。
- 对一致性要求更高的场景，可以使用延迟双删、消息队列重试、Binlog 订阅等方式兜底。
- 本地缓存和多级缓存要设置更短 TTL，或者通过消息通知主动失效，避免本地旧值长期存在。
- 对强一致要求很高的数据，不建议完全依赖缓存，读请求应走数据库或使用更强的一致性设计。

注意点：

- **缓存一致性通常追求最终一致，不要把 Redis 缓存当成强一致存储** 。
- 主动更新缓存容易受并发写顺序影响，工程上更常见的是更新数据库后删除缓存，再依赖读请求重新加载。

**热 key 问题**：某个 key 被极高并发访问，即使缓存命中，也可能把单个 Redis 节点打满。典型场景是秒杀商品、热点新闻、明星事件。

解决方式：

- 本地缓存加 Redis 二级缓存，减少 Redis 压力。
- 将热 key 拆分成多个副本 key，读请求随机访问不同副本。
- 对热点接口做限流、排队和降级。
- 对热点数据做实时探测，发现访问量异常升高后动态延长 TTL、增加本地缓存或切换降级策略。

**大 key 问题**：单个 key 的 value 过大，或者集合元素过多，导致网络传输慢、Redis 阻塞、内存倾斜和删除成本高。典型场景是一个 Hash 存几十万个字段，一个 String 存几 MB 数据。

解决方式：

- 拆分大 key，比如按业务维度、时间片、hash 分片拆成多个小 key。
- 避免一次性读取或删除大 key，删除时使用 `UNLINK` 异步释放内存。
- 定期扫描和治理大 key，避免线上突然阻塞。

**分级缓存问题**：在高并发业务中，如果只有 Redis 一层缓存，Redis 失效或抖动时请求仍然可能直接打到数据库。分级缓存的思路是 **由近到远建立多层缓存，用本地缓存、应用缓存、Redis 缓存逐层抵挡请求，尽量不让流量直接打到数据库**。

常见做法：

- 浏览器、CDN、网关、应用本地缓存、Redis、数据库按访问链路逐层缓存。
- 越靠近前端的缓存粒度通常越粗，越靠近数据库的缓存粒度通常越细。
- 上层缓存失效时，优先从下一层缓存快速重建，而不是直接访问数据库。
- 对本地缓存设置短 TTL 或主动失效机制，避免多级缓存放大一致性问题。

总结来说，面试时可以这样回答：**缓存穿透是查不存在的数据，缓存击穿是热点 key 过期，缓存雪崩是大量 key 同时失效或 Redis 不可用，缓存并发是多个请求同时回源或重建缓存，缓存污染是冷数据挤掉热数据，缓存一致性是缓存和数据库短暂不一致，热 key 和大 key 是缓存使用不合理带来的性能问题，分级缓存是用多层缓存减少系统抖动**。这些问题的核心治理思路都是：减少无效回源、避免集中失效、控制并发重建、保护数据库、提升缓存命中率，并通过高可用、限流降级、异步刷新和多级缓存兜底。
