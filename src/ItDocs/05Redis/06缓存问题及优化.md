# 1. 常见的缓存问题有哪些

常见的缓存问题主要围绕 **缓存没有命中、缓存集中失效、缓存数据不一致、缓存本身被无效数据占用、热点访问压力过大** 这几类展开。面试中通常重点回答缓存穿透、缓存击穿、缓存雪崩，再补充缓存并发、缓存污染、缓存一致性、热 key、大 key 和分级缓存问题。

**缓存穿透**：请求查询的数据在缓存和数据库中都不存在，导致每次请求都绕过缓存直接打到数据库。典型场景是恶意请求不存在的 id，或者业务上大量查询已删除、无效的数据。它的风险是 **缓存层完全失去保护作用，请求直接打到数据库**，如果被恶意高频请求，可能把数据库压垮。

解决方式：

- 对不存在的数据缓存空值，并设置较短过期时间，避免相同 key 反复打库。
- **使用布隆过滤器**，把可能存在的 key 提前加载进去，请求先经过布隆过滤器判断，不存在的 key 直接拦截。
- 使用白名单策略，比如提前把合法 id 加载到 bitmap 中，id 作为 bitmap 的 offset，查询前先判断是否在白名单内，不在白名单内直接拦截。
- 对入口参数做合法性校验，比如 id 范围、格式、租户权限校验，过滤明显非法请求。

注意点：

- 空值缓存适合应对短时间内针对同一个不存在 key 的重复请求，但 TTL 不能太长，避免真实数据后来写入后仍被空值挡住。
- 布隆过滤器本质是空间效率很高的概率型集合结构，不是普通哈希表；它可能误判存在，但不会误判不存在，适合拦截一定不存在的数据。
- 白名单或 bitmap 适合 id 范围相对可控的场景，如果 id 非连续、范围极大或数据变化频繁，维护成本会比较高。

**缓存击穿**：某个热点 key 过期的瞬间，大量并发请求同时发现缓存未命中，然后一起访问数据库。它可以看成缓存雪崩的一个特例，特点是 **单个热点 key 失效导致数据库瞬时压力过大**。

解决方式：

- 热点 key 设置不过期，后台异步刷新缓存。
- 使用互斥锁或分布式锁，只允许一个线程回源数据库并重建缓存，其他线程等待、短暂重试或返回旧值。
- 采用逻辑过期，在缓存值中保存过期时间，发现逻辑过期后由一个线程异步重建，其他线程先返回旧数据。
- 使用二级缓存或本地缓存，热点 key 在 Redis 失效时还能由本地缓存承接一部分流量。
- 结合热点探测和 LFU/LRU 思路识别热点数据，对热点 key 做特殊缓存策略，比如延长 TTL、异步刷新、多副本缓存。

注意点：

- 线程互斥能保护数据库，但会让部分请求等待，极端情况下会降低接口 QPS，所以更适合和逻辑过期、返回旧值一起使用。
- 锁要设置超时时间，避免死锁；重建缓存要做异常处理，避免缓存一直无法恢复。
- LRU 只能说明最近访问的数据更可能继续被访问，适合做淘汰或热点识别参考，不能单独解决击穿问题。

**缓存雪崩**：大量 key 在同一时间过期，或者 Redis 集群整体不可用，导致请求大面积回源数据库。它的特点是 **大量缓存同时失效或缓存服务不可用，数据库被整体压垮**。

解决方式：

- 给不同 key 的过期时间增加随机值，避免同一批缓存同时过期。
- 对核心热点数据设置永不过期或逻辑过期，后台刷新。
- 做缓存预热，系统上线或大促前提前加载核心数据，避免冷启动时请求全部打数据库。
- Redis 做高可用部署，比如主从、哨兵、Cluster，避免单点故障。
- 使用缓存水平切分，把不同 key 分散到多个缓存实例，单个实例故障时不至于所有流量都打到数据库。
- 服务侧增加限流、熔断、降级，Redis 故障时优先保护数据库。
- 提前做容量预估和压测，确认缓存失效或部分缓存实例故障时，数据库、连接池和应用线程池不会被瞬间打满。

注意点：

- 雪崩不只是 key 同时过期，也包括 Redis 宕机、网络故障、集群大面积不可用。
- 高可用缓存集群只能降低缓存整体不可用概率，不能替代限流、降级和容量预估。
- 水平切分可以降低单实例故障影响面，但如果没有副本和故障转移，故障分片上的流量仍然可能回源数据库。

**缓存并发问题**：缓存过期或缓存重建时，多个请求同时回源数据库并写缓存，可能造成数据库压力突增，也可能出现旧值覆盖新值、重复重建缓存等问题。它和击穿、雪崩有交集，但更强调 **并发读写缓存时的重建控制和一致性控制**。

解决方式：

- 对同一个 key 的缓存重建加互斥锁，避免多个线程同时查库和写缓存。
- 使用逻辑过期加异步刷新，减少并发请求直接回源数据库。
- 对写缓存操作增加版本号、更新时间或 CAS 判断，避免旧数据覆盖新数据。
- 对热点接口做请求合并，把同一时间对同一个 key 的多个回源请求合并成一次。

**缓存污染**：大量只访问一次或很少访问的冷数据进入缓存，把真正的热点数据挤出去，导致缓存命中率下降。典型场景是批量查询、爬虫扫描、低频报表数据写入缓存。

解决方式：

- 对低频数据不缓存，或者设置更短 TTL。
- 使用合适的淘汰策略，比如 `allkeys-lfu` 更适合保留高频访问数据，减少偶发访问数据长期占用缓存。
- 对批量任务、爬虫流量和普通在线流量做隔离，避免污染核心业务缓存。

**缓存一致性问题**：数据库和缓存中同一份数据可能短时间不一致。典型原因是更新数据库和删除缓存不是原子操作，或者并发读写导致旧值重新写入缓存。除了数据库和缓存之间的一致性，还要关注缓存主从、副本、本地缓存和 Redis 之间的数据差异。

常见方案：

- 优先使用 **先更新数据库，再删除缓存**，让下一次读请求重新加载新值。
- 对一致性要求更高的场景，可以使用延迟双删、消息队列重试、Binlog 订阅等方式兜底。
- 本地缓存和多级缓存要设置更短 TTL，或者通过消息通知主动失效，避免本地旧值长期存在。
- 对强一致要求很高的数据，不建议完全依赖缓存，读请求应走数据库或使用更强的一致性设计。

注意点：

- **缓存一致性通常追求最终一致，不要把 Redis 缓存当成强一致存储** 。
- 主动更新缓存容易受并发写顺序影响，工程上更常见的是更新数据库后删除缓存，再依赖读请求重新加载。

**热 key 问题**：某个 key 被极高并发访问，即使缓存命中，也可能把单个 Redis 节点打满。典型场景是秒杀商品、热点新闻、明星事件。

解决方式：

- 本地缓存加 Redis 二级缓存，减少 Redis 压力。
- 将热 key 拆分成多个副本 key，读请求随机访问不同副本。
- 对热点接口做限流、排队和降级。
- 对热点数据做实时探测，发现访问量异常升高后动态延长 TTL、增加本地缓存或切换降级策略。

**大 key 问题**：单个 key 的 value 过大，或者集合元素过多，导致网络传输慢、Redis 阻塞、内存倾斜和删除成本高。典型场景是一个 Hash 存几十万个字段，一个 String 存几 MB 数据。

解决方式：

- 拆分大 key，比如按业务维度、时间片、hash 分片拆成多个小 key。
- 避免一次性读取或删除大 key，删除时使用 `UNLINK` 异步释放内存。
- 定期扫描和治理大 key，避免线上突然阻塞。

**分级缓存问题**：在高并发业务中，如果只有 Redis 一层缓存，Redis 失效或抖动时请求仍然可能直接打到数据库。分级缓存的思路是 **由近到远建立多层缓存，用本地缓存、应用缓存、Redis 缓存逐层抵挡请求，尽量不让流量直接打到数据库**。

常见做法：

- 浏览器、CDN、网关、应用本地缓存、Redis、数据库按访问链路逐层缓存。
- 越靠近前端的缓存粒度通常越粗，越靠近数据库的缓存粒度通常越细。
- 上层缓存失效时，优先从下一层缓存快速重建，而不是直接访问数据库。
- 对本地缓存设置短 TTL 或主动失效机制，避免多级缓存放大一致性问题。

总结来说，面试时可以这样回答：**缓存穿透是查不存在的数据，缓存击穿是热点 key 过期，缓存雪崩是大量 key 同时失效或 Redis 不可用，缓存并发是多个请求同时回源或重建缓存，缓存污染是冷数据挤掉热数据，缓存一致性是缓存和数据库短暂不一致，热 key 和大 key 是缓存使用不合理带来的性能问题，分级缓存是用多层缓存减少系统抖动**。这些问题的核心治理思路都是：减少无效回源、避免集中失效、控制并发重建、保护数据库、提升缓存命中率，并通过高可用、限流降级、异步刷新和多级缓存兜底。

# 2. 如何进行缓存预热

缓存预热是指 **在业务流量真正到来之前，提前把核心数据、热点数据加载到缓存中**。它解决的是系统启动、版本发布、Redis 故障恢复、大促开始前缓存为空的问题，避免大量请求在冷启动阶段直接打到数据库。

常见预热方式：

- **系统启动预热**：应用启动后加载基础配置、类目、字典、权限、热门商品等核心数据到 Redis 或本地缓存。适合数据量不大、访问频率高、变化不频繁的数据。
- **定时任务预热**：通过定时任务周期性统计最近一段时间的热点数据，比如热门商品、热门文章、热门搜索词，然后提前刷新缓存。
- **发布前预热**：服务上线、Redis 重启、缓存迁移或大促活动开始前，通过脚本、任务平台或运营后台主动触发预热，把关键 key 提前写入缓存。
- **热点探测预热**：根据访问日志、埋点、网关统计、Redis 访问频率等识别热点 key，对访问量突然升高的数据动态预热。
- **增量预热**：数据发生变更后，通过消息队列、Binlog 订阅或业务事件把变更数据提前写入缓存，避免等用户请求时再回源加载。

执行流程：

- 先确定预热范围，只预热高频、核心、即将被访问的数据，不要把全量数据库数据都塞进缓存。
- 从数据库、搜索引擎、离线统计结果或配置中心读取预热数据。
- 按业务规则组装缓存 key 和 value，设置合理 TTL，必要时给 TTL 增加随机值，避免后续集中失效。
- 分批、限速写入 Redis，避免预热任务本身把数据库、Redis 或网络打满。
- 预热完成后校验关键 key 是否存在、数量是否符合预期、命中率是否提升。

注意点：

- **缓存预热不是全量加载**，全量预热会浪费内存，还可能造成缓存污染。
- 预热任务要限流、分批执行，避免还没承接业务流量就先把数据库打满。
- 预热数据要设置过期时间和随机抖动，避免预热的一批 key 后续同时过期，引发缓存雪崩。
- 对一致性要求高的数据，预热后仍要配合更新数据库后删除缓存、消息通知或 Binlog 订阅，避免缓存长期保存旧值。
- 多实例本地缓存也要考虑预热或懒加载，否则 Redis 已预热但应用本地缓存仍然是冷的。

面试时可以这样回答：**缓存预热就是在流量到来前提前加载核心热点数据，常见方式有启动预热、定时任务预热、发布前脚本预热、热点探测预热和增量事件预热。关键不是预热越多越好，而是只预热高频核心数据，并且要分批限速、设置 TTL 随机值、校验命中率，避免预热变成数据库压力或缓存污染。**

# 3. 如何解决大 key 问题

大 key 指的是 **value 体积很大，或者集合类型成员数量很多，或者集合中成员本身体积很大** 的 key。它没有绝对统一的标准，要结合 Redis 实例规格、网络带宽、业务访问频率、延迟要求和数据结构综合判断。

常见判断示例：

- `String` 类型的 key，value 达到几 MB，比如一个 key 存了 5MB 的 JSON 字符串。
- `List` 类型的 key，列表元素数量很多，比如一个列表有 20000 个元素。
- `Set` 或 `ZSet` 类型的 key，成员数量很多，比如一个 ZSet 有 10000 个成员。
- `Hash` 类型的 key，字段数量不一定特别多，但字段 value 总体积很大，比如 1000 个字段总大小达到 100MB。

大 key 的危害：

- **阻塞 Redis 主线程**：读取、删除、序列化、网络发送大 key 都可能耗时较长，影响其他请求。
- **网络传输变慢**：一次返回几 MB 甚至几十 MB 数据，会占用带宽，拉高客户端和 Redis 的响应时间。
- **内存倾斜**：在 Cluster 模式下，大 key 所在节点内存压力明显高于其他节点，导致分片不均衡。
- **删除成本高**：直接 `DEL` 大集合可能同步释放大量内存，造成 Redis 短时间卡顿。
- **迁移和备份变慢**：主从同步、AOF 重写、RDB 生成、Cluster 迁槽时，大 key 会放大复制、持久化和迁移成本。
- **容易叠加热 key 问题**：如果大 key 同时是热点 key，会同时带来单点高并发和大对象传输问题。

排查方式：

- 使用 `redis-cli --bigkeys` 扫描各数据类型中元素数量最大的 key，适合快速发现集合成员数过多的 key。
- 使用 `redis-cli --memkeys` 或 `MEMORY USAGE key` 查看 key 的内存占用，适合判断 value 体积是否过大。
- 使用 `SCAN` 分批遍历 key，再结合 `STRLEN`、`LLEN`、`HLEN`、`SCARD`、`ZCARD`、`MEMORY USAGE` 做离线分析。
- 结合慢日志、客户端监控、网络流量和实例内存分布，判断大 key 是否已经影响线上延迟。
- 在 Cluster 模式下关注各节点内存、QPS、带宽和槽位分布，判断是否存在大 key 导致的节点倾斜。

解决方式：

- **拆分大 String**：不要把一个超大的 JSON 或对象整体塞进一个 key，可以按业务字段、模块、用户维度或时间片拆成多个小 key。
- **拆分大 Hash**：把一个大 Hash 按字段 hash 分片，比如拆成 `user:profile:0` 到 `user:profile:15` 多个 key，降低单 key 字段数量和体积。
- **拆分大 List / Set / ZSet**：按时间、分页区间、业务分组或 hash 分片拆分，避免一个集合无限增长。
- **限制单次读取数量**：读取集合时不要一次性 `LRANGE 0 -1`、`HGETALL`、`SMEMBERS`、`ZRANGE 0 -1`，优先使用分页、游标或按范围读取。
- **异步删除大 key**：删除大 key 优先使用 `UNLINK`，或者对集合分批删除，避免 `DEL` 同步释放内存阻塞主线程。
- **压缩或精简 value**：对确实需要存储的大对象，可以精简字段、压缩内容，但要评估压缩和解压带来的 CPU 成本。
- **设置合理 TTL**：临时性、低频或中间结果类大 key 要设置过期时间，避免长期占用内存。

预防方式：

- 在代码层限制单个 key 的最大 value 大小、集合最大成员数和单次写入数量。
- 对排行榜、消息列表、行为日志这类天然增长的数据设置长度上限，比如只保留最近 N 条或最近一段时间的数据。
- 建立定期巡检任务，扫描大 key、热 key、内存倾斜和慢命令。
- 在设计缓存模型时提前考虑拆分维度，不要把数据库中的一整张表、一整个用户全量关系或超大 JSON 直接塞进一个 key。
- 大促、迁移、上线前提前扫描大 key，避免在主从同步、RDB 备份、Cluster 迁槽时暴露问题。

注意点：

- 大 key 的判断不能只看成员数量，也要看成员 value 总体积；一个 Hash 只有 1000 个字段，也可能因为每个字段很大而成为大 key。
- 拆分大 key 后要注意多 key 读写的一致性、批量查询效率和 Cluster 跨 slot 问题。
- 大 key 治理通常要结合业务改造，不只是 Redis 运维命令能完全解决。

面试时可以这样回答：**大 key 本质是单个 key 承载的数据太大，可能是 String value 很大，也可能是 List、Hash、Set、ZSet 成员太多或成员总体积太大。分析时先用** `--bigkeys`、`MEMORY USAGE`、`SCAN` 加类型长度命令找出大 key，再看它是否导致慢查询、网络带宽高、内存倾斜、删除阻塞和迁移变慢。解决思路是拆分 key、限制单次读取、避免 `HGETALL` / `SMEMBERS` 这类全量操作、用 `UNLINK` 或分批方式删除，并在业务设计上限制集合长度和单个 value 大小。

# 4. 如何分析解决热 key 问题

热 key 指的是 **某个 key 的访问次数、带宽占用或 CPU 消耗显著高于其他 key**。它不一定是大 key，一个很小的 String 如果每秒被访问几万次也是热 key；一个大 Hash 或大 ZSet 如果被频繁全量读取，也会因为带宽或 CPU 开销成为热 key。

常见判断示例：

- 某个 Redis 实例每秒总访问量是 10000，其中一个 key 每秒访问量达到 7000，说明这个 key 的访问次数明显偏高。
- 一个总大小 1MB、包含上千个成员的 Hash key，每秒被大量执行 `HGETALL`，说明这个 key 的带宽占用明显偏高。
- 一个包含数万个成员的 ZSet key，每秒被大量执行 `ZRANGE`，说明这个 key 的 CPU 时间和网络返回成本明显偏高。

热 key 的危害：

- **单节点压力过高**：Redis Cluster 中一个 key 只落在一个 slot 上，热 key 会把所在节点打满，其他节点即使空闲也分担不了。
- **响应时间升高**：大量请求集中访问同一个 key，会导致 Redis 命令排队、网络带宽升高、客户端超时增加。
- **影响其他请求**：Redis 单线程执行命令，热 key 上的高频或重命令会拖慢同节点上的其他 key。
- **容易引发缓存击穿**：热 key 如果过期，大量请求会同时回源数据库，导致数据库瞬时压力过大。
- **和大 key 叠加时影响更严重**：大 key 频繁读取会同时消耗 CPU、内存带宽和网络带宽。

排查方式：

- 使用业务埋点、网关日志或应用日志统计 key 维度访问量，这是最准确的方式。
- 使用 Redis `MONITOR` 临时观察线上命令，但它开销较大，只适合短时间排查，不适合长期运行。
- 使用 `redis-cli --hotkeys` 结合 LFU 淘汰策略观察热点 key，但依赖 Redis 的访问频率统计，适合辅助判断。
- 从客户端 SDK、代理层或缓存中间件统计 key 的 QPS、命令类型、平均响应时间和返回字节数。
- 结合慢日志、网络流量、CPU 使用率和节点 QPS，判断是否是某个 key 或某类命令导致热点。

解决方式：

- **本地缓存**：把热点 key 缓存在应用本地，比如 Caffeine、Guava Cache，减少 Redis 请求量。适合读多写少、允许短暂不一致的数据。
- **多副本 key**：把一个热点 key 拆成多个内容相同的副本 key，比如 `hot:goods:1:0` 到 `hot:goods:1:9`，读请求随机访问不同副本，分散到不同 slot 和节点。
- **读写分离**：对于读多写少的热点数据，可以让读请求走从节点或副本缓存，但要注意主从延迟。
- **限流和降级**：对热点接口做限流、排队、熔断或返回兜底数据，避免流量继续放大。
- **逻辑过期和异步刷新**：热点 key 不直接依赖物理过期时间，发现逻辑过期后后台刷新，前台先返回旧值，避免击穿数据库。
- **避免重命令**：不要对热点大 Hash 使用高频 `HGETALL`，不要对热点大 ZSet 高频执行大范围 `ZRANGE`，应改成按字段、分页、TopN 或局部范围读取。
- **提前预热**：大促、热点活动、突发流量前提前加载热点 key，并适当延长 TTL 或设置不过期加后台刷新。

预防方式：

- 对秒杀商品、热门榜单、热点新闻、配置字典等天然热点数据，设计阶段就引入本地缓存、多副本或逻辑过期。
- 建立热点探测机制，发现访问量突增的 key 后自动提升缓存等级、延长 TTL 或增加副本。
- 对返回体较大的热点接口做字段裁剪和分页，避免一次返回过多数据。
- 对热点 key 的更新通过消息通知或版本号控制本地缓存失效，避免多级缓存长期不一致。

热 key 和大 key 的区别：

- **热 key 关注访问频率和资源消耗是否集中**，小 key 也可能是热 key。
- **大 key 关注单个 key 承载的数据体积或成员数量是否过大**，不热的大 key 也会带来删除、迁移、持久化风险。
- 最危险的是 **又大又热的 key**，既会造成单点高并发，又会放大网络、CPU 和阻塞问题。

面试时可以这样回答：**热 key 是访问次数、带宽占用或 CPU 消耗明显高于其他 key 的 key。分析时不能只看 QPS，还要看命令类型和返回数据量，比如高频** `HGETALL`、大范围 `ZRANGE` 都可能造成热点。排查可以从业务日志、代理层统计、`--hotkeys`、短时间 `MONITOR`、慢日志和节点监控入手。解决思路是本地缓存、多副本 key、读写分离、限流降级、逻辑过期异步刷新，并避免对热点大集合执行全量读取命令。

# 5. Redis内存暴涨怎么处理

线上 Redis 内存暴涨要按 **先止血、再定位、再治理、最后复盘** 的顺序处理，不能一上来就执行 `KEYS *`、`FLUSHDB` 或直接 `DEL` 大 key，否则可能把 Redis 主线程打挂，导致故障扩大。

第一步是先止血，保护 Redis 和数据库：

- 观察是否已经接近 `maxmemory`，如果 Redis 开始频繁淘汰、响应变慢或 OOM，要先评估是否需要临时扩容内存、增加分片或切流。
- 对非核心写入入口做限流、降级或临时关闭，避免异常流量继续写入大量 key。
- 如果是某个新功能、新版本导致内存快速上涨，优先回滚发布或关闭对应开关。
- 如果是缓存穿透、爬虫、攻击流量导致大量无效 key 写入，要先在网关或业务层拦截异常请求。
- 如果 Redis 已经影响核心链路，要优先保证核心业务 key，必要时降级非核心缓存。

第二步是确认内存到底涨在哪里：

- 看 `INFO memory`，关注 `used_memory`、`used_memory_rss`、`mem_fragmentation_ratio`、`maxmemory`、`evicted_keys`、`expired_keys`。
- 看 `INFO keyspace`，确认哪个 DB 的 key 数量增长明显。
- 看 `INFO commandstats`、慢日志和客户端监控，判断是否有异常写命令、批量写入或大对象操作。
- 看业务发布记录、流量曲线、定时任务、消息堆积、活动入口，判断内存暴涨是否和业务变更有关。
- 在 Cluster 模式下对比各节点内存和 key 数量，判断是全局增长还是单个分片异常。

第三步是定位具体原因：

- **key 数量暴涨**：通常是没有设置 TTL、动态 key 生成失控、缓存空值过多、爬虫或攻击流量写入大量无效 key。
- **value 体积变大**：通常是大 key、缓存对象字段膨胀、把整页结果或大 JSON 塞进 Redis。
- **过期策略失效**：业务忘记设置 TTL，或者 TTL 设置过长，导致历史缓存长期堆积。
- **淘汰策略不合理**：`maxmemory-policy` 设置为 `noeviction` 可能导致写入报错，设置不当也可能导致热点数据被淘汰后反复回源。
- **内存碎片过高**：`used_memory_rss` 明显大于 `used_memory`，可能是频繁申请释放内存导致碎片严重。
- **客户端缓冲区膨胀**：慢客户端、大量 Pub/Sub 或复制积压可能导致 client buffer 占用内存。
- **复制和持久化开销**：AOF 重写、RDB fork、主从复制积压缓冲区也可能让内存压力短时间升高。

第四步是安全排查 key：

- 不要在线上执行 `KEYS *`，应该使用 `SCAN` 分批遍历，避免阻塞 Redis。
- 用 `redis-cli --bigkeys` 快速找各类型成员数量最大的 key。
- 用 `redis-cli --memkeys` 或 `MEMORY USAGE key` 查看 key 的实际内存占用。
- 对 String 用 `STRLEN`，对 List 用 `LLEN`，对 Hash 用 `HLEN`，对 Set 用 `SCARD`，对 ZSet 用 `ZCARD` 判断数据规模。
- 根据 key 前缀统计数量和内存占比，定位是哪个业务模块写入异常。

第五步是治理问题 key 和异常写入：

- 对无用的普通 key 可以分批删除，避免一次性删除过多 key。
- 对大 key 优先使用 `UNLINK` 异步删除，集合类型也可以按批次逐步裁剪后再删除。
- 对缺少 TTL 的缓存补充过期时间，临时治理可以按 key 前缀分批设置 TTL。
- 对持续增长的 List、Set、ZSet 增加长度上限，只保留最近 N 条或必要的 TopN 数据。
- 对大对象缓存做拆分、字段裁剪或压缩，避免单个 value 过大。
- 对异常写入来源做限流、修复代码、关闭任务或回滚版本，避免删完又涨回来。

第六步是处理内存碎片和容量问题：

- 如果 `mem_fragmentation_ratio` 明显偏高，可以评估开启或触发 active defrag，但要注意它会消耗 CPU。
- 如果内存增长是业务真实增长导致的，需要扩容 Redis、增加分片或拆分业务实例，而不是只靠删除 key。
- 如果接近容量上限，要检查 `maxmemory` 和 `maxmemory-policy` 是否符合业务预期。
- 对重要业务要提前做容量水位告警，比如内存使用率、key 数量、单节点倾斜、大 key、淘汰次数、碎片率。

注意点：

- **不要用** `KEYS *` 查全量 key，线上可能直接阻塞 Redis。
- **不要直接** `DEL` 超大 key，同步释放内存可能造成 Redis 卡顿，优先用 `UNLINK` 或分批删除。
- **不要只删 key 不查写入源**，否则内存会继续涨回来。
- **不要盲目调大内存上限**，如果根因是异常写入或无 TTL，扩容只会延后故障。
- 处理过程中要保留现场指标，包括内存曲线、key 数量、异常前缀、慢日志、发布记录，方便复盘。

面试时可以这样回答：**线上 Redis 内存暴涨，我会先止血，限流或关闭异常写入，必要时扩容、切流或回滚；然后看** `INFO memory`、`INFO keyspace`、慢日志、命令统计和节点内存分布，判断是 key 数量暴涨、value 变大、TTL 缺失、碎片率高、客户端缓冲区还是复制持久化开销；排查 key 时不能用 `KEYS *`，要用 `SCAN`、`--bigkeys`、`--memkeys`、`MEMORY USAGE` 和类型长度命令；治理时对无用 key 分批删，大 key 用 `UNLINK`，缺 TTL 的补 TTL，持续增长的数据加长度上限，并最终修复异常写入源和补监控告警。
