# 1. 常见的缓存问题有哪些

常见的缓存问题主要围绕 **缓存没有命中、缓存集中失效、缓存数据不一致、缓存本身被无效数据占用、热点访问压力过大** 这几类展开。面试中通常重点回答缓存穿透、缓存击穿、缓存雪崩，再补充缓存污染、缓存一致性、热 key 和大 key 问题。

- **缓存穿透**：请求查询的数据在缓存和数据库中都不存在，导致每次请求都绕过缓存直接打到数据库。典型场景是恶意请求不存在的 id，或者业务上大量查询已删除、无效的数据。
  - 解决方式：
    - 对不存在的数据缓存空值，并设置较短过期时间，避免相同 key 反复打库。
    - 使用布隆过滤器，把可能存在的 key 提前加载进去，请求先经过布隆过滤器判断，不存在的 key 直接拦截。
    - 对入口参数做合法性校验，比如 id 范围、格式、租户权限校验，过滤明显非法请求。
  - 注意点：空值缓存要设置短 TTL，避免真实数据后来写入后仍被空值挡住；布隆过滤器有误判率，适合拦截一定不存在的数据，但不能替代数据库最终判断。

- **缓存击穿**：某个热点 key 过期的瞬间，大量并发请求同时发现缓存未命中，然后一起访问数据库。它的特点是 **单个热点 key 失效导致数据库瞬时压力过大**。
  - 解决方式：
    - 热点 key 设置不过期，后台异步刷新缓存。
    - 使用互斥锁或分布式锁，只允许一个线程回源数据库并重建缓存，其他线程等待或短暂返回旧值。
    - 采用逻辑过期，在缓存值中保存过期时间，发现逻辑过期后由一个线程异步重建，其他线程先返回旧数据。
  - 注意点：锁要设置超时时间，避免死锁；逻辑过期牺牲了一点实时性，但能保护数据库。

- **缓存雪崩**：大量 key 在同一时间过期，或者 Redis 集群整体不可用，导致请求大面积回源数据库。它的特点是 **大量缓存同时失效或缓存服务不可用，数据库被整体压垮**。
  - 解决方式：
    - 给不同 key 的过期时间增加随机值，避免同一批缓存同时过期。
    - 对核心热点数据设置永不过期或逻辑过期，后台刷新。
    - Redis 做高可用部署，比如主从、哨兵、Cluster，避免单点故障。
    - 服务侧增加限流、熔断、降级，Redis 故障时保护数据库。
    - 预热核心缓存，避免系统刚启动时所有请求直接打数据库。
  - 注意点：雪崩不只是 key 同时过期，也包括 Redis 宕机、网络故障、集群大面积不可用。

- **缓存污染**：大量只访问一次或很少访问的冷数据进入缓存，把真正的热点数据挤出去，导致缓存命中率下降。典型场景是批量查询、爬虫扫描、低频报表数据写入缓存。
  - 解决方式：
    - 对低频数据不缓存，或者设置更短 TTL。
    - 使用合适的淘汰策略，比如 `allkeys-lfu` 更适合保留高频访问数据。
    - 对批量任务、爬虫流量和普通在线流量做隔离，避免污染核心业务缓存。

- **缓存一致性问题**：数据库和缓存中同一份数据可能短时间不一致。典型原因是更新数据库和删除缓存不是原子操作，或者并发读写导致旧值重新写入缓存。
  - 常见方案：
    - 优先使用 **先更新数据库，再删除缓存**，让下一次读请求重新加载新值。
    - 对一致性要求更高的场景，可以使用延迟双删、消息队列重试、Binlog 订阅等方式兜底。
    - 对强一致要求很高的数据，不建议完全依赖缓存，读请求应走数据库或使用更强的一致性设计。
  - 注意点：缓存一致性通常追求最终一致，不要把 Redis 缓存当成强一致存储。

- **热 key 问题**：某个 key 被极高并发访问，即使缓存命中，也可能把单个 Redis 节点打满。典型场景是秒杀商品、热点新闻、明星事件。
  - 解决方式：
    - 本地缓存加 Redis 二级缓存，减少 Redis 压力。
    - 将热 key 拆分成多个副本 key，读请求随机访问不同副本。
    - 对热点接口做限流、排队和降级。

- **大 key 问题**：单个 key 的 value 过大，或者集合元素过多，导致网络传输慢、Redis 阻塞、内存倾斜和删除成本高。典型场景是一个 Hash 存几十万个字段，一个 String 存几 MB 数据。
  - 解决方式：
    - 拆分大 key，比如按业务维度、时间片、hash 分片拆成多个小 key。
    - 避免一次性读取或删除大 key，删除时使用 `UNLINK` 异步释放内存。
    - 定期扫描和治理大 key，避免线上突然阻塞。

总结来说，面试时可以这样回答：**缓存穿透是查不存在的数据，缓存击穿是热点 key 过期，缓存雪崩是大量 key 同时失效或 Redis 不可用，缓存污染是冷数据挤掉热数据，缓存一致性是缓存和数据库短暂不一致，热 key 和大 key 是缓存使用不合理带来的性能问题**。这些问题的核心治理思路都是：减少无效回源、避免集中失效、保护数据库、提升缓存命中率，并通过高可用、限流降级和异步刷新兜底。
